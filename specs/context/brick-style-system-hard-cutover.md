## Current contract evidence

### Core style and node model

- `packages/core/core/src/nodes.ts:93-166` exposes four small direct-style
  shapes. Box/text/media/input/richtext use them, while table/chart/list/
  keyValue/progress/loading expose `variant` and/or `tone` instead of a common
  direct-style contract (`:294-405,413-487`).
- `packages/core/core/src/nodes.ts:178-213` puts `variant` on the base node and
  splits active visuals among `active`, `activeVariant`, and `activeStyle`.
- `packages/core/core/src/tokens.ts` is the closed token source, but it contains
  the retired abbreviations and authored subtree `ColorScheme`; token meanings
  and property-local usage are not represented as data.
- `packages/core/core/src/primitive-node-validation.ts` and
  `packages/core/core/src/brick-validation.ts` own independent style/variant/
  tone allowlists. `packages/core/core/src/brick-registry.ts:78-157` maps brick
  validation and rendering roles but not the complete style vocabulary.
- `packages/core/core/src/tree-validation.ts:326-355` preserves a top-level
  Theme name in the Facet tree. The shared RFC-6902 fold applies patches and then
  runs this fail-soft validator identically on server/client
  (`packages/core/core/src/stage-fold.ts:33-117`).

### Theme and reusable style model

- `packages/core/core/src/theme-types.ts:34-56` defines `FacetTheme` as a partial
  override with optional token groups, `colorDark`, and `recipes`.
- Recipe style shapes and renderer-owned parts are separate public types at
  `packages/core/core/src/theme-types.ts:58-112`; validation is split across
  `theme-token-validation.ts` and `theme-recipe-validation.ts`.
- `packages/core/assets/src/theme.ts` ships one partial default Theme plus sparse
  recipes. `packages/core/react/src/theme.ts:275-332` overlays a selected custom
  Theme on bundled token floors, then resolves `variant`, `tone`, and `default`
  recipe names.
- React's central Theme module is 551 lines and also owns token-to-CSS helpers;
  adding eleven brick-local style targets there would mix resolution, CSS
  mapping, and rendering responsibilities.

### Composition and Catalog model

- `packages/core/core/src/composition-validation.ts:48-77` exposes a concrete
  `FacetComposition` plus rich metadata. It sanitizes invalid node/style pieces
  into a surviving reference (`:79-149`) and still admits legacy `image`
  (`:15-17,172-177`). It has no effective-Theme/Preset compatibility check.
- Default reference data is concentrated in
  `packages/core/assets/src/compositions.ts` (974 lines), with role siblings for
  controls, containers, chart, and table.
- Catalog is a public policy/allow-list layer in
  `packages/core/core/src/catalog-types.ts:3-40`, with separate defaults and
  validation modules. It selects Themes, bricks, Compositions, and editing
  guidance even though the target makes all eleven bricks universally
  available.

### Assets and storage

- `packages/core/runtime/src/asset-store.ts:5-11` requires `themes` and
  `compositions` arrays and optionally accepts `catalog` and `initialTree`.
  `LoadedAssets` returns all three validated concepts
  (`packages/core/runtime/src/assets.ts:148-158`).
- Runtime seeds defaults, merges custom lists by name, and allows custom Theme/
  Composition shadowing (`packages/core/runtime/src/assets.ts:163-178,240-375`).
  Only the result arrays are shallow-frozen (`:427-434`).
- `FileAssets` reads sorted `*.theme.json` and `*.composition.json` collections
  plus `catalog.json` (`packages/core/runtime/src/file-assets.ts:51-56,92-128`).
  Absence and an explicitly empty Pattern list cannot be distinguished.
- Postgres persists `themes`, `compositions`, `catalog`, and `initial_tree`
  (`packages/extensions/store-postgres/src/postgres-assets.ts:4-75`), while
  `initSchema` creates/adds the same retired columns
  (`packages/extensions/store-postgres/src/postgres-store.ts:40-52`).
- Durable stage rows are returned without new-contract conversion
  (`postgres-store.ts:54-62`); old stage/session JSON is therefore also an
  operator migration concern, not just asset rows.

### Renderer and color state

- `StageRenderer` takes a Theme registry and selects by `tree.theme`
  (`packages/core/react/src/StageRenderer.tsx:41-80,128-140`).
- Device `scheme` is currently report-only view data
  (`packages/core/core/src/view.ts:13-15,52-60` and
  `packages/core/react/src/view-snapshot.ts:101-134`). Authored subtree
  `BoxStyle.scheme` instead changes the palette in
  `packages/core/react/src/renderer-render.tsx:394-408`.
- `renderer-render.tsx` (612 lines) is the central budgeted render walk. Existing
  role renderers already provide the correct scaffold for brick-owned styling;
  the central walk must not absorb every target/property resolver.
- `styleOf` currently casts an object (`brick-renderer-shared.ts:42-44`). The new
  nested target/state path must use total closed reads and never spread
  untrusted style objects into CSS.
- Flow containment is centralized in `layout-contract.ts`; renderer-owned
  overlay/backdrop placement is the only sanctioned absolute/fixed/z layer.

### Agent authoring and discovery

- Tool schemas are intentionally generic (`agent-tools/src/specs.ts:8-17`) and
  currently advertise `get_composition` and `set_theme` (`:19-164`). A complete
  eleven-brick style schema embedded here would be too large.
- `parseNodeInput` sanitizes a proposed node through `validateTree` and returns
  issues (`executor-input.ts:32-76`); `render_page` emits a patch even after
  sanitation (`executor-page.ts:5-47`); result formatting calls that success
  `applied_with_warnings` (`executor-result.ts:137-155`). This conflicts with
  strict atomic author errors.
- `StageToolAssets` contains Theme/Composition arrays plus Catalog
  (`agent-tools/src/types.ts:87-97`). The reference agent snapshots these once
  when its factory is created (`reference-agent/src/agent.ts:90-105`), not once
  per provider turn.
- Prompt guidance exposes Theme/Composition/Catalog indexes but not exact
  brick/property/token/Preset discovery (`agent-tools/src/prompt-kit.ts`).
- Composition reads are correctly read-only, but selection truncates and has no
  Theme/Preset compatibility (`composition-references.ts:23-47`).

### Downstream public consumers

- `@facet/agent` exposes `Stage.theme()` and writes `/theme`
  (`packages/extensions/agent/src/stage.ts:103-114`). CLI and bridge surface the
  same command (`packages/extensions/cli/src/commands.ts:50-69`,
  `packages/extensions/bridge/src/persistent.ts:86-98`).
- Quickstart passes Theme arrays/Catalog/Compositions to the reference agent and
  browser; its page re-selects by `tree.theme`. Its built guide contains the
  retired vocabulary, so it is production data rather than docs-only cleanup.
- Client, server, AG-UI, reference-agent message formatting, and tests carry the
  report field `scheme`; all must move atomically to `colorMode`.
- The literal sweep for Theme/Composition/Catalog/recipe/variant/tone/scheme and
  related names found candidates in 147 files across all source groups. This is
  a repository-wide public cut, not a Core-only rename.

## Locked technical resolutions for specification

1. Public Theme nesting is `name`, optional `description`, `tokens`, `defaults`,
   and optional `presets`.
2. `tokens` contains complete non-paint maps plus `paint.light` and
   `paint.dark`; each paint branch completely maps color, shadow, gradient,
   scrim, and highlight token names. Theme validation rejects any missing,
   unknown, unsafe, prototype, or over-limit definition.
3. `defaults` is exhaustive over all eleven bricks and their published nested
   targets. `presets` is brick-scoped; each record value has required
   `description`, `useWhen`, optional `avoidWhen`, and a `style` bundle. Record
   keys are the Preset names; names are not duplicated inside values.
4. Authored `style` and Theme style bundles use the same shape. Theme defaults/
   Presets cannot select another Preset. Authored base style may use `preset`.
   `style.active` may select a same-brick Preset and provide base target choices,
   but cannot recursively contain active/ephemeral state blocks.
5. Raw `AssetDocuments.patterns` is `unknown` so presence is preserved. Missing
   uses `DEFAULT_PATTERNS`; explicit `[]` exposes none; present malformed or
   over-64 rejects the explicit source to none. Individual invalid/incompatible
   Patterns are hidden whole with bounded issues.
6. File assets are exactly optional `theme.json`, optional `patterns.json`
   (array), and optional `initial.tree.json`. Retired recognized filenames are
   reported but never opened as a compatible source.
7. Postgres uses a replacement `facet_assets` schema with singular `theme`,
   exact `patterns`, and `initial_tree`. `initSchema` does not alter an old table
   into the new one; operators replace/migrate it explicitly before deployment.
8. View snapshots report effective `colorMode: "light" | "dark"`. Renderer host
   input is `ColorModePreference = "system" | "light" | "dark"`; SSR system
   fallback is light until hydrated. Forced modes ignore media-query changes.
9. No new renderer diagnostics callback is added. Agent repair comes from strict
   tool results; Core fold/asset APIs already return issues; final raw rendering
   remains a silent last defense.
10. A reference-agent asset source is acquired once at the start of each
    provider turn and deeply snapshotted. Static sources remain valid; dynamic
    hosts may return a new snapshot next turn but are responsible for passing
    the matching Theme to their renderer. Quickstart uses one static loaded
    snapshot; no browser asset fetch or hot-reload protocol is added.
11. The release uses one pre-1.0 `minor` changeset covering all affected
    published packages. No alias, dual reader/writer, feature flag, runtime
    migration, or mixed-version support ships.
