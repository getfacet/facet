## Scope

This context covers a feature change to make LLM-facing Facet stage tool results
structured, explicit, bounded, and documented. Implementation has not started.

## Risk Probe: INV

### RISK-INV-1 - Structured observations must not widen the stage boundary

Evidence:
- Declarative brick and token-only rules are load-bearing:
  `AGENTS.md:15-24`.
- This feature changes observation metadata only; no node schema or token
  schema is targeted.

Resolution requirement:
- The contract may describe patch application and visibility, but must not add
  raw HTML, JS, CSS, pixel values, new node kinds, or provider/domain tools.

### RISK-INV-2 - Visibility claims can overstate browser-local state

Evidence:
- Browser-local `navigate`/`toggle` state runs without the agent and is not the
  server-authoritative stage: `docs/ARCHITECTURE.md` describes local actions in
  the event-loop section.
- The tool buffer only knows the local stage shadow, not each browser tab's
  local view state.

Resolution requirement:
- Define visibility as stage reachability from the active server render root,
  plus visible metadata changes such as `/theme`. Document the limitation.
- Do not claim knowledge of per-tab local navigate/toggle state.

### RISK-INV-3 - Pending/buffered edits must stay fail-safe

Evidence:
- Buffer currently queues forward-referenced boxes and returns text-only
  `queued` observations without emitted messages:
  `packages/agent-stack/agent-tools/src/buffer.ts:127-167`.

Resolution requirement:
- Pending must be explicit and not treated as success. It emits no patch and
  keeps bounded missing-id previews.

## Risk Probe: API

### RISK-API-1 - Observation string changes are visible behavior

Evidence:
- Executor tests pin exact observation text:
  `packages/agent-stack/agent-tools/src/executor.test.ts:43-46`.
- Provider transcript passes observation strings directly:
  `packages/agent-stack/reference-agent/src/harness/loop.ts:326-337`.

Resolution requirement:
- Treat this as an intentional behavior change in `@facet/agent-tools`.
- Preserve a string form for provider compatibility, but make it structured and
  tested.
- Update docs and add a changeset if published package behavior changes.

### RISK-API-2 - Existing result types must remain usable by external authors

Evidence:
- `StageToolResult`, `StageToolObservation`, and related types are exported from
  `packages/agent-stack/agent-tools/src/index.ts:22-44`.

Resolution requirement:
- Additive types/helpers are preferred. If `StageToolObservation.text` becomes
  JSON, keep the field present and bounded.
- Update public README examples to show how to parse/read the result.

## Risk Probe: PKG

### RISK-PKG-1 - Agent-tools must remain provider-free and core-only

Evidence:
- `@facet/agent-tools` README says it does not choose a model, make provider
  requests, read env vars, or own reference policy:
  `packages/agent-stack/agent-tools/README.md:8-17`.

Resolution requirement:
- Implement formatter and visibility helpers inside `@facet/agent-tools` using
  only `@facet/core` data.
- No new dependency or provider import.

### RISK-PKG-2 - Reference-agent prompt updates must not fork the tool contract

Evidence:
- Reference-agent imports `TOOLS` from `@facet/agent-tools` at
  `packages/agent-stack/reference-agent/src/prompt/system.ts:5`.

Resolution requirement:
- Put the durable contract in `docs/AGENT-TOOL-RESULT-CONTRACT.md` and point
  package docs/prompt tests at the same behavior, avoiding duplicate policy
  strings that can drift.
