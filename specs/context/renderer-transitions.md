## Risk Probe: INV

- RISK-INV-1: Transition bookkeeping can bypass the renderer fail-safe.
  Evidence: `StageRenderer.tsx:938-974` contains the existing null/depth/budget
  guards; lifecycle visible-id collection must use the same semantics and never
  do an unbounded secondary traversal.
- RISK-INV-2: Exit visuals and previous-stage snapshots can become a second
  interactive writer or leak stale field values.
  Evidence: `StageRenderer.tsx:151-263` collects mounted fields by scanning
  `[data-facet-field-id]` under `stageRootRef`; `StageRenderer.tsx:777-789`
  sends agent actions with collected fields. Any stale visual inside the scope
  must be inert and must not stamp field ids.
- RISK-INV-3: Stage-level crossfade can violate flow-only layout or remount
  stable subtree state.
  Evidence: `AGENTS.md:14-18` forbids agent-authored absolute layout;
  `StageRenderer.tsx:840-852` currently uses `display: contents` to avoid
  changing layout; `theme.ts:188-189` maps `grow`/`width` onto normal flow CSS.
- RISK-INV-4: Same-id updates could accidentally animate/remount and lose DOM
  view-state.
  Evidence: `StageRenderer.interaction.test.tsx:2035-2055` asserts scroll
  identity survives an unrelated patch; `StageRenderer.interaction.test.tsx:2090-2121`
  asserts adding/removing an appear stylesheet does not remount the stage.
- RISK-INV-5: Rapid patches can leave stale exit/crossfade state after the
  authoritative tree has moved on.
  Evidence: `useFacet.ts:51-77` folds every incoming patch frame immediately;
  renderer visual state must never block or reorder patch folding.

## Risk Probe: API

- RISK-API-1: `FacetState` and `StageRendererProps` are published through the
  `@facet/react` barrel; any new fields must be additive and optional for
  existing consumers.
  Evidence: `useFacet.ts:17-28`, `StageRenderer.tsx:652-680`, and
  `index.ts:1-4`.
- RISK-API-2: Browser View Transitions or React `flushSync` would pull
  `react-dom` into `@facet/react` runtime dependencies.
  Evidence: `packages/react/package.json:31-33` peers only `react`; grep found
  no current `flushSync`/`startViewTransition` source usage.
- RISK-API-3: Existing `appear` and static output are pinned; motion CSS/classes
  must not make token-free first render byte-different.
  Evidence: `StageRenderer.test.ts:443-651`.
- RISK-API-4: Root-replace crossfade needs patch-shape metadata; consumers that
  omit the new prop can still get brick lifecycle diffs but cannot reliably
  identify `render_page` when ids are reused.
  Evidence: `useFacet.ts:47-85` sees raw patch messages, while `StageRenderer`
  currently receives only the folded tree.

## Risk Probe: PKG

- RISK-PKG-1: The v1 scope must not extend the core token/protocol vocabulary or
  server/runtime transport fields. A core fold-result metadata bit is acceptable
  only if it keeps the patch wire format unchanged and preserves salvage
  semantics as the single source of truth.
  Evidence: `AGENTS.md:37-49` package map; `ServerMessage` remains
  patches/say/reset at `protocol.ts:133-136`.
- RISK-PKG-2: Internal CSS helpers must preserve package publish boundaries.
  Evidence: `packages/react/package.json:17` has `"sideEffects": false`;
  existing `appear.ts` is internal and not barrel-exported.
- RISK-PKG-3: Touching `packages/react` and `packages/quickstart` triggers the
  live-link hard gate.
  Evidence: `AGENTS.md:89-103` and `AGENTS.md:111-115`.
