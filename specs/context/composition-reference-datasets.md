## 1. Task Intent

Replace Facet's functional composition system with a read-only reference-data
model. A composition becomes one validated concrete native subtree
`{ name, metadata, root, nodes }`. The model sees only a name/short-description
index, may call one name-only `get_composition` tool to read the full concrete
JSON, and then authors ordinary native Facet nodes through the existing stage
tools. Remove slots, markers, composition references, graph validation,
expansion, `use_composition`, and `Stage.useComposition` with no compatibility
alias. This is a pre-1.0 hard cutover; implementation remains blocked pending
owner approval of the dev spec.

## 5. Current-State Evidence

### 5.1 Functional composition is encoded in core

- `FacetComposition` has top-level `description?`, `slots?`, and
  `nodes: FacetNode | CompositionRef`
  (`composition-validation.ts:61-68`).
- `CompositionRef` is a public non-node reference (`:70-81`).
- Composition validation explicitly enables `allowSlotMarkers` and
  `allowReference` (`:155-158`).
- `sanitizeNode` admits `{use,slots}` only under that option
  (`primitive-node-validation.ts:780-805`).
- Expansion-specific fill/string-leaf handlers are attached to every registry
  entry (`brick-registry.ts:61-98`, `:142-168`).
- Public graph and expansion names are re-exported through `validate.ts` and
  `index.ts`.

### 5.2 Runtime load currently has two composition gates

- Each raw document runs through `validateComposition` and dedupe/shadow
  handling (`runtime/src/assets.ts:310-376`).
- The accepted array then runs through `validateCompositionGraph`
  (`:378-395`).
- The new model retains the first gate and ordinary native subtree bounds, but
  removes the second graph-specific gate.

### 5.3 Prompt and lookup can drift today

- The reference agent already clones once and passes one `compositions`
  variable to both prompt assets and executor assets (`agent.ts:89-103`).
- The prompt indexes every supplied composition (`prompt-kit.ts:281-306`),
  while execution separately applies catalog composition policy
  (`executor-policy.ts:99-109`). One new pure selector must solely own exposure
  filtering for both consumers; executor policy must lose its composition
  predicate and stale authoring guidance.
- Current catalog authoring order is a public literal
  `['composition','component','primitive']` (`catalog-types.ts:39`), which
  contradicts the locked decision that composition is not an authoring tier.
  The composition allow-list remains useful as reference-dataset exposure
  policy, but authoring order must become `['component','primitive']`.

### 5.4 Full JSON is impossible under the current observation path

- Generic observation `data` is replaced with `{"truncated":true}` above
  2,048 characters (`agent-tools/src/observation.ts:20`, `:127-135`).
- The reference harness then truncates each tool result at 4K/8K/12K by preset
  (`reference-agent/src/harness/budget.ts:72-121`;
  `harness/transcript.ts:54-106`).
- The largest current default is already 1,763 serialized characters before
  required reference inlining; the general composition node cap is 1,023
  (`composition-validation.ts:23`).
- Required resolution: `get_composition` has a dedicated exact-data mode that
  bypasses both observation truncation layers. The normal hard context check
  remains; if the complete result cannot fit, the loop stops with
  `context_limit` and never sends a partial composition.
- `harness/in-turn-compaction.ts` can retain zero recent groups when
  `minRecentStepsVerbatim` is zero, so the newest exact read could otherwise be
  summarized before its first provider delivery. The implementation plan must
  pin that newest group verbatim through the first handoff or stop
  `context_limit` when the whole group cannot fit.

### 5.5 Additional public and persisted consumers

- `@facet/agent` exposes the omitted public expansion method
  `Stage.useComposition` (`stage.ts:91-132`). There is no replacement; code
  agents already use `render`, `set`, and `append`.
- Postgres stores raw unknown JSONB and should remain unchanged; `loadAssets`
  stays the semantic gate. Its fixture uses top-level `description` and old
  catalog order (`postgres-assets.test.ts:75-101`).
- FileAssets also stores raw unknown documents; its canonical fixture currently
  omits metadata (`runtime/src/file-assets.test.ts:111-121`).

## 11. Assumptions and Scope Locks

1. The new concrete shape requires `metadata.description`; other metadata
   fields remain optional and are returned only by `get_composition`.
2. “Native Facet node” means the current `FacetNode` union, including deliberate
   intrinsic components already in core. This task does not perform a separate
   box-only node-model migration.
3. A composition may have any native leaf or container root, matching current
   fragment semantics; it is not required to be a full stage/container root.
4. Marker-looking text such as `{{label}}` receives no special parsing. Official
   datasets contain none; a custom literal string is ordinary content.
5. Historical `specs/**` files are archival evidence and are excluded from the
   shipping-surface zero-hit gate. Production code/tests, current docs, examples,
   fixtures, package READMEs, and unreleased changesets are not exempt.
6. `get_composition` returns exact serialized JSON in the standard structured
   observation's data field. It bypasses data/transcript truncation only for this
   tool; context overflow remains a fail-safe stop.
7. The public selector signature is exactly
   `selectCompositionReferences(compositions: readonly unknown[], catalog?: unknown): readonly FacetComposition[]`;
   omitted catalog exposes all valid references, supplied malformed catalog
   exposes none, and the returned array/nested composition objects retain no
   caller references and are deeply frozen.
