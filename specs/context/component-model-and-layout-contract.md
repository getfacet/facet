## Code Shape Evidence

- `packages/core/core/src/nodes.ts` defines the closed node vocabulary and still
  names common UI nouns as `HIGH_LEVEL_NODE_TYPES` around its exported node type
  constants. It also states the vocabulary grows only through typed node shapes
  plus matching validation and renderer support.
- `packages/core/core/src/nodes.ts` defines primitive nodes `box`, `text`,
  `media`, and `field`, existing components including `button`, `section`,
  `card`, `tabs`, `table`, `chart`, `stat`, `badge`, `progress`, `alert`,
  `list`, and `divider`, and `isContainer` currently narrows only `box`,
  `section`, and `card`.
- `packages/core/core/src/catalog.ts` exposes the public catalog shape as
  `bricks`, `stamps`, `primitiveFallback`, and `policy.order:
  ["stamp","brick","primitive"]`. `DEFAULT_CATALOG` advertises current brick
  types and `stat`.
- `packages/core/core/src/spec.ts` exports `STAGE_SPEC`, which prompt consumers
  embed. It currently teaches "high-level brick" and "stamp -> high-level brick
  -> primitive fallback".
- `packages/core/core/src/theme.ts` exposes `FacetTheme.recipes`,
  `ComponentRecipe`, and `RECIPE_COMPONENTS`; these are style recipes, so
  structural user-defined components must not reuse the name `recipe`.
- `packages/core/core/src/validate.ts` is already large and centralizes
  sanitizer switch cases for every node type plus stamp validation. New
  component validation should use role-specific sibling modules where that
  improves ownership and testability, not generic helpers.
- `packages/core/runtime/src/assets.ts` loads raw operator documents for themes,
  stamps, catalog, and initial tree; it validates with core validators and never
  throws. This is the right runtime boundary for immutable composition assets.
- `packages/core/react/src/StageRenderer.tsx` applies live patches through a raw
  local coercion path before render and enumerates known node types for discovery
  and rendering. Each new intrinsic must validate, render, and survive this live
  path fail-safely.
- `packages/core/react/src/brick-renderers.tsx` centralizes component renderers
  and defaults unknown nodes to `null`.
- `packages/core/react/src/theme.ts` contains `boxStyle`, `textStyle`,
  `mediaStyle`, and `fieldStyle`. `boxStyle` owns flex/grid style conversion and
  has partial scroll containment, but no shared child-root containment contract
  currently applies to every primitive/component root.
- `packages/agent-stack/agent-tools/src/prompt-kit.ts` still says "stamp ->
  high-level brick -> primitive fallback" and renders catalog "allowed bricks"
  plus stamp policy into the model prompt.
- `packages/extensions/agent/src/stage.ts` exposes `Stage.useStamp`, so stamps
  are a compatibility API and cannot be silently removed in this feature.

## Consumer Sweep

Searches run across `packages`, `apps`, `docs`, `README.md`, and the intake
brief:

- `STAGE_SPEC`
- `stat`, `"stat"`, `type: "stat"`

  `.stamp.json`
- `recipes`, `ComponentRecipe`, `RECIPE_COMPONENTS`, `RecipeComponentName`,
  `DEFAULT_THEME.recipes`

Findings:

- Public vocabulary and catalog terms are spread across core, assets, runtime,
  react, agent-tools, reference-agent, quickstart, bridge, docs, and tests.
- `stat` is a persisted stage node and default asset value. A hard rename to
  `metric` would silently drop existing stages or default stamps.
- `FacetStamp`, `expandStamp`, `validateStamp`, `DEFAULT_STAMPS`, `.stamp.json`,
  `Stage.useStamp`, and the `use_stamp` tool are public/storage compatibility
  surfaces.

## Implementation Direction Chosen

- Additive migration, not hard rename.
- Agent-facing terms become `Primitive Brick`, `Component`, and `Catalog`.
- Primitive bricks stay exactly `box`, `text`, `media`, and `field`.
- New public core aliases should prefer `INTRINSIC_COMPONENT_TYPES` and
  `CatalogComponent`; legacy `HIGH_LEVEL_NODE_TYPES`,
  `HighLevelNodeType`, `CATALOG_BRICK_TYPES`, and old catalog fields stay
  accepted.
- Current stamps remain as compatibility APIs. New composition/component
  definitions may normalize to the same safe expansion model, but theme
  `recipes` remain style recipes.
- `metric` is added as the preferred intrinsic; `stat` remains a legacy alias.
- `modal`, `drawer`, `menu`, `timeline`, `steps`, `calendar`, `kanban`, and
  `appShell` stay deferred.
- Renderer layout contract is implemented as private renderer-owned containment
  helpers plus tests, not only documentation.
