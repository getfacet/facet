# Feature Intake Brief: Reference Agent Harness PR2

> Generated by `/feature-intake`. Plain-language, testable, invariant-checked.

## Goal

- Rebuild `@facet/reference-agent` from the current single-file tool loop into a
  stable reference harness that external developers can read, copy, and extend
  without re-implementing context assembly, transcript handling, compaction,
  budget/limit policy, tracing, and fail-safe stop behavior from scratch.
- Keep PR2 scoped to the reference-agent package, using PR1's
  `@facet/agent-tools` as the canonical stage-tool mechanism.

## Current Problem

- The current reference agent works, but too much critical behavior lives inside
  `packages/agent-stack/reference-agent/src/agent.ts`: sink history loading,
  provider loop control, tool execution, tool-result feedback, streamed batch
  coalescing, fallback chat, and unresolved-buffer handling.
- Prompt context is still assembled by directly embedding the current stage JSON
  and capped history count, but there is no explicit context budget, budget
  profile, deterministic compaction policy, observation cap, trace surface, or
  named stop reason contract.
- A developer can see that the reference agent works, but cannot easily identify
  the reusable harness pieces that make a tool-calling UI agent reliable.

## User Scenario

### Who

- Primary: an agent author evaluating Facet's reference agent as the pattern for
  their own LLM-backed Facet brain.
- Secondary: Facet maintainers who need the quickstart/live-test path to remain
  stable while the reference-agent internals become more modular.

### Trigger

- A visitor sends a `visit`, `message`, or forwarded `tap` event to a quickstart
  page backed by `@facet/reference-agent`.

### Steps

1. The runtime calls `createQuickstartAgent` / `createReferenceAgent` with the
   current event, session stage, sink, provider, guide, themes, and stamps.
2. The reference agent assembles a bounded provider context from recent sink
   history, the current event, a bounded stage summary / stage JSON payload, and
   deterministic compaction when limits would be exceeded.
3. The harness runs provider steps, executes stage tools through
   `@facet/agent-tools`, appends every tool observation to the transcript before
   the next provider call, and yields streamed batches as they become ready.
4. The budget policy retries only explicitly retryable provider failures, then
   stops the loop on clean provider stop, step/tool/context limits, provider
   failure, or unresolved buffered edits, recording a stable stop reason.
5. The public factory preserves existing behavior and compatibility exports, so
   `@facet/quickstart` and existing tests keep working.

## Input/Output Examples

### Example 1 (Normal)

- Input: provider step 1 calls `render_page`, provider step 2 calls `say`, then
  provider step 3 returns no tool calls.
- Output: the stage patch is yielded before the say batch; both tool
  observations are present in the provider transcript before the final provider
  call; the trace ends with `stop_reason: provider_stop`; no retry is attempted.

### Example 2 (Edge/Error)

- Input: the current stage is too large for the configured context budget, or a
  provider step emits a huge observation / too many tool calls.
- Output: context assembly emits a bounded summary plus an inspect-tool hint
  instead of unbounded stage JSON; oversized observations are truncated with an
  explicit marker; retryable provider failures retry once by default; the loop
  stops or continues according to the configured budget without throwing, and
  prior streamed stage changes are preserved.

## Constraints (What Not to Do)

- Do not add a dependency on Pi, OpenClaw, Hermes, LangGraph, or any other agent
  framework.
- Do not change `@facet/core`, runtime patch semantics, renderer behavior, or the
  protocol.
- Do not move stage-tool schema/execution back into `@facet/reference-agent`;
  `@facet/agent-tools` remains canonical after PR1.
- Do not expose raw HTML, JS, CSS, pixel values, or any backend/domain fetch
  capability to the model.
- Do not make the browser a second writer of stage content.
- Do not do cross-provider fallback inside the harness; switching providers is
  an operator policy decision, not an automatic reference-agent behavior.
- Do not break existing root exports such as `createQuickstartAgent`,
  `createReferenceAgent`, `QuickstartProvider`, `TOOLS`, `buildSystem`,
  `buildInitialMessages`, or the quickstart re-export path.

## Policy & Edge Cases

| Scenario Step | Failure Policy | Boundary/Validation | Concurrency / Duplicate Trigger |
|---|---|---|---|
| Load sink history and current stage | Sink failure degrades to the existing single failure say if nothing was emitted; no throw escapes the agent. | History entries are capped by count and char budget; corrupt stored events describe as `(unknown event)`. | Runtime still serializes per visitor; harness treats one event at a time. |
| Assemble context | If context exceeds budget, compact deterministically before provider call. | Stage JSON is included only when under the configured cap; otherwise summary plus inspect-tool hint is used. | No cross-turn mutable global context; each turn starts from `session.stage`. |
| Execute provider step | Retryable provider rejection is retried up to the configured retry count; non-retryable rejection stops the loop while preserving prior yielded batches. | Provider output shape remains adapter-owned; tool calls are parsed into `ToolCall`; retryable statuses are fixed in the dev spec. | Max steps and max tool calls bound runaway loops. |
| Execute tools and feed results | Bad tool input becomes a tool-result observation, not a thrown turn. | Observations are capped; stage mutations go through `@facet/agent-tools` and RFC 6902 patches. | Tool calls in one provider step are executed in model order; observations are appended before the next provider run. |
| Stop / fallback | Clean stop surfaces final prose only when no `say` tool was emitted; empty/no-op turn emits one failure say. | Stop reasons are stable string values for tests and trace. | Repeated empty provider steps stop by max step policy. |

## Done Criteria (Testable)

- [ ] DC-001 (Structure): reference-agent has named `provider/`, `prompt/`, and
  `harness/` modules, while compatibility barrels keep existing public exports.
  -> Verify: `rg`/vitest export checks.
- [ ] DC-002 (Tool feedback): every executed tool result is appended to the
  provider transcript before the next provider call, including mixed success and
  error observations. -> Verify: reference-agent loop tests.
- [ ] DC-003 (Context limits): context assembly applies bounded history, bounded
  stage JSON, deterministic compaction, and inspect-tool hints for large stages.
  -> Verify: context/compaction unit tests.
- [ ] DC-004 (Budget/stop reasons): max steps, max tool calls per step, max
  context chars, and max observation chars are enforced with stable stop
  reasons and no uncaught throw. The allowed stop reason strings are closed and
  test-pinned. Budget profile defaults are explicit: `quickstart` is the safe
  default, while `hosted` and `local-dev` are opt-in. -> Verify: budget/loop
  unit tests.
- [ ] DC-005 (Fail-safe): provider/sink failures, malformed tool calls,
  unresolved buffered edits, and oversized outputs preserve prior emitted edits
  and produce at most one fallback say when needed. -> Verify: existing and new
  agent tests.
- [ ] DC-006 (Compatibility): quickstart imports, stub behavior, provider
  adapters, and public package-root type aliases remain source-compatible.
  -> Verify: reference-agent, quickstart, provider, and index tests.
- [ ] DC-007 (Boundary): no new Facet core/runtime/client protocol surface is
  introduced; `@facet/agent-tools` remains the only stage-tool executor package.
  -> Verify: import guards and package tests.
- [ ] DC-008 (Retry/trace contract): provider retry policy, fallback policy,
  trace event names, and trace redaction rules are exact and covered by tests.
  -> Verify: budget/loop/trace tests.

## Invariant Fit (Facet-specific - REQUIRED)

| # | Invariant | Status | Note / mitigation |
|---|---|---|---|
| 1 | Responsibility boundary (UI-out/UI-in only; no backend/domain) | OK | PR2 only restructures the reference brain harness and provider adapters. It adds no domain tool, data fetch helper, or backend capability. |
| 2 | Mechanism vs policy (agent authors; Facet provides capability) | TOUCHES | The harness is reference policy/mechanism for LLM turns, but remains in `@facet/reference-agent`. Core/runtime stay mechanism-only, and external authors can still replace the brain. |
| 3 | Fail-safe (never throw/inject; degrade to plain) | TOUCHES | Context, budget, and loop modules must return typed outcomes / fallback messages on bad input, provider failure, and malformed history rather than throwing through the agent turn. |
| 4 | Declarative + tokens only (no raw HTML/JS/CSS/pixels) | OK | Stage changes still come only from `@facet/agent-tools` and existing `FacetNode` / `FacetTree` validation. |
| 5 | Flow-only safety (overlap only via a dedicated overlay brick) | OK | No layout vocabulary or renderer behavior changes. |
| 6 | Two-writers coherence (local execution vs agent stage) | TOUCHES | The harness starts each event from runtime-provided `session.stage`; local shadows exist only inside a provider turn and are realigned after yielded patch batches through `@facet/agent-tools`. |
| 7 | Backend calls go through the agent, never client fetch | OK | Provider HTTP calls remain server-side reference-agent adapter behavior; no client fetch or page-authored backend path is added. |
