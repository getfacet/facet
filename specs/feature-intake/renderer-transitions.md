# Feature Intake Brief: Renderer Transitions

> Generated by `/feature-intake`. Plain-language, testable, invariant-checked.

## Goal

- Make live agent-driven stage updates feel intentional instead of flickery by
  adding a small, renderer-owned transition policy:
  1. brick lifecycle changes get short enter/exit motion,
  2. whole-stage redraws get a short crossfade,
  3. same-id content/style updates stay immediate by default.
- Preserve Facet's current contract: agents still emit only declarative bricks
  and JSON patches; no raw animation values, no new backend behavior, and no
  protocol changes.

## Current Problem

- Today, `useFacet` folds each incoming patch immediately into React state, and
  `StageRenderer` renders the new tree immediately.
- The existing `appear` token helps only when a box mounts or re-shows. It does
  not cover exit, whole-page redraws, or default lifecycle transitions for newly
  visible bricks.
- End-users see abrupt UI jumps when an agent adds/removes UI or calls
  `render_page`; agent authors have to overuse `appear` and still cannot smooth
  removal or full redraws.

## User Scenario

### Who

- **End-user of an agent-built page**: sees live UI updates while chatting.
- **Agent author / framework consumer**: wants smoother updates without
  micromanaging animation code.
- **Agent brain**: keeps emitting ordinary Facet patches; it does not learn a new
  animation DSL for this feature.

### Trigger

- A visitor sends a message or taps an agent action, and the agent updates the
  stage through existing patch messages.

### Steps

1. The current rendered stage has a set of visible brick ids.
2. A patch arrives and the client folds it into the next tree.
3. Facet compares the previously visible brick ids with the next visible brick
   ids.
4. If a brick id is newly visible, it gets a short brick-level enter transition.
5. If a brick id is no longer visible, it gets a short brick-level exit
   transition before its DOM disappears.
6. If the patch is a root document replace, or too many bricks enter/exit at
   once, Facet uses one stage-level crossfade instead of many brick animations.
7. If the same brick id remains visible and only its content/style changes, it
   updates immediately with no default animation.
8. If the user has reduced-motion enabled, all transition motion is suppressed
   and the final UI still appears correctly.

## Input/Output Examples

### Example 1 (Normal — incremental append)

- Input: the agent sends `append_node` that adds a new card under an existing
  list box.
- Output: the new card appears with a short brick enter transition; existing
  list items stay still and keep DOM identity.

### Example 2 (Normal — remove)

- Input: the agent removes a visible card node.
- Output: the card briefly fades out, becomes inert while exiting, then is
  removed from the DOM. Other visible bricks are not remounted.

### Example 3 (Normal — root replace)

- Input: the agent calls `render_page`, producing a patch with `replace ""`.
- Output: Facet treats this as a major redraw and crossfades the stage as a
  whole instead of trying to animate every brick individually.

### Example 4 (Edge/Error)

- Input: a malformed patch folds to a sanitized tree, or a brick is removed and
  then re-added with the same id before its exit finishes.
- Output: the renderer never throws. Latest tree state wins; stale exit visuals
  are cancelled or dropped so there is only one live representation of that id.

## Constraints (What Not to Do)

- No new raw animation values in stage trees: no durations, delays, easing
  curves, transforms, CSS strings, pixels, or keyframes authored by the agent.
- No new backend, storage, fetch, event, or transport capability.
- No new agent-authored animation DSL in v1.
- No default animation for same-id text/style/media updates.
- No remounting stable visible bricks just to animate; field values, focus, and
  scroll offsets must survive unrelated updates.
- No persistent overlay/z-index capability. Any stage-level crossfade is a
  renderer-owned temporary visual effect, not an agent-authored layout feature.
- Do not remove or break the existing `box.style.appear` behavior. Existing
  trees using `appear: "fade"` or `appear: "slide"` must keep working.

## Policy & Edge Cases

| Scenario Step | Failure Policy | Boundary/Validation | Concurrency / Duplicate Trigger |
|---|---|---|---|
| Newly visible brick id | Apply renderer-owned enter transition; if motion is disabled, render instantly | Visible ids are derived from the same render walk/fail-safe rules as StageRenderer | Rapid successive patches may shorten/cancel a visual transition; final tree wins |
| Existing visible brick id remains visible | No default transition; update in place | Same id means update, regardless of text/style/media changes | DOM identity, focus, field values, and scroll offsets must not be reset by unrelated updates |
| Visible brick id disappears | Keep a temporary inert exit visual, then remove it | Exit visual is renderer state only; no stage mutation | If the same id reappears before exit ends, cancel stale exit and show the live brick |
| Root document replace (`replace ""`) | Use stage-level crossfade, not brick-by-brick lifecycle transitions | Patch detection is client-side; server/protocol unchanged | Newer patches win; crossfade must never block patch folding or event sending |
| Many simultaneous enter/exit ids | Use stage-level crossfade to avoid noisy per-brick animation | Default threshold is a renderer constant; no tree-authored number | Threshold prevents a large redraw from animating dozens of independent bricks |
| Existing `appear` token | Preserve current explicit enter behavior for boxes | Existing core vocabulary remains valid | Existing tests and behavior stay compatible |
| Reduced motion | Suppress motion, keep final content visible | CSS/media-query or equivalent browser check is renderer-owned | n/a |
| Malformed or hostile tree shape | Render fail-safe as today; transition layer must not throw | No new trust in raw patch data | Transition bookkeeping must tolerate missing, null, duplicated, or cyclic nodes |

## Done Criteria (Testable)

- [ ] DC-001 (Happy path — brick enter): when a patch makes a previously absent
  visible brick id appear, the rendered brick receives a short enter transition
  without remounting unrelated visible bricks → Verify: jsdom interaction/render
  test.
- [ ] DC-002 (Happy path — brick exit): when a visible brick id disappears, the
  old brick remains briefly as an inert exit visual, then is removed → Verify:
  jsdom fake-timer test.
- [ ] DC-003 (Happy path — stage replace): a root document replace triggers one
  stage-level crossfade path and does not run noisy per-brick enter/exit for the
  whole tree → Verify: `useFacet`/renderer test with a `replace ""` patch.
- [ ] DC-004 (Policy boundary — same-id update): changing text value, media src,
  or box style for an id that remains visible does not trigger default motion or
  remount unrelated stateful DOM → Verify: jsdom test preserving field value,
  focus or scroll offset.
- [ ] DC-005 (Policy boundary — many lifecycle changes): when simultaneous
  enter+exit count exceeds the renderer threshold, the renderer uses stage
  crossfade rather than per-brick lifecycle transitions → Verify: unit/jsdom test
  with threshold-crossing patch.
- [ ] DC-006 (Reduced motion): reduced-motion mode disables transition motion
  while still rendering the final tree → Verify: CSS/unit test for scoped reduced
  motion handling.
- [ ] DC-007 (Fail-safe): malformed, null, dangling, duplicated, cyclic, or
  budget-truncated tree shapes never make transition bookkeeping throw → Verify:
  renderer fail-safe tests around the existing StageRenderer malformed-tree
  cases.
- [ ] DC-008 (Back-compat): existing `appear` behavior remains supported and
  existing appear-free static output remains compatible except where client-only
  transition state is explicitly under test → Verify: existing `appear` tests and
  full test suite.
- [ ] DC-009 (Package/protocol boundary): the first implementation requires no
  new core token, no new server/runtime/client protocol message, and no backend
  call surface → Verify: public API/consumer sweep in `/spec-bridge` plus tests.
- [ ] DC-010 (Live-link gate): because `@facet/react` live rendering changes, the
  feature hard gate includes `/live-test` to verify quickstart still boots and
  streams updates → Verify: `/live-test` result in the implementation PR.

## Invariant Fit (Facet-specific — REQUIRED)

| # | Invariant | Status | Note / mitigation |
|---|---|---|---|
| 1 | Responsibility boundary (UI-out/UI-in only; no backend/domain) | OK | Pure rendering behavior. It does not fetch, compute domain data, store data, or add backend affordances. |
| 2 | Mechanism vs policy (agent authors; Facet provides capability) | TOUCHES | The renderer adds a default visual policy. It stays presentation-only, deterministic, and bounded; agents still author structure/behavior. Existing explicit `appear` remains the agent-authored enter hint. |
| 3 | Fail-safe (never throw/inject; degrade to plain) | TOUCHES | Transition diffing must run over the same sanitized/fail-safe renderable surface and tolerate bad raw patch shapes. Bad transition bookkeeping must degrade to immediate render, not crash. |
| 4 | Declarative + tokens only (no raw HTML/JS/CSS/pixels) | TOUCHES | No new tree-authored animation values are added. Durations/easing/thresholds are renderer constants or static CSS, never agent output. |
| 5 | Flow-only safety (overlap only via a dedicated overlay brick) | TOUCHES | Brick enter/exit must not create persistent floating UI. Stage crossfade may use a temporary renderer-owned visual snapshot/layer, with no agent-authored positioning and no final layout overlap. |
| 6 | Two-writers coherence (local execution vs agent stage) | TOUCHES | Transition state is browser-only view/presentation state. The server remains the only writer of stage content; latest folded tree always wins over in-flight visual effects. |
| 7 | Backend calls go through the agent, never client fetch | OK | No fetch surface. Agent-routed actions are unchanged. |
