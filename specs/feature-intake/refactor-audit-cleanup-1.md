## Current Problem

- Local coding agents can see `theme` in `STAGE_SPEC` and `Stage.theme()` exists,
  but the `facet` CLI and bridge tool surfaces do not expose a matching action.
  Affected users: operators using `@facet/bridge` with Claude/Codex.
- Several docs/process files describe old package state (`@facet/kit`, deferred
  transport/assets, `.claude`-only live-test policy, nested tree wording). Affected
  users: maintainers and coding agents following repo instructions.
- Minor hygiene findings remain in source: `ChatDock` bypasses the shared palette
  for two colors, `core/theme.ts` reuses a misleading range name for font sizes,
  and the playground carries two local flat-map authoring helpers.

## User Scenario

### Who

- The agent author/operator using Facet's local bridge and CLI.
- The maintainer or coding agent relying on repo docs and skill files.

### Trigger

- A coding agent uses `facet-bridge` and wants to switch a stage theme by name.
- A maintainer runs `/refactor-audit` and follows the generated cleanup plan.

### Steps

1. Local agent reads bridge instructions and `STAGE_SPEC`.
2. Local agent calls a documented `theme` action with a theme name.
3. Facet emits a normal `/theme` patch when the name is valid, or fails closed
   before sending a bad theme name.
4. Maintainer reads repo/package docs and sees current package boundaries,
   public APIs, and live-test policy.
5. Audit reruns no longer rediscover the same P2/P3 drift.

## Input/Output Examples

### Example 1 (Normal)

- Input: local agent runs `facet theme midnight`.
- Output: CLI returns a patch message with `{ "op": "add", "path": "/theme",
  "value": "midnight" }`; bridge prompts/tools document the same capability.

### Example 2 (Edge/Error)

- Input: local agent attempts `facet theme "Dark Mode!"`.
- Output: the CLI/tool refuses the name before emitting a patch, avoiding a live
  vs stored-stage divergence when `validateTree` later strips invalid values.

## Constraints (What Not to Do)

- Do not add a new brick, token, protocol message, renderer behavior, backend
  fetch, or browser-side writer.
- Do not change `.claude/**`; this pass updates Codex/repo-facing files only.

  command; existing commands must remain byte-for-byte compatible in behavior.
- Do not use raw CSS values in any agent-authored tree. `theme` remains a name.

## Policy & Edge Cases

| Scenario Step | Failure Policy | Boundary/Validation | Concurrency / Duplicate Trigger |
|---|---|---|---|
| CLI `theme` command | Missing/empty/invalid name throws a descriptive CLI error and emits no patch | Reuse `isValidThemeName`; name only, no CSS values | Multiple theme commands are ordinary ordered patches; last delivered patch wins |
| Bridge spawn prompt | If the model ignores `theme`, existing commands still work | Prompt lists only supported `facet` commands | No new concurrency path; bridge still serializes event handling |
| Persistent bridge tool | Invalid name returns an error tool result and records no stage op | Reuse `isValidThemeName`; tool schema remains a string name | Same persistent queue; no browser writer added |
| Docs/process drift | Stale references are updated or removed | Do not edit Claude-only workflows/skills | Documentation-only; no runtime effect |
| Palette/range/helper hygiene | Preserve current behavior while reducing drift | Use existing token maps and TypeScript types | No shared mutable state; playground helper only builds local demo trees |
