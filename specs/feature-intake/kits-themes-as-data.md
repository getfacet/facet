# Feature Intake Brief: Kits & Themes as Data (Bundle B)

> Generated by `/feature-intake`. Plain-language, testable, invariant-checked.
> Four parts, one flow: theme documents + `validateTheme` ‚Üí theme injection &
> named selection ‚Üí stamps (kit fragments) as prompt data ‚Üí per-agent initial
> tree. Everything feeds prompt layer ‚ë° of the quickstart built-in agent
> (Bundle A), which is why it can all be verified on a live link.

## Goal

- An operator reskins or pre-shapes an agent's page **with data, not code**: a
  theme document (token name ‚Üí CSS value), a stamp library (named brick
  fragments), and an initial tree (first-paint skeleton) are per-agent runtime
  data behind a pluggable registry ‚Äî the same adapter posture as `StageStore`.
- The LLM gains **theme choice without value authoring**: it selects a theme
  by name via a normal patch (`theme?: string` on the tree); it never emits a
  hex code or pixel. Unknown names fall back to the default, fail-safe.
- Any theme document ‚Äî operator-written today, customer-uploaded in a SaaS
  tomorrow ‚Äî passes through one core validator (`validateTheme`) before it can
  ever reach a page, so "safe by construction" survives the move from
  hardcoded constants to untrusted data.
- `@facet/kit` presets become instantiable more than once per page (id
  namespacing), and double as the default stamp library.
- First paint stops being gated on a model call when an initial tree exists
  (absorbs the README "static skeleton for fast first paint" item).

## Current Problem

- **Themes are compile-time constants.** `react/src/theme.ts` hardcodes every
  hex/px; reskinning means forking the renderer. There is no way to give two
  agents two looks, and no path to SaaS-style per-customer theming. Affected:
  every agent author/operator; blocks the Bundle C release story.
- **The model can't influence look-and-feel at all** ‚Äî not even safely. The
  design goal ("the agent styles freely, always on a coherent scale") stops at
  one fixed palette. Affected: the quickstart agent (prompt layer ‚ë° has
  nothing to advertise), every visitor who asks "make it darker".
- **Kit presets collide with themselves.** `Builder` mints sequential
  `k1‚Ä¶kn` ids and `page()` hardcodes `"root"`, so composing one preset twice
  (or dropping a preset into an existing tree) silently overwrites nodes.
  Affected: any consumer of `@facet/kit`; makes presets unusable as stamps.
- **No reusable composition vocabulary for the LLM.** Every page is composed
  from scratch, token by token; there is no way for an operator to say "this
  is what our product card looks like ‚Äî reuse it." Affected: quickstart page
  quality and one-shot reliability.
- **Every first paint waits for the model.** A visit event always runs a full
  LLM turn before anything renders. Affected: every end-user's first seconds;
  cost per visit.

## User Scenario

### Who
- **S1 (theming):** the operator (agent author) who writes theme documents;
  the built-in agent (brain) that selects one; the end-user who sees it.
- **S2 (stamps):** the operator who curates stamp fragments; the built-in
  agent that copies them into pages.
- **S3 (initial tree):** the operator who ships a first-paint skeleton; the
  end-user who sees an instant page.
- **S4 (kit):** an agent author using `@facet/kit` presets in code.

### Trigger
- S1‚ÄìS3: `npx facet-quickstart --assets ./my-assets` (a directory of theme /
  stamp / initial-tree documents), or any host embedding the registry.
- S4: `page([hero(...), card(...)])` composed twice, or a preset added into
  an existing tree.

### Steps (S1 ‚Äî theming)
1. Operator writes `midnight.theme.json` (name, one-line description, token
   values) into the assets directory and boots.
2. Facet: the registry loads it; `validateTheme` runs; invalid documents are
   skipped with logged issues; valid ones join the theme map. The map (default
   theme always included) ships to the browser **once, at page boot** ‚Äî the
   renderer resolves names locally from then on.
3. The quickstart prompt layer ‚ë° advertises the theme **names + one-line
   descriptions only** (never values).
4. End-user asks "make it darker"; the agent patches `theme: "midnight"` (one
   normal RFC 6902 op on the tree); the browser re-resolves tokens against the
   already-shipped document and repaints instantly ‚Äî no reload, no round trip.
5. Expected result: a live theme switch; an unknown name would render the
   default theme instead, without error.

### Steps (S2 ‚Äî stamps)
1. Operator drops `product-card.stamp.json` (name, description, a brick
   fragment: nodes + fragment root) into the assets directory.
2. Facet: the registry loads it; the fragment is validated (brick shapes,
   token membership ‚Äî same posture as `validateTree`); invalid stamps are
   excluded from the prompt with a logged warning.
3. Prompt layer ‚ë° injects the stamp fragments with a mandatory rule: **copy
   the fragment, prefix every id with a unique instance prefix**.
4. The agent, asked for "a page with two products", copies the stamp twice
   with distinct prefixes; the tree passes `validateTree` server-side as
   always. Expected result: consistent, operator-curated composition with
   zero spec/protocol change.

### Steps (S3 ‚Äî initial tree)
1. Operator ships `initial.tree.json` (a full `FacetTree`).
2. On a `visit` for a fresh session, the server seeds the session's stage from
   the initial tree **before** any model call ‚Äî the snapshot the browser gets
   already contains it (server-authoritative, same `applyPatch` path).
3. The agent's first turn then refines the seeded stage instead of painting
   from nothing. An invalid initial tree is ignored (logged), falling back to
   today's model-first paint.

### Steps (S4 ‚Äî kit namespacing)
1. Author composes the same preset twice in one `page()`, or instantiates a
   preset with an explicit id prefix to graft into an existing tree.
2. Facet: builder ids are namespaced per instantiation ‚Äî no collisions, no
   hardcoded `"root"` overwrite. Expected result: presets are safe to reuse,
   and the default stamp library can be generated from them.

## Input/Output Examples

### Example 1 (Normal ‚Äî live theme switch)
- Input: assets contain `midnight` (valid); stage tree gets a patch
  `{op:"add", path:"/theme", value:"midnight"}`.
- Output: browser repaints with midnight's values immediately (map already
  shipped at boot); server stage stores `theme:"midnight"`; chat and layout
  untouched.

### Example 2 (Edge ‚Äî unknown / non-string theme)
- Input: `theme:"no-such-theme"` (LLM typo), or `theme: 42` (malformed).
- Output: unknown name ‚Üí renderer resolves the **default** theme, nothing
  throws, page stays styled; non-string ‚Üí dropped by `validateTree`
  sanitization before it ever reaches the client (same posture as other
  malformed tree input).

### Example 3 (Edge ‚Äî hostile theme document)
- Input: a customer-uploaded document with
  `{"accent": "url(https://evil.example/x)", "bg": "expression(alert(1))",
  "__proto__": {"fg": "red"}}` and a `space.md` of `"9999999px"`.
- Output: `validateTheme` rejects `url()`/`expression()` values and malformed
  colors as **errors** (document refused), clamps out-of-range dimensions,
  and builds token maps on a null prototype so `__proto__`/`constructor`
  never resolve. The renderer can only ever see a validated map.

### Example 4 (Edge ‚Äî low contrast, legitimate)
- Input: a pastel brand theme where `fg` on `bg` computes below the WCAG
  contrast threshold, all values well-formed.
- Output: `validateTheme` reports a **warning** issue naming the failing pair
  and ratio; the document is still accepted. The caller (operator log today,
  SaaS dashboard later) decides policy ‚Äî Facet supplies the measurement.

### Example 5 (Edge ‚Äî stamp copied without a fresh prefix)
- Input: the LLM copies a stamp but reuses ids from an earlier instance.
- Output: nothing crashes ‚Äî `set_node`/`add` semantics replace those nodes
  (a wrong-looking but safe page), `validateTree` still passes, and the
  fail-safe renderer skips any dangling refs. The id-prefix rule in the
  prompt is the mitigation; the failure mode is cosmetic, never structural.

## Constraints (What Not to Do)

- **The LLM never authors theme values in v1.** No tool, no tree field, no
  prompt affordance accepts CSS values from the model ‚Äî it selects names
  only. (LLM-authored themes, if ever, go behind the same `validateTheme`.)
- **No client-side stamp expansion ‚Äî permanently.** Expansion in the browser
  would bypass `validateTree`, break the same-`applyPatch` invariant, and
  invite version skew. Phase 1 is prompt-copy only; server-side
  `{use, id, slots}` macro expansion is phase 3, **only if** copy error rates
  or token costs prove it necessary, and always before persist/forward.
- **No raw CSS passthrough into the tree.** The tree carries one new field:
  a theme *name*. Token discipline for agents is unchanged.
- **No new protocol messages.** Theme documents ride the existing page boot
  (v1); live pushing of edited documents is out of scope.
- **No hot reload of assets** ‚Äî registry contents are read at boot (v1);
  editing a document requires a reboot to take effect.
- **Out of scope for this bundle:** a Postgres/DB assets adapter (interface
  only; file + memory references ship), SaaS dashboards/admin UI, a shared
  stamp marketplace (but the import rules are locked now: strip `onPress` on
  import, re-mint instance ids), animation/overlay tokens (Bundle D), theme
  editing UI, per-visitor (as opposed to per-agent) assets.

## Policy & Edge Cases

| Scenario Step | Failure Policy | Boundary/Validation | Concurrency / Duplicate Trigger |
|---|---|---|---|
| Registry load at boot | Invalid theme/stamp/tree document ‚Üí skipped + logged issue; boot proceeds; default theme always present | `validateTheme` (colors: format allowlist; deny `url()`/`var()`/`expression()`/`javascript:`; dimensions clamped; null-proto maps; contrast ‚Üí warning); stamps/initial tree: brick-shape + token-membership validation | Assets read once at boot ‚Äî no mid-flight mutation races (v1) |
| Theme map ships to browser | Missing/empty map ‚Üí renderer falls back to built-in default theme | Only **validated** documents are ever serialized to the client | One map per page boot; per-visitor sessions independent |
| Agent sets `theme` | Unknown name ‚Üí default theme (fail-safe, silent to the visitor) | `validateTree` keeps `theme` only if a string; drops otherwise | Rapid switches = normal patch ordering, last-write-wins; no new writer |
| Prompt ‚ë° injection | Oversized/invalid stamp ‚Üí excluded + warning; theme with no description ‚Üí name alone | Names + one-line descriptions only, never values | Prompt built per turn from boot-loaded assets ‚Äî stable within a boot |
| Initial tree seeding | Invalid tree ‚Üí ignored (logged), model-first paint as today | Full `validateTree` before seeding | Seed happens once per fresh session, before the first agent turn; same serialized event loop as all stage writes |
| Kit multi-instantiation | ‚Äî | Ids namespaced per instantiation; no hardcoded `"root"` collision | Two instantiations of one preset never share ids |
| LLM copies stamp, bad prefix | Node replacement (cosmetic damage), never a crash or invalid tree | Prompt mandates unique instance prefix; server `validateTree` unchanged | Same-id copies resolve by patch order |

## Done Criteria (Testable)

- [ ] DC-001 (Happy path ‚Äî theming): a registered valid theme document +
  a `theme:"<name>"` patch renders with that document's values.
  ‚Üí Verify: vitest render check in `@facet/react` (DOM style assertion).
- [ ] DC-002 (Boundary ‚Äî unknown theme): `theme:"no-such-theme"` renders the
  default theme; nothing throws. ‚Üí Verify: vitest render check.
- [ ] DC-003 (Error ‚Äî hostile document): `validateTheme` rejects `url()`,
  `var()`, `expression()`, `javascript:`, and malformed color values as
  errors; clamps out-of-range dimensions; hostile keys (`__proto__`,
  `constructor`) never resolve (null-prototype maps).
  ‚Üí Verify: vitest table test in `@facet/core`.
- [ ] DC-004 (Boundary ‚Äî contrast): a low-contrast `fg`/`bg` pair yields a
  **warning** issue naming the pair + ratio, and the document is still
  accepted. ‚Üí Verify: vitest in `@facet/core`.
- [ ] DC-005 (Boundary ‚Äî tree sanitization): `validateTree` preserves a
  string `theme` and drops a non-string one. ‚Üí Verify: vitest in
  `@facet/core`.
- [ ] DC-006 (Happy path ‚Äî kit): one preset instantiated twice in a page
  produces zero id collisions; a preset can be grafted with an explicit
  prefix without touching `"root"`. ‚Üí Verify: vitest in `@facet/kit`.
- [ ] DC-007 (Happy path ‚Äî registry): the file-backed assets reference
  round-trips themes/stamps/initial tree per agent; invalid files are
  skipped with issues; the in-memory reference behaves identically.
  ‚Üí Verify: vitest in `@facet/runtime` (or the registry's home package).
- [ ] DC-008 (Happy path ‚Äî prompt ‚ë°): with assets present, the built system
  prompt contains theme names + descriptions (no values), stamp fragments,
  and the id-prefix rule; with no assets, it is byte-identical to today's.
  ‚Üí Verify: vitest on `@facet/quickstart` prompt building.
- [ ] DC-009 (Happy path ‚Äî initial tree): booting quickstart with an initial
  tree, the first `visit` snapshot already contains the seeded stage before
  any provider call; an invalid initial tree falls back to model-first
  paint. ‚Üí Verify: Tier 1 stub E2E + vitest.
- [ ] DC-010 (Happy path ‚Äî live switch E2E): on a live link, a stub-agent
  turn that sets `theme` visibly restyles the page without reload.
  ‚Üí Verify: Tier 1 stub E2E (DOM assertion on a themed style).
- [ ] DC-011 (Invariant preserved): the renderer never throws for any theme
  name / any token lookup against a validated map (arbitrary strings
  probed). ‚Üí Verify: vitest fail-safe sweep in `@facet/react`.
- [ ] DC-012 (Invariant preserved ‚Äî spec surface): `STAGE_SPEC` documents
  `theme?: string` as select-by-name-only, and no public surface accepts
  model-authored CSS values. ‚Üí Verify: grep + review checklist in
  `/code-review`.

## Invariant Fit (Facet-specific ‚Äî REQUIRED)

| # | Invariant | Status | Note / mitigation |
|---|---|---|---|
| 1 | Responsibility boundary (UI-out/UI-in only) | OK | Themes/stamps/initial trees are UI data. The registry is a pluggable adapter exactly like `StageStore`; no domain/backend capability is added. |
| 2 | Mechanism vs policy | OK | Facet ships the validator, the registry interface, and the selection mechanism. The operator authors the documents; the agent picks a name. Contrast is measured (mechanism) but not enforced (policy stays with the caller). |
| 3 | Fail-safe | TOUCHES (mitigated) | New failure surfaces, each degraded: invalid document ‚Üí skipped at load, never rendered; unknown theme name ‚Üí default; non-string `theme` ‚Üí dropped by `validateTree`; invalid stamp ‚Üí excluded from prompt; invalid initial tree ‚Üí model-first paint. Renderer lookups stay null-proto + undefined-skipping (DC-011). |
| 4 | Declarative + tokens only | TOUCHES (mitigated) | Raw CSS values now exist as *operator data* ‚Äî but only behind `validateTheme`'s allowlist, never in the tree, never authorable by the model. The agent-facing vocabulary is unchanged: token names + one new *name* field. This moves the pixel boundary from "one code file" to "one validator", not into the spec. |
| 5 | Flow-only safety | OK | No layout capability added. Dimension values in theme documents are clamped so a theme cannot push content off-screen. |
| 6 | Two-writers coherence | OK | `theme` is part of the server-authoritative tree, written only via agent patches through the normal event loop. The browser resolves name ‚Üí values (pure lookup on boot-shipped data) ‚Äî it never writes stage state. Initial-tree seeding happens server-side through the same serialized stage-write path. |
| 7 | Backend via agent, never client fetch | OK | The theme map ships with the page from Facet's own transport at boot ‚Äî no separate client fetch, no third-party URLs (and `url()` denial in `validateTheme` closes the "theme value as remote loader" hole). |
