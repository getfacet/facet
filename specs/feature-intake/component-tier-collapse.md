# Feature Intake Brief: PR-6 â€” Component Tier Collapse

> Generated by `/feature-intake`. Plain-language, testable, invariant-checked.
> Owner-approved for `/spec-bridge` translation on 2026-07-15.

## Goal

- Remove Facet's `component` node tier completely so the final authoring model has
  exactly two concepts: validated native bricks and optional reference
  compositions made from those bricks.
- Demote `button`, `form`, `filterBar`, `metric`, `tabs`, and `nav` from node types
  to concrete native-brick references; delete legacy `stat`; and reclassify
  `table`, `chart`, `list`, `keyValue`, `progress`, and `loading` as ordinary
  bricks without changing their behavior.
- Make the transition an atomic, unreleased breaking cutover across core,
  rendering, agent tooling, built-in assets, seeds, prompts, docs, and public
  type names.

## Current Problem

- Facet currently teaches agents three overlapping ideas: primitive bricks,
  intrinsic/legacy components, and composition references. Most remaining
  structural/control components can already be expressed by composing boxes,
  text, inputs, bindings, and validated actions.
- Six data/feedback nodes still need dedicated native data shapes, but calling
  them components creates a category distinction that no longer communicates a
  useful authoring rule.
- Component-specific arrays, unions, validators, catalog sections, theme type
  names, render registries, prompts, and documentation duplicate the same tier
  boundary and can drift independently.
- Removing only the public node types would invalidate shipped references and
  seeds that still contain retired nodes, and stale raw nodes could reach a
  renderer that is expected to fail safely.

## User Scenario

### Who

- The agent author using Facet's closed vocabulary.
- The end-user interacting with a page the agent authored.
- The framework maintainer extending or validating Facet's native brick set.

### Trigger

- The agent needs a familiar button, form, fixed-choice filter, metric, tab, or
  navigation pattern after PR-6 lands, or needs a native data/feedback brick.

### Steps

1. Facet exposes one native brick roster and a separate list of optional
   reference compositions; it exposes no component tier or component-first
   fallback policy.
2. The agent optionally reads a relevant reference and authors the pattern as
   explicit native nodes.
3. A button is a pressable `box` plus label `text`; a form is a `box` containing
   `input` nodes and a pressable submit box; a metric is label/value `text`
   inside a box, with the value optionally bound through `from`.
4. Tabs and navigation use pressable boxes with local `navigate` actions and
   `active: { screen }` plus a selected recipe, so view switching and the active
   look occur without an agent turn.
5. The fixed-choice filter reference uses category-like pressable boxes that
   navigate among agent-preauthored result screens locally; it does not add a
   browser query engine or send an apply event to the agent.
6. `table`, `chart`, `list`, `keyValue`, `progress`, and `loading` validate and
   render exactly as before, now as members of the same brick roster.
7. Invalid or stale retired node types are rejected at authored boundaries and
   skipped safely if raw data reaches the renderer.

## Input/Output Examples

### Example 1 (Normal â€” Local Navigation/Filter)

- Input: The agent preauthors `products-all` and `products-electronics` screens,
  then creates an `Electronics` pressable box with
  `onPress: { kind: "navigate", to: "products-electronics" }`,
  `active: { screen: "products-electronics" }`, and
  `activeVariant: "selected"`.
- Output: Pressing the category switches screens and highlights the selected
  category immediately in the browser, with no event sent to the agent and no
  LLM turn.

### Example 2 (Normal â€” Bound Metric)

- Input: A reference contains a box with a label text and a value text carrying
  `from: "summary"`, `column: "revenue"`, and `row: 0`.
- Output: The composition validates as a native tree fragment; when the authored
  stage supplies that dataset the text renders the bound value.

### Example 3 (Normal â€” Reclassified Brick)

- Input: The agent authors the same valid `table`, `chart`, `list`, `keyValue`,
  `progress`, or `loading` data shape accepted before PR-6.
- Output: Validation output, data binding, rendering, interactions, and motion
  match the pre-cutover behavior; only the type's category and public plumbing
  change.

### Example 4 (Edge/Error â€” Stale Type)

- Input: An old tree or `set_node` request contains `type: "button"`, `"form"`,
  `"filterBar"`, `"metric"`, `"tabs"`, `"nav"`, or `"stat"`.
- Output: Core/agent authoring boundaries return a clean type error without a
  partial stage mutation; a stale raw React node blank-degrades and valid
  siblings still render without a throw.

### Example 5 (Boundary â€” Unsupported Local Query)

- Input: A reference or agent attempts to declare arbitrary browser-side filter
  expressions, JavaScript predicates, remote fetches, or unvalidated `where`
  clauses.
- Output: No such vocabulary is accepted. The agent either preauthors a bounded
  set of result screens for local navigation or handles open-ended domain work
  through its own tools and patches the stage.

## Constraints (What Not to Do)

- Do not retain component aliases, compatibility unions, deprecated validators,
  catalog component sections, component-first policy, legacy `stat`, or public
  component-tier recipe names. This is a hard cut, not a migration window.
- Do not add `input.onChange`, a browser query/filter expression language, client
  fetch, arbitrary resolver, or compound programmable action in PR-6.
- Do not preserve component-only convenience behavior by silently adding policy
  to the renderer. Dedicated form Enter-submit, authored button `disabled`, and
  tabs/nav landmark semantics are accepted breaking removals unless later added
  as separately designed, general brick capabilities.
- Do not change the runtime patch protocol, server/client `applyPatch`, action
  ordering, or backend ownership.
- Do not emit or accept raw HTML, JavaScript, CSS, pixel values, formulas, or
  absolute positioning. References remain validated token-only data.
- Do not turn a reference composition into a third authoring tier, a stage
  mutation, or a capability grant. The agent authors ordinary native bricks.
- Do not change the observable behavior of `table`, `chart`, `list`, `keyValue`,
  `progress`, or `loading` while reclassifying them.
- Do not require every fixed-choice filter to be local. When result combinations
  cannot be preauthored economically, the agent may use a normal agent action;
  the default reference demonstrates the bounded local case only.
- Do not merge or push without explicit owner approval, and do not implement
  directly on `main`.

| Scenario Step | Failure Policy | Boundary/Validation | Concurrency / Duplicate Trigger |

| Use local tabs/nav/filter | Dangling screen targets fail safely under existing navigate behavior and active look remains plain | Local fixed-choice examples require preauthored screen names; no data query is implied | View state is browser-owned and reported through the existing view snapshot when an event later reaches the agent; the server tree remains unchanged |
| Use a bound metric | Missing or malformed binding renders the existing safe empty result | `from`/`column`/`row` use existing `TextNode` validation; inline `value` is not promised as a fallback when `from` exists | Dataset changes arrive only through normal ordered stage patches |
| Submit a native form pattern | Invalid collect target is omitted by existing action validation; no partial mutation | The submit pressable box authors an agent action and a valid `collect` container id | Repeated submits use the existing agent-event/serial ordering policy; PR-6 adds no new event source |
