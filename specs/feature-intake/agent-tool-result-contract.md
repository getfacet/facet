# Feature Intake Brief: Agent Tool Result Contract

> Generated by `/feature-intake`. Plain-language, testable, invariant-checked.

## Goal

- Make Facet stage tool results easy for an LLM agent to interpret correctly.
- Prevent false success: the model must not conclude that a requested page change
  is complete when a tool was rejected, queued, sanitized, or applied to a node
  that is not visitor-visible.
- Establish a durable, documented contract so future tools return consistent
  status, visibility, warnings, and recovery guidance.

## Current Problem

- `@facet/agent-tools` returns useful internal data, but the reference-agent
  provider loop forwards only a bounded observation string to the LLM.
- Current strings such as `ok: set "pricing"`, `queued: ...`, and
  `ok: page replaced note: dropped invalid node(s): ...` force the model to
  infer whether the stage changed, whether the visitor can see the change, and
  what it should do next.
- Agent authors and first-run quickstart users are affected when the reference
  agent says a page was updated even though the visible stage did not change.

## User Scenario

### Who

- Primary: an LLM agent author using Facet's reusable stage tools.
- Secondary: the reference-agent/quickstart user relying on the built-in tool
  loop.
- End-user impact: visitors should see the page change the agent claims it made.

### Trigger

- The LLM calls a stage tool during a provider turn.

### Steps

1. The model calls a Facet stage tool such as `set_node`, `append_node`,
   `render_page`, `use_stamp`, `inspect_stage`, or `say`.
2. Facet executes the tool against the local stage shadow.
3. Facet returns a bounded, structured observation to the provider transcript.
4. The model reads explicit fields for status, outcome, stage application,
   visitor visibility, warnings, and next action.
5. If the result is not visitor-visible success, the model inspects or repairs
   the stage before claiming completion.

## Input/Output Examples

### Example 1 (Normal)

- Input: The model calls `append_node` with parent `root` and a valid text node.
- Output: The tool returns a structured observation with `status: "ok"`,
  `outcome: "applied_visible"`, `applied: true`,
  `visible_to_visitor: true`, patch count, changed ids, and no next action.

### Example 2 (Edge/Error)

- Input: The model calls `set_node` to create `pricing`, but never attaches that
  node under a visible parent.
- Output: The tool returns `status: "ok"`, `outcome: "applied_not_visible"`,
  `applied: true`, `visible_to_visitor: false`, and a next action telling the
  model to append the node under an existing visible box or inspect the stage.

### Example 3 (Rejected)

- Input: The model calls `append_node` with a missing parent.
- Output: The tool returns `status: "error"`, `outcome: "rejected"`,
  `applied: false`, `visible_to_visitor: false`, and a concrete next action.

## Constraints (What Not to Do)

- Do not add new brick types, raw HTML, raw JS, arbitrary CSS, pixel values, or
  new runtime protocol messages.
- Do not move agent policy or business logic into Facet. Facet reports tool
  outcomes; the agent still decides what UI to build.
- Do not expose full stage JSON in every tool result. Observations must remain
  bounded and provider-neutral.
- Do not leak provider keys, visitor ids, collected secrets, CSS values, stamp
  JSON internals, or unbounded node lists into observations.
- Do not require provider-specific behavior. OpenAI and Anthropic must receive
  the same logical observation text.

## Policy & Edge Cases

| Scenario Step | Failure Policy | Boundary/Validation | Concurrency / Duplicate Trigger |
|---|---|---|---|
| Tool result formatting | Never return a bare ambiguous success string for mutating tools. | Required fields: status, outcome, tool, applied, stage_changed, visible_to_visitor, patch_count, changed_node_ids, warnings, message, next_action. | Deterministic output for the same input/shadow. |
| Successful visible mutation | Mark `applied_visible` only when a patch applied and the changed stage is reachable from the active render root or changes visible stage metadata. | Keep ids and warnings bounded. | Sequential calls observe the latest local shadow. |
| Successful non-visible mutation | Mark `applied_not_visible`; do not imply visitor-visible completion. | Common case: `set_node` creates an orphan or updates a non-reachable node. | Repeated orphan updates keep returning repair guidance. |
| Warning/sanitization | Mark `applied_with_warnings` when data was dropped/sanitized or fold issues exist. | Include bounded warnings and next action to inspect/retry when needed. | No partial unbounded issue dumps. |
| Pending buffered edit | Mark `pending`; emitted messages remain empty until dependencies are defined. | Pending is not success. It must name missing child ids in bounded form. | Later child definitions can flush buffered edits in order. |
| Rejected tool | Mark `rejected`, emit no patch, keep shadow unchanged. | Include code and next action. | Duplicate invalid calls are deterministic. |
| Inspection/say | Use a no-stage-change outcome that clearly says the page did not mutate. | Inspection output remains bounded; chat output is not a stage patch. | No effect on local shadow except transcript order. |

## Done Criteria (Testable)

- [ ] DC-001 (Contract doc): A documented LLM-facing tool result contract defines
  required fields, outcome meanings, false-success rules, and per-tool recovery
  expectations. Verify: docs review and `rg` for the contract file.
- [ ] DC-002 (Structured observation): `@facet/agent-tools` exports typed helpers
  for structured LLM observations while preserving a bounded string form for
  provider transcripts. Verify: unit tests for formatter output.
- [ ] DC-003 (Visible success): Visible mutating tools return
  `applied_visible` with true application/visibility metadata. Verify:
  `agent-tools` unit test.
- [ ] DC-004 (False-success prevention): A successful but non-attached
  `set_node` returns `applied_not_visible` and a concrete repair next action.
  Verify: `agent-tools` unit test.
- [ ] DC-005 (Warnings and rejection): Sanitized `render_page` and invalid
  parent/node calls return structured warnings or `rejected` with no patch and a
  concrete next action. Verify: `agent-tools` unit tests.
- [ ] DC-006 (Pending is explicit): Buffered forward-reference edits return
  `pending`, not success, and later flush when dependencies are defined. Verify:
  `buffer` unit test.
- [ ] DC-007 (Reference prompt alignment): The reference-agent system prompt
  instructs the LLM to read the structured result and not claim completion unless
  the outcome is visitor-visible or intentionally no-stage. Verify:
  reference-agent prompt tests.
- [ ] DC-008 (Compatibility and bounds): Existing provider loop still passes
  bounded string observations, existing quickstart/reference tests remain green,
  and no sensitive values are introduced. Verify: affected package tests and
  full `/verify`.

## Invariant Fit (Facet-specific - REQUIRED)

| # | Invariant | Status | Note / mitigation |
|---|---|---|---|
| 1 | Responsibility boundary (UI-out/UI-in only; no backend/domain) | OK | The feature only reports Facet stage tool outcomes. It adds no domain/backend tools. |
| 2 | Mechanism vs policy (agent authors; Facet provides capability) | TOUCHES | Facet standardizes result mechanics and recovery hints. The agent still decides what UI to build. |
| 3 | Fail-safe (never throw/inject; degrade to plain) | TOUCHES | Rejected, warning, and pending outcomes become explicit normal control flow; invalid model output emits no patch. |
| 4 | Declarative + tokens only (no raw HTML/JS/CSS/pixels) | OK | No stage vocabulary or style-token vocabulary changes. |
| 5 | Flow-only safety (overlap only via a dedicated overlay brick) | OK | No layout, overlay, or positioning change. |
| 6 | Two-writers coherence (local execution vs agent stage) | TOUCHES | Visibility is computed from the server/local stage shadow only; browser-local navigate/toggle state is not made authoritative. |
| 7 | Backend calls go through the agent, never client fetch | OK | No client fetch or backend operation is added. |
