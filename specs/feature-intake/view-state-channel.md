# Feature Intake Brief: View-State Channel

> Generated by `/feature-intake`. Plain-language, testable, invariant-checked.

## Goal

- The live agent brain always knows what the visitor is currently looking at —
  which screen is showing, which toggles were flipped, and the visitor's device
  context (viewport size class, color scheme) — **at the moment it is about to
  act**, and on a return visit it knows where the visitor left off. No stage
  document schema change, no new round-trips, UI-in stays events-only.

## Current Problem

- `navigate`/`toggle` taps run locally in the browser by design (no agent turn).
  They are recorded to the `Sink` via `POST /record` for **replay only** — the
  live agent brain never learns them. The agent can patch a screen the visitor
  is not viewing, or talk about the wrong screen after the visitor navigated.
- The agent also knows nothing about the visitor's device (mobile vs desktop,
  dark vs light), so it cannot adapt what it draws.
- `localStorage` currently persists only the visitor id; where the visitor left
  off is lost between visits.
- Affected: the **end-user** of an agent-built page (gets confused, mistargeted
  updates) and the **agent author** (their brain is blind to view context and
  cannot compensate without hacks).

## User Scenario

### Who
- End-user of an agent-built page (visitor), and the agent brain acting on
  events; the agent author consumes the new context via `@facet/agent-tools`
  observations and the reference-agent prompt.

### Trigger
- Any forwarded `ClientEvent` (`visit`, `message`, `tap` with an agent action)
  is about to be sent to the server.

### Steps
1. Visitor navigates locally (tabs/nav/`navigate` tap) and flips a `toggle`;
   the renderer resolves these instantly with no agent turn (unchanged).
2. The browser keeps a compact **view snapshot**: current screen name, toggle
   overrides (node id → shown/hidden), viewport size class
   (`narrow|medium|wide`), color scheme (`light|dark`). It persists the
   navigation part to `localStorage` (per agent link), best-effort.
3. Visitor sends a chat message (or presses an agent-action box). The client
   attaches the current snapshot as an optional `view` field on that forwarded
   event. **No extra request is made.**
4. The server re-validates `view` at the boundary (bounded, fail-soft: a
   malformed or oversized `view` is dropped while the event itself still
   processes) and the event flows to the runtime/agent as today.
5. `@facet/agent-tools` folds `view` into the structured observation /
   event rendering; the reference agent's prompt shows a short line, e.g.
   `visitor is viewing screen "pricing"; expanded: faq-3; device: narrow, dark`.
6. Agent acts with correct targeting (patches the visible screen, or navigates
   the visitor deliberately).
7. Days later the visitor returns. The `visit` event carries the **persisted**
   last-known view as its `view` snapshot (report-only). The agent may greet
   with continuity ("last time you were on pricing…") and decide what to draw;
   the renderer does **not** auto-restore the old screen.

## Input/Output Examples

### Example 1 (Normal)
- Input: visitor on screen `pricing` (navigated locally), toggled `faq-3` open,
  on a phone in dark mode, sends message `"tell me more"`.
- Output: the forwarded event is
  `{ kind:"message", text:"tell me more", view:{ screen:"pricing", toggled:{"faq-3":"shown"}, viewport:"narrow", scheme:"dark" } }`;
  the reference-agent prompt/observation includes that context; the agent
  patches the pricing screen (not `home`).

### Example 2 (Edge/Error)
- Input: a hostile or buggy client POSTs an event whose `view` has a 10k-char
  screen name, 5,000 toggled entries, and `viewport:"4k-ultrawide"`.
- Output: the server boundary drops/clamps `view` (bounded validation,
  fail-soft — unknown enum values dropped, lists capped, oversized strings
  rejected), the event itself still processes, nothing throws, the agent simply
  receives the event without (or with a clamped) view.

### Example 3 (Edge — no storage)
- Input: page runs where `localStorage` is unavailable or throws (SSR,
  sandboxed iframe — the existing `visitor.ts` precedent).
- Output: events still send with the **live** in-memory snapshot; only revisit
  persistence silently degrades. No crash, no console spam.

## Constraints (What Not to Do)

- **No stage document change**: `FacetTree` schema is untouched; `view` rides
  events only (same treatment as `collect` fields).
- **No new round-trips / no new protocol route**: never fire a network call on
  navigate/toggle/resize/scheme-change just to report state; the snapshot rides
  only already-forwarded events. `POST /record` behavior is unchanged.
- **No in-progress field values and no field names** in the snapshot — typed
  input leaves the browser only via explicit `collect` at press time (existing
  privacy rule). (Field-name presence and scroll position are explicit v2
  candidates, not v1.)
- **No auto-restore on revisit** (report-only; restoring is the agent's call
  via its normal patches/navigation).
- **No renderer behavior change** for navigate/toggle themselves (still
  instant, still local, still logged to `/record`).
- No open-ended values: viewport/scheme are closed enums; screen is a name the
  agent authored; toggled keys are node ids.

## Policy & Edge Cases

| Scenario Step | Failure Policy | Boundary/Validation | Concurrency / Duplicate Trigger |
|---|---|---|---|
| 2 snapshot upkeep | storage write fails → in-memory only, silent | toggled map capped (drop-oldest beyond cap); screen/node-id strings length-capped | multi-tab: each event carries **its own tab's** snapshot; `localStorage` last-write-wins |
| 3 attach on send | snapshot unavailable → event sends without `view` (optional field) | `view` optional on all three event kinds; absent = today's behavior | rapid taps: each event snapshots at its own send moment (same as `collect`) |
| 4 server boundary | malformed/oversized `view` → drop or clamp `view`, event still processes, never reject the event for `view` reasons | closed enums for viewport/scheme; caps shared server+client so they can't drift (constants in `@facet/core`, `MAX_FIELD*` precedent) | boundary validation is pure/stateless |
| 5 agent surface | missing `view` → observation simply omits the line (older clients stay valid) | observation line bounded like other observation fields | stale-by-one is acceptable: the snapshot is truth at send time |
| 7 revisit report | stale names (screen/node no longer exists after agent redrew) are **hints, not references** — agent-tools may annotate unknown ids, nothing dereferences them | persisted payload validated on read; garbage in storage → ignored | returning in two tabs: both visits report the same persisted snapshot |
| logging | forwarded events land in the `Sink` as today; `view` contains only agent-authored names + device enums → log-safe, no new redaction class | Sink `StoredEvent` bodies keep existing redaction; `view` adds no sensitive fields | replay ordering unchanged (`/record` untouched) |

## Done Criteria (Testable)

- [ ] DC-001 (Happy path): a forwarded `message`/`tap` event carries
  `view` with the current screen, toggle overrides, viewport class, and scheme;
  `@facet/agent-tools` renders it into the structured observation/event text.
  → Verify: vitest (renderer/client capture + agent-tools observation).
- [ ] DC-002 (Revisit): after a session with local navigation, a new page load
  sends a `visit` event whose `view` equals the persisted last-known snapshot;
  with empty/corrupt storage, `visit` has no (or only live) `view`.
  → Verify: vitest with mocked `localStorage` (visitor.test.ts precedent).
- [ ] DC-003 (Boundary/hostile): server `/event` with malformed `view`
  (wrong types, oversized strings, > cap toggled entries, unknown enum values)
  drops/clamps `view`, still processes the event, and never throws.
  → Verify: vitest hostile-input suite on the server boundary.
- [ ] DC-004 (Storage-less): with `localStorage` undefined/throwing, events
  still send (live snapshot), nothing crashes.
  → Verify: vitest (SSR-like environment).
- [ ] DC-005 (Invariant preserved — two-writers): the feature adds **zero**
  stage-write paths: `FacetTree` schema unchanged, `validateTree` untouched by
  `view`, no renderer state is written from the server because of `view`.
  → Verify: type-level + grep/test that `view` never reaches patch/fold paths.
- [ ] DC-006 (No new traffic): local navigate/toggle still produce no forwarded
  agent event; no network call fires on resize/scheme change; only existing
  event sends carry `view`.
  → Verify: vitest transport-call assertions.
- [ ] DC-007 (Compatibility): events **without** `view` (older clients) remain
  valid end-to-end (server, runtime, agent-tools, reference agent).
  → Verify: vitest back-compat cases.

## Invariant Fit (Facet-specific — REQUIRED)

| # | Invariant | Status | Note / mitigation |
|---|---|---|---|
| 1 | Responsibility boundary (UI-out/UI-in only) | OK | Pure UI-in enrichment: state the browser already owns, reported on existing events. No backend/domain affordance. |
| 2 | Mechanism vs policy | OK | Facet reports; the agent decides (report-only revisit keeps restore policy with the agent). |
| 3 | Fail-safe | TOUCHES (mitigation) | New untrusted boundary input. Mitigation: bounded, pure validation at `/event`; drop/clamp `view` without rejecting the event; agent surfaces omit the line when absent. |
| 4 | Declarative + tokens only | OK | No markup/styles. `viewport`/`scheme` are closed enums; screen/node ids are agent-authored names. |
| 5 | Flow-only safety | OK | No layout/overlay change. |
| 6 | Two-writers coherence | TOUCHES (mitigation) | Reads browser-owned view-state and reports it; never writes stage. Server stays the only stage writer; renderer never applies server-pushed view state (no auto-restore in v1). Snapshot is truth-at-send-time; ordering rides the existing event `seq`. |
| 7 | Backend via agent | OK | No client fetch; nothing new touches a backend. |
