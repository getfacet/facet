# Feature Intake Brief: Font Family Tokens

> Generated by `/feature-intake`. Plain-language, testable, invariant-checked.

## Goal

Let Facet authors and operators control typography font families through the same
safe token/theme mechanism already used for color, spacing, size, and weight.

Outcome: a theme can define the concrete font stacks for the page, the default
Facet assets ship a safe baseline font family map, and the agent can optionally
choose a bounded text-level font family token without ever emitting raw CSS.

## Current Problem

- Facet already treats text size and weight as tokens, but font family is outside
  the theme contract.
- The quickstart page currently pins the wrapper font with raw renderer/app CSS,
  so operators cannot express a brand/system/mono typography choice through the
  theme asset pipeline.
- Agent-authored pages cannot safely request "make this text monospace" or
  "use the brand serif token" except by asking for a capability Facet does not
  expose.
- Affected users:
  - **Agent authors/operators** using Facet themes who want typography to be part
    of the same asset story as color/spacing.
  - **End-users of agent-built pages**, who should see consistent typography
    without the model ever writing CSS font-family strings.

## User Scenario

### Who

- Primary: the **agent author/operator** who supplies themes and wants typography
  controlled as a safe design token.
- Secondary: the **agent brain** that may select a text-level font token when it
  builds a page.
- Secondary: the **end-user** who sees the rendered page.

### Trigger

- An operator starts quickstart or another host with default or custom themes.
- The agent emits text nodes, optionally with a font family token in `TextStyle`.

### Steps

1. Operator provides no theme, or provides a theme that overrides one or more
   font family token values.
2. Facet validates the theme document and resolves it against the default theme.
3. The agent emits ordinary Facet text nodes; most text omits a font family and
   inherits the theme's default `sans` family.
4. When useful, the agent emits `style.family` with a bounded token such as
   `mono` or `serif`.
5. The renderer maps the token to the resolved theme's concrete font-family
   value. Invalid or unknown values are stripped or ignored fail-safe.

## Input/Output Examples

### Example 1 (Normal)

- Input: a theme includes `fontFamily: { sans: "Inter, system-ui, sans-serif",
  mono: "Menlo, Consolas, monospace" }`, and the agent emits:
  `{ "id":"code", "type":"text", "value":"pnpm test", "style": { "family":"mono" } }`.
- Output: ordinary text inherits the resolved `sans` stack; the `code` text
  renders with the resolved `mono` stack. The tree carries only `"mono"`, never
  the raw font-family string.

### Example 2 (Edge/Error)

- Input: a custom theme tries `fontFamily: { sans: "url(http://bad)" }`, or a
  text node carries `style: { family: "script" }` when `script` is not in the
  allowed token set.
- Output: the dangerous theme value is rejected by `validateTheme` and the
  theme document is dropped or sanitized according to the existing theme
  validator posture; the unknown text style token is stripped by `validateTree`.
  The renderer never throws and falls back to the default `sans` font.

## Constraints (What Not to Do)

- Do not let the agent emit raw `font-family` strings, URLs, CSS variables,
  `@import`, HTML, JS, or arbitrary CSS.
- Do not add a new node type; this is a style-token/theme capability.
- Do not add a backend/domain feature or any client-side fetch for fonts.
- Do not bundle or download font files as part of this feature. Operators may use
  host-provided/system fonts in their theme values; external font loading is
  outside this scope.
- Do not make font selection affect layout safety beyond normal text flow. No
  positioning, z-index, overlay, or measurement-based behavior is added.
- Do not require existing trees/themes to change; missing font-family data must
  fall back to the default theme.

## Policy & Edge Cases

| Scenario Step | Failure Policy | Boundary/Validation | Concurrency / Duplicate Trigger |
|---|---|---|---|
| Theme includes font family values | Dangerous/malformed value rejected; invalid theme handled by existing skip-and-log theme pipeline | Values are operator data only; known token keys only; bounded strings; dangerous CSS substrings/chars refused | Theme resolution is pure and deterministic; same inputs resolve the same font map |
| Tree text uses `style.family` | Unknown/non-string token stripped; text still renders with default/inherited family | Token must be one of the closed `FontFamily` vocabulary | Normal patch ordering; no new local execution or writer |
| Text omits `style.family` | Renderer uses the resolved default family token (`sans`) | Default theme must contain a complete `fontFamily` map | N/A |
| Custom theme omits some font family tokens | Missing tokens fall back to default theme values | Partial theme semantics match other theme groups | N/A |
| Quickstart renders default page | Page continues to render with a safe default system stack | Quickstart may use resolved theme for stage/body font; chat dock behavior is explicitly decided in spec | No effect on transport/runtime concurrency |

## Done Criteria (Testable)

- [ ] DC-001 (Core token vocabulary): `@facet/core` exposes a closed
  `FontFamily` token group with at least `sans`, `serif`, and `mono`, and
  `TextStyle.family?: FontFamily` accepts only that vocabulary. → Verify:
  type-level usage plus `validateTree` unit tests for valid/invalid family
  tokens.
- [ ] DC-002 (Theme contract): `FacetTheme` supports a partial `fontFamily`
  group whose values are validated as safe font-family strings; dangerous values
  such as `url(...)`, CSS variables, control characters, or injection characters
  are rejected fail-safe. → Verify: `@facet/core` `validateTheme` unit tests.
- [ ] DC-003 (Default asset): `@facet/assets` `DEFAULT_THEME` includes a complete
  default font family map and passes `validateTheme` with zero issues. → Verify:
  assets theme tests pin the new group shape and representative values.
- [ ] DC-004 (Renderer application): `@facet/react` resolves font family through
  the theme floor and applies it to text rendering; `style.family: "mono"` maps
  to the resolved mono stack, while omitted family falls back to `sans`. → Verify:
  react theme/style tests and StageRenderer render tests.
- [ ] DC-005 (Prompt/spec coherence): `STAGE_SPEC` advertises the new text
  family token vocabulary and continues to state that style values are tokens,
  never raw CSS. → Verify: core spec tests/string assertions.
- [ ] DC-006 (Compatibility/fallback): Existing trees and themes that do not
  mention font families continue to render with the previous effective system
  stack or a consciously pinned compatible default; no migration is required. →
  Verify: existing renderer/theme tests plus a targeted fallback test.
- [ ] DC-007 (No unsafe font loading): The feature does not add dependency,
  network font loading, HTML injection, or client fetch behavior. → Verify:
  grep/import checks and package dependency review.

## Invariant Fit (Facet-specific — REQUIRED)

| # | Invariant | Status | Note / mitigation |
|---|---|---|---|
| 1 | Responsibility boundary (UI-out/UI-in only; no backend/domain) | OK | Font family is UI styling. No backend/domain work or data fetching is added. |
| 2 | Mechanism vs policy (agent authors; Facet provides capability) | TOUCHES | Facet provides the bounded token mechanism; operators decide concrete theme values; the agent may select only token names. |
| 3 | Fail-safe (never throw/inject; degrade to plain) | TOUCHES | Theme values are validated by `validateTheme`; invalid text tokens are stripped by `validateTree`; renderer falls back to default `sans`. |
| 4 | Declarative + tokens only (no raw HTML/JS/CSS/pixels) | TOUCHES | This adds a new style token group on purpose. The tree carries only `family:"mono"`-style tokens; raw font-family values stay operator theme data. |
| 5 | Flow-only safety (overlap only via a dedicated overlay brick) | OK | Font choice changes typography only; no positioning/overlay/layout escape hatch is introduced. |
| 6 | Two-writers coherence (local execution vs agent stage) | OK | No browser-local stage mutation or new action path. Font changes travel as normal theme/tree patches. |
| 7 | Backend calls go through the agent, never client fetch | OK | No font fetching or backend call is introduced. |

## Open Questions / Assumptions

- **Assumption (owner: maintainer, checkpoint: `/spec-bridge`):** the first
  token vocabulary is `sans`, `serif`, `mono`. More semantic families like
  `display` or `brand` can be added later if a concrete need appears.
- **Assumption (owner: maintainer, checkpoint: `/spec-bridge`):** `sans` is the
  default family for omitted text style. A custom theme changes the default by
  changing the concrete value behind the `sans` token.
- **Open Question (owner: maintainer, checkpoint: `/spec-bridge`):** should
  quickstart app chrome and `ChatDock` inherit the resolved theme font, or should
  this feature initially apply only to the agent-drawn Stage text?
