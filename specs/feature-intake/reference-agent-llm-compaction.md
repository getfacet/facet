# Feature Intake Brief: Reference-Agent LLM Context Compaction (compaction v2)

> Generated by `/feature-intake`. Plain-language, testable, invariant-checked.

## Goal

- Long-lived visitor conversations and long tool-loop turns keep working at a
  bounded, token-accurate context size: when context passes a configured
  percentage of the model window, the reference agent compacts it with an LLM
  summary (same provider/model as the acting brain) instead of silently
  dropping the oldest turns or aborting the turn with `context_limit`.
- The summary survives across turns (persisted, rolling generations), the
  visitor-perceived latency does not grow (cross-turn compaction runs in the
  background after a turn), and every LLM-compaction failure degrades to the
  existing deterministic truncation — never a broken turn.
- This is the complete feature (both compaction surfaces, persistence,
  token accounting, prompt caching), not a minimal slice.

## Current Problem

- Compaction today is deterministic truncation only, computed once per turn at
  assembly (`harness/context.ts` → `harness/compaction.ts`): the oldest turns
  are dropped wholesale with a one-line note. Long conversations lose visitor
  intent, page decisions (node ids, naming conventions), collected form data,
  and unfinished promises — the model re-asks and re-invents ids, degrading
  the edit-before-append discipline.
- Mid-turn there is NO compaction at all: the tool transcript grows every step
  and the loop hard-stops with `context_limit` (`harness/loop.ts:176-180`),
  abandoning the visitor's request mid-edit (already-yielded patches stay,
  the rest of the work is lost).
- All sizing is character-count estimation. Provider-reported token usage is
  discarded by both adapters, tool schemas and per-provider window sizes are
  not counted at all, so the budget is both inaccurate and disconnected from
  the real model window.
- Affected users: the **agent deployer** (quickstart/reference-agent consumer)
  whose pages degrade on long sessions, and the **end visitor** whose long
  conversation loses continuity or whose complex request dies mid-turn.

## User Scenario

### Who
- Agent deployer running `@facet/reference-agent` (directly or via
  `facet-quickstart`), and the end visitor of the page it drives.

### Trigger
- A conversation or a single tool-heavy turn grows past the configured
  trigger ratio of the effective token budget.

### Steps
1. Visitor chats with a page over many turns; sink history grows.
2. At each provider step the harness reads real token usage from the provider
   response and keeps a calibrated chars-per-token estimate for pre-call sizing.
3. After a turn ends (final batch yielded), the harness projects the NEXT
   turn's context size; if it exceeds the trigger ratio, it runs a background
   summarizer call (same provider/model, forced tool call) over the turns not
   yet covered, producing/extending a rolling structured summary, and persists
   it to the `SummaryStore` with a covered-through marker.
4. On the next visitor event, context assembly loads the persisted summary,
   injects it as a clearly-delimited user-role data block at the head of the
   history layer (layer ③), and replays only the turns after the
   covered-through marker verbatim. The system prompt stays byte-identical.
5. Separately, during a tool-heavy turn: when the transcript passes the
   trigger ratio between steps, the harness synchronously compacts the oldest
   step groups (whole `assistant_tools` + `tool_result` pairs, never split)
   into an in-turn summary message, refreshes the `CURRENT STAGE` block from
   the tool buffer's shadow tree, and continues the loop instead of stopping.
6. If any summarizer call fails, times out, returns invalid output, or wins
   too little space: the harness falls back to the existing deterministic
   truncation for that assembly/step and the turn completes normally.

## Input/Output Examples

### Example 1 (Normal — cross-turn)
- Input: 60-turn conversation; projected next-turn context = 82% of the
  effective budget (trigger 75%).
- Output: after the current turn's reply is sent, a background summarizer call
  produces `generation 1` covering turns 1–52; the store holds
  `{payload, coveredThrough: 52, generation: 1}`. The next turn's prompt is:
  system (unchanged) → `[CONVERSATION SUMMARY g1 …]` user block → turns 53–60
  verbatim → current event + CURRENT STAGE. Estimated context lands under the
  50% target. Trace shows `compaction_done` with before/after tokens.

### Example 2 (Normal — in-turn)
- Input: a "rebuild the whole page" turn hits 30 steps; transcript passes the
  trigger ratio at step 31.
- Output: steps 1–26 (whole tool_use/tool_result groups) are replaced by one
  in-turn summary message (what was built, ids created, failures seen);
  steps 27–31 stay verbatim; the stale turn-start `CURRENT STAGE` block is
  replaced with a fresh one generated from `buffer.shadow`; the loop continues
  and finishes the request instead of stopping with `context_limit`.

### Example 3 (Edge/Error — summarizer failure)
- Input: the background summarizer call times out (or returns tool args that
  fail schema validation twice).
- Output: no summary is persisted (or the previous generation stays); the next
  turn assembles with the existing deterministic truncation exactly as today;
  trace records `compaction_failed` with the reason. The visitor sees a normal
  reply — never an error caused by compaction.

### Example 4 (Edge — store mismatch)
- Input: the persisted summary says `coveredThrough: 80` but the sink now
  returns only 40 entries (external wipe/reset).
- Output: the summary is discarded as inconsistent, assembly proceeds
  deterministically, and the next background compaction rebuilds from scratch
  (generation resets). No throw, no partial injection.

## Constraints (What Not to Do)

- No tokenizer dependency (tiktoken etc.). Token truth = provider-reported
  usage; pre-call sizing = calibrated chars-per-token estimation. The
  reference-agent keeps zero external dependencies.
- Never LLM-summarize the STAGE tree. The stage block stays the existing
  deterministic JSON / deterministic summary (id hallucination risk). LLM
  compaction applies to history and tool transcript only.
- Never split a tool_use/tool_result pair during in-turn compaction; replace
  contiguous whole step groups only (provider APIs reject orphaned pairs).
- The summary is injected as a **user-role data block**, never into the system
  prompt (layer ①② stays byte-stable and cacheable) and never as
  system/assistant authority.
- Deterministic truncation remains the always-available fallback; LLM
  compaction failing must never fail, delay indefinitely, or abort a turn.
- `@facet/core` is untouched (no new node kinds/tokens). `@facet/runtime`
  gains only a brain-agnostic opaque store interface — core/runtime/server
  keep zero LLM awareness (the summary JSON schema is owned and validated by
  `@facet/reference-agent`).
- No cross-visitor or global memory: summaries are strictly per
  `(agentId, visitorId)`.
- Not a hosted-platform feature: no metering, no billing, no admin surface.
- The stub agent path and Tier-1 deterministic gates must remain byte-for-byte
  deterministic (summarizer is injectable; stub runs never call it).

## Policy & Edge Cases

| Scenario Step | Failure Policy | Boundary/Validation | Concurrency / Duplicate Trigger |
|---|---|---|---|
| Usage accounting (step 2) | Provider response without usage → keep calibrated estimate only; trace notes estimate-only | Calibration ratio clamped to sane bounds (e.g. 1.5–6 chars/token); first call uses conservative default | Per-turn state; no cross-turn mutable calibration store needed (recompute per turn) |
| Background compaction (step 3) | Summarizer error/timeout/invalid output → retry once, then skip; previous generation stays; `compaction_failed` trace | Trigger ratio + target ratio (hysteresis); cooldown + min-gain guard prevent re-trigger loops; `maxSummaryTokens` cap with deterministic truncation of the summary itself | Serialized per `(agentId, visitorId)`; `coveredThrough` must be monotonic — a stale writer loses (write rejected/ignored); process restart loses only the in-flight call (acceptable) |
| Summary persistence (step 3) | Store write failure → drop + trace; never blocks | Payload validated against the reference-agent schema on read; invalid/corrupt → discard + rebuild | Last-write-wins guarded by monotonic `coveredThrough`; `NullSink`/`ForwardSink` (no history) → whole cross-turn path is a no-op, zero summarizer calls |
| Assembly with summary (step 4) | Store read failure → assemble without summary (deterministic path) + trace | `coveredThrough` > sink length → discard summary (Example 4); summary + verbatim tail + stage must still fit → existing deterministic compaction runs after injection as the final guard | Background compaction may lag: assembly tops up with verbatim turns after `coveredThrough`, so a stale summary is always safe |
| In-turn compaction (step 5) | Summarizer failure mid-turn → deterministic step-group truncation fallback, loop continues; if still over budget → existing `context_limit` stop remains the last resort | Keep last `minRecentStepsVerbatim` step groups verbatim; group boundary = whole assistant_tools+tool_results unit; CURRENT STAGE refresh uses the same deterministic formatter with the same JSON/summary bounds | Synchronous within the (already serialized) turn lane — no new concurrency; cooldown prevents compacting again immediately after a compaction |
| Redaction (step 3/5) | — | Existing sensitive-field regexes run over summarizer OUTPUT (and persisted payload) in addition to inputs; matches replaced with `[redacted]` | — |
| Prompt caching (step 2) | Cache-control ignored by API → harmless (additive header/body fields) | Cache breakpoints only on stable prefixes (system + tools; summary block); never on per-step content | — |

## Done Criteria (Testable)

- [ ] DC-001 (Happy path, cross-turn): with a fake provider reporting usage and
  a stub summarizer, a conversation crossing the trigger ratio persists a
  generation-1 summary with the correct `coveredThrough`, and the NEXT turn's
  assembled messages = summary block + only-after-marker verbatim turns +
  current event/stage, under the target ratio. → Verify: vitest (harness
  context + loop integration test).
- [ ] DC-002 (Happy path, in-turn): a scripted long tool loop crossing the
  trigger mid-turn replaces the oldest whole step groups with one summary
  message, keeps the last N groups verbatim, refreshes CURRENT STAGE from the
  buffer shadow, and completes with `provider_stop` (not `context_limit`).
  → Verify: vitest (loop test with scripted provider).
- [ ] DC-003 (Fallback): summarizer throw/timeout/invalid-schema (after one
  retry) falls back to deterministic truncation in both surfaces; the turn
  completes; `compaction_failed` trace is emitted. → Verify: vitest.
- [ ] DC-004 (Pair integrity boundary): for every compaction boundary
  placement, the resulting transcript maps to valid Anthropic and OpenAI wire
  shapes (no orphan tool_use/tool_result). → Verify: vitest unit sweep over
  boundary positions + adapter translation assertions.
- [ ] DC-005 (Redaction, both directions): a summarizer output containing
  `sk-…`/`Bearer …`/password-named fields is redacted before injection AND
  before persistence. → Verify: vitest.
- [ ] DC-006 (Injection placement): system prompt is byte-identical with and
  without an active summary; the summary appears exactly once, as a user-role
  block at the head of layer ③. → Verify: vitest.
- [ ] DC-007 (Determinism preserved): stub-agent Tier-1 E2E remains
  deterministic (two runs, deep-equal), and no summarizer is constructed or
  called on the stub path. → Verify: existing quickstart e2e run twice + a
  spy assertion.
- [ ] DC-008 (Token accounting): both adapters parse usage into
  `ProviderStep.usage`; the estimator counts system + messages + tool schemas;
  calibration converges toward reported usage across steps in a fake-provider
  test. → Verify: vitest (provider fixtures + estimator unit tests).
- [ ] DC-009 (Rolling generations): a second background compaction extends the
  summary to generation 2 with a later `coveredThrough`, replacing gen 1;
  a regressive (stale) write is rejected. → Verify: vitest (store contract +
  harness test).
- [ ] DC-010 (No-op stores): with `NullSink`/`ForwardSink` (empty history) or
  no configured `SummaryStore`, the cross-turn path performs zero summarizer
  calls and behaves exactly as today. → Verify: vitest.
- [ ] DC-011 (Hysteresis/cooldown/min-gain): after a compaction lands under
  the target ratio, no re-trigger occurs within the cooldown; a compaction
  that frees less than the min-gain does not loop. → Verify: vitest.
- [ ] DC-012 (Store contract): `MemorySummaryStore`, `FileSummaryStore`, and
  `PostgresSummaryStore` pass one shared contract suite (get/put keyed by
  agentId+visitorId, opaque payload round-trip, monotonic `coveredThrough`,
  corrupt-payload read → undefined). → Verify: vitest (postgres behind the
  same env gating as the existing store-postgres tests).
- [ ] DC-013 (Prompt caching): the Anthropic adapter emits `cache_control`
  breakpoints on the stable prefix (system + tools) and request bodies remain
  schema-valid; OpenAI body unchanged. → Verify: vitest request-body snapshot.
- [ ] DC-014 (Stale-summary safety): assembly with a summary whose
  `coveredThrough` lags the sink by K turns includes those K turns verbatim
  and stays within budget (deterministic compaction as final guard).
  → Verify: vitest.
- [ ] DC-015 (Store mismatch): `coveredThrough` exceeding sink length discards
  the summary, assembles deterministically, and the next compaction rebuilds
  at generation 1. → Verify: vitest.
- [ ] DC-016 (Live smoke): the key-gated provider smoke (Tier 2) passes with
  the new adapters (usage parsing + cache_control live). → Verify:
  `/live-test` Tier 2 (BLOCKING — provider.ts changes).

## Invariant Fit (Facet-specific — REQUIRED)

| # | Invariant | Status | Note / mitigation |
|---|---|---|---|
| 1 | Responsibility boundary (UI-out/UI-in only; no backend/domain) | OK | Compaction is agent-brain internals inside `@facet/reference-agent` (explicitly the pluggable brain). The summarizer call is the brain's own provider call, server-side. `SummaryStore` is a pluggable adapter like `Sink`/`StageStore` — mechanism, not domain. |
| 2 | Mechanism vs policy (agent authors; Facet provides capability) | OK | Trigger/target ratios, verbatim windows, and summary caps are deployer-configurable budget policy; the harness provides the mechanism. Nothing moves page-authoring decisions into the framework. |
| 3 | Fail-safe (never throw/inject; degrade to plain) | OK | Every failure (summarizer error/timeout/invalid output, store read/write failure, corrupt payload, marker mismatch) degrades to the existing deterministic truncation; `context_limit` stays as the last-resort stop. Renderer/validator untouched. |
| 4 | Declarative + tokens only (no raw HTML/JS/CSS/pixels) | OK | No stage-vocabulary change; the summary never enters the page tree. `@facet/core` untouched. |
| 5 | Flow-only safety (overlap only via a dedicated overlay brick) | OK | Not a rendering feature. |
| 6 | Two-writers coherence (local execution vs agent stage) | OK | The stage is never written by compaction. The only new concurrent writer is the background summarizer vs the next turn's reader — made safe by per-visitor serialization, monotonic `coveredThrough`, and stale-summary top-up (assembly replays post-marker turns verbatim). |
| 7 | Backend calls go through the agent, never client fetch | OK | Summarizer runs server-side in the agent harness; no client-side calls added. |
