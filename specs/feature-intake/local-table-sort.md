# Feature Intake Brief: Local Table Sort (v1)

> Generated by `/feature-intake`. Plain-language, testable, invariant-checked.

## Goal

- A visitor can sort a table by clicking a column header and see the rows
  reorder **instantly in the browser — with no agent turn**, the way
  `navigate`/`toggle` already resolve locally. The underlying data is never
  mutated by the browser; only the *view* of it changes.

## Current Problem

- A table's data is display-only: to reorder it, the agent must be asked and must
  re-emit the table (a round-trip that costs an LLM turn and feels laggy). Since
  tree-data-bindings (#37) separated data into a named `data` warehouse, the
  browser now has everything it needs to reorder a view locally — but there is no
  mechanism to do so. This was the explicit follow-up deferred from #37.
- Affected: the **end-user** (waits on the agent for a basic sort) and the
  **agent author** (spends turns/latency on interactions that should be free).

## User Scenario

### Who
- The **end-user** clicks a sortable column header; the **renderer** resolves the
  sort locally; the **agent brain** optionally learns the current sort via the
  view-state channel. The **agent author** opts a column into sortability.

### Trigger
- The visitor clicks a **sortable** column header of a rendered table.

### Steps
1. The agent emits a `table` whose columns opt into sorting (e.g. a per-column
   `sortable` flag), bound to a dataset (`from`) or with inline `rows`.
2. The visitor clicks a sortable column header. The renderer updates a **local
   sort view-state** for that table (column key + direction) and re-renders the
   rows in that order — instantly, no agent turn, no network.
3. Clicking the same header cycles direction (ascending → descending → unsorted);
   clicking a different sortable header sorts by that column instead.
4. The browser never mutates the dataset/`rows`; it holds only the sort spec. When
   the agent later patches the data, the browser re-applies its current sort spec
   to the new rows — no drift.
5. The current sort spec rides the visitor's next forwarded event on the `view`
   snapshot (view-state channel, #36), so the agent can see "sorted by price desc"
   and respond in context.

## Input/Output Examples

### Example 1 (Normal — local sort)
- Input: a `table` with `columns:[{key:"name",label:"Name",sortable:true},
  {key:"price",label:"Price",sortable:true}]`, `from:"products"`. The visitor
  clicks the "Price" header twice.
- Output: rows reorder by `price` ascending, then descending, in the browser with
  no agent turn; a subsequent chat message carries
  `view.sort:{ <tableId>: { column:"price", direction:"desc" } }`.

### Example 2 (Edge — non-sortable / unknown / malformed)
- Input: the visitor's browser somehow carries a sort spec naming a column that
  is not `sortable`, or a key absent from the columns, or a malformed spec; a
  column mixes cell types (numbers and strings).
- Output: the table renders in its natural (unsorted) order for an
  invalid/non-sortable spec; a mixed-type column sorts through a renderer-owned
  total comparator; nothing throws.

### Example 3 (Edge — data changes under a sort)
- Input: the visitor has sorted by `price desc`; the agent patches
  `data/products` (new/changed rows).
- Output: the browser re-applies `price desc` to the new rows; the sort persists
  coherently; the server remains the sole writer of the data.

## Constraints (What Not to Do)

- **No filtering in v1.** Only sort. Filtering (column predicates) is a deliberate
  follow-up — it opens the DSL-line risk (a predicate op set) that v1 avoids
  entirely by shipping sort only (a column key + a direction, no expression).
- **No expression language / no DSL.** The sort is a column key + a direction and
  a renderer-owned comparator. The agent never authors sort LOGIC; it only marks a
  column sortable. No computed sort keys, no custom comparators from the tree.
- **The browser never writes data.** The sort is pure browser **view-state** (like
  which screen is showing / which nodes are toggled). The server stays the sole
  writer of `data`/`rows`; the renderer applies the sort at render time only.
- **No new stage-content writer, no protocol round-trip.** The header click
  resolves locally (no agent turn, no network); the only thing that travels is the
  existing `view` snapshot on already-forwarded events.
- **No sort of non-tabular vocabulary in v1** (list/keyValue/chart local sort is
  out — table only).

## Policy & Edge Cases

| Scenario Step | Failure Policy | Boundary/Validation | Concurrency / Duplicate Trigger |
|---|---|---|---|
| 1 opt-in | a non-boolean `sortable` → treated as not-sortable (drop-with-issue) | `sortable` is a bounded per-column flag; validated like other closed fields | pure validation |
| 2 header click | click on a non-sortable header → no-op; unknown column → no sort | renderer resolves the sort against the actual columns; unknown/invalid → unsorted | rapid clicks cycle deterministically; last click wins |
| 2 sort compute | mixed/absent cell types → renderer-owned TOTAL comparator, never throws; stable sort | comparator is closed (numeric vs string vs boolean vs empty), renderer-owned; row/col caps reuse existing table bounds | pure render-time function of (rows, spec) |
| 4 re-apply on patch | data patch replaces rows → re-apply current spec; spec naming a now-absent column → unsorted | sort spec is view-state keyed by table node id | server stays sole writer; sort spec is not part of `data` |
| 5 ride view snapshot | snapshot omitted (older client) → agent just doesn't see the sort | `view.sort` is a bounded map (cap # of tables, closed direction enum) sanitized at the boundary (reuse #36's sanitizeView path) | rides existing forwarded events, no new traffic |

## Done Criteria (Testable)

- [ ] DC-001 (Happy path — local sort): clicking a sortable column header reorders
  the rendered rows ascending, then descending, with NO agent event/turn fired. →
  Verify: `@facet/react` jsdom interaction test (click → sorted DOM order; assert
  zero transport sends).
- [ ] DC-002 (Cycle): repeated clicks cycle asc → desc → unsorted; clicking a
  different sortable header switches the sort column. → Verify: jsdom test.
- [ ] DC-003 (Two-writers preserved): the browser never mutates `data`/`rows`; the
  sort is view-state only; after an agent `data` patch the current sort re-applies
  to the new rows. → Verify: render/structure test + `applyPatch` interplay.
- [ ] DC-004 (Fail-safe): a non-sortable/unknown/malformed sort spec renders the
  natural order; a mixed-type column sorts through a total comparator; never
  throws. → Verify: fail-safe render tests.
- [ ] DC-005 (View-state channel): the current sort spec appears on the `view`
  snapshot of the next forwarded event, sanitized/bounded; absent when nothing is
  sorted. → Verify: `@facet/core` sanitize test + host-wiring/observation test.
- [ ] DC-006 (Back-compat): a table with no `sortable` columns renders and behaves
  byte-identically; `sortable` and `view.sort` are additive optionals. → Verify:
  existing table render suites stay green + no-sortable case.
- [ ] DC-007 (No round-trip): a header click fires no network/agent event; only a
  later normal event carries the sort. → Verify: transport-call assertion.
