# Feature Intake Brief: Unified Asset Pipeline (themes + stamps, default as base layer)

> Generated by `/feature-intake`. Plain-language, testable, invariant-checked.

## Goal

Give Facet **one asset pipeline** for both themes and stamps, where the
Facet-provided **defaults are a bundled base layer** and any per-agent **custom**
assets **layer on top** (add / refine, never wholesale replace). Management
(store, validate, load, layer) is unified in `AssetsStore`; application stays at
each consumer (a renderer maps a theme → its output; the graft applies stamps).
Facet-provided default assets become renderer-agnostic **data**, not code baked
into a specific renderer or a code-factory package.

Outcome: default and custom assets stop being scattered across packages; a second
renderer (e.g. `@facet/vue`) can consume the same default assets; and the
now-orphaned `@facet/kit` code factory is removed.

## Current Problem

- **Default assets are scattered and consumer-coupled.** The default theme lives
  as `DEFAULT_THEME` inside `@facet/react` (the web renderer); the default stamps
  live as `KIT_STAMPS` inside `@facet/kit` (a code factory). Custom themes/stamps
  load through `@facet/runtime`'s `AssetsStore`. So *custom* is unified but
  *default* is not — an inconsistency, not a principled split.
- **`@facet/kit` is near-dead weight.** Only `apps/playground` imports it
  (`page`, `text` helpers); `KIT_STAMPS` has **no consumer** anywhere; `fragment()`
  is unused outside kit. It is a code factory in a system whose primary consumer
  (the LLM) authors in data (bricks + stamps), never in kit code.
- **The scatter blocks the multi-renderer goal.** A default theme stuck in
  `@facet/react` cannot be consumed by a future `@facet/vue`/`@facet/flutter`
  renderer without depending on the React package.
- Affected: the **maintainer** (drift, an orphan package, a blocked multi-renderer
  path) and, indirectly, **agent authors/operators** (two different mental models
  for "where do default vs custom assets come from").

## User Scenario

### Who
- Primarily the **agent author / operator** (Facet's framework consumer) who
  supplies per-agent assets and relies on sensible defaults.
- Secondarily the **maintainer** (this is largely a structural/refactor feature).

### Trigger
- An agent session starts and its stage must resolve a theme + the stamp library
  it can draw on, from whatever the operator provided (possibly nothing).

### Steps
1. Operator provides zero or more custom asset docs for an agent (`*.theme.json`,
   `*.stamp.json`), or none at all.
2. Facet's `AssetsStore` loads them through the single `loadAssets` gate,
   **on top of the bundled default base layer**.
3. Result per asset type:
   - **theme** → the default theme **merged per-field** with the custom theme
     (custom wins for fields it names; default fills the rest) → one effective theme.
   - **stamps** → the default stamps **unioned** with the custom stamps (both
     usable; a custom stamp whose name collides with a default one shadows it).
4. The agent authors UI using bricks + any stamp from the resolved library; the
   renderer applies the resolved theme. With no operator input, the agent still
   gets a complete default theme and the default stamp library.

## Input/Output Examples

### Example 1 (Normal)
- Input: operator supplies one custom theme `{ color: { bg: "#000000" } }` and one
  custom stamp `"promo-banner"`.
- Output: effective theme = default theme with `color.bg` replaced by `#000000`
  and every other field (space, fontSize, fontWeight, …) still the default; stamp
  library = all default stamps **plus** `promo-banner`.

### Example 2 (Edge/Error)
- Input: operator supplies a malformed theme doc (bad token value) and a stamp doc
  that fails `validateTree`, alongside one valid custom stamp.
- Output: the two bad docs are **dropped with recorded issues**; the valid custom
  stamp and all defaults survive; `loadAssets` **never throws**; the agent still
  renders on a complete theme.

## Constraints (What Not to Do)

- **Non-goal: LLM authoring new stamps at runtime.** This feature keeps the
  current model (operator provides assets as data; the agent *uses/selects* them,
  e.g. switches the active theme). Runtime stamp authoring is a separate feature.
- **Non-goal: changing the brick/token vocabulary.** No new bricks, no new tokens.
- **No raw CSS/HTML** enters the tree; themes remain an operator token→value config
  applied by the renderer, never tree content.
- **No new dependency into `@facet/core`**; the new assets data package must be
  browser-safe / node-free (renderer-agnostic).
- Do **not** make "default" a wholesale override target — default is a base layer
  that always survives; custom only adds (stamps) or refines specific fields (theme).

## Policy & Edge Cases

| Scenario Step | Failure Policy | Boundary/Validation | Concurrency / Duplicate Trigger |
|---|---|---|---|
| Load custom docs over defaults | Bad doc dropped + issue recorded; defaults + valid customs remain; never throw | Each doc passes `loadAssets` (`validateTheme` / `validateTree`) | Re-load is idempotent; same inputs → same resolved assets |
| Theme per-field merge | Missing/invalid custom field → default value used | Only known theme fields honored; unknown dropped | One effective theme per resolve; last-writer per field = custom |
| Stamp union | Malformed stamp dropped | Names must be non-empty/disjoint; reserved `root` prefix untouched | Name collision → custom shadows default deterministically |
| No operator assets at all | Defaults alone resolve (never empty) | Base layer is always present | N/A |

## Done Criteria (Testable)

- [ ] DC-001 (Happy path): With **no** operator assets, `loadAssets` resolves the
  bundled default theme + the full default stamp library. → Verify: unit test on
  `loadAssets` with empty `AssetDocuments`.
- [ ] DC-002 (Theme merge): A custom theme naming only `color.bg` resolves to an
  effective theme with custom `bg` and **default values for all other fields**. →
  Verify: unit test on the theme merge.
- [ ] DC-003 (Stamp union): Default stamps and a custom stamp are both resolvable;
  a custom stamp sharing a default's name **shadows** it. → Verify: unit test on
  the stamp resolve.
- [ ] DC-004 (Fail-safe): Malformed theme/stamp docs are dropped with issues while
  defaults + valid customs survive, and `loadAssets` never throws. → Verify: unit
  test with bad docs.
- [ ] DC-005 (Kit removed): `@facet/kit` no longer exists in the workspace and is
  imported nowhere; the full build passes. → Verify: `git grep '@facet/kit'` = 0
  (outside history) + `pnpm build`.
- [ ] DC-006 (Renderer-agnostic default): default assets live in a browser-safe /
  node-free data package (`@facet/assets`) with **no renderer dependency**, and
  `@facet/react` no longer owns the default-theme **data**. → Verify: import-graph
  / grep check + package `exports` review.

## Invariant Fit (Facet-specific — REQUIRED)

| # | Invariant | Status | Note / mitigation |
|---|---|---|---|
| 1 | Responsibility boundary (UI-out/UI-in only; no backend/domain) | OK | Assets are UI styling (theme) + UI structure (stamps); no backend/domain surface added. |
| 2 | Mechanism vs policy (agent authors; Facet provides capability) | OK | Facet provides the pipeline (mechanism); operators/agents provide + select assets (policy). No authoring decision moves into the framework. |
| 3 | Fail-safe (never throw/inject; degrade to plain) | TOUCHES | `loadAssets` must validate every doc and drop bad ones with issues; defaults always survive; never throw. Mitigation: extend the existing `loadAssets` gate + DC-004 test. |
| 4 | Declarative + tokens only (no raw HTML/JS/CSS/pixels) | OK | Themes stay an operator token→value config (not tree content); stamps are `validateTree`-checked brick trees. No raw CSS in the tree. |
| 5 | Flow-only safety (overlay only via dedicated brick) | OK | No overlay/positioning involved. |
| 6 | Two-writers coherence (local execution vs agent stage) | OK | No new local execution. Theme selection is a stage edit driven by the agent; ordering/version unchanged. |
| 7 | Backend calls go through the agent, never client fetch | OK | No backend calls; assets are static data loaded by the runtime. |

## Public API / Package Surface

| `@facet/*` package | Change | Additive or Breaking |
|---|---|---|
| `@facet/kit` | **Removed** entirely | Breaking (pre-1.0; only `apps/playground` consumed it) |
| `@facet/assets` (new) | Default stamps (+ renderer-agnostic default theme) as **data** | Additive (new package) |
| `@facet/react` | Default-theme **data** (`DEFAULT_THEME`) moves out to `@facet/assets`; react keeps only theme→CSS **application** | Breaking (for direct `DEFAULT_THEME` importers; mostly internal) |
| `@facet/runtime` | `AssetsStore`/`loadAssets` gain base-layer seeding + documented per-type merge (theme field-merge, stamp union) | Additive behavior; possibly breaking to `LoadedAssets` semantics |
| `@facet/core` | Possibly a small theme-merge helper; `FacetStamp`/`FacetTheme` types unchanged | Additive (if any) |
| `apps/playground` | Replace `@facet/kit` `page`/`text` imports with local helpers or bricks | N/A (unpublished) |

## Open Questions / Assumptions

- **OQ-1 (owner: maintainer, checkpoint: /spec-bridge):** Are default **theme
  values** renderer-agnostic enough to live in `@facet/assets`, or does only the
  default **theme** (vs default **stamps**) stay web-flavored? Options: (a) keep
  theme values semantic and let each renderer interpret; (b) ship a web default
  theme in `@facet/assets` now and let native renderers ship their own baseline
  later. Stamps (brick trees) are cleanly renderer-agnostic regardless.
- **Assumption (owner: maintainer, checkpoint: /spec-bridge):** the current
  `AssetsStore` / `loadAssets` gate can be extended to seed a base layer + merge,
  without changing the `FacetStamp` / `FacetTheme` core types.
- **Assumption:** `apps/playground`'s use of `@facet/kit` (`page`, `text`) is small
  enough to inline or replace with bricks without a behavior change.
