# Feature Intake Brief: Catalog + Design System v1

> Generated by `/feature-intake`. Plain-language, testable, invariant-checked.
> Status: owner-approved direction from the 2026-07-09 design discussion.

## Goal

- Let Facet agents build rich app/web screens inside a brand-controlled design
  system, instead of hand-assembling every screen from low-level `box`/`text`
  nodes.
- Introduce a first-class **catalog**: the agent-facing "toolbox + usage manual"
  that tells an agent which theme, bricks, variants, stamps, and policies are
  available for the current project/page.
- Expand Facet from a low-level primitive-only vocabulary into a safe UI
  vocabulary with high-level app bricks, while keeping primitive fallback for
  novel layouts.
- Keep branding stable: active theme is locked by default; agents use component
  variants and stamps rather than raw CSS or arbitrary style values.

## Current Problem

- Facet now has themes and stamps, but the agent only receives names,
  descriptions, and schemas. It is not told clearly when to use a theme, stamp,
  high-level brick, or primitive fallback.
- Current themes are mostly token value maps. They do not yet describe component
  recipes such as `button.primary`, `card.default`, `table.dense`, or
  `chart.line`, so the agent still chooses low-level token combinations.
- Default stamps are too few and too low-level (`hero`, `card`, `cta-button`).
  They do not cover dashboards, pricing, settings, data display, forms, or common
  product pages.
- The current primitive vocabulary is safe, but rich SaaS screens, dashboards,
  tables, charts, metrics, tabs, badges, alerts, and progress views require too
  much manual composition to reliably produce a "wow" result.
- Affected users:
  - agent authors using Facet, who need consistent UI output without writing a
    bespoke renderer or prompt harness;
  - end-users of an agent-built page, who expect useful, branded, compact UI
    rather than a long hand-built page;
  - operators/hosts who need to expose only the UI capabilities that match their
    brand, product surface, and permission model.

## User Scenario

### Who

- The **operator / host** configures the active theme, allowed bricks, allowed
  variants, stamps, and catalog policy.
- The **agent author / LLM harness** consumes the catalog and generates UI within
  that allowed vocabulary.
- The **end-user** sees a branded, useful page and can interact with predeclared
  UI controls.

### Trigger

- A visitor asks an agent for a rich UI: pricing, dashboard overview, comparison,
  settings, onboarding, support triage, product summary, chart/table view, or
  a multi-step flow.

### Steps

1. Host loads an asset/catalog registry for the agent. It contains an active
   theme, allowed brick set, allowed variants, stamp library, and usage policy.
2. Facet validates the catalog, themes, and stamps fail-safely. Invalid entries
   are skipped or downgraded with issues; boot continues with safe defaults.
3. The reference agent prompt/tool layer exposes the catalog in compact form:
   active theme and policy, available bricks/variants, stamps with metadata, and
   rules such as stamp-first, brick-second, primitive fallback.
4. The agent chooses the best available stamp or high-level brick instead of
   hand-building everything. If no catalog item fits, it uses primitives within
   token/flow constraints.
5. Tool execution enforces the catalog: unknown bricks, disallowed variants,
   unknown stamps, and locked theme switches are rejected with structured
   observations the LLM can recover from.
6. The renderer renders known high-level bricks using the active theme recipes,
   and skips/downgrades invalid or unknown content without throwing.

## Input/Output Examples

### Example 1 (Normal)

- Input: catalog allows `section`, `card`, `button`, `table`, `chart`, `stat`,
  and stamps `dashboard-overview` and `pricing-section`; active theme is locked
  to `acme-default`.
- Agent request: "Show a compact revenue dashboard with KPIs, a line chart, and
  the latest accounts."
- Output: agent uses `dashboard-overview` or `stat` + `chart` + `table` bricks;
  no theme switch; all component variants resolve through `acme-default` recipes.

### Example 2 (Edge/Error)

- Input: agent tries to emit `chart` when catalog does not allow `chart`, or
  tries `set_theme("playful")` while theme policy is locked.
- Output: tool result is `rejected` with a clear reason and next action; no patch
  is emitted. The agent can retry with an allowed brick/stamp or keep the active
  theme.

### Example 3 (Fallback)

- Input: user asks for a niche UI pattern that has no matching stamp or high-level
  brick, but it can be represented with flow-safe primitives.
- Output: agent uses `box`/`text`/`media`/`field` fallback with token styles.
  Catalog policy marks this as allowed fallback, not a failure.

## Constraints (What Not to Do)

- Do not add raw HTML, JS, arbitrary CSS, arbitrary pixels, arbitrary colors, or
  arbitrary z-index/positioning.
- Do not move domain/backend work into Facet. Charts/tables display data supplied
  by the agent; Facet does not fetch analytics, query databases, or call external
  APIs.
- Do not make the browser fetch catalog/theme/stamp data from arbitrary URLs.
  Hosts load and validate assets through Facet adapters.
- Do not expose every core brick to every agent by default. Catalog chooses the
  allowed subset per project/page.
- Do not let the agent switch themes unless the catalog policy explicitly allows
  it.
- Do not require high-level bricks for every possible UI. Primitive fallback stays
  available and safe.
- Do not implement unconstrained overlays. Modal/drawer-like behavior, if added,
  must be a dedicated constrained brick with two-writers and flow/overlay
  discipline documented.

## Policy & Edge Cases

| Scenario Step | Failure Policy | Boundary/Validation | Concurrency / Duplicate Trigger |
|---|---|---|---|
| Catalog load | Invalid entries skipped with issues; safe defaults remain | names capped; arrays bounded; unknown keys dropped; no Node-only dependency in core | loaded once per boot/session scope; deterministic order |
| Active theme | missing/unknown active theme falls back to default; locked policy blocks agent switch | `switchPolicy` in allowed enum; theme name validates with existing name rule | theme patch ordering remains normal stage patch ordering |
| Brick use | disallowed or unknown brick rejected by tools; renderer skips unknown node types fail-safe | core node validator bounds shapes; catalog validator bounds exposed subset | repeated tool calls append/update through existing serialized runtime |
| Variant use | unknown variant falls back to default recipe or tool rejects when catalog-enforced | variant name capped; recipe tokens validated | no local writer |
| Stamp use | unknown/disallowed stamp rejected; malformed stamp skipped at load | stamp metadata bounded; expansion still server-side before patches | expansion ids remain fresh per call |
| Primitive fallback | allowed only when no stamp/high-level brick fits; still token/flow-only | existing primitive validation applies | same as existing stage patches |
| Local UI interactions | only existing safe actions, plus any new brick-local view state must define coherence | no backend fetch; local state must not mutate authoritative content | two-writers policy required for any local execution |

## Done Criteria (Testable)

- [ ] **DC-001 (Catalog contract):** Facet exposes a validated catalog document
  shape that can name active theme policy, allowed bricks, allowed variants,
  stamps, and usage policy. → Verify: `@facet/core` unit tests for valid and
  malformed catalog documents.
- [ ] **DC-002 (Catalog enforcement):** agent tools reject a brick, stamp, theme
  switch, or variant not allowed by catalog with structured recovery guidance and
  no patch. → Verify: `@facet/agent-tools` tool executor tests.
- [ ] **DC-003 (Theme recipes):** theme documents can include bounded component
  recipes for high-level bricks, using token names rather than raw CSS in the
  tree; invalid recipe tokens are dropped with issues. → Verify: `@facet/core`
  `validateTheme` tests and renderer style tests.
- [ ] **DC-004 (Locked theme):** when catalog policy says theme is locked, the
  reference agent prompt explains that the active theme must be kept and
  `set_theme` rejects switches. → Verify: prompt snapshot/unit test and
  `set_theme` executor test.
- [ ] **DC-005 (Expanded brick vocabulary):** the v1 high-level brick set covers
  action, layout, data display, and feedback: `button`, `section`, `card`, `tabs`,
  `table`, `chart`, `stat`, `badge`, `progress`, `alert`, `list`, and `divider`.
  Each renders fail-safely from declarative data. → Verify: core validation tests
  and React renderer tests per brick group.
- [ ] **DC-006 (Primitive fallback):** existing `box`/`text`/`media`/`field`
  remain valid and can be used when catalog policy allows fallback. → Verify:
  existing tests plus a catalog prompt/tool test.
- [ ] **DC-007 (Stamp metadata):** stamps support bounded metadata such as
  `category`, `useWhen`, `avoidWhen`, `variants`, `tags`, `repeatable`,
  `preferredParent`, `composedOf`, `dataRequirements`, and `followUpEdits`.
  Metadata reaches the prompt; full stamp node JSON does not. → Verify:
  `validateStamp` tests and prompt-kit tests.
- [ ] **DC-008 (Default stamp coverage):** default assets include a practical
  common stamp library for product pages and app screens, including pricing,
  FAQ, dashboard, settings, feature, CTA, empty-state, and summary patterns.
  → Verify: `@facet/assets` tests validate all default stamps and metadata.
- [ ] **DC-009 (Catalog prompt quality):** reference-agent system prompt teaches
  the order `stamp -> high-level brick -> primitive fallback`, edit-before-append,
  locked theme behavior, and compact-screen policy. → Verify: prompt-kit tests.
- [ ] **DC-010 (Renderer fail-safe):** malformed high-level nodes, unknown
  variants, too-large chart/table payloads, and unknown catalog entries never
  throw and degrade to skipped/default/plain UI. → Verify: React renderer and
  core validation tests.
- [ ] **DC-011 (Docs/spec):** `STAGE_SPEC`, README/architecture/package docs
  describe catalog, design-system theme recipes, high-level bricks, stamp
  metadata, and catalog-gated agent behavior. → Verify: docs update review and
  spec tests/grep.
- [ ] **DC-012 (Quickstart/live behavior):** quickstart real/stub paths expose the
  catalog to the reference agent and can build a compact dashboard/pricing UI
  using stamps or high-level bricks without arbitrary theme changes. → Verify:
  quickstart unit/E2E tests and live-test policy.

## Invariant Fit (Facet-specific — REQUIRED)

| # | Invariant | Status | Note / mitigation |
|---|---|---|---|
| 1 | Responsibility boundary (UI-out/UI-in only; no backend/domain) | OK | Catalog/theme/stamp/brick capabilities are UI vocabulary and policy. Data shown in charts/tables is supplied by the agent; Facet does not fetch or compute domain data. |
| 2 | Mechanism vs policy (agent authors; Facet provides capability) | TOUCHES | Catalog can restrict available capability, but the agent still authors the page. Mitigation: Facet provides mechanism/policy declaration and enforcement; operators choose catalog contents. |
| 3 | Fail-safe (never throw/inject; degrade to plain) | TOUCHES | New catalog/theme/brick surfaces add malformed-input risk. Mitigation: bounded validators, unknown entries skipped/rejected, renderer fallback, structured tool observations, tests DC-001/002/010. |
| 4 | Declarative + tokens only (no raw HTML/JS/CSS/pixels) | TOUCHES | High-level bricks and recipes must remain typed declarative data. Recipes use existing/expanded token names, not tree-authored raw CSS. Raw CSS remains operator theme data behind validators only. |
| 5 | Flow-only safety (overlap only via dedicated overlay brick) | TOUCHES | High-level layout/data bricks must render in normal flow. Modal/drawer-like bricks are out of v1 unless implemented as dedicated constrained overlays with their own mitigation. |
| 6 | Two-writers coherence (local execution vs agent stage) | TOUCHES | Most bricks are display/action wrappers over existing stage writes. Any brick-local interaction uses existing `navigate`/`toggle`/field collection rules or must document coherence before inclusion. |
| 7 | Backend calls go through the agent, never client fetch | OK | Catalog does not add client fetch. Tables/charts render provided data only; external data remains the agent's own tools. |

## Decision Lock

| Item | Type (Decision/Assumption/Question) | Impact if wrong | Owner | Resolution checkpoint | Status |
|---|---|---|---|---|---|
| Catalog is the agent-facing toolbox + usage manual, not the same thing as a brick or stamp | Decision | Agent receives too little or too much context | owner | intake | fixed |
| Core may support more bricks than a catalog exposes | Decision | Per-project UX control impossible if reversed | owner | intake | fixed |
| Theme equals design system: base tokens plus component recipes; recipes are the agent-facing styling unit | Decision | Agent keeps hand-picking tokens and brand consistency remains weak | owner | spec-bridge | fixed |
| Active theme is locked by default | Decision | Agent can break service branding | owner | spec-bridge | fixed |
| Agent usage order is stamp first, high-level brick second, primitive fallback last | Decision | Agent keeps building long low-level pages | owner | spec-bridge | fixed |
| v1 high-level brick list is broad but still safe and declarative | Decision | Too small fails "wow"; too large risks unstable contract | owner | spec-bridge | fixed |
| Overlay-class bricks are not part of v1 unless separately constrained | Decision | Flow-only invariant risk | owner | spec-bridge | fixed |
| Chart/table data is display-only and supplied by agent | Decision | Backend responsibility leaks into Facet if reversed | owner | spec-bridge | fixed |
| Exact recipe schema shape | Decision | Poor shape can cause public API churn | main-agent + owner | spec-bridge | fixed: `ComponentRecipes` maps component/variant names to validated token style bundles |
| Whether to implement all listed v1 bricks in one PR or split into foundation + brick batches | Decision | One PR may be too large; splitting may delay "wow" | main-agent + owner | spec-bridge gate | fixed: one feature PR decomposed into bounded WUs, with high-risk contracts first |

## Resolved Questions / Assumptions

- **Resolved assumption:** the first implementation can be a large feature only if the
  spec decomposes it into independently testable work units with disjoint files.
  Owner: main-agent. Checkpoint: `/spec-bridge`.
- **Resolved assumption:** the catalog can live in the existing assets pipeline rather than
  a separate backend/control-plane package. Owner: main-agent. Checkpoint:
  `/spec-bridge`.
- **Resolved decision:** `chart` is a display-only v1 data brick with simple
  bounded variants. It may render SVG internally, but it does not fetch data or
  run arbitrary chart code. Owner: main-agent + owner. Checkpoint:
  `/spec-bridge`.
- **Resolved decision:** `table` is display-only in v1. Sorting/filtering local
  interactions are out of v1 unless separately specified with two-writers
  coherence. Owner: main-agent + owner. Checkpoint: `/spec-bridge`.
