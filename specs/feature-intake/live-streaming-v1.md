# Feature Intake Brief: Live Streaming v1 (per-step stage delivery)

> Generated by `/feature-intake`. Theme 4 of `facet-brick-vocab-roadmap`.
> **Implementation in Codex — this chain stops at `/spec-bridge`.**

## Goal

Make the page **build live as the agent works** — stream each provider-step's stage
changes to the client mid-turn instead of one flush at turn end — so the visitor
watches the model render the page (Facet's "UI a model renders itself, live"
thesis) and perceived latency drops on multi-step turns. Stay fail-safe and keep
turn-ordered, server-authoritative coherence.

## Current Problem

There is **no streaming**. The quickstart agent loop accumulates all ops across its
tool steps and flushes **once at turn end** (`packages/quickstart/src/agent.ts`:
`[ provider.run → exec tools → observe ]* → flush`), and `defineAgent`
(`packages/agent/src/define-agent.ts`) does `await logic(...) → stage.flush()` once.
So on a multi-step (multi-second) turn the visitor sees **nothing until the whole
turn completes**, then the page pops to final — no "live building" feel, which
contradicts the product thesis and the market's "progressive render" expectation.
**Affected:** the end-user (perceived latency, no live feel) and the core "live"
claim.

## User Scenario

### Who
The end-user of an agent-built page (and the agent author whose turns now stream).

### Trigger
A visitor sends a message that triggers a multi-step agent turn.

### Steps
1. The agent's provider loop runs; after **each step** (a `provider.run` + its tool
   batch), the agent yields that step's `ServerMessage[]` batch.
2. The runtime applies the batch to the server-authoritative session and delivers it
   as a frame — **during** the turn.
3. The page folds each frame and updates incrementally — header appears, then cards,
   then pricing — as the model works.
4. The final step completes the page; one turn = several frames, one per step.

## Input/Output Examples

### Example 1 (Normal)
- Input: a 3-step agent turn (build skeleton → add cards → add pricing).
- Output: **3 frames delivered during the turn** (not one at the end); the visitor
  sees the page build in three live steps.

### Example 2 (Edge/Error)
- Input: the agent throws mid-stream after 2 batches delivered; or a batch adds a
  child whose parent arrives in a later batch.
- Output: the 2 delivered batches **stay** (server-authoritative stage keeps them —
  a coherent partial page); the turn errors without crashing; a dangling child is
  **fail-safe skipped** until its parent's batch arrives. Never throws in render.

## Constraints (What Not to Do)

- **No per-op streaming** — per-**step** granularity; the agent decides when to yield
  (the framework only iterates).
- **No change to LOG granularity** — the `Sink` still records **ONE `StoredEvent`
  per turn** (the turn's accumulated messages), not per batch — event-layer-v1's
  turn-granular replay log stays intact.
- **No breaking the per-visitor serial order** — the whole generator runs inside the
  existing per-(agent,visitor) serial lane; batches within a turn deliver in yield
  order; different visitors stay parallel.
- **No client coherence change** — the same `foldPatchIntoStage`/`applyPatch` folds
  each frame; the fail-safe renderer handles partial trees (invariant #2/#3).
- **No new transport channel** — batches fan out via the existing `/stream` SSE
  (`era:seq` frames); the `/event` POST still acks 202 immediately.

## Policy & Edge Cases

| Scenario Step | Failure Policy | Boundary/Validation | Concurrency |
|---|---|---|---|
| yield a batch | empty batch → no frame | each batch = valid `ServerMessage[]` | within the serial lane, in yield order |
| apply+deliver batch | apply fail-safe (fold salvages) | patches validated by `foldPatchIntoStage` (existing) | server is the single stage writer |
| agent throws mid-stream | delivered batches persist; turn errors, no crash | — | later visitor turns unaffected |
| dangling ref across batches | fail-safe renderer skips until parent arrives | existing dangling-skip | frames ordered by `era:seq` |
| Sink record | record ONE turn row with accumulated messages | — | recorded at turn end (existing per-turn record) |

## Done Criteria (Testable)

- [ ] **DC-001 (streaming delivery):** a generator agent yielding N batches causes
  the runtime to deliver N frames during ONE turn (not one at end). → Verify: runtime test.
- [ ] **DC-002 (per-batch apply+persist):** each batch applies to the session +
  delivers; final stage = fold of all batches. → Verify: runtime test.
- [ ] **DC-003 (ordering):** batches deliver in yield order via `era:seq`; different
  visitors stay parallel. → Verify: server/runtime test.
- [ ] **DC-004 (defineAgent back-compat):** a `defineAgent` agent yields once (final
  flush) = today's single-frame behavior. → Verify: agent test.
- [ ] **DC-005 (fail-safe partial):** agent throws after 2 batches → those 2 persist,
  coherent partial page, no crash. → Verify: runtime test.
- [ ] **DC-006 (Sink one-row-per-turn):** the `Sink` records ONE `StoredEvent` with
  the turn's accumulated messages (not per-batch) — event-layer intact. → Verify: runtime test.
- [ ] **DC-007 (quickstart per-step):** the quickstart loop yields per provider step
  (a multi-step chat turn streams). → Verify: quickstart test.
- [ ] **DC-008 (contract):** `FacetAgent` is `AsyncIterable<ServerMessage[]>` (or a
  compatible union); all importers updated. → Verify: full typecheck.

## Invariant Fit (Facet-specific — REQUIRED)

| # | Invariant | Status | Note / mitigation |
|---|---|---|---|
| 1 | Responsibility boundary | **OK** | Only changes WHEN stage patches are delivered; no backend/domain surface. |
| 2 | Mechanism vs policy | **OK** | The agent decides yield points (policy); the framework only iterates+delivers (mechanism). |
| 3 | Fail-safe | **TOUCHES** | Mid-stream partial trees must render fail-safe (they already do — dangling skip, salvage fold). A mid-stream agent throw persists delivered batches and errors without crashing. |
| 4 | Declarative + tokens only | **N/A** | No tree/token change. |
| 5 | Flow-only safety | **N/A** | No rendering/overlay change. |
| 6 | Two-writers coherence | **TOUCHES** | Streaming delivers server-authoritative stage patches incrementally. The SERVER stays the single stage writer; the whole generator runs in the per-visitor serial lane so batch order == yield order; frames carry `era:seq`; client folds each. No second writer introduced. |
| 7 | Backend via agent | **OK** | No client fetch. |

## Public API / Package Surface

| Package | Change | Additive/Breaking |
|---|---|---|
| `@facet/core` | `FacetAgent` return type `Promise<ServerMessage[]>` → `AsyncIterable<ServerMessage[]>` (or union) | **Breaking** (pre-1.0) |
| `@facet/agent` | `defineAgent` yields once at end (back-compat single-batch); `Stage.flush` reused per batch | Additive (+ contract) |
| `@facet/runtime` | `handle` iterates the generator, applies+delivers per batch within the serial lane, accumulates the turn's messages into ONE Sink record | Additive (behavior) |
| `@facet/server` | deliver each yielded batch as a frame via the existing `/stream` era:seq path | Additive (likely reuses `deliver`) |
| `@facet/quickstart` | agent loop yields per provider step (flush moves inside the loop) | Additive (behavior) |

## Open Questions / Assumptions

- **Open (spec-bridge/Codex):** exact reuse of the async-delivery-1 machinery — does
  each batch go through `persist` + the frame-log `deliver` (`era:seq`) like a normal
  turn frame, and does the `applyMessages` late-seam need changes? (Assume: reuse
  `persist`/`deliver` per batch; the generator is driven inside the lane task.)
- **Open (spec-bridge/Codex):** the transport wait/timeout — the `/event` POST acks
  202 and batches stream via `/stream`; confirm the existing wait/late-result path
  (`FACET_MAX_CONCURRENT`, resume) composes with mid-turn frames.
- **Open (spec-bridge/Codex):** mid-stream error — record the delivered batches in
  the turn's Sink row (accumulated so far) vs drop. (Assume: record what was
  delivered; fail-safe keeps the partial coherent.)
- **Assumption:** `agentMutated`/seed-frame logic (persistWithSeed) applies to the
  FIRST batch of a turn; later batches are plain patches. Owner. Checkpoint: spec-bridge.
