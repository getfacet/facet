# Feature Intake Brief: Tree Data Section + Bindings

> Generated by `/feature-intake`. Plain-language, testable, invariant-checked.

## Goal

- An agent can declare a dataset ONCE in a named `data` warehouse on the stage
  tree and have multiple display nodes (table, chart, list, keyValue,
  metric/stat) render FROM it by name, instead of copying the same rows/series
  into each node. Updating the data is then a single patch at one path, and every
  bound view reflects it. Inline data stays fully supported; binding is opt-in.

## Current Problem

- Data is inlined in each node: `table.rows`, `chart.series`, `list.items`,
  `keyValue.items`, `metric/stat.value`. The same numbers shown as both a chart
  and a table must be authored twice and updated twice — a partial update leaves
  the two views showing different numbers (a real inconsistency), and every
  refresh re-sends the full payload into each node.
- Affected: the **agent author** (must duplicate + keep N copies in sync; larger
  prompt/context footprint) and, indirectly, the **end-user** (can see a chart
  and table disagree after a partial edit).

## User Scenario

### Who
- The **agent brain** authoring a stage (emits the tree + patches); the
  **end-user** views the rendered result. The **agent author** (framework
  consumer) benefits via a smaller, single-source data model.

### Trigger
- The agent draws a page that shows one dataset in more than one view, or expects
  to update a dataset repeatedly.

### Steps
1. The agent emits a tree with `data: { sales: [ {month,revenue}, … ] }` and two
   nodes: a `table` and a `chart`, each carrying `from: "sales"` instead of
   inline `rows`/`series`.
2. The renderer resolves each node's data from `data["sales"]`, projecting the
   row-records into that node's shape (table rows as-is; chart series from
   numeric columns; etc.).
3. Later the agent patches `data/sales` (whole dataset) or a single cell
   (`data/sales/3/revenue`). Both the table and chart re-render from the one
   change — no per-node duplication.
4. If a node's `from` names a dataset that is absent, dangling, or malformed, the
   node renders as an EMPTY node (empty table/chart/list) — never throws, never
   blank-crashes; when the data later arrives (buffered patch), it fills in.

### Steps (edge — precedence)
1. A node carries BOTH inline `rows` and `from: "sales"`.
2. `from` wins: the warehouse dataset is used and the inline value is ignored
   (declaring `from` is treated as the explicit intent to bind).

## Input/Output Examples

### Example 1 (Normal — one dataset, two views)
- Input: `{ data: { sales: [{month:"Jan",revenue:10},{month:"Feb",revenue:20}] },
  nodes: { t:{type:"table",columns:[…],from:"sales"},
  c:{type:"chart",kind:"bar",from:"sales"} , … } }`
- Output: the table shows two rows; the chart shows a 2-point series — both from
  the single `sales` dataset. A patch `replace /data/sales/1/revenue 25` updates
  both.

### Example 2 (Edge — dangling reference)
- Input: a node `{ type:"table", columns:[…], from:"ghost" }` with no
  `data.ghost`.
- Output: an empty table (headers, no rows) renders; nothing throws;
  `validateTree` keeps the node; `treeHasContent` treats an empty data-backed
  node as non-content (consistent with today's empty-table rule).

### Example 3 (Edge — malformed dataset)
- Input: `data.sales` is `"not an array"` / `[42, null]` / a 10^6-row array /
  rows with nested objects or forbidden keys.
- Output: the value is sanitized to the closed row-record shape (non-conforming
  rows/cells dropped, capped at the existing per-node row/cell bounds); the node
  renders the surviving rows or empty; never throws.

## Constraints (What Not to Do)

- **No local sort/filter engine in v1.** Binding does NOT add browser-side
  reordering/filtering; those remain agent round-trips (filterBar unchanged).
  This is a deliberate scope cut — the two-writers design for local view-state
  over server-owned data is a separate follow-up.
- **No backend/data fetching.** `data` is authored by the agent (UI-OUT) and
  patched by the agent; there is NO `from: "https://…"`, no resolver, no
  client-side fetch. A dataset is inert declared data, never a data source.
- **Closed value schema only.** A dataset is
  `Array<Record<string, string | number | boolean>>` — the same cell type as
  today's `table.rows`. No nested objects, no arbitrary JSON, no functions.
- **No expression/computed columns / no DSL.** Nodes project columns; they do not
  compute. Aggregation/derivation stays the agent's job.
- **No general repeater / template / binding-expression layer.** Projection stays
  renderer-owned INSIDE the closed display nodes (a table iterates its own rows).
  Facet does NOT add a "bind any subtree over an array" primitive, a named-function
  registry, or a `${…}` interpolation language. (This is exactly A2UI's `ChildList`
  template + `FunctionCall`/`formatString` surface — deliberately avoided; see
  Prior Art.)
- **No new client writer of stage content.** Resolution is pure render-time
  lookup; the browser writes no stage/data state. `data` is patched only by the
  server (the agent), exactly like `nodes`.
- Inline data stays valid and unchanged for nodes that don't use `from`.

## Policy & Edge Cases

| Scenario Step | Failure Policy | Boundary/Validation | Concurrency / Duplicate Trigger |
|---|---|---|---|
| 1 author `data` | malformed `data` map → drop bad datasets, keep valid ones (fail-soft, like nodes) | `data` optional; keys are dataset names (string, length-capped); each dataset is a row-record array capped at existing per-node row bound | server is sole writer; `data` patched on the same serialized stage-write path as `nodes` |
| 2 resolve `from` | dataset absent/dangling/malformed → node renders empty, never throws | `from` is a string dataset name; unknown → empty node | pure render-time lookup; no state written |
| 3 patch data | a stale/partial data patch salvages per-op like any patch (`foldPatchIntoStage`) | `/data/<name>` and `/data/<name>/<i>/<col>` are ordinary JSON-Pointer paths into the tree | same `applyPatch` on server + client; buffered forward-ref data fills a node that referenced it early |
| 4 empty render | empty data-backed node is non-content (`treeHasContent`/seed gate) | consistent with empty table/chart/list today | data arriving later re-renders the node (idempotent) |
| precedence | both inline + `from` → `from` wins deterministically | rule is static (presence of `from`), no ambiguity | — |
| projection | a node whose columns don't exist in the dataset → those cells render empty, node does not throw | each node reads only the columns it declares/needs | — |

## Done Criteria (Testable)

- [ ] DC-001 (Happy path — one dataset, many views): a tree with `data.sales` and
  a `table`+`chart` both `from:"sales"` renders the table rows and the chart
  series from the single dataset. → Verify: vitest render tests (`@facet/react`)
  + `validateTree` accepts `data` + `from`.
- [ ] DC-002 (Single-source update): a patch to `data/sales` (whole) and to
  `data/sales/<i>/<col>` (cell) both change every bound view; no per-node inline
  edit needed. → Verify: vitest `applyPatch`/render test.
- [ ] DC-003 (Dangling/absent `from`): a node referencing a missing dataset
  renders EMPTY and never throws; `treeHasContent` counts it as non-content. →
  Verify: vitest fail-safe render + tree tests.
- [ ] DC-004 (Malformed data sanitized): non-array data, junk rows/cells, nested
  objects, forbidden keys, and over-cap row counts are dropped/clamped to the
  closed row-record schema; the node renders survivors or empty; never throws. →
  Verify: vitest hostile-input suite on the data validator.
- [ ] DC-005 (Precedence): a node with both inline data and `from` uses the
  warehouse dataset and ignores the inline value. → Verify: vitest render test.
- [ ] DC-006 (Two-writers preserved): binding adds NO client-side stage/data
  writer — `data` is only mutated via server patches through the same
  `applyPatch`; the renderer writes no data state. → Verify: type/structure test
  + grep that resolution is read-only.
- [ ] DC-007 (Back-compat): existing trees with inline `rows`/`series`/`items`
  and no `data`/`from` render byte-identically; `data`/`from` are additive
  optionals. → Verify: existing render suites stay green + explicit no-`data`
  case.
- [ ] DC-008 (Buffered forward reference): a node that references a dataset added
  in a LATER patch batch resolves once the data arrives (no permanent prune). →
  Verify: vitest patch-ordering test.

## Invariant Fit (Facet-specific — REQUIRED)

| # | Invariant | Status | Note / mitigation |
|---|---|---|---|
| 1 | Responsibility boundary (UI-out/UI-in only; no backend) | TOUCHES (mitigation) | `data` is agent-authored declared data (UI-OUT), NOT a fetch/source. No URL binding, no resolver, no client fetch. It is the same trust tier as inline `rows` — just relocated to a named section. |
| 2 | Mechanism vs policy | OK | Facet provides the warehouse + projection mechanism; the agent decides what data exists and which nodes bind. No computation/aggregation moved into the framework. |
| 3 | Fail-safe | TOUCHES (mitigation) | New validated surface (`data` map + `from`). One pure data validator (closed row-record schema, caps, forbidden-key safe, non-recursive) + dangling/absent/malformed → empty node, never throws. Mirrors `validateTree`/theme fail-safe. |
| 4 | Declarative + tokens only | OK | `data` is typed declarative values (cells are string/number/boolean); no markup/CSS/scalars-as-style. |
| 5 | Flow-only safety | OK | No layout/overlay change; nodes render in flow exactly as today. |
| 6 | Two-writers coherence | OK | No local execution added (sort/filter explicitly deferred). `data` is server-written only, patched via the same serialized path and the same `applyPatch` on both sides — identical coherence story to `nodes`. |
| 7 | Backend via agent, never client fetch | OK | No client fetch; datasets are authored/patched by the agent through its normal tools. |

- Three Assumptions routed to `/spec-bridge` (projection per node — especially

## Prior Art — Google A2UI data model (deep-research 2026-07-12, primary specs v0.8/v0.9/v1.0)

A2UI independently made the **same structure↔data split** Facet is pursuing (a
first-class Data Model object separate from the component tree), which validates
the direction. But its later design choices are precisely the hazards Facet's
invariants avoid — so the research **confirms** our decisions and sharpens two:

**Borrow / confirmed by A2UI's converging choice**
- First-class separate data object with small incremental data updates
  (structure sent once, data patched) — our `data` warehouse + `/data/...` patches.
- Fail-safe "unresolved reference → degrade" — A2UI makes this an advisory
  *should*; Facet already makes it a **hard invariant** (dangling `from` → empty
  node). Facet's posture is one step stronger; keep it.
- By-name `from:"sales"` is an A2UI-unexplored simplification that fits
  whole-dataset display nodes well (A2UI only has JSON-Pointer path binding).

**Deliberately AVOID (A2UI accepted these for expressiveness; they break Facet invariants)**
- **Two-way binding / dual-writer**: A2UI v0.9+ input components write the LOCAL
  data model and ship the ENTIRE model back to the server as convergence metadata.
  Facet stays **server-sole-writer**; `data` is patched only by the agent. (This is
  the same reason local sort/filter is deferred — invariant #6.)
- **Whole-model snapshot transport**: A2UI returns the full model on every action.
  Facet travels **only RFC-6902 deltas** through the one pure `applyPatch`.
- **Embedded compute layer**: A2UI's `ChildList` repeater templates, named
  `FunctionCall` registry (`required`/`regex`/`add`/`formatString`/`and`/`or`/`not`),
  and `${…}` interpolation language. Facet keeps projection **renderer-owned inside
  closed nodes** and adds **no DSL** (see Constraints). A2UI keeps this "safe" only
  by name-referencing pre-registered functions — a surface Facet does not want.
- **Over-flexible value schema**: A2UI drifted to arbitrary nested JSON. Facet's
  closed `Array<Record<string,scalar>>` row-records are stricter at every version.

**Sharpened by the research** (folded into Constraints + the projection Assumption):
no general repeater/template/expression primitive; and scalar nodes (`metric`/`stat`)
are the one place A2UI's fine-grained path binding does something Facet's coarse
by-name `from` can't — spec-bridge must add a row/column selector or defer them.
