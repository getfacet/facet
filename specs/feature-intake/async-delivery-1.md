# Feature Intake Brief: Async Delivery & Scale Round 1

> Generated by `/feature-intake`. Plain-language, testable, invariant-checked.
> Source: the deferred P2 umbrella in `docs/HARDENING-BACKLOG.md` (post-campaign-1).

## Goal

- An agent turn's result is **never silently lost**: however long the turn takes
  and whatever the visitor's connection does in the meantime, the completed
  result reaches the session and every connected tab exactly once, in order.
- The reference deployment survives load and abuse gracefully: spawn-mode brain
  processes are bounded, a wedged visitor POST cannot silently stall the page,
  and the server's remote-agent contract is pinned by HTTP-level tests.

## Current Problem

- **Late results are discarded** (`packages/server/src/server.ts` `remoteAgent`):
  the per-event 120s timeout resolves the pending request with an offline note;
  when the real `/agent/control` result arrives later, `pending.get(requestId)`
  is gone and the messages are dropped with a 202 — nothing is applied to the
  session, nothing reaches the browser. Routine in persistent mode, where turns
  queue serially behind a long turn. Affected: the end-user (their answer
  vanishes) and the agent author (work done, invisible).
- **Reconnect can lose frames**: the `/stream` rehydrate window is documented
  (RISK-HRD-4) to drop a `say`/patch that lands between the snapshot read and
  the fan-out join under an async sink. Affected: the end-user on flaky
  networks.
- **Spawn mode is unbounded** (`packages/bridge/src/bridge.ts`): each new
  `visitorId` can spawn a brain CLI concurrently with no cap — a resource
  exhaustion vector on a public page. Affected: the operator.
- **A wedged visitor POST stalls silently** (`packages/client/src/sse-transport.ts`,
  RISK-HRD-3): the ordered send chain has no fetch timeout, so one black-holed
  POST queues everything behind it while the SSE channel looks healthy.
- **No HTTP-level regression tests** for the remote-agent handshake and event
  round-trip — the exact seams the above changes touch.

## User Scenario

### Who
- The **end-user** of an agent-built page (slow turns, reconnects, rapid input),
  the **operator/agent author** running the bridge on a public link.

### Trigger
- A visitor event starts an agent turn that outlives the server's per-event
  timeout; and/or the visitor's connection drops and reconnects mid-turn.

### Steps
1. Visitor sends a message; the agent (persistent driver) starts a long turn;
   a second visitor event queues behind it.
2. The server's per-event timeout fires for the queued event → the visitor sees
   an interim "still working" note (not a terminal error).
3. The turn completes; the driver POSTs `/agent/control` with the result.
4. **Expected:** the server applies the late result to the session through the
   normal per-visitor path and pushes it to every connected tab — the visitor
   sees the real answer arrive.
5. Meanwhile the visitor's tab reconnects (`/stream` with `Last-Event-ID`).
6. **Expected:** the server joins the tab to the live fan-out immediately and
   replays exactly the frames after the tab's last-seen seq — no lost `say`,
   no stale full-replace overwriting a newer stage, no duplicates.
7. On a public spawn-mode page, 50 new visitors arrive at once.
8. **Expected:** at most N (default 4, env-tunable) brain CLIs run concurrently;
   the rest queue FIFO (per-visitor ordering preserved); waiting visitors get a
   brief notice; nothing is dropped.
   _Amendment (spec-bridge): the notice is the server's interim-timeout say, so
   under default knobs it arrives after `agentTimeoutMs` (120s) — operators
   wanting a snappier queue notice lower that knob. A bridge-originated
   immediate notice needs a protocol change and is deferred._

## Input/Output Examples

### Example 1 (Normal — late result delivered)
- Input: event `E2` queued behind a 3-minute turn; server timeout 120s fires;
  driver posts `E2`'s result at t=180s.
- Output: at t=120s the visitor sees an interim note; at t=180s the real
  patches/says apply to the session and render in the open tab. `/agent/control`
  answers 202 both before and after the timeout.

### Example 2 (Edge — reconnect during the late window)
- Input: the tab's EventSource reconnects (sends `Last-Event-ID: 41`) while the
  late result (frames 42–43) is being applied.
- Output: the server joins the tab to the fan-out first, replays frames 42–43
  from the per-session log, and live frames continue from 44 — the tab shows
  every frame exactly once, in seq order. A malformed/ancient `Last-Event-ID`
  degrades to today's full snapshot rehydrate (never an error page).

### Example 3 (Edge — spawn flood)
- Input: 50 concurrent first-visit events in spawn mode, cap 4.
- Output: 4 CLIs run; 46 events wait FIFO; each waiting visitor receives one
  notice say; as slots free, turns run in arrival order. Process count never
  exceeds the cap; no event is silently dropped.

## Constraints (What Not to Do)

- No delivery-infrastructure dependency (no Redis/queue service) — the seq log
  rides the existing `Sink`/`StageStore` interfaces; in-memory and file
  references stay dependency-free.
- No protocol vocabulary change: bricks/tokens/actions untouched (`STAGE_SPEC`
  unaffected). The seq is transport framing (SSE `id:`), not tree content.
- No client-side fetch of domain data (invariant #7) — this is all UI-out/UI-in
  plumbing.
- Do not break the in-process (`LocalTransport`/playground) path, which has no
  reconnect or timeout concerns.
- Do not turn the interim-timeout note into a terminal state — the pending entry
  may resolve the HTTP wait, but the turn stays live.
- Spawn cap must not reorder a single visitor's events (per-visitor serialization
  is a correctness requirement from campaign 1).

## Policy & Edge Cases

| Scenario Step | Failure Policy | Boundary/Validation | Concurrency / Duplicate Trigger |
|---|---|---|---|
| Late-result apply (step 4) | Result applies via the existing per-visitor serial queue + per-op salvage; a throwing op drops, turn survives | `/agent/control` body shape-checked as today; unknown `requestId` after timeout is the LATE path, not a 400 | Two late results for the same visitor apply in arrival order through the serial queue |
| Interim timeout note (step 2) | Note is a normal `say`; if the agent stream died meanwhile, existing `dropAgent` offline path wins | Timeout stays operator-tunable (`agentTimeoutMs`) | Timeout firing and result arriving race: `pending` resolution is atomic — exactly one of {in-time resolve, interim note + late apply} |
| Seq replay (step 6) | Missing/garbled `Last-Event-ID` → full snapshot rehydrate (today's behavior); replay read failure → end the stream so EventSource retries (campaign-1 fix preserved) | Seq is per-session, monotonic, starts at 1; log retention = existing sink history (no new store) | Frame written vs tab joining: join-first + replay-after-join makes the window loss-free; duplicates prevented by seq comparison |
| Spawn cap (step 8) | CLI crash frees its slot (existing close handler); queue entry for a closed session is skipped | Cap ≥ 1, env `FACET_MAX_CONCURRENT` validated like `FACET_BRIDGE_PORT` (bad value → clear error, exit 1) | Same visitor's second event never overtakes their first (per-visitor chain kept); cap applies across visitors |
| Client POST timeout | `AbortSignal.timeout` abort logs + chain continues (rejected-POST behavior from campaign 1) | Default bounded (Assumption A-2); no retry (at-most-once from the client, as today) | Ordering unchanged — timeout only bounds how long the chain head can block |

## Done Criteria (Testable)

- [ ] DC-001 (Happy path — late delivery): a result posted to `/agent/control`
  after the per-event timeout is applied to the session and pushed to a
  connected `/stream` tab. → Verify: HTTP-harness vitest (short `agentTimeoutMs`,
  delayed control POST, assert frame arrives + store updated).
- [ ] DC-002 (Boundary — no duplicate on in-time result): a result arriving
  before the timeout is applied exactly once (no interim note, no double push).
  → Verify: harness test asserting frame count and absence of the interim say.
- [ ] DC-003 (Happy path — seq replay): a tab reconnecting with `Last-Event-ID`
  receives exactly the missed frames, in order, no duplicates, while a
  concurrent event lands in the gap. → Verify: harness test (delayed store,
  concurrent `/event`, assert seq-ordered frames).
- [ ] DC-004 (Boundary — bad resume token): reconnect with a malformed or
  out-of-retention `Last-Event-ID` degrades to the full snapshot rehydrate;
  never a 4xx/5xx on `/stream`. → Verify: harness test.
- [ ] DC-005 (Happy path — spawn cap): with cap=2 and 5 concurrent visitors,
  at most 2 brain processes exist at any moment and all 5 turns complete FIFO.
  → Verify: bridge vitest with a stub command recording concurrent count.
- [ ] DC-006 (Boundary — cap misconfig): `FACET_MAX_CONCURRENT=0`/`"abc"` exits
  with a clear error (no uncaught crash). → Verify: env vitest matrix.
- [ ] DC-007 (Client timeout): a black-holed POST aborts after the bound and the
  next queued event is sent (chain not wedged). → Verify: sse-transport vitest
  with a never-resolving fetch + fake timers.
- [ ] DC-008 (Invariant preserved — fail-safe): a late result whose patches no
  longer apply (stage moved on) salvages per-op and never kills the turn or the
  stream. → Verify: harness test (late result with one stale op + one good op).
- [ ] DC-009 (Handshake regression): agent connect with bad token → 403; second
  concurrent agent → 409; heartbeat keeps the slot; reaped slot accepts a new
  dial. → Verify: harness tests (pins campaign-1 behavior at the HTTP level).
