# Feature Intake Brief: Event Layer v1 — 3-Layer Model + Ordered Replay Log

> Generated by `/feature-intake`. Plain-language, testable, invariant-checked.
> Companion design memory: `facet-event-layer-redesign` (owner's private memory).

## Goal

Give Facet a coherent, extensible event model where **every** visitor interaction —
not just the ones the agent answers — is captured into **one durable, correctly
ordered log per session**, so a visitor's journey can later be **faithfully
replayed**. Do this without growing the agent-facing surface: the agent still
sees a minimal set of semantic events. (v1 delivers the *capture/log*; the replay
player itself is a follow-up.)

## Current Problem

- **"Recorded" and "sent to the agent" are the same thing today.** The runtime
  `Sink` (`packages/runtime/src/sink.ts`) records only interactions that reach
  `runtime.handle` — the forwarded subset (`visit`, `message`, agent `action`).
- **Local interactions vanish.** `navigate`/`toggle` taps are handled entirely in
  the renderer (`StageRenderer.handlePress`) and never reach the server; `scroll`
  is a style token, not an event. None are logged.
- **Consequence:** replay is impossible — the log shows "arrived, sent a message"
  but misses the screen the visitor navigated to and what they toggled, so a
  replay reconstructs the wrong stage.
- **Naming smell:** the single `action` wire kind conflates trigger-agnostic
  agent actions with local view commands; as more triggers arrive, lumping them
  all under `action` erases "what actually happened."
- **Affected:** the agent author / operator who wants session replay, analytics,
  or audit of a Facet page; and Facet's own coherence (the model doesn't yet
  separate collection from forwarding).

## User Scenario

### Who
- **Primary:** the agent author / operator running a Facet server who wants a
  complete, ordered, replayable record of each visitor session.
- **Secondary:** the end-user of an agent-built page (their local interactions —
  navigate/toggle — must now be captured without changing how the page feels).

### Trigger
A visitor interacts with a Facet page (arrives, taps, navigates, toggles, chats).

### Steps
1. Visitor arrives → `visit` event → forwarded to agent AND logged.
2. Visitor taps a button wired to an agent action → `tap` event → forwarded AND logged.
3. Visitor taps a `navigate`/`toggle` button → renderer applies the effect
   **instantly (optimistic)**, AND fires a **record-only** send in the background.
4. Server records the local `tap` (with its resolved effect) to the `Sink` on the
   **same serial lane**, but does **not** invoke the agent.
5. Visitor sends a chat message → `message` event → forwarded AND logged.
6. Result: `Sink.history()` returns every interaction in true send order, each
   paired with the agent's reply patches (empty for record-only rows) — a complete
   bidirectional timeline keyed by the append id.

## Input/Output Examples

### Example 1 (Normal — local navigate tap)
- Input: visitor taps box `#go-pricing` whose `onPress` is `{kind:"navigate", to:"pricing"}`.
- Output: screen switches to `pricing` immediately (no round-trip); a background
  `POST /record` stores a `CollectedEvent` `{kind:"tap", target:"#go-pricing",
  effect:{navigate:"pricing"}, seq:N}` with `messages:[]`; the agent is not called.

### Example 2 (Edge/Error — record-only send fails)
- Input: the background `POST /record` for a toggle tap fails (offline blip).
- Output: the toggle still applied locally (UI unaffected); no throw, no retry in
  v1; the visitor's next successful record carries a higher client `seq`, so the
  gap (`seq N missing`) is **detectable** in `history()` — silent loss is avoided.

## Constraints (What Not to Do)

- **No hold removal** — `onHold` stays as-is in v1 (separate future bundle).
- **No scroll/drag/swipe collection** — `scroll` remains a style token; drag/swipe
  are not added. Only `visit`/`message`/`tap` are collected in v1.
- **No replay player/UI** — v1 produces the *log*, not the playback.
- **No retry/queue/dedup for record-only** — best-effort send in v1.
- **No new durable store** — reuse existing `Sink` backends (File/Postgres).
- **No client-side backend fetch** — `/record` targets Facet's own reference
  server (UI-IN transport), never arbitrary domain data.
- **No blocking the UI on logging** — local effects stay instant.

## Policy & Edge Cases

| Scenario Step | Failure Policy | Boundary/Validation | Concurrency / Duplicate Trigger |
|---|---|---|---|
| `visit` forward+log | agent offline → `offlineFor` fallback (existing) | one per session open | serial lane per (agent,visitor) |
| agent `tap` (`/event`) | existing validation; 202 ack | `MAX_FIELDS_KEYS`/`MAX_FIELD_VALUE_CHARS` (existing) | same serial lane; ordered by append id |
| local `tap` (`/record`) | send fails → dropped (best-effort), gap detectable via client `seq` | reject malformed body like `/event`; unknown target already no-ops in renderer | MUST share the same client send channel + server serial lane so order = append id |
| `message` forward+log | existing | existing | serial lane |
| ordering across `/event`+`/record` | — | — | interleaved by one monotonic append id; `at` timestamp is NOT the order key |

## Done Criteria (Testable)

- [ ] **DC-001 (Happy — local capture):** a `navigate`/`toggle` tap produces a
  `Sink` record whose `event` is a `tap` `CollectedEvent` with the resolved
  `effect`, and the agent is **not** invoked. → Verify: vitest on the `/record`
  server path — `Sink.record` called once, agent mock not called.
- [ ] **DC-002 (Ordering):** interleaved sends A(`/event`), B(`/record`),
  C(`/event`) from one visitor appear in `history()` in send order (append id
  ascending), independent of `at`. → Verify: vitest sequence assertion.
- [ ] **DC-003 (Fail-safe):** a rejected `/record` send does not throw client-side
  and does not alter renderer view state (local effect already applied). →
  Verify: vitest — record transport rejects; renderer state unchanged; no throw.
- [ ] **DC-004 (Gap detection):** every interaction carries a per-session client
  monotonic `seq`; dropping one record leaves a detectable gap in `history()`. →
  Verify: vitest — seqs strictly increasing; omit one → gap assertion.
- [ ] **DC-005 (Record-only isolation):** the `/record` endpoint never calls
  `runtime.handle`/the agent. → Verify: vitest — agent mock uninvoked on `/record`.
- [ ] **DC-006 (Type shape):** `CollectedEvent` is the log currency (`visit |
  message | tap`); `tap` carries `{target, name?, payload?, fields?, effect?}`;
  `ClientEvent` (forward) is derived. → Verify: type-level + unit test on the
  record boundary mapping.
- [ ] **DC-007 (Boundary):** a malformed `/record` body is rejected (4xx) with no
  `Sink` write and no crash. → Verify: vitest — bad body → reject, no record.
- [ ] **DC-008 (Rename / breaking):** the wire + agent-facing kind is `tap` (not
  `action`); core/agent/quickstart consumers compile against the renamed surface.
  → Verify: `pnpm typecheck` + updated unit tests referencing `tap`.

## Invariant Fit (Facet-specific — REQUIRED)

| # | Invariant | Status | Note / mitigation |
|---|---|---|---|
| 1 | Responsibility boundary (UI-out/UI-in only; no backend/domain) | **OK** | Pure UI-IN capture + persistence of interactions; no domain fetch/compute added. `Sink` is existing conversation persistence. |
| 2 | Mechanism vs policy (agent authors; Facet provides capability) | **OK** | Agent still authors triggers in the tree; Facet only records. v1 forward policy (visit/message/tap all forward) is deliberate framework mechanism. |
| 3 | Fail-safe (never throw/inject; degrade to plain) | **OK** | Record-only send is best-effort/fire-and-forget; a failure never throws or blocks the renderer. `/record` body validated like `/event`. |
| 4 | Declarative + tokens only (no raw HTML/JS/CSS/pixels) | **OK** | `tap`/`CollectedEvent` are data types; no new markup or escape hatch. |
| 5 | Flow-only safety (overlay only via dedicated brick) | **OK** | Nothing floats/overlaps; not touched. |
| 6 | Two-writers coherence (local execution vs agent stage) | **TOUCHES** | Local navigate/toggle execute in the browser AND are now logged. Mitigation: record-only events pass the SAME client send channel (`sendChain`) + SAME server per-(agent,visitor) serial lane; order = Sink **append id** (join key), so the ordered bidirectional log reconstructs the server-authoritative stage faithfully. Client monotonic `seq` makes any dropped record detectable, not silent. |
| 7 | Backend calls go through the agent, never client fetch | **OK** | `/record` is Facet's own reference-transport endpoint (UI-IN), not a page-initiated domain fetch. |

## Open Questions / Assumptions

- **Assumption:** existing `Sink` backend durability (File JSONL / Postgres
  `facet_event`) is sufficient for v1 replay; no cross-restart durability work
  beyond what the chosen backend already gives. Owner: owner. Checkpoint: spec-bridge.
- **Assumption:** the ephemeral frame-log (`<era>:<seq>`, resume-only) is left
  untouched; the durable join key is the Sink append id (separate concern). Owner:
  owner. Checkpoint: spec-bridge.
- **Open (spec-bridge):** exact wire shape of `CollectedEvent.tap.effect` and how
  `ClientEvent` (forward) is derived from a `CollectedEvent` at the record boundary.
- **Open (spec-bridge):** whether the client `seq` also travels on `/event` (not
  just `/record`) so gap detection spans the whole timeline — recommended yes.
