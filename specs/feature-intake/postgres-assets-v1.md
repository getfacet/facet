# Feature Intake Brief: PostgresAssets — DB backend for AssetsStore

> Generated by `/feature-intake`. Theme 3 of `facet-brick-vocab-roadmap`.
> Runs in PARALLEL with theme 1 (isolated: `@facet/store-postgres` only).
> **Implementation will be done in Codex — this chain stops at `/spec-bridge`.**

## Goal

Make custom per-agent **themes and stamps** (and the optional initial tree) durable
in a database, by adding a **`PostgresAssets`** backend for the existing
`AssetsStore` seam — closing the one asymmetry among Facet's three persistence
seams (`StageStore` and `Sink` already have Postgres backends; `AssetsStore` does
not). It mirrors `PostgresStageStore`/`PostgresSink` exactly.

## Current Problem

- `AssetsStore` (`packages/runtime/src/assets.ts`) has only **`MemoryAssets`** (docs
  injected in the constructor) and **`FileAssets`** (reads a directory). There is
  **no DB backend** — `@facet/store-postgres` implements `PostgresStageStore` +
  `PostgresSink` but NOT assets.
- So custom themes/stamps can't be stored in a DB → not durable across restarts,
  not shareable across instances, no operator UI can persist them centrally.
- **Affected:** the operator running a Facet deployment who wants custom
  themes/stamps to live in their DB (Postgres/Supabase) alongside stage + events.
  Owner: "커스텀 테마/스탬프는 DB 지원을 해줘야 해요."
- This also unblocks **theme 2** (stamp reference-expand): the runtime expands a
  named stamp from the AssetsStore — a DB-backed store is the scalable source.

## User Scenario

### Who
The operator (Facet deployment owner) — and, at boot, the runtime's `loadAssets`.

### Trigger
An operator has stored custom themes/stamps for an agent in Postgres; a Facet
server boots (or an admin path writes a new custom stamp).

### Steps
1. (write) The operator persists an agent's asset documents to Postgres via
   `PostgresAssets` (a concrete write method) — themes, stamps, initial tree.
2. (read) At boot, the runtime calls `loadAssets(new PostgresAssets(pool), agentId)`
   → `PostgresAssets.load(agentId)` reads the agent's `AssetDocuments` from the DB.
3. `loadAssets` runs them through the SAME `validateTheme`/`validateStamp` gate as
   defaults (the store returns RAW docs; validation stays in the runtime).
4. The agent's page resolves with the default base layer + the agent's custom
   themes/stamps layered on top (existing `loadAssets` behavior).

## Input/Output Examples

### Example 1 (Normal)
- Input: `PostgresAssets(pool).load("agent-42")` where `facet_assets` holds a row
  for `agent-42` with `themes`/`stamps`/`initial_tree` jsonb.
- Output: an `AssetDocuments` `{ themes, stamps, initialTree }` from the row →
  `loadAssets` validates → `LoadedAssets`.

### Example 2 (Edge/Error)
- Input: `load("agent-with-no-row")`; or a row whose `stamps` jsonb is malformed.
- Output: **no row → empty `AssetDocuments`** (the default base layer still
  resolves — DC-001 of `loadAssets`); **malformed doc → dropped by `loadAssets`'s
  validator with a recorded issue**, other docs survive. Never throws.

## Constraints (What Not to Do)

- **No hot reload** — `loadAssets` runs once at boot (existing Decision Lock). The
  store just reads; no watch/subscribe.
- **No change to the `AssetsStore` interface** — it stays read-only (`load`). The
  write API is a concrete `PostgresAssets` method, NOT added to the interface (so
  `MemoryAssets`/`FileAssets` are unaffected).
- **No validation in the store** — `PostgresAssets.load` returns RAW
  `AssetDocuments`; `loadAssets` owns validation/fail-safe (single gate).
- **`pg` stays a peer dependency** (as in the existing store-postgres package).
- **No new domain affordance in the tree** — this is a persistence adapter behind
  an existing interface, not a new brick/token/backend binding.
- Mirror `PostgresStageStore`/`PostgresSink` (constructor takes `pool`; `initSchema`
  creates the table).

## Policy & Edge Cases

| Scenario Step | Failure Policy | Boundary/Validation | Concurrency |
|---|---|---|---|
| `load(agentId)` | no row → empty docs; DB error → propagates (boot-time, operator-visible) | agent_id text key | read-only, idempotent |
| write custom docs | invalid input rejected before write (shape check) OR left to loadAssets on read | jsonb size (DB-bounded) | upsert per agent |
| malformed stored jsonb | `loadAssets` validator drops the bad doc, records an issue | per-doc validation in runtime | n/a |
| `initSchema` | idempotent (`create table if not exists`) | — | safe to run repeatedly |

## Done Criteria (Testable)

- [ ] **DC-001 (load round-trip):** `PostgresAssets.load(agentId)` returns the
  `AssetDocuments` persisted for that agent (themes+stamps+initialTree). → Verify:
  vitest with the package's existing pg mock/double pattern.
- [ ] **DC-002 (empty agent):** `load` for an agent with no row returns empty docs;
  `loadAssets` still resolves the default base layer. → Verify: vitest.
- [ ] **DC-003 (write persists):** the `PostgresAssets` write method upserts an
  agent's docs and a subsequent `load` returns them. → Verify: vitest round-trip.
- [ ] **DC-004 (fail-safe malformed):** a malformed stored stamp/theme is dropped by
  `loadAssets` (issue recorded), others survive; no throw. → Verify: vitest.
- [ ] **DC-005 (initSchema):** `initSchema(pool)` creates the assets table
  idempotently (mirrors facet_stage/facet_event). → Verify: schema test / SQL check.
- [ ] **DC-006 (barrel export + interface):** `PostgresAssets implements AssetsStore`
  and is exported via the package barrel; the `AssetsStore` interface is unchanged.
  → Verify: typecheck + export test.

## Open Questions / Assumptions

- **Open (spec-bridge/Codex):** write shape — bulk `putAssets(agentId, docs)`
  (mirror `load`'s whole-docs shape) vs granular per-doc (`saveStamp`/`deleteStamp`).
  Assume **bulk put** first; granular is a follow-up. Owner.
- **Open (spec-bridge/Codex):** schema — one jsonb row per agent (mirror
  `facet_stage`) vs per-doc rows (`facet_theme`/`facet_stamp`). Assume **one row per
  agent** (matches `load(agentId)`-whole). Owner.
- **Assumption:** `AssetDocuments` is the raw (pre-validation) shape the store
  round-trips; confirm its exact fields (themes/stamps/initialTree) in spec-bridge.
