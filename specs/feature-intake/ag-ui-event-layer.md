# Feature Intake Brief: AG-UI Event Layer

> Generated by `/feature-intake`. Plain-language, testable, invariant-checked.
> Status: approved by owner request for `/spec-bridge` → `/worktree-prep` → `/implement`.

## Goal

- Move Facet's public client/server/agent communication onto the official AG-UI
  event model, while keeping Facet's safe UI semantics: `FacetTree`, RFC 6902
  stage patches, `validateTree`, shared stage folding, and the fail-safe renderer.
- Use AG-UI for the broad event envelope Facet is likely to need anyway:
  run lifecycle, message streaming, tool lifecycle, state sync, activity,
  reasoning, custom events, interrupts, and capabilities.
- Avoid inventing a parallel "AG-UI-like" protocol. Facet should add the
  Facet-specific safety layer around AG-UI, not reimplement AG-UI.

## Current Problem

- Facet currently has a small native event protocol: browser `ClientEvent`
  (`visit` / `message` / `tap`) and agent `ServerMessage` (`patch` / `say` /
  server `reset`), carried by reference SSE+POST transports.
- That protocol is intentionally simple, but it is starting to overlap with
  event categories AG-UI already standardizes: run lifecycle, message streaming,
  tool calls, state snapshots/deltas, activity, reasoning, custom events,
  interrupts, and capabilities.
- If Facet keeps expanding its own event vocabulary, it risks creating a
  home-grown protocol that is hard to interoperate with CopilotKit, A2UI, and
  other AG-UI-compatible agent stacks.
- Affected users are future Facet agent authors and platform integrators. There
  are no external Facet users yet, so the public protocol can change now with
  less migration cost.

## User Scenario

### Who

- **Primary:** a Facet framework consumer building a client app or hosted
  platform that wants to use standard AG-UI-compatible clients and agents.
- **Secondary:** the end-user of a Facet page; they should still see a safe,
  live, per-visitor UI rendered from Facet bricks.
- **Internal:** Facet maintainers, who need a protocol that can grow without
  copying AG-UI's event surface by hand.

### Trigger

An owner decides to make AG-UI the public event layer before Facet has external
users, instead of continuing to grow the native `ClientEvent`/`ServerMessage`
wire protocol as the primary external surface.

### Steps

1. A visitor opens or interacts with a Facet page.
2. The browser/app uses an AG-UI-facing Facet client adapter to send a run input
   or Facet UI event to the Facet server.
3. The server extracts the Facet UI-IN event from the AG-UI input envelope,
   validates it, opens the authoritative `(agent, visitor)` session, and invokes
   the Facet runtime.
4. The runtime keeps applying Facet stage patches through the existing
   `foldPatchIntoStage` / `validateTree` path; the server remains the
   authoritative stage writer.
5. The server emits AG-UI events back to the client: lifecycle events, text
   message streaming for chat, and `STATE_SNAPSHOT` / `STATE_DELTA` for Facet
   stage state.
6. The client AG-UI adapter accepts only the reserved Facet state payloads for
   stage updates, applies them with the shared Facet fold, and renders through
   `StageRenderer`.
7. External or in-process agents can be connected through AG-UI-compatible
   adapter boundaries, while Facet's safe UI model stays Facet-owned.

## Input/Output Examples

### Example 1 (Normal - message updates the stage)

- Input: a visitor sends "show pricing" through the AG-UI client path. The
  Facet adapter encodes the visitor message and Facet visitor context into a
  `RunAgentInput` / AG-UI request understood by the Facet server.
- Output: the server streams AG-UI events such as `RUN_STARTED`,
  `TEXT_MESSAGE_*`, `STATE_DELTA` containing Facet stage JSON Patch operations,
  and `RUN_FINISHED`. The client extracts the Facet state delta, folds it into
  the current `FacetTree`, validates it, and renders pricing UI using only
  `box` / `text` / `media` / `field`.

### Example 2 (Normal - agent action tap)

- Input: the visitor presses a Facet box whose `onPress` is an agent action with
  `name`, optional `payload`, and collected `fields`.
- Output: the client sends the tap through the AG-UI-facing Facet adapter. The
  server converts it back to a Facet UI-IN event, validates action payload and
  field caps, invokes the agent, and streams AG-UI lifecycle/text/state events
  back. Local `navigate` / `toggle` actions remain local view-state and may be
  logged through an AG-UI-compatible record/custom path without invoking the
  agent.

### Example 3 (Edge/Error - malformed AG-UI state delta)

- Input: an AG-UI stream contains a `STATE_DELTA` whose patch targets an
  unexpected state namespace, has invalid patch ops, or tries to introduce raw
  HTML/JS into the Facet stage.
- Output: Facet ignores or safely salvages the invalid stage update through the
  same fail-safe fold/validation path. The renderer does not throw, does not
  inject markup, and keeps the last valid stage.

## Constraints (What Not to Do)

- Do not put `@ag-ui/*` dependencies into `@facet/core`; core must remain
  dependency-free and browser-safe.
- Do not let arbitrary AG-UI `state`, `custom`, or `raw` events write directly
  to the renderer. Only Facet-recognized stage payloads can mutate `FacetTree`.
- Do not replace `FacetTree`, brick validation, token style constraints, or
  shared patch folding with generic AG-UI state handling.
- Do not add backend/domain capabilities to Facet. Domain work still belongs to
  the agent's own tools.
- Do not require browser clients to connect directly to an agent process. A
  Facet server/proxy may be the AG-UI endpoint and may route to in-process or
  external agents.
- Do not implement a private clone of AG-UI event schemas when official SDK
  types/schemas can be used.
- Do not remove fail-safe native behavior until AG-UI parity is test-covered.

## Policy & Edge Cases

| Scenario Step | Failure Policy | Boundary/Validation | Concurrency / Duplicate Trigger |
|---|---|---|---|
| Browser/app sends AG-UI input | malformed AG-UI input is rejected or converted to a safe no-op/error event | Facet extracts only known UI-IN payloads; visitor identity remains app/server responsibility | same visitor events must preserve current per-session ordering semantics |
| Server invokes runtime | agent/runtime failure yields AG-UI error/text events without wedging the stream | server ignores client-provided stage as authority; StageStore remains source of truth | same `(agent, visitor)` lane stays serialized |
| Server emits stage update | invalid patch is dropped/salvaged; valid chat/lifecycle can continue | `STATE_DELTA`/`STATE_SNAPSHOT` stage payloads must be Facet-shaped and pass fold/validation | streamed batches must preserve patch/text order |
| Client receives AG-UI events | unknown AG-UI events are ignored or surfaced as non-stage activity; renderer never crashes | only reserved Facet state namespace mutates stage | duplicate/replayed snapshots must be idempotent or replace current state coherently |
| External agent adapter | disconnected/timeout agent produces AG-UI lifecycle/error feedback | AG-UI agent output is normalized to Facet patch/text before stage mutation | late results must not overwrite newer server-authoritative stage changes |

## Done Criteria (Testable)

- [ ] **DC-001 (Happy path - AG-UI client to Facet server):** a visitor message
  sent through the AG-UI-facing client path reaches the Facet runtime as the
  correct Facet UI-IN event and returns an AG-UI event stream containing
  lifecycle, text, and Facet stage state events. -> Verify: integration vitest
  using the AG-UI client/server adapter.
- [ ] **DC-002 (Stage state sync):** a Facet patch is encoded as AG-UI
  `STATE_DELTA` and applied on the client with the same Facet fold/validation
  semantics as the current native path. -> Verify: unit test comparing final
  client tree to server saved tree.
- [ ] **DC-003 (Rehydrate/snapshot):** a reconnect or fresh client can receive a
  full Facet stage through AG-UI `STATE_SNAPSHOT` and chat/message history
  through AG-UI-compatible message events without duplicating chat. -> Verify:
  reconnect/rehydrate vitest.
- [ ] **DC-004 (Message streaming):** Facet chat output maps to AG-UI
  `TEXT_MESSAGE_*` events while preserving visible ordering relative to stage
  updates. -> Verify: stream-order unit/integration test.
- [ ] **DC-005 (Boundary/error):** malformed or hostile AG-UI state/custom/raw
  events cannot inject raw markup, cannot bypass `validateTree`, and cannot
  crash the renderer. -> Verify: malformed event matrix test.
- [ ] **DC-006 (Agent side compatibility):** an in-process Facet agent can be
  driven through the server-mediated AG-UI adapter without duplicating stage
  safety logic. External NAT-safe AG-UI dial-out is a follow-up adapter, not this
  implementation slice. -> Verify: adapter contract tests and explicit waiver
  docs for external dial-out.
- [ ] **DC-007 (No direct browser-to-agent requirement):** the supported AG-UI
  flow works with the Facet server as the AG-UI endpoint/proxy; browser clients
  are not required to hold credentials for or connect directly to an agent. ->
  Verify: test and docs example with server-mediated AG-UI.
- [ ] **DC-008 (Package boundary):** `@facet/core` remains dependency-free and
  AG-UI dependencies are isolated to adapter/edge packages. -> Verify:
  package.json/import boundary checks.
- [ ] **DC-009 (Official SDK use):** Facet uses official `@ag-ui/*` packages for
  AG-UI types/schemas/encoding instead of hand-rolled equivalent event schemas.
  -> Verify: dependency/import tests and code review.
- [ ] **DC-010 (Native migration posture):** because there are no external users,
  the new AG-UI public path may replace or demote the native reference wire path,
  but all Facet demos/tests must pass through the chosen AG-UI path or an
  explicitly documented internal-only fallback. -> Verify: updated package tests
  and docs grep.

## Invariant Fit (Facet-specific - REQUIRED)

| # | Invariant | Status | Note / mitigation |
|---|---|---|---|
| 1 | Responsibility boundary (UI-out/UI-in only; no backend/domain) | OK | AG-UI carries UI-IN/UI-OUT events only. Domain/backend work still belongs to the agent's own tools. |
| 2 | Mechanism vs policy (agent authors; Facet provides capability) | TOUCHES | Facet standardizes the event envelope, not the UI behavior. The agent still authors stage structure/actions; AG-UI only provides transport/lifecycle vocabulary. |
| 3 | Fail-safe (never throw/inject; degrade to plain) | TOUCHES | AG-UI events are untrusted at the Facet boundary. Only recognized Facet stage payloads reach `foldPatchIntoStage`/`validateTree`; malformed events degrade to no-op/error events. |
| 4 | Declarative + tokens only (no raw HTML/JS/CSS/pixels) | OK | AG-UI does not change the stage vocabulary. The rendered UI remains `FacetTree` bricks with token styles. |
| 5 | Flow-only safety (overlap only via a dedicated overlay brick) | OK | No new visual/layout capability is introduced. |
| 6 | Two-writers coherence (local execution vs agent stage) | TOUCHES | Client-side AG-UI adapters may see state deltas, local view actions, and replay/snapshot events. Mitigation: server remains the authoritative stage writer; client applies only ordered server-emitted Facet state payloads through the shared fold. Local navigate/toggle remain view-state and must preserve existing record/order semantics. |
| 7 | Backend calls go through the agent, never client fetch | OK | AG-UI tool events may describe agent tool work, but Facet does not execute domain fetches from the page. |

## Public API / Package Surface

| `@facet/*` package | Change | Additive or Breaking |
|---|---|---|
| `@facet/core` | Keep `FacetTree`, patch, validator, and internal protocol primitives; no `@ag-ui/*` dependency | Internal/public surface may change, but dependency boundary must remain |
| `@facet/client` | Add or replace public browser transport with AG-UI-facing transport/adapter | Breaking allowed pre-user; exact removal vs demotion decided in spec |
| `@facet/server` | Add AG-UI endpoint/stream adapter that maps AG-UI input/events to Facet runtime and back | Breaking allowed pre-user; exact native route posture decided in spec |
| `@facet/react` | `useFacet` may consume an AG-UI-backed transport while retaining safe stage folding | Additive or breaking depending on transport API |
| `@facet/runtime` | Continue to own stage/session application; may expose adapter seams for AG-UI event conversion | Mostly additive |
| `@facet/agent-client` | Native dial-in may be replaced, wrapped, or demoted in favor of AG-UI-compatible agent adapter | Breaking allowed pre-user |
| `@facet/quickstart` | Should exercise the AG-UI public path so first-run behavior proves the new protocol | Breaking/internal migration |
| New `@facet/ag-ui` (candidate) | Adapter package containing AG-UI mapping and official SDK dependencies | Additive package, candidate surface |

## Open Questions / Assumptions

- **Assumption:** AG-UI remains pre-1.0 but active enough to justify adoption;
  version churn is mitigated by isolating AG-UI imports to adapter/edge packages.
  Owner: main-agent. Checkpoint: spec-bridge context.
- **Assumption:** Facet can encode stage sync as AG-UI `STATE_DELTA` /
  `STATE_SNAPSHOT` under a reserved Facet state namespace. Owner: main-agent.
  Checkpoint: spec-bridge risk probe.
- **Open Question:** should native `SseTransport`/`/event`/`/stream` remain as an
  internal fallback for tests, or should quickstart/playground move entirely to
  AG-UI immediately? Owner: owner. Checkpoint: spec-bridge.
- **Decision / Waiver:** the external agent path preserves current native
  dial-out ownership for this implementation slice. A Facet-specific
  `@facet/ag-ui/agent` adapter can be designed later after the client/server
  AG-UI edge is stable. Owner: owner + main-agent. Checkpoint: spec-bridge.
