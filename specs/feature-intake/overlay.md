# Feature Intake Brief: Overlay (modal / drawer) — the flow-only exception, done safely

> Generated by `/feature-intake`. Plain-language, testable, invariant-checked.
> PR-3 of the node-model restructure. A **genuine new behavioral capability**:
> the one sanctioned way a `box` floats ABOVE flow content.

## Goal

Give an agent a **safe, bounded way to float a `box` above the page** as a modal
or a drawer — the single sanctioned exception to flow-only layout — without ever
handing the author coordinates, size, z-index, or an absolute-positioning escape
hatch. The renderer owns all placement, the scrim, stacking, focus, and close.

## Current Problem

- **Who is affected:** the agent author (needs dialogs/confirmations/side panels
  — a universal UI need) and the end-user of an agent-built page (expects a modal
  to actually overlay, trap focus, and close on Esc/scrim).
- **Concrete pain:** Facet is deliberately flow-only — boxes stack and never
  overlap — so today an agent literally **cannot** draw a modal confirm dialog or
  a slide-in drawer. Faking it inline (a big highlighted box pushed into the
  flow) shoves the rest of the page down and doesn't behave like an overlay
  (no scrim, no focus trap, no "on top"). The invariant reserves overlap for "a
  constrained, dedicated brick/descriptor" — which does not exist yet. `overlay`
  is that descriptor.
- This is the **last** flow-only escape valve in the locked node-model design;
  after it, the vocabulary's overlap story is complete.

## User Scenario

### Who
The agent author declares which box is an overlay; the end-user opens/closes it.

### Trigger
The agent wants a dialog/panel; it tags a box and wires a toggle to open it.

### Steps
1. Agent draws a `box` normally, tags it `overlay: { kind: "modal" }`, and starts
   it hidden (`hidden: true`).
2. Agent places a trigger elsewhere: `button` with
   `onPress: { kind: "toggle", target: <overlay-box id> }`.
3. End-user taps the trigger → the **existing local toggle** flips that box's
   `view.toggled` to visible — **no agent turn**.
4. The renderer sees "this box is visible AND has `overlay`", lifts it out of
   flow, and paints it: **modal = screen-center + full-viewport scrim**;
   **drawer = right (end) edge + scrim**; above all content, in a
   renderer-fixed positive-z band.
5. End-user closes via scrim-click, `Esc`, or a close `button` — **all dispatch
   the SAME toggle** (target = the overlay box) → box hidden → overlay gone.
6. Because close routes through the toggle, `view.toggled` flips back too, so the
   agent's view snapshot stays coherent (it never thinks a closed overlay is
   open).

## Input/Output Examples

### Example 1 (Normal — a modal)
- Input: `box{ id:"dlg", overlay:{kind:"modal"}, hidden:true, children:[text "Delete?", button "Confirm", button "Cancel"(onPress: toggle "dlg")] }` + a trigger `button "Delete"(onPress: toggle "dlg")`.
- Output: tapping "Delete" floats `dlg` centered over a scrim, above the page;
  `Esc`/scrim/"Cancel" all close it. No agent round-trip.

### Example 2 (Normal — a drawer)
- Input: `box{ overlay:{kind:"drawer"}, hidden:true, … }`, opened by a toggle.
- Output: the box slides in from the right (end) edge over a scrim; same close
  affordances.

### Example 3 (Edge — unknown/malformed overlay → fail-safe)
- Input: `box{ overlay:{kind:"lightbox"} }` (unknown kind) or `overlay:{}` (no
  kind) or `overlay:"modal"` (wrong type).
- Output: the `overlay` descriptor is dropped in validation; the box renders as a
  **normal inline flow box** (no float, no scrim). Never throws, never a broken
  floating element.

### Example 4 (Edge — visible overlay is the only "open" signal)
- Input: an overlay box authored `hidden:false` (starts open) with no scrim
  interaction wired.
- Output: it floats immediately; `Esc`/scrim still close it via the renderer
  dispatching the box's toggle. If nothing can ever hide it, that is the agent's
  authoring choice (mechanism-vs-policy), not a framework failure.

## Constraints (What Not to Do)

- **No author coordinates / size / z-index / offset / absolute positioning.** The
  author supplies ONLY the `kind` name. Any pixel/z/position leak is a hard fail
  (invariant #4/#5).
- **No general overlap.** Overlap is allowed ONLY through this `overlay`
  descriptor with renderer-fixed placement — not a reusable z-index token, not a
  `position` field.
- **No new open/close state channel.** Reuse the existing local `toggle` +
  `view.toggled`. Do NOT invent an overlay-specific "isOpen".
- **No backend / no fetch** (untouched).
- **Renderer must never throw** on a malformed/unknown overlay — degrade to an
  inline flow box.
- **popover + `anchor` are OUT of v1** (deferred). The `kind` union stays closed
  but extensible so `"popover"` + an optional `anchor?` can be added additively.
- **drawer edge is fixed (right/end) in v1** — no author `edge` field yet
  (deferrable additive token).
- **No focus/scroll-lock library dependency** beyond what the renderer already
  ships; keep it declarative + fail-safe.

## Policy & Edge Cases

| Scenario Step | Failure Policy | Boundary/Validation | Concurrency / Duplicate Trigger |
|---|---|---|---|
| Author tags `overlay` | Unknown/malformed `kind` → drop the descriptor, render inline box (never throw) | `kind ∈ {"modal","drawer"}` (closed); anything else dropped | N/A (validation) |
| Open via toggle | If the target id is missing/dangling → existing toggle fail-safe (no-op) | Only a box with a valid `overlay` floats when visible | Rapid toggles just flip visibility; last state wins (existing toggle semantics) |
| Render floating | Renderer owns center/edge placement, scrim, positive-z band; no author input | Placement fixed per kind; drawer = end edge | Multiple visible overlays → **stack in tree order** (later = on top) within the fixed band |
| Close (Esc/scrim/button) | All dispatch the box's **toggle** (not a renderer-internal hide) so `view.toggled` stays coherent | Esc + scrim-click + close button all map to the same toggle target | Double-close is idempotent (toggle to hidden twice = hidden) |
| overlay + backdrop same box | Independent Layered concerns; both apply (backdrop paints inside the floated box) | Allowed; no mutual-exclusion rule | N/A |
| SSR / no-JS | Overlay content still validates + exists in the tree; floating/scrim/focus are browser-only | N/A | N/A |

## Done Criteria (Testable)

- [ ] **DC-001** (Happy — modal floats + closes): A visible `overlay:{kind:"modal"}`
  box renders centered above content with a scrim; `Esc`, scrim-click, and a
  close button each dispatch the box's toggle and hide it. → Verify: jsdom
  interaction test (renders scrim + centered node; simulated Esc/scrim/click call
  the toggle → `onAction`/view-state) **and** the mandatory live-journey
  real-browser tier for actual stacking/paint.
- [ ] **DC-002** (Happy — drawer edge): `overlay:{kind:"drawer"}` renders from the
  right (end) edge with a scrim. → Verify: jsdom test asserts the drawer
  placement class/style is the renderer's end-edge preset (not an author value) +
  live-journey.
- [ ] **DC-003** (Fail-safe — unknown/malformed): `overlay:{kind:"lightbox"}`,
  `overlay:{}`, `overlay:"modal"` → descriptor dropped, box renders inline, no
  throw. → Verify: `validateTree` unit test (descriptor sanitized/removed) +
  renderer never-throws test.
- [ ] **DC-004** (No author positioning leak — invariant #4/#5): the sanitized
  `overlay` carries ONLY a `kind` from the closed set; no coordinate/size/z field
  is accepted; the emitted style has no author-controlled z-index/inset/position.
  → Verify: unit test feeding junk extra keys (`{kind:"modal", z:999, top:10}`) →
  only `kind` survives; renderer-output test asserts no author z/position.
- [ ] **DC-005** (Two-writers coherence): closing via Esc/scrim flips the same
  `view.toggled` entry as the trigger toggle (open state single-sourced; the
  agent's snapshot cannot read "open" after a local close). → Verify: jsdom test
  asserting close dispatches the toggle for the box id (same key the trigger
  uses).
- [ ] **DC-006** (Stacking — multiple overlays): two visible overlay boxes render
  both, stacked in tree order within one renderer-fixed positive-z band; neither
  uses an author z. → Verify: jsdom presence/order test + live-journey visual.
- [ ] **DC-007** (STAGE_SPEC taught): the LLM-facing `STAGE_SPEC` documents
  `overlay` on box (kind set, renderer-owned placement, toggle-to-open/close, no
  coords). → Verify: `spec.test.ts` / grep.

## Priority
High — completes the flow-only overlap story and unblocks dialogs/drawers, a
universal UI need. It is the riskiest vocabulary addition (first paint above
flow), so the live-browser gate is non-negotiable.
