# Feature Intake Brief: Interaction Phase 2 — `appear` animation tokens + `onHold` + scroll regions

> Generated by `/feature-intake`. Plain-language, testable, invariant-checked.
> (Bundle D, descoped 2026-07-04: **overlay and drag are deferred by the owner** —
> this brief covers animation tokens and hold/scroll only.)

## Goal

- Grow the agent-facing interaction vocabulary by three small, token-shaped
  words — with **zero new protocol messages** and the existing two-writers split
  untouched:
  1. **`appear` animation token** — a node can declare how it enters
     (e.g. `fade` / `slide`); the theme maps the token to CSS, the agent never
     touches raw animation values.
  2. **`onHold`** — a box can react to a long-press, reusing the *same* action
     union as `onPress` (`agent` / `navigate` / `toggle`), so "press = primary,
     hold = secondary" becomes expressible.
  3. **Scroll region** — a box can declare itself a bounded, internally
     scrollable area (long lists stop stretching the page), via a closed token,
     never raw CSS.

## Current Problem

- Every stage change **pops** into place: nodes appear with no transition, so
  agent-driven live edits read as glitches rather than intentional updates. The
  agent has no vocabulary to say "ease this in".
- A box has exactly one gesture (`onPress`). Secondary interactions (peek a
  detail, reveal admin/extra options) must burn the single press slot or clutter
  the layout with extra buttons.
- A long list (chat history, product grid, feed) grows the page indefinitely —
  the flow-only guarantee keeps content on-page but gives the agent no way to
  cap a region and scroll inside it. Layouts with a long middle section push
  everything below off the first viewport.
- Affected: **the agent (LLM)** whose expressiveness is capped, and **the
  end-user** who sees janky updates and endless pages.

## User Scenario

### Who
- **The agent (LLM)** authoring a stage through Facet (primary consumer), and
- **the end-user (visitor)** interacting with the page it drew.

### Trigger
- The agent draws a storefront: a hero, a long product list, and product cards
  with a primary action and a secondary "quick peek".

### Steps
1. Agent renders the hero box with `appear: "fade"` and each product card with
   `appear: "slide"` → new nodes ease in instead of popping; the theme decides
   duration/curve.
2. Agent marks the product-list box as a scroll region → the list scrolls
   inside a bounded area; the footer stays reachable without scrolling past 100
   products.
3. Visitor taps a card → `onPress` fires exactly as today (e.g. `navigate` to
   the product screen).
4. Visitor **long-presses** the same card → `onHold` fires its declared action
   (e.g. `toggle` a pre-drawn in-flow detail panel — instant, browser-local; or
   a `kind:"agent"` action when new reasoning is needed).
5. Agent patches new cards into the scrolling list mid-conversation → they
   animate in with their `appear` token; scroll position and view-state are
   untouched (browser-owned).
6. A visitor with `prefers-reduced-motion` set sees instant (non-animated)
   updates — the token is advisory; the theme's CSS honors the OS setting.

## Input/Output Examples

### Example 1 (Normal — appear + scroll)
- Input: `{ id: "list", type: "box", style: { scroll: true, ... }, children: [...] }`
  with children carrying `appear: "slide"`; agent later `add`s a new child.
- Output: the list renders as a bounded scrollable region; the new child
  animates in; the page body itself does not grow or gain a scrollbar because
  of the list.

### Example 2 (Normal — hold)
- Input: a box with `onPress: { kind: "navigate", to: "detail" }` and
  `onHold: { kind: "toggle", target: "peek_panel" }`; visitor quick-taps, then
  separately long-presses.
- Output: quick tap → navigate only; long press → toggle only (the two never
  both fire for one gesture). `agent`-kind holds carry `payload`/`collect`
  fields exactly like presses.

### Example 3 (Edge/Error)
- Input: `appear: "explode"` (unknown token), `onHold: 42` (malformed),
  `scroll: "sideways"` (unknown), a hold released at 300ms (below threshold),
  a hold on a box whose `onHold` was removed by a patch mid-press.
- Output: unknown/malformed values are **stripped by `validateTree`** — the
  node renders plain (no animation, non-holdable, normal flow box); a
  below-threshold hold falls through to a normal press; a mid-press removal
  no-ops. The renderer never throws.

## Constraints (What Not to Do)

- **No raw values anywhere**: no durations, easings, pixel heights, or CSS in
  the tree. `appear` and `scroll` are closed token sets; concrete CSS lives
  only in the renderer theme.
- **No overlay** (badges/modals/toasts) and **no drag/sort** — descoped from
  this bundle by the owner (2026-07-04); flow-only stands unchanged.
- **No scroll telemetry**: scroll position is never sent to the agent (chatty,
  surveillance-smelling). Scroll state is browser view-state, full stop.
- **No animation orchestration DSL**: no sequences, delays, per-node timing,
  or exit animations in v1 — one enter-token per node, that's the whole word.
- **No new gesture kinds beyond hold**: no swipe, no double-tap (future tokens
  if ever needed).
- **No new protocol messages / transport changes**: everything rides existing
  patches and existing action events.
- Trees that don't use the new fields must render byte-identically to today.

## Policy & Edge Cases

| Scenario Step | Failure Policy | Boundary/Validation | Concurrency / Duplicate Trigger |
|---|---|---|---|
| `appear` on a new node | unknown token → stripped, renders instantly | closed set (e.g. `none/fade/slide`) enforced by validateTree | node re-added rapidly → animation replays per mount; agent patch during animation → CSS animation is purely visual, applyPatch unaffected |
| `appear` under reduced motion | animation suppressed, content still appears | theme CSS gated on `prefers-reduced-motion` | n/a |
| long-press a box | below threshold → treated as press; `onHold` absent → plain press behavior | `onHold` normalized by the same action-union validation as `onPress` (incl. legacy `{name}` = agent kind) | press-vs-hold mutually exclusive per gesture; pointer move beyond slop cancels the hold; repeated rapid holds → each is one action, same as rapid presses today |
| hold with `kind:"agent"` | offline agent → same failure face as pressed agent actions today | `payload`/`collect` re-validated server-side at the boundary, exactly as for press | serialized per-visitor queue unchanged |
| scroll region rendering | unknown/malformed `scroll` value → stripped, box flows normally | closed token; bounded by theme-owned max-height; horizontal page overflow must never appear | agent patches children while user is mid-scroll → content updates in place; scroll offset is browser-owned view-state and is not reset by unrelated patches |
| mixed: hold-toggle inside a scroll region during an agent redraw | same as phase-1 rules: view-state (visibility, scroll offset) survives content patches; orphaned targets no-op | n/a | content written ONLY by server; view-state written ONLY by browser — unchanged split |

## Done Criteria (Testable)

- [ ] DC-001 (Happy — appear): a node with a valid `appear` token renders with
  the theme's animation binding; a tree without `appear` renders exactly as
  today → Verify: jsdom render test asserting the animation class/style hook +
  snapshot-equality for the no-token case.
- [ ] DC-002 (Happy — hold): long-press fires the declared `onHold` action —
  `toggle`/`navigate` run browser-local with zero transport events; `agent`
  kind emits one ClientEvent with `payload`/`collect` fields intact → Verify:
  jsdom pointer-event test with fake timers + transport `send` mock.
- [ ] DC-003 (Happy — scroll): a box with the scroll token renders as a bounded
  scrollable region (overflow contained, page body unaffected) → Verify: jsdom
  style assertion + real-browser check (DC-009).
- [ ] DC-004 (Gesture split): on one gesture, quick tap fires `onPress` only
  and long press fires `onHold` only — never both → Verify: jsdom fake-timer
  test covering tap, hold, and below-threshold release.
- [ ] DC-005 (Boundary/fail-safe): unknown `appear`/`scroll` tokens and
  malformed `onHold` shapes are stripped by `validateTree`; StageRenderer never
  throws on them → Verify: unit tests on validateTree + renderer fail-safe
  suite.
- [ ] DC-006 (Two-writers preserved): hold-triggered `toggle`/`navigate` write
  view-state only; scroll offset survives an unrelated content patch; the
  server stage is byte-identical after any local-only interaction → Verify:
  jsdom + existing stage-fold tests stay green.
- [ ] DC-007 (Back-compat): the full existing test suite passes without
  modification; a pre-D tree renders identically (no new fields = no behavior
  change) → Verify: `pnpm test` green + DC-001's snapshot-equality case.
- [ ] DC-008 (Reduced motion): with `prefers-reduced-motion`, appear animations
  are suppressed but content still renders → Verify: theme CSS unit test (media
  query present and scoped to the appear bindings).
- [ ] DC-009 (Vocabulary + live): STAGE_SPEC teaches `appear`/`onHold`/scroll;
  `/live-test` passes; a real-browser session (Bundle B lesson — mandatory for
  interaction work) demonstrates animate-in, hold, and inner scroll → Verify:
  STAGE_SPEC content test + `/live-test` tiers + manual browser check recorded
  in the PR.

| # | Invariant | Status | Note / mitigation |

| 2 | Mechanism vs policy (agent authors; Facet provides capability) | OK | Facet ships the capability; the agent decides what animates, what hold means per box, and what scrolls. | 3 | Fail-safe (never throw/inject; degrade to plain) | TOUCHES | All three enter through `validateTree` normalization: unknown tokens stripped, malformed `onHold` dropped (box stays pressable-or-plain), renderer no-ops on the rest. Pinned by DC-005. |
| 4 | Declarative + tokens only (no raw HTML/JS/CSS/pixels) | TOUCHES | Animation is exactly where raw-value pressure appears (durations/easings). Held: `appear` is a closed token set; duration/curve/max-height live only in the renderer theme. No numeric or CSS values are added to any node type. Pinned by DC-001/003/005. |
| 5 | Flow-only safety (overlap only via a dedicated overlay brick) | OK | Nothing floats. `appear` is a transition of in-flow content; scroll regions clip *inside* flow bounds. (A transient transform during a slide-in never changes layout position.) Overlay itself is explicitly descoped. |
| 6 | Two-writers coherence (local execution vs agent stage) | TOUCHES | No new writer appears. `onHold` reuses the phase-1 action split (local kinds → browser view-state; agent kind → event). | 7 | Backend calls go through the agent, never client fetch | OK | No fetch surface; `agent`-kind holds route through the existing event path. |
