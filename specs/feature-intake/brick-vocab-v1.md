## Goal

Let an agent draw the common web/app UI it currently **cannot** â€” video, real form
controls, horizontal carousels, precise grids â€” **without growing the brick count**.
Do it the Facet-native way: a unified **media** brick (typed by `kind`), an expanded
**`field.input`** set, and two **layout tokens** â€” mirroring how `field` already
discriminates input types, so the palette stays minimal while coverage jumps.

## Current Problem

Facet's 4 bricks + current tokens under-cover real UI (confirmed by deep research â€”
these gaps are exactly what every gen-UI competitor treats as first-class):
- **No video.** `image` is static (`src`+`alt`) only. Motion/media is impossible.
- **Form controls are text-only.** `FIELD_INPUTS = [text, number, email, password,
  search]` â€” no checkbox, radio, select, switch. A signup/survey/settings form
  can't be expressed.
- **No horizontal carousel.** `scroll` is vertical-only.
- **No precise grid.** `width` is `auto`/`full` only â€” no N-column grid.

**Affected:** the agent author / the LLM brain (can't compose these) and the
end-user of the built page (no video, forms, carousels, grids).

## User Scenario

### Who
The agent author (framework consumer) â€” in practice the LLM brain drawing a page.

### Trigger
The agent wants a page with a video, a signup form (radio/select/checkbox), a
horizontal image carousel, or a 3-column feature grid.

### Steps
1. Agent emits new bricks/tokens: a `media`(kind:"video") node, `field`(input:"select",
   options:[â€¦]) nodes, a `box`(columns:3), a `box`(scroll:"x", direction:"row").
2. The renderer renders them safely (native video element, native form controls,
   CSS grid, horizontal scroll region).
3. The end-user interacts â€” plays the video, picks a radio/select option, scrolls
   the carousel.
4. On a `collect` press, the current field VALUES (incl. select/checkbox state) are
   snapshotted into the event's `fields` (existing mechanism).

## Constraints (What Not to Do)

- **No `audio`** â€” deferred (kind is image|video only this round).
- **No `range`(slider) / `date`** â€” deferred to a later form-control set.
- **No stamp library** â€” that's theme 2 (couples with stamp reference-expand).
- **No overlay/floating brick.** Select/date popups use **native browser controls**
  only â€” never a custom z-index/overlay escape hatch (invariant #5).
- **No raw CSS/HTML.** Every new capability is a typed prop or a token (video
  controls = booleans, columns/scroll = tokens).
- **No backend/data-binding.** `media.src` is a static URL exactly like `image.src`
  today â€” no fetch, no arbitrary-URL data binding.
- **No new bricks beyond `media`** (which replaces `image`). Net brick count stays 4.
