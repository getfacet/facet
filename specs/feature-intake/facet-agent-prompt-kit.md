# Feature Intake Brief: Facet Agent Prompt Kit

> Generated by `/feature-intake`. Plain-language, testable, invariant-checked.

## Goal

- Provide a reusable, provider-neutral Facet prompt kit so agent authors can give
  LLMs a consistent, compact guide for building useful Facet pages.
- Refactor `@facet/reference-agent` to consume that kit while keeping the
  reference agent's public prompt entrypoints compatible.

## Current Problem

- The generic Facet guidance currently lives inside
  `@facet/reference-agent/src/prompt/system.ts`, even though external agent
  authors building their own loops need the same rules.
- The current system prompt is also too narrow: it tells the model which tools
  exist, but it does not give enough reusable UX guidance about compact screens,
  editing existing page state before appending, avoiding duplicates, and
  recovering from non-visible or rejected tool results.
- Affected users are agent authors using Facet and end-users of agent-built
  pages who experience overly long, repetitive, append-only pages.

## User Scenario

### Who

- Agent author: a developer building an OpenAI/Anthropic/custom provider loop
  with `@facet/agent-tools`.
- Reference-agent user: a developer running `@facet/reference-agent` or
  `facet-quickstart`.
- End-user: the visitor interacting with a live Facet page.

### Trigger

- An agent author wants a sane default Facet prompt without copying prompt text
  from the reference agent.
- A reference-agent turn asks the model to update an existing page.

### Steps

1. The agent author imports a prompt-kit helper or section constants from
   `@facet/agent-tools`.
2. The helper builds a system prompt containing the Facet stage contract,
   page-experience guidance, state-editing strategy, tool playbook, asset/privacy
   rules, and the deployer's page brief.
3. `@facet/reference-agent` calls the same helper from its existing
   `buildSystem(guide, assets?)` API.
4. During a turn, the LLM sees guidance to inspect and edit existing state before
   appending, to keep pages compact, to use `render_page` only for first paint or
   major restructure, and to treat non-visible/rejected/pending tool outcomes as
   repair work instead of completion.

## Input/Output Examples

### Example 1 (Normal)

- Input: `buildSystem("# Onboarding assistant", { themes, stamps })` from
  `@facet/reference-agent`.
- Output: a system prompt containing `STAGE_SPEC`, reusable page-experience and
  tool-playbook sections, theme/stamp names and descriptions only, and a final
  `PAGE BRIEF` section containing the supplied guide.

### Example 2 (External Author)

- Input: an external agent imports `buildFacetAgentSystemPrompt` from
  `@facet/agent-tools` and passes a custom page brief.
- Output: the external loop receives the same Facet-safe guidance without
  importing provider adapters, quickstart code, or the reference-agent harness.

### Example 3 (Edge/Error)

- Input: no themes or stamps, or empty asset arrays.
- Output: the prompt omits injected asset sections, does not throw, and does not
  include CSS values, full stamp JSON, API keys, visitor ids, or provider keys.

## Constraints (What Not to Do)

- Do not change Facet stage vocabulary, node types, tokens, validation, renderer
  behavior, runtime behavior, or transport protocol.
- Do not make `@facet/agent-tools` depend on `@facet/reference-agent`,
  provider SDKs, Node-only APIs, or quickstart.
- Do not make prompt guidance an enforcement layer in core/runtime; it is
  authoring guidance for the LLM.
- Do not expose raw theme CSS values, full stamp JSON, visitor ids, provider
  keys, or collected secret-looking values to the prompt.
- Do not remove or rename `@facet/reference-agent` public exports such as
  `DEFAULT_GUIDE`, `PromptAssets`, `buildSystem`, or `TOOLS`.

## Policy & Edge Cases

| Scenario Step | Failure Policy | Boundary/Validation | Concurrency / Duplicate Trigger |
|---|---|---|---|
| Agent author imports prompt kit | Missing optional assets produce no asset sections; no throw for empty arrays. | Package remains provider-neutral and exports through the barrel. | N/A; prompt building is pure string assembly. |
| Reference agent builds system prompt | Keep existing `buildSystem(guide, assets?)` API compatible. | `STAGE_SPEC` remains the core source for tree vocabulary. | System prompt is built once per agent creation, same as today. |
| Assets are injected | Include names/descriptions/slot names only. | Theme values and stamp node JSON are not serialized into the prompt. | Assets are an immutable per-agent snapshot at boot. |
| LLM receives tool result | Guidance says only `applied_visible` proves visible completion. | `pending`, `rejected`, `applied_not_visible`, and warnings require inspect/repair. | Multiple tool calls in one turn still follow the existing harness transcript order. |
| Existing page needs a small change | Prompt recommends `inspect_stage`/`inspect_node`, `set_node`, or `remove_node` before appending. | Guidance only; runtime/tool validation still decides actual safety. | Repeated requests should update or replace prior content rather than duplicate it. |

## Done Criteria (Testable)

- [ ] DC-001 (Prompt kit API): `@facet/agent-tools` exports a provider-neutral
  prompt-kit builder and/or section constants for Facet role, stage contract,
  page experience, state editing, tool playbook, asset/privacy rules, and page
  brief composition. -> Verify: `@facet/agent-tools` unit and barrel tests.
- [ ] DC-002 (Reference compatibility): `@facet/reference-agent buildSystem`
  consumes the prompt kit while preserving its existing signature and public
  exports. -> Verify: reference-agent prompt/index tests.
- [ ] DC-003 (UX guidance): the composed prompt explicitly tells the model to
  keep pages compact, avoid duplicate append-only growth, inspect/edit existing
  state before appending, use `render_page` only for first paint or major
  restructure, and keep chat short. -> Verify: prompt content tests.
- [ ] DC-004 (Tool-result recovery): the composed prompt explains
  `applied_visible`, `applied_not_visible`, `applied_with_warnings`, `pending`,
  `rejected`, and `no_stage_change`, and says not to claim visible completion
  without a visible applied change when a page change was requested. -> Verify:
  prompt content tests.
- [ ] DC-005 (Asset/privacy boundary): theme/stamp prompt sections include only
  names, descriptions, and stamp slot names; they do not include theme CSS
  values, full stamp JSON, visitor ids, API keys, or provider keys. -> Verify:
  prompt content and negative leakage tests.
- [ ] DC-006 (Docs): package docs explain that `@facet/agent-tools` is the
  reusable LLM tool/prompt mechanism, while `@facet/reference-agent` remains the
  runnable reference brain. -> Verify: docs diff and `/update-docs`.
- [ ] DC-007 (Boundary non-regression): core stage validation, reference-agent
  provider failure handling, and runtime same-visitor ordering remain unchanged
  because this feature only changes prompt composition. -> Verify: existing core
  validate, reference-agent harness, and runtime concurrency tests.

## Invariant Fit (Facet-specific - REQUIRED)

Mark each: `OK` / `TOUCHES (mitigation)` / `CONFLICT (must re-scope)`.

| # | Invariant | Status | Note / mitigation |
|---|---|---|---|
| 1 | Responsibility boundary (UI-out/UI-in only; no backend/domain) | OK | The kit only assembles prompt text for existing stage tools; it adds no backend/domain tools. |
| 2 | Mechanism vs policy (agent authors; Facet provides capability) | TOUCHES (mitigation) | The kit provides default guidance, not runtime policy. Agent authors can supply/replace their page brief and still own behavior. |
| 3 | Fail-safe (never throw/inject; degrade to plain) | TOUCHES (mitigation) | Prompt assembly is pure and bounded to safe summaries; runtime/tool validation remains the safety boundary. Tests cover empty assets and leakage. |
| 4 | Declarative + tokens only (no raw HTML/JS/CSS/pixels) | TOUCHES (mitigation) | The prompt reiterates token-only stage authoring and never serializes theme CSS values as model-authored content. |
| 5 | Flow-only safety (overlap only via a dedicated overlay brick) | OK | No new layout capability or overlay behavior is introduced. |
| 6 | Two-writers coherence (local execution vs agent stage) | OK | No new local browser actions or stage writers are introduced. |
| 7 | Backend calls go through the agent, never client fetch | OK | No client fetch, backend connector, or domain operation is added. |
