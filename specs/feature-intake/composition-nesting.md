# Feature Intake Brief: Composition Nesting (a composition can reference another composition)

> Generated by `/feature-intake`. Plain-language, testable, invariant-checked.
> PR-5.0 of the node-model restructure — the PREREQUISITE that unblocks the
> component→composition demotions (PR-5.1+). New capability, additive.

## Goal

Let a catalog composition **reference another composition** (not just embed
primitive/native-brick nodes), so that when the component tier is demoted to
compositions (PR-5.1+), a composition that used to embed a component (e.g. a card
with a badge) can **point at** the demoted thing's composition instead of
inlining its structure — keeping the catalog DRY. Both stay available: a
composition may inline primitives AND/OR reference other compositions. Safety is
enforced by **extending the existing load-time `validateComposition`** to check
the reference graph (no cycles, all references resolve, bounded depth) — so an
invalid nesting is refused when the catalog loads, never reaching a visitor.

## Current Problem

- **Who is affected:** the operator/catalog author (and the maintainer executing
  PR-5.1+). End-users see no change.
- **Concrete pain:** the catalog is currently 3-tier — `composition → component →
  primitive`. Existing compositions (`hero` = section+button; card-family embeds
  `badge` at `compositions.ts:173,253`) are built out of COMPONENT nodes, not
  primitives. When PR-5.1+ demotes those components to compositions, every
  composition that embedded a component must be rewritten. Without nesting, the
  only option is to **inline** each demoted structure (duplication that grows as
  ~9 components collapse and embed each other). With nesting, a composition just
  references the demoted composition — reuse is preserved. Composition references
  don't exist yet; a composition's `nodes` can only hold node objects.

## User Scenario

### Who
The operator/catalog author defining reusable compositions (and the maintainer
demoting components in PR-5.1+). The agent still just calls `use_composition` by
name — nesting is invisible to it.

### Trigger
A composition needs to include another composition's structure (e.g. a card
composition that contains a badge composition).

### Steps
1. The operator authors composition `card` whose `nodes` include a **reference
   node** to composition `badge` (with slot values threaded from the card).
2. At catalog **load**, `validateComposition` (extended) checks the reference
   graph: `badge` exists, no cycle (`card→badge→…→card`), depth ≤ limit, total
   expanded size within the node cap. An invalid graph → the catalog is **refused
   at load** (fail-safe), exactly as an invalid single composition is today.
3. At use time, the agent calls `use_composition("card", at:{parent})`.
   `expandComposition` recursively expands: the card's nodes, and where a
   reference node appears, the referenced `badge` composition is expanded in place
   with its slots filled — down to **primitive/native-brick nodes only**. Only the
   canonical primitive patch travels to the visitor (PR #30 canonicalization).
4. The visitor sees the fully-expanded primitive tree; the nesting is a
   server-side authoring convenience, never shipped as a "composition" node.

## Input/Output Examples

### Example 1 (Normal — a composition references another)
- Input: composition `card` has a node `{ use: "badge", slots: { label: "{{status}}" } }`; the catalog also defines `badge`.
- Output: `use_composition("card", …)` expands to card primitives with the badge's box+text expanded inline (label filled). No `badge`/`card` node reaches the visitor — only primitives.

### Example 2 (Edge — cycle is refused at load)
- Input: composition `a` references `b`, and `b` references `a`.
- Output: at catalog load, the graph validator detects the cycle and **refuses both** (bounded issue logged); they never become usable. No infinite expansion ever runs.

### Example 3 (Edge — missing reference)
- Input: composition `card` references `badge`, but no `badge` composition is in the catalog.
- Output: refused at load (dangling reference); `card` is not registered. (Or: `card` loads but the reference is dropped/degrades — **owner decides the policy**; default = refuse `card`.)

### Example 4 (Edge — depth / size bound)
- Input: a nesting chain deeper than the limit, or one whose full expansion exceeds `MAX_COMPOSITION_NODES` / the patch-op cap.
- Output: refused at load (depth) / guarded at expand (op cap, as `use_composition` already does). Never throws.

## Constraints (What Not to Do)

- **No DSL / no expressions.** A reference node's slot values are **bounded
  strings** filled into the child's named slots (same as today's slot mechanism) —
  never expressions, conditionals, loops, or computed structure. Nesting is
  static reference + string-slot threading, nothing more (invariant #2).
- **No new agent surface.** The agent's authoring stays `use_composition` by name.
  Nesting is composition DATA (operator/catalog), invisible to the agent. Do NOT
  let the agent author a reference node in the live stage.
- **Expands to primitives only.** The final expanded tree the visitor sees is
  primitive/native-brick nodes; a "composition reference" is never a live node
  type in the stage (it exists only inside composition DEFINITIONS, resolved
  server-side).
- **Bounded.** Reuse the existing `MAX_COMPOSITION_NODES` cap + patch-op cap; add a
  nesting-depth cap. No unbounded recursion.
- **Cycles caught at load, not at expand.** The reference graph is validated when
  the catalog loads (operator-authored, once) — a cyclic/dangling graph is refused
  before any agent can use it. Expand-time keeps a defensive guard but is not the
  primary gate.
- **Fail-safe everywhere.** `validateComposition` and `expandComposition` never
  throw; bad input → refuse/degrade with a bounded issue.

## Policy & Edge Cases

| Scenario Step | Failure Policy | Boundary/Validation | Concurrency / Duplicate |
|---|---|---|---|
| Author reference node | Malformed reference (no name / bad shape) → refuse the composition at load | Reference shape is a closed `{ use: name, slots?: {..strings} }` | N/A (load-time) |
| Load-time graph validation | Cycle / dangling ref / over-depth → refuse the affected composition(s), bounded issue | No cycle; all refs resolve; depth ≤ CAP; total expanded nodes ≤ MAX_COMPOSITION_NODES | Catalog assembled once at load |
| Expand (use_composition) | Op-cap exceeded → the existing `use_composition` op-cap error; defensive skip on any residual bad ref | Only primitives/native bricks in the final tree | Same as current use_composition ordering/buffer rules |
| Slot threading | Missing slot value → child's slot default (as today) | Slot values are bounded strings only | N/A |

## Done Criteria (Testable)

- [ ] **DC-001** (Nesting expands): a composition with a `{ use: <other>, slots }`
  reference expands (via `expandComposition`) to the referenced composition's
  primitives, with slots filled, at the correct place. → Verify: `expandComposition`
  unit test asserting the final tree is primitives with the nested structure inline.
- [ ] **DC-002** (Cycle refused at load): a cyclic reference graph
  (`a→b→a`, and self-`a→a`) is refused by the extended `validateComposition` /
  load path, with a bounded issue; neither composition registers. → Verify:
  validation unit test + the assets/runtime load test refusing a cyclic catalog.
- [ ] **DC-003** (Dangling reference): a reference to a non-existent composition is
  refused at load (per the chosen policy). → Verify: validation unit test.
- [ ] **DC-004** (Depth / size bound): a nesting chain past the depth cap, or whose
  expansion exceeds the node/op cap, is refused at load / guarded at expand, never
  throws. → Verify: unit test at the cap boundary.
- [ ] **DC-005** (No live composition node): the expanded tree that reaches the
  visitor contains only primitive/native-brick node types — no "composition
  reference" node type leaks into the stage. → Verify: expand output + `validateTree`
  never sees a reference node.
- [ ] **DC-006** (Agent surface unchanged): the agent still authors only via
  `use_composition` by name; a reference node is not an authorable stage node. →
  Verify: the executor/validateTree rejects a stage-authored reference node
  (fail-safe), and `use_composition` behavior is unchanged for non-nested comps.
- [ ] **DC-007** (Fail-safe): malformed reference / bad graph never throws;
  `validateComposition` + `expandComposition` degrade to refuse/skip. → Verify:
  hostile-input unit tests.
