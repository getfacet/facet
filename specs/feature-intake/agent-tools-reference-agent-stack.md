# Feature Intake Brief: Agent Tools and Reference Agent Stack

> Generated by `/feature-intake`. Plain-language, testable, invariant-checked.

## Goal

- Make Facet's first-run agent story easy to understand and safe to copy:
  package docs should separate **Core**, **Agent Stack**, and **Optional /
  advanced** packages.
- Migrate the workspace to a physical, expandable package taxonomy so the
  filesystem matches the package story.
- Add a reusable `@facet/agent-tools` package so LLM agent authors can use the
  same Facet stage tools without importing the full reference agent.
- Rebuild `@facet/reference-agent` on top of that reusable tool layer with a
  robust loop for tool results, stage shadow feedback, context limits,
  compaction, budgets, tracing, and provider usage metadata.

## Current Problem

- Facet currently has several agent-facing packages (`@facet/agent`,
  `@facet/agent-client`, `@facet/client`, `@facet/cli`, `@facet/bridge`,
  `@facet/reference-agent`, `@facet/quickstart`) whose roles are easy to mix up.
  This makes the public onboarding path look heavier than it needs to be.
- The current flat `packages/<name>` layout does not visually communicate the
  proposed taxonomy, so first-time contributors have to infer which packages are
  core, which packages are part of the recommended agent stack, and which ones
  are optional extensions.
- LLM-facing stage tool schemas and execution are effectively private to
  `@facet/reference-agent`, so external agent authors cannot reuse Facet's safe
  stage mutation tools directly.
- The current reference agent is too shallow for public copy/paste use: it does
  not have a first-class harness for iterative tool observations, canonical
  within-turn stage shadow updates, context window management, compaction,
  budget enforcement, or traceable stop reasons.
- Affected users:
  - Facet maintainers publishing the quickstart and package docs.
  - Agent authors who want to build their own LLM loop against Facet.
  - First-time users trying Facet locally with their own model API key.

## User Scenario

### Who

- Primary: an agent author using Facet to build an LLM-driven UI agent.
- Secondary: a first-time Facet user running the quickstart locally.
- Internal: Facet maintainers keeping package boundaries and docs coherent.

### Trigger

- Facet is preparing a public reference agent and quickstart that people can run
  locally and later adapt for their own agent stack.

### Steps

1. The first-time user reads the package overview and sees a clear grouping:
   Core packages, Agent Stack packages, and Optional / advanced packages.
2. The user runs the quickstart and gets the reference agent path by default,
   without needing to understand external-agent bridge packages first.
3. An advanced agent author imports `@facet/agent-tools` into their own LLM loop.
4. The agent author executes a Facet stage tool call and receives:
   - safe patch messages to send through Facet,
   - a typed tool observation for the model,
   - an updated local stage shadow,
   - compact stage feedback rather than a full stage dump by default.
5. The reference agent uses the same `@facet/agent-tools` package and adds the
   higher-level provider loop, context assembly, compaction, budgets, and trace
   behavior.
6. Maintainers can review PR1 and PR2 separately:
   - PR1: physical package taxonomy migration, package taxonomy docs, and
     `@facet/agent-tools`.
   - PR2: reference-agent harness rebuild on top of PR1.
7. The grouped directory layout leaves room for future published extensions and
   unpublished lab packages without blurring the first-run path.

## Input/Output Examples

### Example 1 (Normal)

- Input: The LLM calls a Facet stage tool such as `append_node` with a valid
  parent id and a declarative `box` / `text` / `media` / `field` node using
  token values.
- Output:
  - The tool executor returns an `ok` observation with patch count, changed ids,
    and a compact applied-stage summary.
  - It emits ordinary `ServerMessage` patch output.
  - It updates the local stage shadow by folding the patch through Facet's
    canonical stage fold path.
  - It does not return the full stage unless the model explicitly uses an
    inspection tool.

### Example 2 (Edge/Error)

- Input: The LLM calls `append_node` with an unknown parent id, malformed node,
  unsupported raw CSS value, or invalid child reference.
- Output:
  - The executor returns a typed error observation for the model.
  - No patch is emitted for that failed mutation.
  - The stage shadow remains unchanged.
  - The executor does not throw as normal control flow, and the renderer remains
    fail-safe.

### Example 3 (Context Pressure)

- Input: The reference agent has accumulated enough transcript, tool results,
  and stage feedback to exceed the configured context budget.
- Output:
  - The harness compacts older context into a summary before the next provider
    call.
  - Recent user intent, unresolved tool observations, budget status, and compact
    stage state remain visible to the model.
  - The turn stops with a traceable reason if the budget is exhausted.

## Constraints (What Not to Do)

- Do not add raw HTML, raw JS, arbitrary CSS, hex colors, pixel positioning, or
  arbitrary client-side execution to Facet's generated stage surface.
- Do not add backend/domain tools to Facet. Domain fetches, purchases, database
  writes, and business logic remain the operator agent's own tools.
- Do not make `@facet/core` depend on `@facet/agent-tools`,
  `@facet/reference-agent`, Node built-ins, providers, or runtime code.
- Do not require `@facet/agent-tools` users to import `@facet/reference-agent`.
- Do not take Pi or another agent framework as a runtime dependency. Pi may only
  be used as design reference during planning.
- Do not make `@facet/agent`, `@facet/agent-client`, `@facet/cli`, or
  `@facet/bridge` the required first-run path. They remain useful optional /
  advanced packages unless a later brief changes that.
- Do not use a vague catch-all directory for everything non-core. The taxonomy
  must distinguish recommended agent-stack packages from advanced extensions and
  unpublished lab / WIP packages.
- Do not rename npm package names while moving directories. `@facet/core`,
  `@facet/runtime`, and the rest keep their published names.
- Do not move packages without updating workspace globs, TypeScript paths,
  scripts, docs, tests, live-link gate path rules, and release tooling in the
  same PR scope.
- Do not merge PR2 before the public surface from PR1 is available and reviewed.

## Policy & Edge Cases

| Scenario Step | Failure Policy | Boundary/Validation | Concurrency / Duplicate Trigger |
|---|---|---|---|
| Package taxonomy docs | Documentation must not imply optional external-agent packages are required for quickstart use. | Package roles must match actual dependency direction and package exports. | Multiple docs must use the same taxonomy to avoid drift. |
| Agent author imports `@facet/agent-tools` | Import must work without importing reference-agent provider code. | Browser-safe or core-only surfaces must not pull Node built-ins. | Repeated imports should be side-effect free. |
| Valid stage tool call | Return `ok`, patch output, changed ids, and compact stage feedback. | Validate declarative bricks and token-only styles through existing Facet validation/fold behavior. | Tool calls in one turn execute sequentially against the latest stage shadow. |
| Invalid stage tool call | Return a typed error observation, emit no patch, keep shadow unchanged. | Reject unknown ids, malformed nodes, unsupported operations, and unsafe values. | Duplicate invalid calls should produce deterministic errors and no partial stage mutation. |
| Stage inspection request | Return bounded stage or node summaries, not unbounded full dumps by default. | Inspection output must be size limited and structured. | Inspection observes the current local shadow after prior successful tool calls. |
| Reference-agent provider loop | Stop with traceable reason on provider stop, tool limit, retry limit, or budget limit. | Provider results, tool calls, and usage data must be narrowed from unknown input. | Iterative tool calls must feed observations back into the next provider step in order. |
| Context pressure | Compact older context before exceeding configured limits. | Keep recent user intent, unresolved state, and tool results. | Repeated compaction must be deterministic enough for tests. |
| Directory-level grouping | Package moves must be treated as an explicit migration with path tooling updated in the same scope. | Workspace globs, TypeScript paths, package scripts, docs, tests, gates, and release tooling must all be audited. | PR2 must branch from or rebase onto PR1 after the path migration lands. |

## Done Criteria (Testable)

- [ ] DC-001 (PR1, Package taxonomy): Core, Agent Stack, and Optional / advanced
  package groups are reflected in both the workspace directory layout and
  package overview docs. Verify: docs diff review, `find packages -maxdepth 2`,
  and `rg` for package descriptions that contradict the taxonomy.
- [ ] DC-002 (PR1, Reusable tool package): `@facet/agent-tools` exposes Facet LLM
  stage tool specs, a tool executor, stage shadow helpers, and inspection
  helpers without exporting a provider loop. Verify: package barrel exports,
  typecheck, and focused package tests.
- [ ] DC-003 (PR1, Happy path tool execution): A valid stage mutation tool call
  produces `ServerMessage` patch output, an `ok` observation, changed ids,
  patch count, compact stage summary, and an updated stage shadow. Verify:
  vitest unit test in `packages/agent-tools`.
- [ ] DC-004 (PR1, Boundary/error tool execution): An invalid stage mutation returns
  a typed error observation, emits no patch, does not throw as normal control
  flow, and leaves the stage shadow unchanged. Verify: vitest unit test in
  `packages/agent-tools`.
- [ ] DC-005 (PR1, No reference-agent duplication): `@facet/reference-agent` consumes
  `@facet/agent-tools` for Facet stage tool specs/execution instead of keeping a
  duplicate private tool implementation. Verify: import scan with `rg` plus
  reference-agent tests.
- [ ] DC-006 (PR2, Reference harness): `@facet/reference-agent` has explicit loop,
  transcript/context, compaction, budget, and trace modules or equivalent
  package-local structure. Verify: package tests cover provider iterations,
  tool observations, compaction, budget stop, and trace stop reasons.
- [ ] DC-007 (PR1+PR2, Compatibility): Existing public reference-agent factories, stub
  behavior, and quickstart behavior continue to work. Verify: typecheck,
  reference-agent tests, quickstart deterministic E2E, and live-link gate as
  required by touched files.
- [ ] DC-008 (PR1+PR2, Dependency hygiene): `@facet/core` remains dependency-free;
  browser-safe entries do not import Node built-ins; no Pi runtime dependency is
  introduced. Verify: `package.json` review, import scan, typecheck, and build.
- [ ] DC-009 (PR1, External agent author path): A small example or test demonstrates
  using `@facet/agent-tools` without importing `@facet/reference-agent`.
  Verify: docs/example typecheck or vitest compile-time coverage.
- [ ] DC-010 (PR1, Directory migration completeness): The physical package taxonomy
  migration updates workspace globs, root scripts, `tsconfig.base.json` paths,
  docs, tests, live-link gate path rules, and any release tooling that references
  package paths. Verify: `/spec-bridge` includes an impact scan and the PR runs
  typecheck, tests, lint, format check, and build after the move.

## Invariant Fit (Facet-specific - REQUIRED)

Mark each: `OK` / `TOUCHES (mitigation)` / `CONFLICT (must re-scope)`.

| # | Invariant | Status | Note / mitigation |
|---|---|---|---|
| 1 | Responsibility boundary (UI-out/UI-in only; no backend/domain) | TOUCHES | The new package is limited to Facet stage tools. Backend/domain tools are explicitly out of scope and remain operator-owned. |
| 2 | Mechanism vs policy (agent authors; Facet provides capability) | TOUCHES | `@facet/agent-tools` provides reusable mechanisms. `@facet/reference-agent` provides one default policy layer, not a required framework policy. |
| 3 | Fail-safe (never throw/inject; degrade to plain) | TOUCHES | Tool execution returns typed observations for bad model output and folds successful patches through the existing fail-safe stage path. |
| 4 | Declarative + tokens only (no raw HTML/JS/CSS/pixels) | OK | The feature reuses existing bricks, tokens, validation, and patch protocol. No new raw escape hatch is introduced. |
| 5 | Flow-only safety (overlap only via a dedicated overlay brick) | OK | No overlay, absolute positioning, or layout vocabulary is added. |
| 6 | Two-writers coherence (local execution vs agent stage) | TOUCHES | Within a provider turn, successful tool calls update a local stage shadow in order. The runtime remains the authoritative stage writer when patches are yielded. |
| 7 | Backend calls go through the agent, never client fetch | OK | No client fetch or domain backend operation is added. External business tools stay in the agent's own tool set. |

## Open Questions / Assumptions

- Assumption: PR1 can expose enough stage feedback through local shadow folding
  without changing the runtime's generator contract. Owner: Codex. Checkpoint:
  `/spec-bridge` risk probe.
- Assumption: Existing quickstart behavior can remain compatible while the
  reference-agent internals are rebuilt. Owner: Codex. Checkpoint: PR2 tests and
  live-link gate.

  `optional` into a catch-all. A likely shape is:
  - Core: stable Facet platform packages.
  - Agent Stack: recommended reusable/reference LLM agent packages.
  - Extensions: optional advanced adapters and bridge surfaces.
  - Labs: unpublished experiments, test packages, and WIP packages only.
