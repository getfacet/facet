# Feature Intake Brief: Screens + Declarative View Actions (behavior language, phase 1)

> Generated by `/feature-intake`. Plain-language, testable, invariant-checked.

## Goal

- Give the agent a **declarative behavior language** so pre-declared interactions
  run **instantly in the browser with zero LLM round-trips**: (1) the stage can
  hold multiple **named screens** the agent pre-draws, (2) buttons can declare
  `navigate` (switch screen) and `toggle` (show/hide an element), and (3) the
  browser owns that view-state. Open-ended interactions keep routing to the agent
  exactly as today.

## Current Problem

- Every user event — even "open the menu" or "switch tab" — routes to the live
  LLM: seconds of latency, real cost per click, and a dead page when the agent is
  offline. Rapid clicks queue N LLM calls behind the per-visitor serializer.
- The agent has no way to say "this button just shows that screen"; the
  vocabulary can express structure (bricks) but not behavior. This is the biggest
  half-missing piece of the UI-IN thesis (established by the architecture review).

## User Scenario

### Who
- **The agent (LLM)** authoring a page through Facet (primary consumer), and
- **the end-user (visitor)** interacting with the page it drew.

### Trigger
- The agent draws a multi-screen experience — e.g. florist "Fern": a `home`
  screen, an `about` screen, and a collapsible ☰ menu panel.

### Steps
1. Agent renders a stage with **screens** `{ home, about }`, entry `home`, a nav
   button declaring `{ kind: "navigate", to: "about" }`, and a ☰ button declaring
   `{ kind: "toggle", target: "menu_panel" }` (menu panel pre-drawn, initially hidden).
2. Visitor clicks **About** → the browser switches to the `about` screen
   **instantly**; no event is sent; no LLM runs.
3. Visitor clicks **☰** → the menu panel expands in-flow **instantly**; clicking
   again collapses it.
4. Visitor types "do you deliver on Sundays?" in chat → routes to the agent
   exactly as today (`kind:"agent"` path unchanged).
5. While the agent is redrawing content (a slow LLM turn), steps 2–3 still work
   instantly — view-state and content never contend.
6. Agent's redraw arrives: the visitor stays on the screen they were viewing if
   it still exists; otherwise the entry screen shows (fail-safe).

## Input/Output Examples

### Example 1 (Normal — navigate)
- Input: stage `{ screens: { home: <tree>, about: <tree> }, entry: "home" }`; a
  box with `onPress: { kind: "navigate", to: "about" }`; visitor clicks it.
- Output: `about` renders immediately; **no** ClientEvent is emitted; the server
  stage is untouched.

### Example 2 (Normal — toggle)
- Input: a box `menu_panel` marked initially hidden; ☰ box with
  `onPress: { kind: "toggle", target: "menu_panel" }`; visitor clicks twice.
- Output: panel visible after click 1, hidden after click 2 — all browser-local.

### Example 3 (Edge/Error)
- Input: `navigate` to a screen name that doesn't exist; `toggle` targeting a
  removed node id; the agent deletes the screen the visitor is viewing;
  a malformed action (unknown `kind`, wrong field types) arrives via raw patch.
- Output: unknown targets are **no-ops**; deleted current screen **falls back to
  entry**; malformed actions are sanitized/ignored — the renderer never throws
  (degrade to plain).

## Constraints (What Not to Do)

- **No router features**: no route params, no guards, no nested routes, no URL
  sync/deep links in this phase (DSL-creep line).
- **No floating overlay**: the menu toggle is **in-flow** (pushes content).
  Overlay bricks (dropdown floating over content, modal) are a later feature.
- **No `kind:"patch"`** (server-applied content mutations without the LLM) — next
  phase. In this phase browser-executed actions touch **view-state only**, never
  stage content.
- **No field value transport** (phase B), **no kit presets** (follow-up), **no
  transitions/animation tokens** (follow-up), **no client-side fetch kind** (never).
- Legacy single-tree stages and legacy `{name, payload}` actions must keep working.

## Policy & Edge Cases

| Scenario Step | Failure Policy | Boundary/Validation | Concurrency / Duplicate Trigger |
|---|---|---|---|
| navigate on click | unknown screen → no-op | screen names validated by validateTree; non-string `to` stripped | rapid clicks idempotent (browser-local, no queue) |
| toggle on click | unknown/removed target id → no-op; orphaned view-state entries ignored | non-string `target` stripped | double-click = open+close (last-write-wins, browser-local) |
| agent redraw during viewing | current screen still exists → stay; deleted → entry fallback; entry missing → first screen; no screens → offline-style plain render | validateTree normalizes the screens map fail-safe (cycles/depth/unknown keys per screen) | content written ONLY by server; view-state written ONLY by browser — no contention by construction |
| legacy stage `{root,nodes}` arrives | rendered as a single-screen stage (normalized) | validateTree normalization is the single conversion point | n/a |
| legacy action `{name,payload}` (no kind) | treated as `kind:"agent"` (today's behavior) | unknown `kind` values → action stripped (box renders non-pressable) | n/a |
| refresh / reconnect | view-state resets to entry + defaults (accepted; browser-only state) | n/a | n/a |

  current screen alive, the visitor stays on it → Verify: jsdom.

## Invariant Fit (Facet-specific — REQUIRED)

| # | Invariant | Status | Note / mitigation |
|---|---|---|---|
| 1 | Responsibility boundary (UI-out/UI-in only; no backend/domain) | OK | Pure UI vocabulary; nothing fetches/computes/stores domain data. |
| 2 | Mechanism vs policy (agent authors; Facet provides capability) | OK | Facet ships navigate/toggle + screens capability; the agent decides how many screens to pre-draw (depth), what to wire, and when to fall back to `kind:"agent"`. |
| 3 | Fail-safe (never throw/inject; degrade to plain) | TOUCHES | New shapes (screens map, entry, action union) all normalize through validateTree; renderer guards: unknown screen/target no-op, entry fallback, per-screen cycle/depth caps. DC-004/005 pin this. |
| 4 | Declarative + tokens only (no raw HTML/JS/CSS) | OK | Actions are pure data with enum kinds; no expressions, conditions, or computed values (the `inc`-style op line is explicitly NOT crossed in this phase). |
| 5 | Flow-only safety (overlay only via a dedicated brick) | OK | Toggle shows/hides IN-FLOW panels. No floating/overlap capability is added; overlay remains a future dedicated brick. |
| 6 | Two-writers coherence (local execution vs agent stage) | TOUCHES | Resolved by ownership split: **browser writes ONLY view-state** (current screen, per-node visibility), **server remains the ONLY writer of stage content**. View-state references content by name/id; orphans resolve by no-op/entry-fallback. No stage version machinery needed because the two writers never write the same data. |
| 7 | Backend via agent tools, never client fetch | OK | No fetch/tool kind exists; anything needing data routes `kind:"agent"`. |
