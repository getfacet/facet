# Feature Intake Brief: Box Concern Packs + the Brick-vs-Field Growth Rule

> Generated by `/feature-intake`. Plain-language, testable, invariant-checked.
> PR-2.5 of the node-model restructure. **Pure type-space refactor + a
> governing-rule doc — behavior-preserving, no runtime change.** Framed like
> PR-1.5 node-mixins: existing suites are the oracle.

## Goal

Two outcomes, both structural/documentary — zero change to validation, render,
or prompt output:

1. **A written growth rule** that answers, once and for all, *"when does a new
   capability become a new brick vs a field on an existing brick?"* — so the
   vocabulary stops accreting ad-hoc fields.
2. **`box`'s remaining ad-hoc fields grouped into named concern packs**
   (`Pressable`, `Layered`), extending PR-1.5's module-private field-pack shape,
   so `box` reads as a composition of named concerns instead of a growing list
   of loose fields.

## Current Problem

- **Who is affected:** the Facet maintainer + coding agents evolving
  `@facet/core`'s vocabulary. The end-user/agent-author sees nothing change.
- **Concrete pain:** `box` and `text` kept gaining optional fields with no stated
  rule for *when* that is legitimate. `box` today carries four ad-hoc behavior
  fields (`onPress`, `onHold`, `hidden`, `backdrop`) declared inline, and PR-3
  will add `overlay`. Without a rule, every new capability is a judgment call and
  the primitive base drifts toward a grab-bag. PR-1.5 tidied the *cross-cutting*
  fields into packs (`BaseNode`/`Styleable`/`ActiveLook`/`ContainerFields`/
  `DataBound`) but left `box`'s behavior fields loose and never wrote down the
  policy that governs future growth.
- **Audit finding (2026-07-14, all bricks checked):** among the 5 primitives
  only `box` has this accumulation problem; `text`/`media`/`field`/`richtext`
  carry only their own content fields (rule-conformant). Components are flat but
  rule-conformant and are absorbed by the PR-5/6 reshape → out of scope here.

## User Scenario

### Who
The agent author's framework maintainer (and future coding agents) evolving the
brick vocabulary. No runtime actor; this is a source-shape + docs change.

### Trigger
A future capability needs to be added to the vocabulary (e.g. PR-3 `overlay`,
PR-4 input consolidation), and the contributor must decide brick-vs-field.

### Steps
1. Contributor consults the **growth rule** in `docs/ARCHITECTURE.md`.
2. The rule's two-axis test (+ the fuzzy-middle line) yields a clear answer:
   new *kind of content* → a new native data brick; a new *way `box`
   behaves/presents* → a named concern pack (not a loose field).
3. For a `box` behavior mode, the contributor adds it to the relevant concern
   pack (`Pressable`/`Layered`), deliberately, like adding a token.
4. `box`'s interface still reads as `BaseNode(+hidden) & Styleable & ActiveLook &
   ContainerFields & Pressable & Layered` — a composition of named concerns.

### Steps (this PR's own change)
1. Extract `Pressable { onPress?; onHold? }` and `Layered { backdrop? }` as
   module-private packs in `nodes.ts`; `box extends … Pressable, Layered`.
   `hidden` stays on `BaseNode` (visibility, not layering).
2. Write the growth rule in `docs/ARCHITECTURE.md`; add a one-line pointer to it
   from `AGENTS.md` Conventions.
3. Existing suites stay green with no test rewrites (mechanical only).

## Input/Output Examples

### Example 1 (Normal — a `box` tree is byte-identical)
- Input: an agent emits a `box` with `onPress`, `onHold`, `hidden`, `backdrop`.
- Output: the validated/rendered result is **identical** to before the refactor
  (packs are interfaces; they erase at compile — no runtime code moves).

### Example 2 (Edge — a would-be drift is caught)
- Input: a contributor accidentally drops `onHold` while extracting the pack, or
  retypes `backdrop` from `NodeId` to `string`.
- Output: the type-level field-set/exhaustiveness guard (PR-1.5) trips at
  `tsc`/test time — the mistake cannot land silently.

### Example 3 (Doc — the rule resolves a real question)
- Input: "Is `overlay` a new brick or a `box` field?"
- Output: the rule's fuzzy-middle line answers: needs renderer computation
  `box`+`text` can't express → brick; a way `box` overlaps/decorates → a `box`
  concern (pack). `overlay` = a `box` concern → goes in `Layered` (PR-3).

## Constraints (What Not to Do)

- **No behavior change.** No validation/render/prompt output may differ; no
  runtime code path changes. This is type-space + docs only.
- **No field added/removed/renamed/retyped** on any node. `box`'s resolved
  field-set must be byte-identical.
- **Packs are module-private** (not exported from the barrel), mirroring PR-1.5,
  to avoid growing the public surface. Exception is only if a later PR needs it.
- **Do NOT touch components** (`button`/`card`/`emptyState` etc.) even though they
  share `onPress`/`onHold`; they are reshaped in PR-5/6. Defining `Pressable` now
  is enough; they adopt it later.
- **Do NOT add `overlay`** here — that is PR-3. `Layered` starts with `backdrop`
  only.
- The `type` literal stays a **direct member** of each node interface (never
  inherited from a pack) — PR-1.5 lesson: discriminated-union narrowing +
  `Record<CoreNodeType,…>` exhaustiveness hold only if `type` is direct.
- The rule text lives in **one** place (`docs/ARCHITECTURE.md`); `AGENTS.md` only
  points to it — no duplicated copy that can drift.
