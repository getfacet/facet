# Feature Intake Brief: Brick Style System Hard Cutover

> Generated by /feature-intake. Plain-language, testable, invariant-checked.
>
> Status: Owner-approved on 2026-07-17; implemented through the approved hard-cut flow.
>
> Implementation amendment: live provider evidence showed that returning full
> value metadata with one or more Brick specs exceeded useful context bounds.
> The approved discovery intent is preserved with one compact Brick per
> `get_brick_spec` call and one exact property domain per
> `get_style_choices` call.
>
> This brief supersedes the style, catalog, theme-selection, recipe/variant, and
> composition-naming decisions in older planning artifacts. Historical specs
> remain archival evidence; this brief is the product source for the next
> /spec-bridge.

## Goal

- Give an LLM one simple way to style every Facet brick: omit style for the
  Theme default, name a Preset for a reusable look, provide direct closed token
  choices for a deliberate adjustment, or combine a Preset with direct
  adjustments.
- Make the agent able to discover what each brick, Preset, token choice, and
  Pattern means before it authors a Facet document, without loading the full
  design system into every mutation tool schema.
- Make one active Theme fully control the brand-specific values, native-brick
  defaults, reusable Presets, light/dark appearance, and renderer-owned internal
  targets while the Facet document remains declarative and token-only.
- Preserve Facet's strict authoring boundary and fail-safe renderer: normal
  authoring mistakes are rejected atomically with repairable errors; bypassed
  or stale documents still render as much safe content as possible and never
  throw.
- Remove overlapping or misleading concepts rather than adding another layer:
  no public recipe, variant, tone, scheme, composition, Catalog, or DesignGuide.
- Land one atomic hard cutover in one feature PR. The cutover replaces the
  entire public style/design-asset contract at one version boundary and is
  decomposed internally into bounded, testable work units.

## Current Problem

- The current style model has several overlapping paths. A node may use inline
  style tokens, variant/tone selectors, Theme recipes, recipe parts, active
  variants/styles, and a subtree scheme. Their responsibilities and precedence
  are difficult for both people and agents to understand.
- Recipes exist structurally but are sparse and poorly advertised. Patterns
  currently bake many token choices directly into concrete trees, so reusable
  semantic styling does not scale through names.
- Direct styles exist only for part of the eleven-brick roster. Renderer-owned
  targets such as an input control, table header, chart series, progress track,
  or loading indicator do not have one consistent agent-facing style model.
- The agent cannot reliably choose a valid direct style without knowing the
  brick's property paths, the token group allowed at each path, and the meaning
  of each choice. A single generic object schema and long prose specification
  are not an effective discovery surface.
- Composition references are concrete examples, but their current name sounds
  like a runtime composition mechanism. Catalog simultaneously controls theme
  selection, brick restrictions, reference exposure, and authoring advice even
  though the Core bricks are all safe and the per-agent asset result already
  selects the Theme and reference list.
- The current tree can carry a Theme name and the agent can call set_theme,
  although the target model has one operator-selected Theme per agent. The word
  scheme is also used for authored subtree paint and browser color reporting.
- The normal tool path may sanitize or partially apply bad author input in ways
  that can look successful to the LLM. The agent needs an atomic rejection with
  precise repair guidance, while the renderer still needs independent fail-safe
  behavior for stale or bypassed data.
- Affected users:
  - LLM agents authoring and repairing Facet documents;
  - operators supplying a brand Theme and Pattern library;
  - framework consumers maintaining custom agents, transports, and asset stores;
  - end-users who need consistent, legible UI that continues to render safely.

## Vocabulary Lock

| Example | Canonical term | Meaning |
|---|---|---|
| fontSize, gap, color | Style property | A closed visual choice owned by a brick's style vocabulary |
| label, control, track, fill | Style target | A closed nested target owned by that brick, never a global part |
| md, lg, success | Token name | A closed semantic or scale name the agent may author |
| 16px, 24px, #16a34a | Resolved CSS value | Renderer input supplied by a validated Theme, never Facet document syntax |
| space.md = 16px | Theme token definition | The active Theme's concrete value for one Core token name |
| panel, heading | Preset name | A reusable named style for one brick type |
| a validated bundle of token choices and state styles | Preset definition | Theme-owned data with required usage metadata |
| a validated concrete native-brick tree | Pattern | Read-only example data an agent inspects and adapts |
| light, dark, system | Color mode | Client/host display state, never a Facet document style token |

The public vocabulary does not use recipe, variant, tone, scheme, composition,
Catalog, component tier, global part, or raw CSS value as authoring concepts.

## User Scenario

### Who

- The operator configures an agent's asset result: at most one custom Theme, an
  exact Pattern list, and optionally an initial tree.
- The LLM agent discovers available Patterns, bricks, and Presets, then authors
  native Facet bricks and styles.
- The end-user views and interacts with the rendered page. The renderer also
  reacts locally to color mode, current screen, toggles, hover, press, focus,
  checked, and other explicitly supported visual states.

### Trigger

- An agent must create or update a branded Facet page, either from a fitting
  Pattern or directly from the native-brick vocabulary.

### Steps

1. The host loads the per-agent asset result. Facet validates the complete
   custom Theme first, falls back to the default Theme if necessary, then
   validates Patterns against the effective Theme.
2. At the start of a provider turn, the agent receives short indexes for all
   Core bricks, valid active-Theme Presets, and valid compatible Patterns.
3. The agent checks Pattern names, descriptions, and useWhen guidance first. If
   one fits, it calls get_pattern, inspects the concrete tree, and adapts the
   structure rather than inserting it automatically or copying blindly.
4. Before authoring an unfamiliar brick or style, the agent calls
   get_brick_spec for that one brick. It sees the brick's data fields and compact
   style target/property map. Only when one direct value is unfamiliar does it
   call get_style_choices for that exact Brick-owned property. If it needs a
   reusable look's details, it calls get_preset.
5. The agent authors ordinary native bricks. It normally uses a suitable
   Preset, uses direct style only for Pattern-specific layout or an intentional
   adjustment, and combines both when the direct choice must override one
   Preset property.
6. The authoring tool validates the entire requested mutation against the same
   Core contract and active asset snapshot. A valid mutation emits normal RFC
   6902 patches. Any invalid Preset, property, target, token, or fixed value
   rejects the whole mutation with structured repair information and no patch.
7. The renderer resolves Theme defaults, Presets, direct styles, and supported
   state styles. It reacts to client color mode and local active/interaction
   state without asking the LLM and without writing stage data.
8. If stale, hostile, or manually bypassed data reaches the renderer, invalid
   style fragments are ignored and valid bricks/siblings continue rendering.

## Input/Output Examples

### Example 1 (Normal: Pattern-first, Preset-first)

- Input assets:
  - one valid Theme with box Preset panel and text Preset heading;
  - a compatible account-summary Pattern whose metadata explains when to use
    it.
- Agent flow:
  1. sees account-summary in the Pattern index;
  2. reads it with get_pattern;
  3. reads box and text specifications as needed;
  4. authors adapted native bricks:

        {
          "id": "summary",
          "type": "box",
          "style": {
            "preset": "panel",
            "gap": "lg"
          },
          "children": ["summary-title"]
        }

        {
          "id": "summary-title",
          "type": "text",
          "value": "Account summary",
          "style": {
            "preset": "heading"
          }
        }

- Output:
  - no CSS values appear in the Facet document;
  - the Theme default applies first, panel/heading apply next, and gap:lg
    overrides only the panel's gap;
  - the renderer produces the active Theme's light or dark concrete values.

### Example 2 (Normal: no explicit style)

- Input:

        {
          "id": "status",
          "type": "progress",
          "value": 62
        }

- Output:
  - the Progress brick uses the active Theme's complete Progress default;
  - an omitted style does not mean unthemed or unstyled.

### Example 3 (Edge/Error: invalid authoring request)

- Input:

        {
          "id": "status",
          "type": "progress",
          "value": 62,
          "style": {
            "preset": "missing-preset",
            "track": { "height": "4px" },
            "fill": { "background": "magenta" }
          }
        }

- Output on the normal tool path:
  - status is error, outcome is rejected, applied and stage_changed are false,
    patch_count is zero;
  - bounded errors identify the invalid Preset and the invalid property-local
    choices, with allowed choices and a next action;
  - the existing stage remains byte-for-byte unchanged.
- Output if equivalent stale data bypasses authoring validation:
  - the renderer ignores invalid style fragments, uses safe defaults for the
    brick, keeps valid siblings, and never throws. This direct-renderer bypass
    is the silent last defense: normal agent authoring receives structured tool
    errors, while asset/fold boundaries keep their existing host-visible issues.

### Example 4 (Edge/Error: invalid Theme and incompatible Pattern)

- Input:
  - a custom Theme omits required Core token definitions;
  - a Pattern references a Preset absent from the effective Theme.
- Output:
  - the custom Theme is rejected as a whole and the default Theme becomes
    effective;
  - the incompatible Pattern is hidden as a whole and an operator-facing asset
    issue is recorded;
  - no partial Theme merge, Preset stripping, token rewriting, or Pattern
    fallback occurs.

### Example 5 (Local state without an agent turn)

- Input:

        {
          "id": "settings-nav",
          "type": "box",
          "activeWhen": { "screen": "settings" },
          "style": {
            "preset": "navigationItem",
            "active": {
              "preset": "selectedItem",
              "color": "accentForeground"
            }
          },
          "children": ["settings-nav-label"]
        }

- Output:
  - navigating to settings applies style.active locally;
  - navigating away restores the base look;
  - no event, stage patch, or LLM turn is generated by the visual transition.

## Constraints (What Not to Do)

- Do not add raw HTML, JavaScript, CSS, pixels, arbitrary colors, CSS variables,
  expressions, arbitrary URLs, or arbitrary scalar style escape hatches to a
  Facet document.
- Do not add a new native brick. The roster remains box, text, media, input,
  richtext, table, chart, list, keyValue, progress, and loading; only box is a
  container.
- Do not add absolute positioning, general z-index, margin, transform, opacity,
  asymmetric padding, arbitrary dimensions, or other flow-escape styling.
- Do not expose arbitrary DOM parts or selectors. Every style target belongs to
  one brick's closed style vocabulary. No global parts registry appears in
  Facet document syntax.
- Do not initially expose arbitrary table row/cell/column styling, list item
  selection, rich-text run/block selection, chart point selection, or similar
  data-addressed style selectors.
- Do not make a Pattern a runtime node, parameterized template, automatic
  insertion API, provenance marker, or hidden stage writer.
- Do not create a Catalog or DesignGuide replacement. All Core bricks are
  available; the active Theme defines available Presets; AssetsStore returns the
  exact Pattern list.
- Do not let the agent select or switch Theme, author a Theme name in the tree,
  or author light/dark mode in the Facet document.
- Do not partially merge an incomplete custom Theme into the default Theme.
- Do not silently truncate an over-limit Preset or Pattern exposure set.
- Do not add a get_token tool. Token choices and meanings are disclosed only at
  the property paths where they are valid.
- Do not preserve, read compatibly, or automatically migrate old Catalog, composition,
  get_composition, set_theme, tree theme, variant, tone, recipe, scheme,
  activeVariant, activeStyle, or abbreviated style-property authoring shapes.
- Do not dual-read or dual-write old and new asset shapes, export aliases for
  retired types/tools, keep compatibility flags, or ship a transition period.
- Do not reinterpret persisted old Theme/Composition/Catalog JSON at runtime.
  Operators upgrading custom file/Postgres assets must replace or migrate their
  stored data to the new Theme/Pattern shape before using the new release.
- Do not move page purpose, service persona, backend policy, or user-specific
  business context into design assets. Those remain in the agent brain,
  operator Page Brief, or host/client context.
- Do not add client-side domain fetching or business logic. Backend work remains
  the agent's own tools.
- Do not change the patches-only transport contract or create another stage
  writer.
- Do not split the final behavior across independently merged compatibility PRs
  or support mixed old/new Facet package versions. The target is one atomic
  feature PR with reviewable internal work units and one package-version
  cutover.

## Locked Product Contract

### Authoring and resolution

- Every brick may omit style or carry one style object.
- style may contain a Preset name, direct closed properties, nested
  brick-owned targets, and supported state blocks.
- The normal resolution order is:

        Theme brick default
        -> base Preset
        -> base direct style
        -> matching active style
        -> current ephemeral interaction state

- Later layers override earlier layers per property; unspecified properties
  remain inherited from the earlier layer.
- style omitted means the active Theme's brick default, not an unstyled brick
  and not a named default Preset.
- Presets and direct styles use the same brick style vocabulary. Presets may
  define supported state styles; direct styles may override supported state
  properties.
- Preset names are scoped by brick type. An authored box uses panel rather than
  box.panel because the node type provides the namespace.
- The Theme's semantic defaults and Presets should carry repeatable
  single-brick roles. Pattern-specific multi-brick layout normally uses direct
  token/fixed choices. A one-off exception may use Preset plus direct override.

### Core-owned style and token contract

- Core is the single source of truth for:
  - every brick's fields and required/optional data;
  - every brick's closed style targets and property paths;
  - the allowed token group or fixed choices at each property path;
  - token names, ordering, meaning, and property-local usage guidance;
  - Brick metadata used in the short index and detail lookup.
- Validators, agent detail schemas, reference documentation, and public types
  must be generated from or checked against this one contract so they cannot
  drift.
- Several bricks may independently own a property or target with the same name.
  chart.style.title and list.style.title are separate brick-owned vocabulary,
  not a global title part.
- Global Core token groups may be reused by several properties and bricks.
  Core may also define a token group used by only one brick when the visual
  value genuinely belongs in Theme data.
- A value is a Theme token when its concrete brand value may vary. A value is a
  Core fixed choice when its meaning is stable across brands.

### Initial token groups and fixed choices

- Scale/value token groups:
  - space: none, xs, sm, md, lg, xl, 2xl
  - fontSize: xs, sm, md, lg, xl, 2xl, 3xl, 4xl
  - fontFamily: sans, serif, mono
  - fontWeight: regular, medium, semibold, bold
  - radius: none, sm, md, lg, full
  - borderWidth: none, thin, medium, thick
  - shadow: none, sm, md, lg
  - aspectRatio: auto, square, landscape, portrait, wide
  - minHeight: auto, half, screen
  - maxWidth: none, prose, narrow, wide
  - letterSpacing: tight, normal, wide
  - lineHeight: tight, normal, relaxed
  - gradient: none, accent, success, warning, danger, info
  - scrim: none, soft, strong
  - highlight: none, accent, warning
  - controlHeight: sm, md, lg
  - indicatorSize: sm, md, lg
  - progressThickness: sm, md, lg
  - chartThickness: sm, md, lg
- Semantic color names include:
  - background, surface, mutedSurface, foreground, mutedForeground, border;
  - accent, accentSurface, accentForeground, focusRing;
  - success, successSurface, successForeground;
  - warning, warningSurface, warningForeground;
  - danger, dangerSurface, dangerForeground;
  - info, infoSurface, infoForeground;
  - chart1 through chart6;
  - inherit where inheritance is a meaningful property-local choice.
- Core fixed choices include:
  - direction: row or column;
  - alignment: start, center, end, stretch;
  - justification: start, center, end, between, around;
  - wrap, grow, sticky: boolean;
  - width: auto or full;
  - scroll: none, horizontal, vertical;
  - columns: none, 2, 3, 4;
  - textAlign: start, center, end;
  - fontStyle: normal, italic;
  - objectFit: cover, contain;
  - objectPosition: center, top, bottom, start, end;
  - enterAnimation: none, fade, slide;
  - loading animation: none, pulse.
- Semantic text roles such as heading, body, caption, or label are Presets,
  rather than alternative fontSize token names.

The initial surface is deliberately useful but bounded. It can grow later only
through an explicit Core contract change.

### Theme, Presets, and color mode

- One active Theme represents one brand design system and contains:
  - concrete values for the complete Core token space;
  - complete defaults for all eleven bricks and their published style targets;
  - Preset definitions and metadata;
  - complete light and dark paint-sensitive values.
- A custom Theme is complete, not a patch over the default Theme. Missing or
  invalid required definitions reject the whole custom Theme, produce an
  operator-facing issue, and select the default Theme.
- Theme may choose concrete values only. It cannot add/remove token names,
  brick properties, style targets, fixed choices, or native bricks.
- Preset metadata requires name, description, and useWhen; avoidWhen is
  optional detail. Category and tags are not initial agent-facing concepts.
- A Theme may contain at most 16 valid Presets per brick and 64 total. An
  over-limit custom Theme is invalid rather than silently truncated.
- The agent never receives resolved CSS values and never chooses the active
  Theme.
- Color mode belongs to client/host state:
  - colorMode is system, light, or dark;
  - the default is system;
  - a host may force light or dark;
  - mode changes may alter paint-sensitive color, shadow, gradient, scrim, and
    highlight values, but not spacing, dimensions, typography scale, or layout.
- Theme selection, tree theme, set_theme, and authored subtree scheme do not
  exist in the target public contract.

### Pattern contract and usage

- Pattern is a named, validated, read-only concrete Facet tree using native
  bricks and the same style contract as an ordinary document.
- Pattern metadata requires name, description, and useWhen. avoidWhen is
  optional detail.
- Pattern has no parameters, placeholder language, nested reference language,
  automatic insertion, runtime node type, or provenance written to the stage.
- The agent sees a short Pattern index first, calls get_pattern only when a
  Pattern fits, and then adapts/re-authors ordinary native bricks.
- A Pattern may use Theme defaults, Presets, direct styles, or Preset plus
  direct override. It should preserve applicable style choices when adapted,
  but the agent must not copy sample text/data/actions blindly.
- Pattern dependencies on Preset names are derived automatically. A Pattern is
  exposed only when it is valid against the effective Theme.
- A missing or renamed Preset hides the entire dependent Pattern and records an
  operator-facing issue. Facet does not strip, substitute, or remap the Preset.
- At most 64 compatible Patterns are exposed to one agent snapshot. An
  over-limit exact Pattern list is rejected for exposure with an operator issue;
  it is never silently truncated.
- A Pattern earns a name when it demonstrates a reusable multi-brick
  structure, behavior, or data relationship. A visual/tone-only variation is a
  Preset, not another Pattern.

### Assets and exposure

- Catalog is removed without a replacement layer.
- Core's eleven bricks are always available.
- AssetsStore is the per-agent selection boundary. Its logical result contains
  one optional custom Theme, an exact Pattern list, and an optional initial
  tree.
- With no valid custom Theme, the default Theme is effective. Exact default
  Pattern seeding versus an explicitly empty Pattern list is an implementation
  shape to be made unambiguous in /spec-bridge; it must never depend on Catalog.
- Asset validation order is custom Theme, effective Theme fallback, Presets,
  then Patterns against the effective Theme.
- Prompt indexes, read tools, and mutation validation share one immutable asset
  snapshot for a provider turn. An asset update becomes visible on the next
  turn, never halfway through one turn.
- Assets are initially per agent/service, matching AssetsStore.load(agentId).
  Per-user asset selection, hot reload, and hosted control-plane policy are out
  of scope.
- Page Brief, service persona, domain rules, and user business context remain
  outside Theme/Pattern assets.
- The released loader accepts only the new Theme/Pattern asset contract. Old
  files and persisted JSON are invalid input, produce bounded operator issues,
  and are never converted or partially interpreted.

### Agent discovery surface

- The always-visible prompt contains three short indexes:
  - all Core bricks: name, description, useWhen;
  - active-Theme Presets: brick type, name, description, useWhen;
  - compatible Patterns: name, description, useWhen.
- Full details are progressively disclosed through read-only tools:
  - get_brick_spec accepts exactly one brick type and returns its data fields,
    compact style target/property map, state rules, and applicability;
  - get_style_choices accepts one exact Brick-owned target/property path and
    returns only that property's allowed token or fixed choices with meanings;
  - get_preset returns one exact same-brick Preset's metadata and unresolved
    token/fixed style bundle;
  - get_pattern returns one exact compatible Pattern's metadata and concrete
    unresolved tree.
- Read success is no_stage_change and has no patch, shadow mutation, pending
  mutation, or visitor visibility.
- Unknown, hidden, incompatible, or invalid asset lookups all return the same
  bounded not_available class so hidden asset existence is not leaked.
- There is no get_token tool. The agent sees only choices valid at the
  requested brick property path.
- Mutation tools stay compact and generic. They do not inline all eleven full
  brick schemas; the Core validator remains authoritative.

### Failure behavior

- Normal agent authoring is strict and atomic. Any invalid Preset, target,
  property, token name, fixed value, or input-kind applicability error rejects
  the entire requested mutation before any RFC 6902 patch is emitted.
- A rejected authoring observation contains:
  - status error and outcome rejected;
  - applied false, stage_changed false, visible false, patch_count zero;
  - at most 16 path/message/allowed error entries;
  - an omitted-error count when needed;
  - a concrete next_action;
  - no unsafe echo of hostile raw values.
- Normal author validation does not report applied_with_warnings.
- Renderer validation remains an independent last defense. It may drop invalid
  style fragments or skip invalid nodes while preserving valid content and
  never throws. A direct raw `StageRenderer` bypass does not add a diagnostics
  callback; normal agent authoring reports structured tool errors, and
  asset/fold boundaries expose their existing bounded issues.

### Active and interaction styles

- box and text initially support node-level activeWhen predicates for current
  screen or toggled-node state.
- All active visual choices live in style.active. style.active may name a
  same-brick Preset and carry direct style choices.
- activeVariant and activeStyle are removed.
- The renderer evaluates activeWhen read-only. A dangling or invalid predicate
  keeps the base look.
- The effective active look is resolved before ephemeral hover, pressed, and
  focus state so immediate feedback remains visible.
- Existing local navigation/toggle ordering stays authoritative. Visual
  state evaluation emits no event, stage patch, data write, or LLM turn.

## Policy & Edge Cases

| Scenario Step | Failure Policy | Boundary/Validation | Concurrency / Duplicate Trigger |
|---|---|---|---|
| Load custom Theme | Reject the complete custom Theme, report an operator issue, use default Theme | Complete Core token/default coverage; safe concrete CSS validation; Preset caps | One immutable effective Theme per provider turn; update is visible next turn |
| Load Pattern list | Skip invalid/incompatible Pattern as a whole; reject an over-limit exposure set rather than truncate | Concrete native tree; required metadata; effective-Theme Preset dependencies; max 64 exposed | One frozen exact list per provider turn; no mid-turn drift |
| Show indexes | Omit invalid assets; never reveal CSS values | Short bounded metadata only; all eleven Core bricks always indexed | Index and read tools use the same snapshot |
| Read Brick details/choices | Reject an invalid Brick or exact style path without partial data | One exact Core Brick type, then one exact Brick-owned target/property | Read-only; repeated identical calls do not mutate state |
| Read Preset/Pattern | Return bounded not_available for hidden, missing, incompatible, or invalid names | Exact names within active snapshot; no existence leak | Read-only; repeated calls do not mutate stage or pending buffer |
| Author valid document | Emit ordinary RFC 6902 patches after strict validation | Closed brick/style/token/fixed-value contract | Existing server/client patch ordering and per-visitor serialization remain authoritative |
| Author invalid document | Reject the whole tool mutation with repairable errors and zero patch | Max 16 error entries; bounded allowed lists; no hostile value echo | Duplicate bad calls remain no-ops and cannot partially accumulate |
| Render bypassed/stale data | Drop invalid style pieces or invalid nodes, preserve valid siblings, never throw | Same closed contract used as fail-safe sanitation | Rendering is deterministic for the same tree, Theme, view, and mode |
| Change color mode | Re-resolve paint-sensitive values only | system/light/dark; host may force; no document-authored mode | Local re-render only; no stage patch or LLM turn |
| Match activeWhen | Apply style.active; dangling/invalid predicate leaves base style | box/text only initially; closed screen/toggled predicate | Local view ordering remains existing behavior; visual change writes no stage state |
| Apply interaction state | Apply only published state properties on the owning target | Per-brick state allow-list; reduced-motion policy for animation | Ephemeral renderer state; no extra writer |
| Store/update assets | Adapter returns exact per-agent data; validation is centralized | Per-agent key; no user/tenant control-plane semantics | Updates become visible on the next provider turn snapshot |

## Done Criteria (Testable)

- [ ] DC-001 (Vocabulary): Published types, prompts, tools, current docs, and
  default assets use Style property, Style target, Token, Theme, Preset,
  Pattern, and colorMode consistently, with no current public authoring path for
  Catalog, composition, recipe, variant, tone, scheme, activeVariant, or
  activeStyle. → Verify: public-export/type tests plus a hard-cut source/docs
  scanner that excludes archival specs and intentional negative tests.
- [ ] DC-002 (Simple authoring): Every one of the eleven native bricks accepts
  omitted style, Preset-only style, direct style, and Preset-plus-direct style
  wherever its published vocabulary permits. → Verify: Core type/validation
  matrix tests for all brick types.
- [ ] DC-003 (Resolution): Theme default, Preset, direct style, active style,
  and ephemeral state resolve in the locked order with property-level override,
  and omitted style still renders the Theme default. → Verify: React resolver
  and renderer precedence tests.
- [ ] DC-004 (Single source): One Core-owned contract supplies or mechanically
