# Feature Intake Brief: Component Model + Renderer Layout Contract

> Generated by `/feature-intake`. Plain-language, testable, invariant-checked.
> Status: owner-approved direction from the 2026-07-10 component-model discussion.

## Goal

- Make Facet easier for agents to use by simplifying the agent-facing model to
  **Primitive Brick -> Component -> Catalog**.
- Merge the conceptual roles of "high-level brick" and "stamp" into
  **Component**, so agents choose meaningful UI nouns instead of deciding whether
  something is a brick, stamp, or fallback pattern.
- Keep primitive bricks as the universal fallback: `box`, `text`, `media`, and
  `field`.
- Split component implementation internally into:
  - **intrinsic components**: platform/core-managed capabilities the renderer
    directly understands;
  - **composition components**: user/operator-defined components built only from
    allowed primitives and components.
- Define a renderer layout contract so every primitive and component obeys a
  consistent containment rule: parent owns placement, child owns internal layout,
  and the renderer owns clipping/bounds.

## Current Problem

- Facet now has primitives, high-level bricks, stamps, theme recipes, and
  catalog policy. This is powerful, but the boundary between high-level bricks
  and stamps is ambiguous for agents and operators.
- Agents should not need to reason about "is this a stamp or a high-level
  brick?" They should see a catalog of components and use the best available UI
  noun.
- Hosted Facet needs a safe extension model. Platform-managed intrinsic
  components can add renderer/runtime capability, but tenant/user-defined
  components must not inject code or expand the closed vocabulary.
- The current React renderer has many local safety guards, but no explicit
  cross-brick layout contract. Parent/child rules such as alignment inheritance,
  child override behavior, `min-width: 0`, max-width containment, and overflow
  ownership are not centralized or documented.
- Affected users:
  - agent authors and LLM harnesses, who need a small stable mental model;
  - hosted-platform operators, who need a safe user-defined component story;
  - end-users of agent-built pages, who need UI that stays inside bounds and
    does not visually break when nested.

## Intrinsic Component Criteria

An intrinsic component is a UI noun that Facet/platform officially supports as
part of the closed UI language. The test is not "could primitives build this?"
because primitives can build almost anything. The test is whether the noun should
be standardized so every agent can use it safely and consistently.

A component should become intrinsic only when it passes all required criteria:

1. **Generic UI noun:** it appears across many apps/domains and is not a
   business-specific, brand-specific, or page-template concept.
2. **Closed schema:** it can be described with bounded declarative props, token
   styles, and existing action semantics; no raw code, arbitrary CSS, backend
   binding, or expression language.
3. **Renderer-owned value:** the renderer/validator adds real value by owning at
   least one of semantics/accessibility, structured data rendering, interaction
   rules, containment/responsive behavior, or consistent fail-safe behavior.
4. **Invariant-safe:** it preserves declarative/token-only output, flow-only
   layout, fail-safe rendering, and agent-owned backend work.
5. **Stable vocabulary:** the name is likely to remain a durable UI word in the
   spec, prompt, catalog, docs, and tests.

Composition components are for everything else: domain-specific patterns,
brand-specific patterns, app/page templates, and repeated operator-local
assemblies of primitive bricks and intrinsic components.

Intrinsic boundary examples:

| Intrinsic noun | Composition example |
|---|---|
| `form` | `signupForm`, `billingForm` |
| `table` | `invoiceTable`, `customerTable` |
| `card` | `pricingCard`, `profileCard` |
| `emptyState` | `noInvoicesYetState` |
| `keyValue` | `customerDetailsPanel` |

## Initial Intrinsic Set

The v1 intrinsic component set should be:

- `button` — semantic action control over existing `FacetAction` semantics.
- `section` — page/screen region container with title/body slots.
- `card` — bounded grouped-content/action container.
- `divider` — semantic separator.
- `tabs` — local screen/view navigation over existing local navigate semantics.
- `nav` — app/section navigation list over existing local navigate semantics.
- `form` — field group plus submit/collect boundary; no backend ownership.
- `search` — bounded search input/action affordance.
- `filterBar` — bounded filter controls plus apply/reset actions; filtering
  logic stays with the agent/backend tools.
- `table` — display-only tabular data with capped columns/rows/cells.
- `chart` — display-only chart data with capped series/points.
- `list` — bounded repeated text/item display.
- `keyValue` — bounded label/value detail display.
- `metric` — compact KPI/stat display; current `stat` should migrate or alias to
  this noun.
- `progress` — bounded progress indicator.
- `badge` — compact status/label indicator.
- `alert` — semantic feedback/warning/notice.
- `emptyState` — semantic no-data/no-result state with optional action.
- `loading` — semantic pending/busy state with bounded label/skeleton variant.

Deferred from v1:

- `modal`, `drawer`, `menu` — overlay/focus/stacking components; require a
  separate constrained overlay design.
- `timeline`, `steps` — useful but more workflow/domain-shaped; validate through
  composition first unless later evidence says they must be built-in.
- `calendar`, `kanban` — complex domain/interaction surfaces; future separate
  intake if needed.
- `appShell` — app/page template, better expressed as composition over `nav`,
  `section`, `card`, and primitives.

## User Scenario

### Who

- The **platform/operator** defines intrinsic components and exposes allowed
  components through a catalog.
- The **tenant/user** may define composition components using the allowed
  primitive/component vocabulary, but cannot add renderer code.
- The **agent author / LLM harness** consumes the catalog and emits primitives
  and components.
- The **end-user** sees bounded, stable, responsive UI.

### Trigger

- A hosted Facet page needs richer UI than low-level primitives alone can express,
  while keeping the closed vocabulary and renderer safety model.

### Steps

1. The platform exposes a catalog containing primitive bricks and components.
2. Intrinsic components are provided by Facet/platform code and validated in the
   closed vocabulary.
3. Tenant-defined composition components are loaded as declarative definitions
   that expand into allowed primitives/components only.
4. The agent chooses catalog components first and uses primitive bricks only when
   no suitable component exists.
5. The renderer places each child inside its parent according to a shared layout
   contract. Parent layout affects immediate children; children can override
   their own internal layout only within the parent's bounds.
6. Malformed, disallowed, or too-large component definitions are rejected or
   skipped fail-safely, with no raw HTML/JS/CSS and no client-side fetch.

## Input/Output Examples

### Example 1 (Normal)

- Input: catalog exposes primitive bricks plus intrinsic components `form`,
  `table`, `chart`, `keyValue`, `emptyState`, and `metric`.
- Agent request: "Show a compact customer detail page with billing status,
  activity, and an update form."
- Output: agent emits a component-first stage using `section`, `keyValue`,
  `metric`, `table`/`list`, and `form`. The renderer keeps all children inside
  their parent containers, and primitive fallback is used only for small custom
  layout details.

### Example 2 (Hosted user-defined component)

- Input: tenant defines `customerSummaryCard` as a composition component using
  `card`, `badge`, `keyValue`, and `button`.
- Output: Facet validates the definition, exposes it in the catalog, and expands
  or renders it through the same safe primitive/component path. The tenant does
  not provide JavaScript, CSS, HTML, or renderer code.

### Example 3 (Edge/Error)

- Input: tenant attempts to define `customCalendar` with raw HTML, arbitrary CSS,
  or a component name not allowed by the catalog; or a child component emits a
  table wider than its parent.
- Output: invalid definition is rejected/skipped with issues; the page keeps
  rendering. Oversized content is bounded by renderer-owned containment rules
  such as `max-width: 100%`, `min-width: 0`, wrapping, or an explicit scroll
  region.

## Constraints (What Not to Do)

- Do not remove primitive bricks. They remain the universal fallback and the
  lowest-level safe composition layer.
- Do not let hosted tenants add intrinsic components at runtime. Intrinsics are
  Facet/platform-managed capabilities.
- Do not allow user-defined components to contain raw HTML, JS, CSS, pixel
  values, arbitrary colors, arbitrary positioning, arbitrary z-index, or
  arbitrary client fetch/data binding.
- Do not make the agent reason about high-level brick vs stamp. The agent-facing
  noun is `component`.
- Do not overload code-level theme `recipes` with component-definition semantics.
  Theme recipes remain style recipes. User-defined component structure should use
  a distinct internal term such as `componentDefinition` or `composition`.
- Do not ship a large default library of structural composition components from
  `@facet/assets` as if it were core vocabulary. If Facet provides a UI noun
  out-of-the-box, it should be an intrinsic component or an example/demo asset,
  not a hidden default composition contract.
- Do not include unconstrained overlay components in this first pass. `modal`,
  `drawer`, and `menu` require a separate constrained overlay feature.
- Do not move domain/backend work into Facet. Components display data supplied by
  the agent; business calls stay in the agent's own tools.

## Policy & Edge Cases

| Scenario Step | Failure Policy | Boundary/Validation | Concurrency / Duplicate Trigger |
|---|---|---|---|
| Catalog exposure | Missing/malformed catalog falls back to safe defaults | catalog names/components bounded and validated | loaded per asset snapshot; deterministic order |
| Intrinsic component use | Unknown/disallowed intrinsic rejected by tools or skipped by renderer | intrinsic names are in the closed core/platform vocabulary | normal serialized patch flow |
| Composition definition load | Invalid composition skipped with issues; no partial unsafe registration | definitions may reference only allowed primitives/components; no raw markup/code/styles | loaded as immutable asset snapshot for a run/session |
| Composition expansion/render | Expansion overflow rejected before partial output | ids remapped; unreachable/dangling nodes dropped; patch batch caps apply | expansion uses server-side/runtime path, not client execution |
| Hosted tenant extension | Tenant can add composition only, not renderer code | tenant definitions pass validators and catalog policy | no new writer; platform controls deployment |
| Parent/child layout | Child that would push outside parent is bounded or scrolls only in explicit scroll regions | renderer applies shared containment rules to all primitive/component roots | no local state unless existing action model is used |
| Local interactions | Use existing `agent`, `navigate`, `toggle`, and field `collect` semantics | no new client-side logic DSL in component definitions | existing two-writers mitigation applies |
| Deferred overlays | Overlay requests remain unsupported unless a dedicated overlay intrinsic exists | no general `position`/`zIndex` escape hatch | separate feature must address focus/stacking/coherence |

## Done Criteria (Testable)

- [ ] **DC-001 (Agent-facing vocabulary):** docs, catalog prompt text, and stage
  guidance describe the model as `Primitive Brick -> Component -> Catalog`, not
  as separate high-level-brick and stamp concepts for the agent. -> Verify:
  docs/prompt grep and prompt-kit tests.
- [ ] **DC-002 (Primitive fallback preserved):** `box`, `text`, `media`, and
  `field` remain valid fallback primitives and are still accepted by
  `validateTree` and rendered by `@facet/react`. -> Verify: existing primitive
  tests plus a catalog fallback test.
- [ ] **DC-003 (Intrinsic boundary):** intrinsic components are closed,
  validated, renderer-supported capabilities managed by Facet/platform code; a
  hosted tenant cannot add a new intrinsic through catalog assets; each
  intrinsic passes the documented criteria. -> Verify: validator/catalog tests
  that reject unknown intrinsic component kinds and a spec/docs checklist for the
  criteria.
- [ ] **DC-004 (Composition boundary):** user/operator-defined composition
  components can reference only allowed primitives/components and cannot contain
  raw HTML, JS, CSS, scalar style values, arbitrary URLs for data binding, or
  renderer code. -> Verify: core/runtime asset validation tests.
- [ ] **DC-005 (Initial intrinsic set):** the first intrinsic component set is
  explicitly locked before implementation and exposed consistently through
  catalog/spec/prompt/docs: `button`, `section`, `card`, `divider`, `tabs`,
  `nav`, `form`, `search`, `filterBar`, `table`, `chart`, `list`, `keyValue`,
  `metric`, `progress`, `badge`, `alert`, `emptyState`, and `loading`; overlay
  components and app/page templates are deferred. -> Verify: spec-bridge
  checklist and generated tests for every included intrinsic.
- [ ] **DC-006 (Composition extension only):** composition components are an
  operator/tenant extension mechanism, not a default core component library.
  `@facet/assets` may include example/demo compositions for quickstart guidance,
  but the default supported UI nouns exposed by Facet are intrinsic components.
  -> Verify: asset/catalog tests and docs distinguish intrinsic defaults from
  optional operator compositions.
- [ ] **DC-007 (Renderer layout contract):** React renderer has a documented and
  tested containment contract: parent owns immediate-child placement; child owns
  internal layout; renderer enforces `box-sizing`, bounded width, `min-width: 0`
  where needed, wrapping, and no general absolute/fixed positioning. -> Verify:
  `@facet/react` render tests and docs.
- [ ] **DC-008 (No parent overflow by default):** long text, media, charts,
  tables, progress bars, and nested components do not widen the page or escape
  parent bounds unless inside an explicit renderer-owned scroll region. -> Verify:
  React render tests with hostile/large content.
- [ ] **DC-009 (Override discipline):** parent layout settings affect placement
  of immediate children, while child-selected layout/variant/style affects only
  the child's own root/internal layout within the parent bounds. -> Verify:
  React tests for nested row/column/grid cases.
- [ ] **DC-010 (Fail-safe):** malformed components, unknown component names,
  invalid composition references, and hostile nested definitions never throw and
  never inject; they reject, skip, or degrade to plain UI with bounded issues.
  -> Verify: core validator tests and React renderer tests.
- [ ] **DC-011 (Compatibility/migration):** existing public terminology and
  nodes from the current catalog/high-level-brick/stamp system have a documented
  migration path. Existing valid trees should not silently break during the
  transition unless a breaking change is explicitly approved. -> Verify:
  migration tests or compatibility tests selected in spec-bridge.
- [ ] **DC-012 (No backend leak):** components never add client-side domain fetch
  or business operations. Any data-bearing component renders data supplied by the
  agent. -> Verify: schema/spec tests and docs review.

## Invariant Fit (Facet-specific - REQUIRED)

| # | Invariant | Status | Note / mitigation |
|---|---|---|---|
| 1 | Responsibility boundary (UI-out/UI-in only; no backend/domain) | OK | This is a UI vocabulary and renderer-contract change. Data and business calls remain in the agent's tools. |
| 2 | Mechanism vs policy (agent authors; Facet provides capability) | TOUCHES | Catalog can recommend and restrict components, but the agent still authors the page. Mitigation: Facet provides primitives/components/catalog policy; operators choose catalog contents. |
| 3 | Fail-safe (never throw/inject; degrade to plain) | TOUCHES | Component definitions and renderer containment add new malformed-input surfaces. Mitigation: bounded validation, skip/reject behavior, no code injection, renderer tests DC-007 through DC-010. |
| 4 | Declarative + tokens only (no raw HTML/JS/CSS/pixels) | TOUCHES | Composition components must remain declarative and token-only. Mitigation: user-defined structures reference allowed primitives/components only; theme style recipes stay separately validated operator data. |
| 5 | Flow-only safety (overlap only via a dedicated overlay brick) | TOUCHES | Layout contract directly strengthens this invariant. Overlay components are deferred unless introduced as dedicated constrained intrinsics with their own feature intake/spec. |
| 6 | Two-writers coherence (local execution vs agent stage) | TOUCHES | Components reuse existing `agent`, `navigate`, `toggle`, and field `collect` behavior. No new local execution DSL is included in this brief. Any future local component state must document coherence. |
| 7 | Backend calls go through the agent, never client fetch | OK | No component may fetch domain data from the browser. Data-bearing components render agent-supplied payloads only. |

## Open Questions / Assumptions

- **Assumption:** use `composition` or `componentDefinition` internally for
  user-defined structural components, while reserving `recipe` for theme style
  recipes. Owner: owner. Checkpoint: spec-bridge API naming.
- **Assumption:** existing `stat` should become/alias to agent-facing `metric`.
  Owner: owner. Checkpoint: spec-bridge compatibility plan.
- **Open Question:** should composition expansion happen server-side into normal
  nodes, or can the renderer render composition references after validation?
  Owner: spec-bridge. Checkpoint: architecture/design spec.
- **Open Question:** should current stamps be preserved as an internal expansion
  mechanism for operator-defined composition components, or replaced by a new
  composition-definition shape? Owner: spec-bridge. Checkpoint: migration plan.
- **Open Question:** exact renderer containment implementation: global wrapper
  class/style helper vs per-intrinsic root style contract. Owner: spec-bridge.
  Checkpoint: React renderer work-unit design.
