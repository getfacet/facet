# Feature Intake Brief: Coding-Agent Documentation Entrypoint

> Generated by `/feature-intake`. Plain-language, testable, invariant-checked.

## Goal

- Make the root README the clear first entrypoint for a developer or coding agent
  evaluating Facet, so it can understand what Facet is, decide whether it fits,
  choose the right integration path, and follow an explicit link to the exact
  detailed guide it needs without reverse-engineering package source or tests.

## Current Problem

- Facet's current architecture, design-system contract, package roles, and agent
  tool behavior are documented accurately, but the information is distributed
  across the root README, architecture and migration documents, the tool-result
  contract, and package READMEs.
- The root README explains the system but does not yet act as a strong decision
  page: a first-time coding agent must infer which package combination and which
  detailed document match its integration scenario.
- Stable terminology is currently clearest in a migration document, while
  operator-facing Theme/Preset/Pattern examples and complete cross-package
  integration guidance remain thin.
- Historical specs intentionally retain retired concepts and paths. A coding
  agent using repository search can mistake those records for the current
  contract unless the archival boundary is explicit.
- The affected user is a developer using a coding agent such as Codex or Claude
  Code to evaluate or integrate Facet. Maintainers and human developers benefit
  from the same navigation, but the primary usability target is the coding
  agent reading repository documentation.

## User Scenario

### Who

- Primary: a framework consumer's coding agent evaluating or integrating Facet.
- Secondary: the developer directing that coding agent and Facet maintainers
  keeping public documentation consistent.

### Trigger

- The coding agent opens the Facet repository or an npm package README and needs
  to determine what Facet does and how to use it in the consumer's project.

### Steps

1. The coding agent reads the root README first.
2. The README explains Facet's purpose, execution flow, essential concepts,
   safety boundaries, quickstart, and use-case-oriented package choices without
   becoming a full reference manual.
3. The agent selects its scenario, such as trying Quickstart, embedding the
   React renderer, bringing a custom LLM loop, connecting an external agent,
   using AG-UI, or adding durable storage.
4. The README links the agent to one detailed document or package README whose
   responsibility is explicit and non-overlapping.
5. The detailed document provides the complete integration or contract guidance
   and links back to the root entrypoint and to directly related references.
6. The coding agent can produce an implementation plan without searching tests
   or treating historical specs as current authority.

## Input/Output Examples

### Example 1 (Normal)

- Input: A coding agent is asked, "Evaluate Facet for our React application with
  our own LLM loop. Which packages and documents should we use?"
- Output: From the README alone, it identifies `@facet/react`,
  `@facet/agent-tools`, the required Core/default-asset dependencies, and the
  correct Getting Started, Design System, and Agent Integration links. It can
  explain the boundary between Facet Document authoring, Theme resolution, and
  host-owned provider/business policy before proposing code.

### Example 2 (Edge/Error)

- Input: A coding agent searches the repository for an old term such as
  `Composition` or an old package path and finds it in a completed historical
  spec.
- Output: Documentation clearly marks historical specs as archival and points
  to `AGENTS.md`, the root README, `docs/ARCHITECTURE.md`, and current Core
  contracts as authoritative. Current public documentation contains no retired
  term presented as a live Facet concept, and all navigation links resolve.

## Constraints (What Not to Do)

- Do not change runtime behavior, public APIs, npm package names, package
  boundaries, Facet Document syntax, Brick/style contracts, tool schemas, or
  validation behavior.
- Do not make the README an exhaustive copy of Architecture, Design System, or
  Agent Tool Result Contract details.
- Do not maintain a hand-written exhaustive table of every Brick style property
  and token choice; Core and progressive discovery remain authoritative.
- Do not add `llms.txt`, a new runnable example application, or a new package in
  this scope. Embedded examples may be added when they use current public
  surfaces and have a concrete drift check.
- Do not rewrite historical feature/dev specs to current terminology. Mark the
  archival boundary instead.
- Do not introduce a second terminology source that can diverge from Core and
  `docs/ARCHITECTURE.md`.
- Do not add backend fetching, client-side domain logic, raw HTML/JS/CSS, or any
  new UI capability as part of documentation work.

## Policy & Edge Cases

| Scenario Step | Failure Policy | Boundary/Validation | Concurrency / Duplicate Trigger |
|---|---|---|---|
| Read root README | If a claim conflicts with Core or Architecture, the documentation gate fails and the claim must be corrected before merge. | README stays decision-oriented and links to details rather than duplicating full contracts. | Repeated concepts across README and detail docs must use the same canonical terms and ownership statements. |
| Choose an integration path | If a use case has no unambiguous path, add or correct the decision table and its target link; do not make the reader infer from directory names. | Only existing packages and supported integration paths may be recommended. Reference implementations must be labeled accurately. | Multiple valid paths may be shown, but their selection criteria must distinguish them. |
| Follow detail links | Broken, circular-without-content, or misleading links block completion. | Every new or changed local link must resolve from its source file; package READMEs link to the relevant root guide. | Duplicate links are acceptable only when they serve different entry contexts and share the same canonical target. |
| Use examples | An unverifiable or stale example blocks completion or must be reduced to pseudocode explicitly labeled as such. | Code examples use public exports and current package paths; no raw CSS or retired concepts appear as supported syntax. | The same example must not be copied into several documents unless generated or covered by one drift source. |
| Encounter historical specs | Historical content remains unchanged, but the current-vs-archival boundary must be visible before it can be mistaken for authority. | Current docs and package READMEs remain subject to hard-cut scanners; archival specs are not rewritten. | Parallel repository searches may surface old and new terms; the authority order must resolve the ambiguity deterministically. |

## Done Criteria (Testable)

- [ ] DC-001 (Happy path): The root README lets a first-time coding agent explain
      Facet's purpose, main data flow, essential concepts, two safety invariants,
      and pre-1.0 status without opening source files. → Verify: documentation
      review against `AGENTS.md`, `docs/ARCHITECTURE.md`, and current Core terms.
- [ ] DC-002 (Decision path): The root README contains a use-case-oriented table
      covering Quickstart, custom LLM loop, in-process agent, external agent,
      React renderer, default assets, reference transport, AG-UI, and Postgres,
      with a working next link for each. → Verify: table audit plus local-link
      checker.
- [ ] DC-003 (Information architecture): Detailed documentation has explicit,
      non-overlapping homes for getting started/integration, architecture,
      design-system concepts and operator assets, agent integration, tool-result
      protocol, package boundaries, and migration. → Verify: heading/ownership
      review and duplication sweep.
- [ ] DC-004 (Design-system clarity): A coding agent can distinguish Facet
      Document, Brick, style property/target, token, fixed choice, Theme, Preset,
      Pattern, Patch, Runtime, and Renderer, including who authors or resolves
      each. → Verify: canonical terminology table and cross-document consistency
      review.
- [ ] DC-005 (Integration guidance): Detailed guides include enough current,
      copy-oriented guidance for a coding agent to plan a minimal React setup and
      a custom-agent-tools setup without reading implementation tests. → Verify:
      public-export/package-path audit and example review.
- [ ] DC-006 (Navigation): Root and package READMEs link to the relevant detailed
      guides, and detailed guides link back to the root entrypoint and directly
      related references. → Verify: automated repository-local Markdown link
      check plus manual navigation matrix.
- [ ] DC-007 (Boundary/error): Historical specs are explicitly marked archival,
      while current documentation and package READMEs contain no retired concept,
      API, asset shape, or package path presented as current. → Verify:
      `check-style-hard-cut`, `check-package-layout`, targeted `rg`, and the
      archival notice review.
- [ ] DC-008 (No behavior change): No production TypeScript, public export,
      package manifest contract, runtime behavior, or generated protocol changes
      are introduced. → Verify: changed-file audit, `pnpm verify`, and
      `pnpm package:smoke` if package README/package surfaces are included by the
      final spec.
- [ ] DC-009 (No broken examples): New concrete code examples use current public
      imports and cannot silently drift. → Verify: executable/typechecked doc
      snippet test where practical, otherwise an explicit spec decision to label
      non-executable material as pseudocode and validate symbols/paths.
- [ ] DC-010 (Scope control): README remains a concise entrypoint; exhaustive
      style vocabulary, full result schemas, and migration mechanics stay in
      their canonical detailed locations. → Verify: section responsibility
      checklist and review for copied contract blocks.

## Invariant Fit (Facet-specific — REQUIRED)

| # | Invariant | Status | Note / mitigation |
|---|---|---|---|

| 3 | Fail-safe (never throw/inject; degrade to plain) | OK | The existing strict-author/fail-soft-render distinction is preserved and explained; no behavior changes. |

| 5 | Flow-only safety (overlap only via a dedicated overlay brick) | OK | No new layout or rendering capability is proposed. |
| 6 | Two-writers coherence (local execution vs agent stage) | OK | Documentation preserves the existing ownership explanation; no local execution is added. |
| 7 | Backend calls go through the agent, never client fetch | OK | Integration guidance must explicitly keep provider/business/backend tools outside Facet and add no browser fetch surface. |
