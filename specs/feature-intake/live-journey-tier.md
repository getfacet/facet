# Feature Intake Brief: /live-test "Live Journey" Tier (real browser + real LLM + judged screenshots)

> Generated by `/feature-intake`. Plain-language, testable, invariant-checked.

## Goal

Add a new top tier to `/live-test` that **proves the actual Facet experience
works end-to-end** — a real LLM draws a page in a **real (headless) browser**, a
visitor drives a fixed journey (load → live chat edits → an interaction),
**screenshots are captured, and LLM/vision judges assess them** — so a change
that visually or interactively breaks the product is caught before merge, which
the current jsdom/stub tiers cannot catch.

Outcome: a maintainer can run one command pre-merge and get a trustworthy
PASS/FAIL (plus a GIF) that the live, generative, per-visitor, interactive Facet
experience actually renders and responds — not just that the code compiles and a
tree has the right shape.

## Current Problem

- **The gate never exercises the real product.** `/live-test` Tier 1b runs the
  built page bundle in **jsdom** (a simulated DOM) — it proves the bundle mounts
  without throwing, NOT that it visually renders or that interactions work. Tier 2
  makes ONE real-LLM turn and only asserts the returned tree is a box with
  children (shape, not experience). Nothing drives a real browser through a user
  journey or looks at what actually painted.
- **The published bin's standalone run isn't gated.** `node dist/cli.js` fails in
  the dev monorepo (workspace `@facet/*` resolve to `src/*.ts`) — a real gap hit
  in practice — and no tier catches it.
- **"Different per visitor", "live updates", and "interactions" are claimed but
  never verified as an experience.**
- Affected: the **maintainer** (can merge a change that silently breaks the real
  experience) and, downstream, **OSS visitors** (a broken quickstart is Facet's
  first impression).

## User Scenario

### Who
- The **maintainer** running the gate before merging a change (especially one
  touching `packages/quickstart/`, `@facet/react`, the renderer, or the protocol).

### Trigger
- Pre-merge / on-request (heavier than the per-change vitest tiers — a real LLM +
  browser + vision judging).

### Steps
1. Maintainer runs `/live-test` (pre-merge).
2. The existing fast vitest Tiers 1a/1b/2 run as today (unchanged).
3. `/live-test` then invokes the new **live-journey Workflow**.
4. **Preflight**: the workflow's harness boots the real-LLM quickstart server
   (OpenAI, gpt-5.4-mini) and runs a `node dist/cli.js` standalone-run smoke.
5. **Fan-out**: 3 "visitor" runs, each a FRESH `visitorId`, each driving the
   FIXED journey via **Playwright headless** and capturing screenshots:
   (1) load → shot; (2) chat "Add a section titled Pricing with three simple plan
   cards" → wait for the UI to update → shot; (3) chat "Now restyle the whole page
   with a playful, cat-themed look" → wait → shot; (4) click the most prominent
   button/link → shot.
6. **Judge**: LLM/vision judges assess the captured screenshots (semantic, loose
   criteria — NOT golden-image).
7. **Synthesize**: aggregate all visitor + judge verdicts → PASS / FAIL + a
   summary + an optional GIF; teardown the server.
8. Maintainer sees the verdict + artifacts and decides to merge.

## Input/Output Examples

### Example 1 (Normal)
- Input: a healthy branch; `/live-test` run pre-merge with `OPENAI_API_KEY` set.
- Output: 3 visitors render coherent, distinct pages; each chat visibly updates
  the UI; the click responds; no broken/blank/raw state anywhere → **tier PASS**
  + a GIF of one journey + the per-lens judge summary.

### Example 2 (Edge/Error)
- Input: a change that makes the renderer paint a blank canvas / raw JSON / an
  overlapping-off-screen layout on one visitor.
- Output: the **safety/fail-safe judge flags it** → **tier FAIL**, naming the
  offending visitor + screenshot; the GIF is still produced for inspection.
- Input variant: the LLM simply doesn't add a "Pricing" section (ignored the
  request) but rendering + safety + responsiveness are fine → **tier PASS with a
  request-fidelity WARNING** (soft lens), not a FAIL.
- Input variant: the server won't boot, or the `node dist/cli.js` smoke fails →
  **preflight FAIL** (no visitor runs).

## Constraints (What Not to Do)

- **Non-goal: run in CI / on every push.** This tier is LOCAL, owner-run,
  pre-merge/on-request (real LLM cost + browser). The per-change gate stays the
  vitest Tiers 1-2.
- **Non-goal: golden-image / pixel comparison.** Judging is semantic (LLM/vision)
  because the LLM output is non-deterministic.
- **Non-goal: replace Tiers 1-2.** This is additive; the fast deterministic tiers
  remain the first line.
- **Non-goal: agentic browser driving.** Driving is scripted Playwright headless
  (deterministic infra); only the JUDGING is LLM.
- **No change to any `@facet/*` runtime/protocol/renderer surface.** This is TEST
  TOOLING — it boots and observes the existing quickstart; it adds no brick,
  token, action, protocol message, or backend affordance.
- **No secrets in artifacts.** Screenshots/GIF must not embed the provider key or
  the raw `visitorId` beyond what the page already shows.

## Policy & Edge Cases

| Scenario Step | Failure Policy | Boundary/Validation | Concurrency / Duplicate Trigger |
|---|---|---|---|
| Preflight boot + node-bin smoke | Boot fail or smoke fail → preflight FAIL, no visitors run | Server must answer HTTP 200 within a timeout | One shared server; each visitor a separate browser context |
| Visitor journey (Playwright) | Page won't load / step times out → that visitor FAIL (recorded), others continue | ≥4 screenshots captured per visitor; per-step wait for a settled DOM (bounded timeout) | 3 fresh `visitorId`s → 3 independent sessions/stages, no shared client state |
| Judge assessment | A judge agent dies/abstains → treated as abstain; a lens needs a quorum of valid votes (no silent pass) | Loose semantic criteria; verdict per lens per visitor | Judges run in parallel; independent per (visitor × lens) |
| Verdict aggregation | HARD lens fail on ANY visitor → tier FAIL; SOFT lens fail → WARNING only | HARD = safety + render + responsiveness; SOFT = request-fidelity + cross-visitor diversity | N/A (single synthesis) |
| Teardown | Server killed even on failure/exception (no orphan process) | Port freed | N/A |

## Done Criteria (Testable)

- [ ] DC-001 (Happy path): on a healthy `main`, the live-journey tier boots the
  server, runs 3 visitors through the fixed journey capturing ≥4 screenshots each,
  the HARD lenses (safety/render/responsiveness) pass for all 3, and the tier
  reports **PASS** with artifacts. → Verify: run the tier on `main`; observe PASS
  + the screenshot/GIF artifacts.
- [ ] DC-002 (Boundary/Error — safety catches a break): given a deliberately
  broken render (a negative fixture that paints blank/raw/overlap), the safety
  judge flags it and the tier reports **FAIL** naming the offending screenshot. →
  Verify: a negative-fixture run (or an injected broken stage) → tier FAIL.
- [ ] DC-003 (Soft lens does NOT hard-fail): given a run where request-fidelity is
  low (the LLM ignored the request) but safety+render+responsiveness pass, the
  tier reports **PASS with a WARNING** (not FAIL). → Verify: unit test of the
  verdict-aggregation function with soft-lens-fail + hard-lens-pass inputs → PASS+warn.
- [ ] DC-004 (Judge quorum, no silent pass): if judge agents for a HARD lens
  abstain/die below quorum, the aggregation does NOT report PASS for that lens. →
  Verify: unit test of the aggregation function with abstaining votes.
- [ ] DC-005 (Preflight bin smoke): the `node dist/cli.js` standalone-run smoke
  runs and its result is reported in the tier output (pass = boots; fail = the
  documented dev-resolution error surfaced). → Verify: the preflight step output.
- [ ] DC-006 (Additive + teardown): the existing vitest Tiers 1a/1b/2 still run
  and pass unchanged, and the tier leaves no orphan server process. → Verify: run
  `/live-test`; confirm Tiers 1-2 results + no listening process on the port after.

## Invariant Fit (Facet-specific — REQUIRED)

This feature is **test tooling** — it boots and OBSERVES the existing quickstart;
it changes no framework surface. Every invariant is therefore `OK`.

| # | Invariant | Status | Note / mitigation |
|---|---|---|---|
| 1 | Responsibility boundary (UI-out/UI-in only; no backend/domain) | OK | Adds no backend affordance; boots the existing server and drives the browser. No `kind:"fetch"`, no data binding. |
| 2 | Mechanism vs policy (agent authors; Facet provides capability) | OK | No authoring moved into the framework; the tier only exercises + judges the existing agent/renderer. |
| 3 | Fail-safe (never throw/inject; degrade to plain) | OK | The tier *verifies* fail-safe (the safety judge). It adds nothing to the render path. |
| 4 | Declarative + tokens only (no raw HTML/JS/CSS/pixels) | OK | No new brick/token/action; the Playwright script + judges are dev tooling, not tree content. |
| 5 | Flow-only safety (overlay only via dedicated brick) | OK | No overlay added; the tier checks for off-screen/overlap as a failure signal. |
| 6 | Two-writers coherence (local execution vs agent stage) | OK | No new local execution; the tier drives the existing chat/press events through the real transport. |
| 7 | Backend calls go through the agent, never client fetch | OK | The tier makes no client-side domain fetch; it drives the browser against the local server. |

## Open Questions / Assumptions

- **OQ-1 (owner: maintainer, checkpoint: /spec-bridge):** exact per-step
  wait/settle strategy after a chat send — poll the DOM for a change with a
  bounded timeout, vs a fixed sleep. Affects flakiness.
- **Assumption:** the workflow can boot/teardown the server via a preflight
  agent (the workflow script itself has no shell); the `/live-test` skill or a
  preflight agent owns lifecycle. Checkpoint /spec-bridge.
- **Assumption:** `OPENAI_API_KEY` present locally (owner has it); a missing key
  makes this tier a SKIP-with-reason for a local run (it is not a CI gate).
