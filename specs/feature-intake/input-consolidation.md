# Feature Intake Brief: Input Consolidation (`field`+`search` → one `input` brick)

> Generated by `/feature-intake`. Plain-language, testable, invariant-checked.
> PR-4 of the node-model restructure. A **hard-cutover consolidation** (breaking
> OK — pre-deploy, no shims), atomic across core + react + agent + assets.

## Goal

Collapse the input-capturing vocabulary from two node types to one. Rename the
primitive `field` brick to **`input`**, and **remove the separate `search`
component type** — `input` already carries a `"search"` input kind, so a search
box is just `input` with that kind; a search box *with a submit button* becomes a
composition (`input` + a `button`), not a bloated single node. Fewer node types,
one clear input brick, consistent with the brick-vs-field growth rule.

## Current Problem

- **Who is affected:** the agent author (two overlapping ways to ask for input is
  confusing) and the framework's own coherence (vocabulary sprawl the cutover is
  meant to remove).
- **Concrete pain:** today `field` (primitive: `name`, `input?` kind incl. a
  `"search"` kind, `options`, `label`, `placeholder`, styleable) and `search`
  (component: `name`, `label`, `placeholder`, `value`, `submitLabel`, `variant`,
  `onSubmit`) both capture user input. `field` already covers the search *input*
  via its `"search"` kind; `search` only adds a submit affordance
  (`submitLabel`/`onSubmit`/`value`). Two node types for one job, and the name
  `field` is less clear than `input`. This is the first of the cutover PRs that
  removes the "component" tier duplication.

## User Scenario

### Who
The agent author drawing an input; the end-user typing into it.

### Trigger
The agent wants to capture text/selection/search input from the visitor.

### Steps
1. Agent emits an `input` node (`{ type:"input", name, input?:<kind>, ... }`) —
   the same shape `field` had, just renamed. For a search box it uses
   `input:"search"`.
2. The renderer draws the native input; the visitor types; the value is captured
   through the existing field/collect mechanism (unchanged).
3. For "search + submit", the agent composes: an `input` (kind `"search"`) next
   to a `button` whose `onPress` fires the agent action — a composition, not a
   special node.

### Steps (this PR's own change — hard cutover)
1. Rename `field` → `input` everywhere: `type:"field"`→`type:"input"`,
   `FieldNode`→`InputNode`, `FieldInput`→`InputKind`, `FieldStyle`→`InputStyle`,
   `FIELD_INPUTS`→`INPUT_KINDS`, validators/renderers/registry/prompt/spec.
2. Remove the `search` node type entirely (type, `SearchNode`, its validation,
   renderer, executor entry, stage-summary handler, STAGE_SPEC line, catalog
   entry, print-tree case).
3. Migrate any seed/playground/quickstart authored trees that used `field` or
   `search` to `input`.
4. Existing `field`/`search` tests become `input` tests (mechanical); behavior
   preserved under the new name; the removed `search` submit affordance is
   dropped (expressible as `input`+`button`).

## Input/Output Examples

### Example 1 (Normal — a renamed input)
- Input: `{ type:"input", name:"email", input:"email", label:"Email" }`.
- Output: renders identically to the old `field` with the same props — value
  captured the same way. (Byte-identical behavior, new type name.)

### Example 2 (Normal — search input)
- Input: `{ type:"input", name:"q", input:"search", placeholder:"Search…" }`.
- Output: a search-styled input (what `field input:"search"` already produced);
  no submit button (add a `button` for that).

### Example 3 (Edge — removed `search` type degrades)
- Input: a stale tree still containing `{ type:"search", … }`.
- Output: the renderer treats `search` as an unknown type and **skips it
  (blank-degrades), never throws** — acceptable pre-deploy (hard cutover, no
  compat alias).

### Example 4 (Edge — unknown input kind)
- Input: `{ type:"input", name:"x", input:"telepathy" }`.
- Output: the unknown kind degrades to the default text input (carried over from
  `field`'s existing fail-safe); never throws.

## Constraints (What Not to Do)

- **No compat alias for `search` or `field`.** Hard cutover — the old type names
  are removed, not aliased. (Pre-deploy; breaking is free.)
- **No submit affordance on `input`.** Do NOT carry `submitLabel`/`onSubmit`/
  `value` onto `input` — "search with submit" is a composition (`input`+`button`),
  keeping `input` a single input control.
- **filterBar is OUT of scope** — it becomes a composition in PR-5/6, untouched
  here.
- **No repo-wide uniform-styling change** — `field`/`input` is already a styleable
  primitive; the "components can't take inline style" asymmetry is fixed in PR-6
  when the surviving component bricks are reclassified. Do not sweep styling here.
- **Atomic removal.** The `search` removal + `field`→`input` rename must land in
  ONE PR across core + assets + react + agent (+ seed/playground), or the renderer
  blank-degrades a half-removed type (PR-0 lesson).
- **No new capability.** `input` = `field`'s exact shape, renamed. No new fields.

## Policy & Edge Cases

| Scenario Step | Failure Policy | Boundary/Validation | Concurrency / Duplicate Trigger |
|---|---|---|---|
| Author emits `input` | Malformed → sanitized like `field` was (drop bad props), never throw | `input` kind ∈ the existing closed `INPUT_KINDS` set (incl. `search`) | Value capture unchanged from `field` |
| Unknown input kind | Degrade to default text input (carried from `field`) | Closed kind set | N/A |
| Stale `search`/`field` type in a tree | Renderer skips unknown type, blank-degrades, never throws | No `search`/`field` type remains in the vocabulary | N/A |
| Removal atomicity | If any package lags, that surface blank-degrades — so land atomically | core+assets+react+agent+seed all updated together | N/A |
| "search + submit" need | Expressed as `input`+`button` composition (no single-node submit) | — | Button's `onPress` fires the action |
