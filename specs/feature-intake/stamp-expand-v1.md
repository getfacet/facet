# Feature Intake Brief: Stamp Reference-Expand v1 (use_stamp + slot fill)

> Generated by `/feature-intake`. Theme 2 of `facet-brick-vocab-roadmap`.
> **Implementation in Codex — this chain stops at `/spec-bridge`.**

## Goal

Make stamps **scale as reusable, parameterized building blocks**: the LLM
references a stamp **by name** via a `use_stamp(name, params, at)` tool, and the
runtime **expands it server-side into ordinary brick patches**, filling **named
slots** with `params`. Seed a few default stamps (Card/Hero/…) to prove it and give
immediate value. This lets Facet keep a **minimal brick palette** while covering
rich UI through **stamp composition** — without bloating the prompt.

## Current Problem

Stamps today are **prompt-copy**: the full `{root, nodes}` brick JSON of every
stamp is injected into the prompt (cap `MAX_STAMP_PROMPT_CHARS`, oversized
**dropped**), and the LLM **copies the bricks itself** — there is **no server
expansion** (`packages/core/src/validate.ts` ~ "prompt-copy only: stamps are never
expanded server-side or client-side"). This does **not scale**: many/long stamps
blow the prompt budget and degrade the model, oversized stamps silently vanish, and
"the model re-copies bricks" is unreliable vs a deterministic macro. **Affected:**
the agent author (can't ship a real stamp library) and the model (prompt bloat).

## User Scenario

### Who
The agent author (defines stamps as branded, reusable pieces) and, indirectly, the
end-user (sees consistent stamped UI).

### Trigger
The agent, mid-turn, wants a known composite (e.g. a product Card) filled with
specific content.

### Steps
1. The agent calls `use_stamp("card", { title: "iPhone 15", body: "$999", img: "…" }, at)`.
2. The runtime **loads** the `card` stamp from the AssetsStore, **fills its named
   slots** with `params`, **remaps its internal node ids to fresh unique ids**, and
   emits ordinary **brick patches** that insert the subtree at `at`.
3. The expansion is **coalesced into the turn's flush** (and, under streaming, rides
   the current per-step batch as a **referentially-closed** subtree).
4. `use_stamp` **returns the new node ids** so the agent can further edit the stamped
   subtree with ordinary `set_node`/`append` (that is where full freedom lives —
   slots cover the common fill; anything else is a normal follow-up edit).

## Input/Output Examples

### Example 1 (Normal)
- Input: `use_stamp("card", {title:"iPhone 15", body:"$999", img:"iphone.jpg"}, {parent:"root"})`
- Output: the Card subtree (box→image+title+body) appended under `root`, slots filled,
  fresh ids; returns `{root:"n12", slots:{title:"n13", body:"n14", img:"n12/img"}}`.

### Example 2 (Edge/Error)
- Input: `use_stamp("nope", …)` (unknown), or a call missing the `body` param, or a
  `params.title` that is a huge/typed-wrong value.
- Output: **unknown stamp → no-op** (issue logged, turn continues, like an unknown
  theme name); **missing slot → its default/empty** (fail-safe); **bad param →
  sanitized** (string/token, length-capped) or dropped with an issue. Never throws.

## Constraints (What Not to Do)

- **No stamp node in the tree** — the reference travels on the **tool channel**
  (like `set_theme`), never as a node type. The 4-brick tree invariant is untouched;
  the expanded output is only ordinary bricks.
- **No new brick.** Stamps are compositions of the existing four.
- **Slots only (v1)** — params fill **named slots** the stamp author declared. **No
  override-patch params** in v1 (full freedom is already available via post-stamp
  `set_node`/`append`).
- **Server-side expansion**, never client-side. The client keeps folding ordinary
  patches; it never learns what a stamp is.
- **Fail-safe** — unknown/malformed stamp or params **degrade to a no-op**, never a
  throw, never a broken partial (renderer already skips dangling).
- **Keep the reference out of persisted history semantics** beyond the patches it
  produces — the durable log records the resulting patches (one per turn), not a new
  "stamp used" concept (unless spec decides otherwise).

## Policy & Edge Cases

| Scenario Step | Failure Policy | Boundary/Validation | Concurrency |
|---|---|---|---|
| resolve stamp name | unknown → no-op + issue | name is a bounded string | reads the turn's loaded assets snapshot |
| fill slots | missing → default/empty; extra param → ignored | param value = string/token, length-capped (reuse field caps) | — |
| remap ids | fresh unique ids per call | stamp used N× on one screen → N disjoint id-sets, no collision | within the turn's serial lane |
| emit patches at `at` | invalid parent → no-op + issue | `at` = {parent, index?} mirroring append | coalesced into the turn flush / per-step batch |
| streaming interaction | expanded subtree stays referentially closed in its batch | (child-ref + node in same batch — reuse streaming closure) | server single writer |
| size | very large stamp still expands (no prompt cap now) | optional node-count cap on expansion | — |

## Done Criteria (Testable)

- [ ] **DC-001 (expand):** `use_stamp(name, params, at)` produces ordinary brick
  patches inserting the stamp subtree at `at`. → Verify: agent/runtime test.
- [ ] **DC-002 (slot fill):** declared slots are filled from `params`; a missing slot
  falls back to its default/empty. → Verify: test.
- [ ] **DC-003 (unknown → no-op):** an unknown stamp name yields no patch + an issue,
  turn continues. → Verify: test.
- [ ] **DC-004 (id remap):** stamping the same stamp twice on one tree yields disjoint
  fresh ids (no collision). → Verify: test.
- [ ] **DC-005 (fail-safe):** malformed stamp doc / bad params → no-op, never throws;
  tree stays valid. → Verify: test.
- [ ] **DC-006 (tree invariant):** after expansion the tree contains only the 4 brick
  types; no "stamp" node exists. → Verify: validateTree/test.
- [ ] **DC-007 (returns ids):** `use_stamp` returns the new subtree/slot ids so the
  agent can follow up with `set_node`/`append`. → Verify: test.
- [ ] **DC-008 (default stamps):** a few default stamps are seeded (DEFAULT_STAMPS) and
  usable via `use_stamp` out of the box. → Verify: test.
- [ ] **DC-009 (STAGE_SPEC):** the drawing vocab advertises available stamp **names +
  slot names + description** (not full brick JSON), so the model knows what to call.
  → Verify: spec.ts + prompt test.
- [ ] **DC-010 (atomic/coalesced):** the expansion lands in the turn's single flush
  (and, with streaming, a referentially-closed batch) — no intermediate broken paint.
  → Verify: runtime/streaming test.

| 1 | Responsibility boundary | **OK** | Pure UI composition; no backend/domain surface. |
| 2 | Mechanism vs policy | **OK** | Stamp CONTENT is the agent-author's (policy, in AssetsStore); the framework only provides the expand/slot-fill MECHANISM. |
| 3 | Fail-safe | **TOUCHES** | Unknown/malformed stamp or params → no-op + issue, never throws; expanded output validated like any tree; dangling skipped. |
| 4 | Declarative + tokens only | **OK** | Stamps are brick data; slot values are strings/tokens; expansion yields only bricks — no raw markup, no new escape hatch. |
| 5 | Flow-only safety | **OK** | Stamps compose flow bricks; no overlap/positioning introduced. |
| 6 | Two-writers coherence | **TOUCHES** | Expansion is server-side, coalesced into the turn flush; under streaming the expanded subtree must be **referentially closed within its batch** (reuse the streaming closure so a forward child-ref isn't split). Server stays the single writer. |
| 7 | Backend via agent | **OK** | No client fetch; stamp defs come from the AssetsStore the runtime already loads. |

## Open Questions / Assumptions

- **Open (spec-bridge):** slot **syntax** in the stamp template — a text/attr value
  convention (e.g. `{{title}}`) vs a dedicated `slot` field on nodes. Prefer a
  bounded, validate-able marker. (Assumption: an explicit slot marker, not free
  string interpolation.)
- **Open (spec-bridge):** exact **return shape** of `use_stamp` (root id + slot→id
  map, or full id map) so the agent can address non-slotted nodes for follow-up edits.
- **Open (spec-bridge):** **streaming interaction** — the expanded subtree must be a
  single referentially-closed batch (reuse theme-4 closure); confirm `Stage.useStamp`
  emits one closed op-group. **(Depends on live-streaming-v1 being merged first — see
  sequencing note.)**
- **Open (spec-bridge):** prompt **advertisement** format + cap (names+slots+short
  desc), and an optional **node-count cap** on expansion.
- **Assumption:** param values reuse the existing field-value caps
  (`MAX_FIELD_VALUE_CHARS`, etc.) for the sanitizer. Owner. Checkpoint: spec-bridge.
