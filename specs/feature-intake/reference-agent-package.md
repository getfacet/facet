# Feature Intake Brief: Reference Agent Package

> Generated by `/feature-intake`. Plain-language, testable, invariant-checked.

## Goal

- Split the current quickstart built-in brain into a dedicated reference-agent package so `@facet/quickstart` can compose it instead of owning it, while preserving the existing `facet-quickstart` user experience.
- Create a stable home for future improvements to the reference LLM/stub agent without mixing those changes into the quickstart wrapper server, CLI shell, or browser bundle.

## Current Problem

- Today `@facet/quickstart` owns two responsibilities at once: the one-command local server/page shell and the reference LLM/stub agent implementation.
- The built-in agent code (`agent.ts`, `provider.ts`, `prompt.ts`, `stub.ts`) is large and will be improved next, but keeping it inside quickstart makes package ownership, test targeting, and public API wording blurry.
- Affected users are Facet maintainers and agent authors who want to import or iterate on the reference agent separately from the quickstart boot path. End-users of a quickstart page should see no behavior change.

## User Scenario

### Who
- Facet maintainer improving the built-in reference agent.
- Agent author using the framework who wants a reusable reference agent package.
- End-user of a `facet-quickstart` page, who should experience the same page boot and chat behavior.

### Trigger
- A maintainer decides to extract the reference brain from `@facet/quickstart` before improving it further.

### Steps
1. Maintainer adds a new package, assumed name `@facet/reference-agent`, whose responsibility is the reference LLM/stub `FacetAgent` and provider adapters.
2. `@facet/quickstart` keeps the CLI/server/page-bundle role and imports the reference agent package when resolving `--stub` or provider-backed mode.
3. Running `facet-quickstart --stub` still boots the same deterministic local fixture.
4. Running `facet-quickstart` with `OPENAI_API_KEY` or `ANTHROPIC_API_KEY` still resolves the same provider, model string, prompt/tool loop, asset names, stamps, themes, and sanitized provider errors.
5. Future work can change the reference agent package with focused tests, while quickstart tests mainly verify composition, CLI flags, shell serving, and the wrapper server.

## Input/Output Examples

### Example 1 (Normal)
- Input: `OPENAI_API_KEY=sk-... facet-quickstart --guide ./my-page.md`
- Output: The command still prints the quickstart URL and `Brain: openai (...)`; the page is still built by the same streaming tool-calling reference agent, but the agent implementation is imported from `@facet/reference-agent`.

### Example 2 (Normal Import)
- Input: A framework consumer imports `createReferenceAgent` or an explicitly documented compatibility alias from `@facet/reference-agent`, supplies a provider, `Sink`, `agentId`, guide, themes, and stamps.
- Output: The returned value is a normal `FacetAgent` usable anywhere a Facet server/runtime accepts an in-process agent.

### Example 3 (Edge/Error)
- Input: `facet-quickstart` runs with no provider key and without `--stub`.
- Output: The command exits with the same current no-key message. No key value is logged, persisted, or echoed.

### Example 4 (Boundary/Error)
- Input: `facet-quickstart --stub` after extraction.
- Output: The same deterministic stub tree and replies are produced; the Tier-1 quickstart E2E fixture remains stable except for import paths.

## Constraints (What Not to Do)

- Do not change the Facet brick vocabulary, tokens, patch protocol, renderer behavior, server protocol, or runtime event-loop semantics.
- Do not add backend/domain affordances to Facet. Provider calls remain inside the agent package as operator-supplied brain behavior, not a generic Facet data-fetch feature.
- Do not move provider SDK dependencies into the repo; keep the current raw-`fetch` adapter posture unless a separate approved feature changes it.
- Do not expose provider keys beyond environment resolution and request auth headers. Error messages must continue naming env vars only, never values.
- Do not make the browser page call providers or fetch domain data directly.
- Do not regress `facet-quickstart` CLI flags, default guide fallback, assets behavior, stub mode, or live-test expectations.
- Do not implement agent quality improvements in this extraction. The follow-up work can improve the separated agent after the package boundary is clean.

| Public API migration | Existing quickstart consumers must have either a compatibility path or documented migration. | New package should expose the canonical reference-agent surface; quickstart may re-export compatibility symbols if chosen by the dev spec. | Duplicate imports from both packages should reference the same implementation, not forked logic. |
