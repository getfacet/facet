## Overview
- Feature slug: `unified-asset-pipeline`
- Intake brief: `specs/feature-intake/unified-asset-pipeline.md`
- Context evidence: `specs/context/unified-asset-pipeline.md`
- Affected `@facet/*` packages: `@facet/assets` (NEW), `@facet/react`, `@facet/runtime`, `@facet/quickstart`, `@facet/kit` (REMOVED), `apps/playground`. `@facet/core` is **unchanged** (see RISK-PKG-4 waiver).
- One-paragraph technical approach:

  Create a browser-safe, node-free **`@facet/assets`** data package (depends on
  `@facet/core` only) that becomes the single source of truth for the default
  theme data (the five web-flavored value maps `SPACE`/`FONT_SIZE`/`FONT_WEIGHT`/
  `RADIUS`/`RATIO` + the `COLOR` map derived from core `DEFAULT_COLORS`, plus the
  assembled `DEFAULT_THEME` document) and the default stamp library (`DEFAULT_STAMPS`,
  re-expressed as literal validated `FacetStamp` trees, replacing kit-factory-derived
  `KIT_STAMPS`). `@facet/react` stops **owning** that data: its `resolveTheme` floor
  and `DEFAULT_RESOLVED` import the value maps back from `@facet/assets`, and it
  re-exports `DEFAULT_THEME`/`COLOR` for back-compat ‚Äî so zero-arg style output stays
  byte-identical. `@facet/runtime`'s `loadAssets` **seeds those defaults as a base
  layer** (validated through the same `validateStamp`/`validateTheme` gate, never
  thrown), then layers custom docs on top with **symmetric** collision rules ‚Äî
  both loops are **defaults-first**, a custom whose name collides with a seeded
  default **shadows** it (custom-last in the list), and custom-vs-custom stays
  first-wins. For **themes** the shadow is a load-time list swap and custom fields
  additionally refine the default per-field **at render** (the single merge site,
  `resolveTheme`); for **stamps** the shadowing custom replaces the default entry
  outright. A custom theme named `"default"` therefore reskins the default itself.
  `@facet/quickstart` calls `loadAssets` on **every** boot so the default stamp/theme
  library always reaches the agent + shell. Finally, the orphaned code-factory
  **`@facet/kit` is deleted** (playground migrated to a tiny local brick helper).

## Fail-safe & Boundary Checklist
- **Malformed / empty / null input:** a bad custom theme/stamp doc is dropped with an issue; a bad **seeded default** (should never happen, but the gate treats it identically) is dropped with an issue; remaining defaults + valid customs survive. Empty/absent `AssetDocuments` ‚Üí defaults alone resolve (never empty). `loadAssets` never throws (per-doc `try/catch` retained).
- **Deeply nested / cyclic tree:** every default + custom stamp still passes `validateStamp` (`packages/core/src/validate.ts:699`) which sanitizes the node map, prunes dangling children, and `breakCycles`. Seeding defaults does not bypass this.
- **Offline agent / rapid events / racing agent turn:** unchanged. `loadAssets` runs once at boot (no hot reload); `resolveTheme` is a zero-write pure lookup on the browser path; `withInitialStage` seed semantics untouched.
- **Name collisions (themes and stamps are SYMMETRIC ‚Äî default-collision shadows, custom-vs-custom first-wins):** theme ‚Äî a custom theme named `"default"` **shadows** the seeded default: `loadAssets` removes the seeded `default` document from the themes list and appends the custom one in its place (defaults-first, custom-last), with a recorded issue, so the list holds exactly one `default` entry (the custom). `resolveTheme("default")`'s `themes.find` therefore returns the custom document and overlays it onto the imported default value-map floor ‚Üí custom fields win, defaults fill the rest (DC-007). stamp ‚Äî a custom stamp whose name equals a default's **shadows** the default the same way (defaults-first, custom-last), with a recorded issue (DC-003). Custom-vs-custom (two customs sharing a name, neither a seeded default) keeps today's first-wins for both. The shadow is a load-time **list** operation, never a merge ‚Äî the single theme-merge site (`resolveTheme`) is untouched, so custom-over-default is still applied exactly once, at render (Invariant #6).
- **Prototype-key tokens:** default value maps stay null-proto (moved verbatim into `@facet/assets`); `resolveTheme`'s `overlayGroup` re-establishes null-proto after any JSON round trip. Hostile keys (`__proto__`, `constructor`) resolve to nothing (pinned by existing react tests).
