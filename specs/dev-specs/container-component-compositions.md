## Overview

- Feature slug: `container-component-compositions`
- Intake brief: `specs/feature-intake/container-component-compositions.md`
- Context evidence: `specs/context/container-component-compositions.md`
- Behavior/release-note packages: `@facet/core`, `@facet/assets`, `@facet/react`,
  `@facet/agent-tools`, `@facet/reference-agent`, `@facet/quickstart`
- Test-only/transitive consumers: `@facet/runtime`, `@facet/store-postgres`,
  `apps/playground`
- Documentation-only published consumer: `@facet/store-postgres` README (no
  direct changeset entry; fixed-group policy and evidence are recorded below)
- Technical approach: first make default reference/theme data independent of the
  retired types, then narrow the core union/registry/validators/catalog/spec, and
  finally migrate every exhaustive consumer and seed in one indivisible
  cross-package cutover. `card`, generic `section`, and `empty-state` remain only
  as read-only reference names backed by concrete `box`/`text`/`button` nodes.
  The references bake closed token names directly; no box-to-component recipe
  alias is added. `ContainerNode` narrows to `BoxNode | FormNode`, preserving form
  behavior. Stale raw retired nodes are rejected by validated authoring and
  skipped as whole subtrees by React.

### Explicit semantic waiver proposed for owner approval

The approved heading/landmark downgrade is broader for `emptyState`: its old
renderer emitted a native `<button>`, while the surviving Facet `button` currently
uses the shared `div role="button"` pressable and does not implement Enter/Space
activation. PR-5b preserves pointer/action dispatch but does **not** repair that
pre-existing shared-button accessibility gap. Approving this spec accepts that
keyboard-parity downgrade and defers the shared button fix to a separate change.
Adding raw markup or expanding PR-5b into a button redesign is rejected.
