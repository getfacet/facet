## Overview
- Feature slug: `input-consolidation`
- Intake brief: `specs/feature-intake/input-consolidation.md`
- Context evidence: `specs/context/input-consolidation.md`
- Affected `@facet/*` packages: `@facet/core`, `@facet/react`, `@facet/agent-tools`,
  `@facet/reference-agent`, `@facet/assets`, `@facet/quickstart`, plus `apps/playground` (unpublished).
- One-paragraph technical approach: A **hard-cutover consolidation** (PR-4 of the
  node-model restructure), landed **atomically in one PR**. (1) Rename the `field`
  primitive brick → `input` across the node type discriminant, the exported
  symbols (`FieldNode`→`InputNode`, `FieldInput`→`InputKind`, `FieldStyle`→`InputStyle`,
  `FIELD_INPUTS`→`INPUT_KINDS`), `PRIMITIVE_BRICK_TYPES`, validators, renderers,
  the brick registry, the executor/prompt kit, the stage-summary, the STAGE_SPEC,
  catalog defaults, seed/playground trees, and the composition remap/fill switches.
  (2) **Remove the `search` component node type entirely** (type, `SearchNode`,
  union membership, validator branch, renderer, executor entry, stage-summary
  handler, STAGE_SPEC line, catalog entry, composition remap/fill arms, theme
  recipe). The `"search"` token survives **only as an input KIND** (member of the
  renamed `INPUT_KINDS`), never as a node `type`. (3) "Search + submit" is
  re-expressed as an agent-authored `input`+`button` composition — no built-in
  recipe — so the STAGE_SPEC/prompt-kit must teach the submit button's `collect`
  wiring. The **styling layer keeps its `field` naming** (see RISK-API-6): the
  `fieldStyle` fn, the `field` styleable-slot key, and the `componentRecipe(theme,
  "field")` literal are a separate namespace decoupled from the node type; renaming
  them would collide with the existing `input` recipe-part key and is deferred to
  PR-6. Only the imported core *symbols* those files reference are updated.

## Fail-safe & Boundary Checklist
- **Malformed / empty / null `input` node:** sanitized exactly as `field` was (bad
  props dropped by `validateInput`); never throws.
- **Unknown `input` kind** (e.g. `"telepathy"`): degrades to the default text input
  (carried from `field`'s existing fail-safe); never throws.
- **Stale `type:"search"` or `type:"field"` node in a live tree:** unknown type →
  renderer skips it (blank-degrades), `collectFieldValues` returns `{}` for it — no
  throw. (Hard cutover, no compat alias; acceptable pre-deploy.)
- **Password value:** excluded from `collectFieldValues` output (never rides the
  action event into the agent / third-party LLM / history replay) — preserved
  verbatim under the `input` branch (RISK-INV-2). The authoritative assertion is
  `StageRenderer.interaction.test.tsx:732` ("never harvests a password field's
  value"), whose `type:"field"` authoring WU-4 migrates → `type:"input"` so it
  keeps enforcing the exclusion; the new `renderer-press.test.ts` adds a focused
  restatement, it does not replace it.
- **Deeply nested / cyclic tree:** `collectFieldValues` keeps its existing
  `MAX_DEPTH` cap, ancestor cycle guard, and `RENDER_BUDGET`/`gatherRefsBudget`
  ceilings unchanged (only the type-guard string changes). Pinned verbatim by
  `StageRenderer.interaction.test.tsx:1075` (cyclic collect terminates) and
  `StageRenderer.budget.test.tsx:114`/`:139` (deep/lattice collect budget) —
  WU-4 migrates their `field` authorings → `input` so these ceilings stay tested.
- **Offline agent / rapid events / racing turn:** value capture is a read-only
  press-time DOM snapshot only; nothing is written back to the tree — unchanged.
- **Search+submit composition with no `collect` wired:** submit fires, harvests
  nothing (agent-authoring error, not a framework throw) — mitigated by DC-005
  teaching, not by a runtime guard.

> Atomicity note: this is a hard cutover — the repo only fully typechecks once ALL

>
> Registry-coupling note (why the two core WUs serialize AND why WU-2's search
> cutover is scoped to 5 files with a dead-code follow-on): `brick-registry.ts` is the single `BRICK_REGISTRY` dispatch
> hub and the **only** consumer of the `field`/`search` behavior functions
> (`validateField`, `rendersField`/`rendersSearch` in `tree.ts`,
> `fillField`/`fillSearch`+`leavesField`/`leavesSearch` in `expand-composition-fill.ts`);
> it is imported by the whole `validateTree`/`expand` graph, so **every core red_check
> transpiles it**. It therefore cannot be co-edited by a *parallel* WU. The split:
>
> - **WU-1 (field→input, self-contained):** rename ONLY the registry `field:` entry
>   KEY → `input:` with `validate: validateInput`, and **keep** the
>   `rendersField`/`fillField`/`leavesField` function NAMES (only `validateField`→
>   `validateInput`, WU-1's own file, is renamed — the other three references resolve
>   against symbols already on `main`). WU-1 **leaves the `search:` entry intact.**
> - **WU-2 (search removal, 5 files):** `component-nodes.ts` (`SearchNode` +
>   `COMPONENT_NODE_TYPES` membership), `component-validation-control.ts`,
>   `expand-composition-remap.ts`, `expand-composition-fill.ts`
>   (`fillSearch`/`leavesSearch`), AND `brick-registry.ts` (the `search:` entry + its
>   `rendersSearch`/`fillSearch`/`leavesSearch` imports). The registry `search:`
>   removal MUST be atomic with the `COMPONENT_NODE_TYPES` `search` removal: component
>   validation dispatches `BRICK_REGISTRY[rawType]` at `component-validation.ts:93`
>   AFTER an `isComponentNodeType` guard keyed on `COMPONENT_NODE_TYPES` — and there is
>   NO fail-safe try/catch. If the registry entry were dropped first (WU-1) while
>   `search` stayed in the type set, a `type:"search"` node would pass the guard then
>   hit `undefined.established` and **throw**, breaking fail-safe transiently and making
>   WU-2's `'search node type removed'` red_check unable to go cleanly RED→GREEN. This
>   binds exactly `{brick-registry.ts, component-nodes.ts}`. The other three arms
>   (control-validation `case "search"`, the remap arm, and the
>   `fillSearch`/`leavesSearch` definitions) blank-degrade if they lag rather than
>   throw — but `brick-registry.ts` *imports* `fillSearch`/`leavesSearch`, so their
>   definitions are removed in the same WU (else the import dangles); that is 5 files.
> - **WU-10 (dead `rendersSearch` cleanup, 1 file, `depends_on [WU-2]`):** `tree.ts`'s
>   exported `rendersSearch(node: Record<string, unknown>)` predicate is dispatched
>   ONLY through the `BRICK_REGISTRY` `search:` entry. Once WU-2 drops that entry AND
>   the `rendersSearch` import, the predicate is unreferenced dead code with **no throw
>   risk** and **no `SearchNode` type dependency** — so its removal is pure cleanliness
>   and is split into this tiny follow-on WU. This keeps WU-2 at the Gate-7 ≤5-files
>   limit without weakening the atomic cutover: `tree.ts` was never part of the
>   hard-bound `{brick-registry.ts, component-nodes.ts}` pair, and the transient
>   unused-export between WU-2 and WU-10 rides the same hard-cutover atomicity model as
>   the whole-repo typecheck (final gate, not per-WU).
>
