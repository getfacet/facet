## Overview

- Feature slug: `appear-hold-scroll`
- Intake brief: `specs/feature-intake/appear-hold-scroll.md`
- Context evidence: `specs/context/appear-hold-scroll.md`
- Affected `@facet/*` packages: `@facet/core`, `@facet/react`, `@facet/kit`,
  `@facet/quickstart` (test-only Tier-2 anchor) (+ `apps/playground` demo)
- One-paragraph technical approach: Three token-shaped words, zero protocol
  change. **`appear`** is a new closed token group in `@facet/core`
  (`APPEARS = ["none","fade","slide"]`) landing on `BoxStyle` only; the concrete
  animation lives in a **framework-owned static CSS constant** in a new internal
  `@facet/react` module (`appear.ts`: `@keyframes` + a
  `prefers-reduced-motion: reduce` gate), rendered by `StageRenderer` as one
  `<style>` element **only when the tree actually uses a classifiable appear
  token** — so appear-free trees stay byte-identical to today. The token maps to
  a class NAME only; durations/curves never enter the tree, `FacetTheme`
  documents, or `boxStyle`'s published `CSSProperties` signature. **`onHold`**
  is a second optional `FacetAction` on `BoxNode`, normalized by the SAME
  (field-label-parameterized) `asAction` in `validateTree` and classified by the
  SAME `classifyPress` in the renderer; gesture detection is
  pointerdown/up/move/cancel + a timer inside a new internal `HoldableBox`
  component (used ONLY when a classifiable `onHold` exists — no-hold trees keep
  today's exact element), with a suppress-next-click latch so press and hold
  never both fire, and both gestures route through the ONE existing
  `handlePress` switch (single writer path; hold-agent events are byte-identical
  in shape to press events — no gesture discriminator). **`scroll`** is a
  boolean on `BoxStyle` (matching `wrap`/`border`/`grow`); `theme.ts#boxStyle`
  maps literal `true` inline to
  `overflowY:"auto"; overflowX:"hidden"; maxHeight:SCROLL_MAX_HEIGHT; minHeight:0`
  (renderer constant, never overflow-x, flex-safe). All three enter through BOTH
  validation layers (stored `validateTree`/`validateStamp`/fold AND the raw
  live-patch renderer path), degrade to plain on junk, and are taught by
  `STAGE_SPEC` with a quickstart prompt-test anchor that makes the `/live-test`
  Tier-2 path heuristic fire.

### Decision 1 — scroll token shape (resolves brief open question 1)

`scroll?: boolean` on `BoxStyle`; only literal `true` activates. Rationale:
(a) it reuses the existing boolean-style pattern (`wrap`/`border`/`grow` via
`asBool`, `packages/core/src/validate.ts:210-215`) — no new token group for a
single-axis knob; (b) vertical-only by construction (there is no value that
could mean "sideways", killing the horizontal-overflow class of bugs at the
type level); (c) the bounded height is **not** a token or theme datum — it is a
non-exported renderer constant `SCROLL_MAX_HEIGHT = "20rem"` in
`packages/react/src/theme.ts` (RISK-API-5: `ResolvedTheme`/`FacetTheme`/
`DEFAULT_THEME`/`validateTheme` stay byte-identical). If size variants are ever
demanded, `scroll` widens additively to a closed union (`true | "sm" | "lg"`).
Exact CSS (RISK-INV-6a): `overflowY:"auto"`, `overflowX:"hidden"` (never a
horizontal page scrollbar), `maxHeight:SCROLL_MAX_HEIGHT`, `minHeight:0` (a
flex child defaults to `min-height:auto` and would silently refuse to clip
inside a `grow` column — jsdom can't see that, hence the DC-009 real-browser
check is load-bearing).

### Decision 2 — appear set + replay semantics (resolves brief open question 2)

`APPEARS = ["none", "fade", "slide"] as const` in `packages/core/src/tokens.ts`
(same const-array pattern as `SPACES`, `tokens.ts:16`); `appear?: Appear` on
`BoxStyle` ONLY (text/image/field wrap in a box — confirmed against STAGE_SPEC
ergonomics: the box is already the unit of composition and the only brick with
per-node behavior fields). `"none"` is a valid explicit token (renders with no
class, same as absence). **Replay semantics are pinned as replay-on-mount**:
the animation runs whenever the element mounts — first paint, node re-add,
toggle re-show, and screen navigation (the renderer unmounts hidden/off-screen
nodes, `StageRenderer.tsx:446-449`, so these ARE mounts). This makes tests
unambiguous and matches the CSS-animation mechanism exactly (no JS bookkeeping,
no played-state map). The animation is purely visual: `applyPatch`, the fold,
and layout position are untouched mid-animation; slide uses a transient
`translateY(6px)` + opacity (sub-gap scale — nothing perceptually overlays,
RISK-INV-6b).

### Decision 3 — hold gesture protocol (resolves brief open question 3)

Renderer constants (not tokens, not theme data — confirming the brief's
assumption): `HOLD_MS = 500`, `HOLD_SLOP_PX = 8`, non-exported in
`StageRenderer.tsx`. Detection: `pointerdown` arms a timer; `pointermove`
beyond slop, `pointerup` before threshold, `pointercancel`, or `pointerleave`
disarms it. Timer fire ⇒ arm a one-shot **window-level capture-phase click
interceptor**, then execute the hold through `handlePress`. The subsequent
browser-synthesized `click` (browsers fire it after pointerup regardless) is
swallowed at window capture — press and hold NEVER both fire (DC-004,
RISK-INV-1/RISK-API-4a). **Interceptor lifecycle (pinned, revised in review
r2):** SET when the hold timer fires; CONSUMED by the next `click` anywhere
(window capture runs before any target/bubble handler); RESET by the next
`pointerdown` anywhere — whichever comes first. Window+capture scope is
structural, not stylistic: a component-scoped latch provably fails three ways
— a pressable DESCENDANT's click handler runs target-first (before the held
box's bubble handler), a release-outside targets the synthesized click at the
common ancestor (the held box never sees it), and a nested holdable's
pointerdown `stopPropagation` defeats a per-box arm-time reset. Teardown is
deliberately NOT tied to component unmount (a hold that unmounts its own box
must still swallow the click that follows). Consecutive gestures are exact:
hold-then-tap ⇒ one hold + one press; hold-then-hold ⇒ exactly two holds.
Below-threshold release ⇒
timer disarmed, native `click` runs `onPress` as today. Touch disambiguation
inside scroll regions is free: when the browser claims the gesture for
scrolling it fires `pointercancel`, which disarms the hold; `contextmenu` is
`preventDefault`-ed only while a hold is armed on a box that HAS `onHold` (so
long-press context menus don't race the hold on touch devices, and boxes
without `onHold` keep native behavior). Mid-press `onHold` removal no-ops: the
gesture state lives in a small internal `HoldableBox` component whose timer
callback reads the LATEST classification via a ref updated each render — a
patch that removes `onHold` re-renders with a null classification before the
timer fires. `HoldableBox` is used ONLY when `classifyPress(node.onHold)` is
non-null; press-only and plain boxes keep today's exact inline elements
(byte-identical DOM, DC-007). Keyboard: hold has NO keyboard path in v1 —
recorded as accepted, mitigated by STAGE_SPEC policy guidance ("hold is a
secondary gesture; never make it the only path to critical content" — prompt
advice, not framework enforcement, per invariant #2). A hold-only box (no
`onPress`) is focusable/pressable-styled but a quick tap on it no-ops.

### Decision 4 — appear CSS mechanism (RISK-INV-4 / RISK-API-2)

All animation CSS is one **static, framework-owned string constant**
`APPEAR_CSS` in a new package-internal module `packages/react/src/appear.ts`
(NOT exported through the `@facet/react` barrel — no new public surface),
containing the `facet-appear-fade`/`facet-appear-slide` keyframes + classes
(fade: 160ms ease-out opacity; slide: 200ms ease-out translateY(6px)+opacity)
and the `@media (prefers-reduced-motion: reduce)` block that sets
`animation: none` on exactly those classes (DC-008). No theme-document data and
no tree data is ever interpolated into it — `validateTheme` stays closed to
animation CSS (Decision Lock). `StageRenderer` renders `<style>{APPEAR_CSS}</style>`
once per stage IFF the current tree contains at least one node with a
classifiable appear token. **Prescan design (pinned, TOTAL on the raw path):**
the prescan is a single FLAT iteration over the `tree.nodes` record values that
null-guards EVERY value before dereferencing it:
`Object.values(tree.nodes).some(n => n != null && appearClass(styleOf((n as {style?: unknown}).style)) !== undefined)`
using the same total classifier. It is safe on both raw-path axes: (a)
**cycle/depth** — it NEVER follows `children`, so no recursion and no
reachability walk exists to hang on a cyclic or over-deep tree; (b) **junk node
VALUES** — the raw live path can hold `null` (or scalar) node values (a patch
can legally set a node — even the root — to JSON null; the existing renderer
guards exactly this with the `node == null` check at `StageRenderer.tsx:428`,
and `isTreeShaped` — the only guard before render, `packages/core/src/tree.ts:49-54`
— checks only that `nodes` is an object, never its values), so the prescan's
`n != null` guard mirrors renderNode's guard and no property is ever
dereferenced off a null/scalar value (`styleOf`/`appearClass` are already total
on whatever survives the guard). This matters because the prescan runs on the
raw live path, which bypasses `validateTree` (RISK-INV-3) and can receive
cyclic/junk trees; reachability is deliberately ignored because an extra
idempotent `<style>` element for an unreachable appear node is harmless, while
a reachability walk would be a new unguarded traversal. Pinned by DC-005
raw-path tests: (1) a cyclic raw tree containing an appear token renders
without hanging or throwing; (2) a raw tree whose `nodes` record contains a
null AND a scalar node value alongside an appear-token node renders without
throwing. Appear-free
trees emit no style element, keeping the existing static suite byte-identical.
Duplicate style elements from two stages on one page are idempotent (namespaced
class names, identical constant). `boxStyle`'s published
`(style?, theme?) => CSSProperties` signature is unchanged: `scroll` IS mapped
inline there (a complete mapping for direct consumers), `appear` is documented
as renderer-bound (a class, inexpressible as inline CSSProperties).

### Decision 5 — hold event shape (RISK-INV-5, zero protocol delta)

A hold-triggered `kind:"agent"` action emits a `ClientEvent` **byte-identical
in shape** to a press-triggered one — no `gesture` field, no new event type;
the agent differentiates by the action `name` it authored. `collect`/`payload`
ride exactly as for press (same `handlePress` arm, same `collectFieldValues`).
`packages/server/src/server.ts` is deliberately UNTOUCHED — an empty diff there
is the proof of "zero new protocol messages"; any diff appearing there during
review is out of spec. Scroll emits nothing ever: no scroll handler exists, and
`handlePress`'s agent arm remains the sole `onAction` call site.

## Fail-safe & Boundary Checklist

- Malformed / empty / null input: `appear:"explode"`/`scroll:"sideways"` ⇒
  stripped by the boxStyle whitelist (stored) and mapped to nothing
  (raw/theme.ts) — box renders instantly / flows normally. `onHold:42`,
  `onHold:{kind:"swipe"}` ⇒ stripped with an issue naming `onHold` (stored) and
  classified null (raw) — box stays pressable-or-plain, hold handlers never
  attach. `onHold:{name:"x"}` legacy shape ⇒ normalized to `kind:"agent"`
  exactly like onPress. `validateTree(null | 42)` behavior unchanged.
- Deeply nested / cyclic tree / junk node values: the render/gather walks keep
  their unchanged guards (ancestor-set at `StageRenderer.tsx:456`, `MAX_DEPTH`
  `:428`, `RENDER_BUDGET` `:33`). The appear-style prescan IS one new
  traversal, but by pinned design (Decision 4) it is a non-recursive flat
  iteration over the `tree.nodes` record values that never follows `children`
  AND null-guards every value (`n != null`) before touching `.style` — a
  cyclic or over-deep RAW tree cannot hang it, and a raw tree carrying null or
  scalar node VALUES (legal on the live path: a patch can set any node to JSON
  null, and `isTreeShaped` never checks node values) cannot make it throw (the
  prescan runs before the guarded walks, on unvalidated input). Pinned by
  DC-005 raw-path tests: a cyclic raw tree containing an appear token renders
  without hanging or throwing, and a raw tree with a null and a scalar node
  value alongside an appear-token node renders without throwing. `HoldableBox`
  wraps a single node's element only; the new words add no other recursion.
- Offline agent / rapid events / racing agent turn: hold-toggle/navigate are
  browser-local and work with the agent offline; a hold with `kind:"agent"`
  fails exactly like a pressed agent action today (serialized per-visitor queue
  unchanged). Rapid repeated holds ⇒ one action per completed gesture (latch
  cleared on consume-by-click OR next pointerdown, whichever first — a hold
  released off-box never swallows the following tap; covered by the
  consecutive-gesture tests in WU-3). Below-threshold (e.g. 300ms) release ⇒
  plain press. Patch
  removing `onHold` mid-press ⇒ timer callback reads the latest classification
  and no-ops. Agent patches children while the user is mid-scroll ⇒ content
  updates in place; container identity (and thus scrollTop) survives (DC-006
  test); node re-added rapidly ⇒ appear replays per mount (pinned semantic).
