# Dev Spec: Brick Style System Hard Cutover

> Generated by `/spec-bridge`. Human approval spec. The execution manifest is
> canonical; this document is its condensed human mirror for Work Unit ids,
> canonical titles, owned files, dependencies, RED checks, test plans, and final
> gates. Objectives remain fully detailed in the manifest.
>
> Approved implementation amendment: provider/live probes split detailed value
> metadata out of `get_brick_spec`. It now returns one compact Brick, while
> `get_style_choices` returns one exact Brick-owned property domain. This keeps
> each discovery observation within 4,000 characters without adding a global
> token lookup.

## Overview

- Feature slug: `brick-style-system-hard-cutover`
- Intake: `specs/feature-intake/brick-style-system-hard-cutover.md`
- Context: `specs/context/brick-style-system-hard-cutover.md`
- Base: `main` / `4c89b56`
- Published packages: all current `@facet/*` packages are versioned atomically;
  direct production edits are expected in core, assets, runtime, react, server,
  agent-tools, reference-agent, quickstart, agent, cli, bridge, store-postgres,
  client-facing view contracts, and ag-ui-facing tests/typing.
- Approach: replace the shipped styling/asset contract in one hard cut. Core
  becomes the one source for brick-local style vocabularies, property-local
  choices, Theme/Preset/Pattern contracts, strict author validation, and
  fail-soft sanitation. Runtime resolves one complete Theme plus one exact
  Pattern list. React renders all eleven vocabularies and host/client colorMode.
  Agent tools provide progressive discovery and reject invalid authoring before
  patch creation. All old terms, selectors, files, persistence fields, tools,
  and aliases disappear at the same package-version boundary.

## Locked Technical Contract

### 1. Public vocabulary and author syntax

The current public meanings are replaced, not aliased:

| Meaning | Public term / shape |
|---|---|
| `fontSize`, `gap`, `color` | Style property |
| `label`, `control`, `track`, `fill` | Brick-owned style target |
| `md`, `lg`, `success` | Token name |
| `16px`, `#16a34a` | Resolved CSS value; Theme/operator-only |
| `space.md = "1rem"` | Theme token definition |
| `panel`, `heading` | Brick-scoped Preset name |
| named single-brick style bundle + usage metadata | Preset definition |
| named concrete Facet tree + usage metadata | Pattern |
| `system`, `light`, `dark` | colorMode preference; never document syntax |

Every `FacetNode` has optional `style` of its own brick type. The four supported
forms are omitted, `{preset}`, direct properties/targets, and `{preset,...direct}`.
There is no `variant`, `tone`, `recipe`, `scheme`, tree `theme`, `activeVariant`,
or `activeStyle` authoring path.

### 2. Core value domains

Core exports ordered, metadata-bearing token domains. Each name has a bounded
`description`, `useWhen`, and optional `avoidWhen`; concrete values are absent.

| radius | none, sm, md, lg, full |
| borderWidth | none, thin, medium, thick |
| aspectRatio | auto, square, landscape, portrait, wide |
| minHeight | auto, half, screen |
| maxWidth | none, prose, narrow, wide |
| letterSpacing | tight, normal, wide |

| semantic color | background, surface, mutedSurface, foreground, mutedForeground, border, accent, accentSurface, accentForeground, focusRing, success, successSurface, successForeground, warning, warningSurface, warningForeground, danger, dangerSurface, dangerForeground, info, infoSurface, infoForeground, chart1..chart6, inherit |

`alignment(start,center,end,stretch)`,

meaningful; its Theme concrete value must be exactly `inherit`.

### 3. Brick-local style vocabularies

Core's `BRICK_CONTRACT` is serializable metadata and the single runtime registry
for Brick discovery and author validation. It owns the eleven Brick names,
Brick description/`useWhen`/`avoidWhen`, authored data fields, and each local
style property path, domain, state support, description, `useWhen`, and
input-kind applicability. `BRICK_TYPES`/`BrickType` and public style types derive
from it where TypeScript permits and are otherwise guarded by exhaustive parity
tests. `get_brick_spec` projects this same registry; it does not maintain a
second description or style table.

Common property bundles below are only notation in this spec; they do not create
global style targets in the public API:

- `typography` = fontFamily/fontSize/fontWeight/fontStyle/color/textAlign/
  letterSpacing/lineHeight where listed.
- `surface` = background/color/borderColor/borderWidth/borderRadius/shadow where
  listed.
- paint states use only the explicitly listed subset of their owning target.

| Brick | Root properties | Owned targets and states |

| box | direction, gap, padding, alignItems, justifyContent, wrap, columns, grow, width, minHeight, maxWidth, scroll, sticky, background, color, backgroundGradient, borderColor, borderWidth, borderRadius, shadow, backdropScrim, enterAnimation | root hover/pressed: background,color,borderColor,shadow; root focus: borderColor,borderWidth,shadow |
| text | typography + highlight | no nested target/state; color falls back to parent box inheritance, then Theme default |

The registry asserts that position, inset, z-index, margin, transform, opacity,
arbitrary width/height scalars, selector syntax, raw CSS, and data-addressed
targets are absent from every target and state.

Public type roles:

- `BrickStyleDefinition<B>`: direct root/target properties plus supported
  ephemeral state blocks; no Preset selection and no `active`.
- `BrickStyle<B>`: `BrickStyleDefinition<B>` plus optional `preset` and optional
  `active`.
- `BrickActiveStyle<B>`: optional same-brick `preset` plus direct root/target
  properties only; it cannot recursively contain `active` or ephemeral states.
- `BrickStyleByType`, `BoxStyle`, `TextStyle`, `MediaStyle`, `InputStyle`,
  `RichTextStyle`, `TableStyle`, `ChartStyle`, `ListStyle`, `KeyValueStyle`,
  `ProgressStyle`, and `LoadingStyle` are exported.

Only box/text accept node-level `activeWhen: ViewPredicate` initially. All active
appearance lives at `style.active`. Unknown/dangling predicates use base style.

### 4. Theme and Preset shape

```ts
interface FacetTheme {
  readonly name: string;
  readonly description?: string;
  readonly tokens: FacetThemeTokens;
  readonly defaults: BrickStyleDefinitionMap;
  readonly presets?: FacetPresets;
}

interface FacetThemeTokens {
  readonly space: CompleteMap<Space, string>;
  readonly fontSize: CompleteMap<FontSize, string>;
  readonly fontFamily: CompleteMap<FontFamily, string>;
  readonly fontWeight: CompleteMap<FontWeight, number>;
  readonly radius: CompleteMap<Radius, string>;
  readonly borderWidth: CompleteMap<BorderWidth, string>;
  readonly aspectRatio: CompleteMap<AspectRatio, string>;
  readonly minHeight: CompleteMap<MinHeight, string>;
  readonly maxWidth: CompleteMap<MaxWidth, string>;
  readonly letterSpacing: CompleteMap<LetterSpacing, string>;
  readonly lineHeight: CompleteMap<LineHeight, string>;
  readonly controlHeight: CompleteMap<ControlHeight, string>;
  readonly indicatorSize: CompleteMap<IndicatorSize, string>;
  readonly progressThickness: CompleteMap<ProgressThickness, string>;
  readonly chartThickness: CompleteMap<ChartThickness, string>;
  readonly paint: {
    readonly light: FacetPaintTokens;
    readonly dark: FacetPaintTokens;
  };
}

interface FacetPreset<B extends BrickType> {
  readonly description: string;
  readonly useWhen: string;
  readonly avoidWhen?: string;
  readonly style: BrickStyleDefinition<B>;
}
```

`FacetPaintTokens` completely maps semantic color, shadow, gradient, scrim, and
highlight. `defaults` contains all eleven brick keys and every published nested
target (an empty target object is valid); default/Preset bundles use token/fixed
names only. Preset record keys are their names and are brick-scoped. Names and
metadata are bounded by the existing safe slot/prose rules.

A custom Theme is valid only if every required token key, paint branch, brick
default, and nested target exists; every concrete value passes group-specific
CSS safety; every style bundle passes the Core vocabulary; every unknown/
prototype key is rejected; Presets are at most 16 per brick and 64 total. Any
error rejects the whole Theme. There is no partial merge. The bundled default is
validated through the same gate and is complete.

Concrete Theme values use these exact, non-clamping grammars. A decimal has at
most four fractional digits; every string is at most 512 bytes and surrounding
whitespace is rejected.

- `space`: `0`, or `0..256px`, or `0..16rem|em`.
- `fontSize`: `8..256px` or `0.5..16rem|em`.
- `radius`: `0`, or `0..9999px`, or `0..625rem|em`.
- `borderWidth`: `0`, or `0..16px`, or `0..1rem|em`.
- `controlHeight`: `16..256px` or `1..16rem|em`;
  `indicatorSize`: `4..128px` or `0.25..8rem|em`;
  `progressThickness`: `1..64px` or `0.0625..4rem|em`;
  `chartThickness`: `1..32px` or `0.0625..2rem|em`.
- `aspectRatio.auto` is exactly `auto`; every other aspect-ratio value is two
  finite positive decimals in `a / b`, each `0.01..100`.
- `minHeight`: `auto`, `0`, `0..2000px`, `0..125rem|em`, or `0..100svh`;
  `maxWidth`: `none`, `0`, `0..4096px`, or `0..256rem|em|ch`.
- `letterSpacing`: `0`, `-16..16px`, or `-1..1rem|em`;
  `lineHeight`: a unitless decimal `0.8..3`.
- `fontWeight`: a finite integer `1..1000`; `fontFamily`: 1–200 characters
  from letters, digits, spaces, comma, quote, hyphen, and underscore only.
- opaque color is exactly `#RGB`, `#RRGGBB`, `#RGBA`/`#RRGGBBAA` with an `f`/`ff`
  alpha, `rgb(r,g,b)` with three `0..255` numbers or `0..100%` channels,
  `hsl(h,s%,l%)` with finite hue and `0..100%` saturation/lightness, or one of
  Core's explicit named-color allow-list. `inherit` and `transparent` are
  accepted only for token paths that explicitly own those keywords. Scrim and
  shadow colors additionally accept `rgba`/`hsla` with alpha `0..1`.
- gradient/highlight is `none`, or `linear-gradient(angle, stops)` with angle
  `-360..360deg`, or `radial-gradient(circle at x% y%, stops)` with `x`/`y`
  `0..100`. There are 2–8 comma-separated stops; each is one validated opaque
  color or `transparent` followed by a `0..100%` position, and positions are
  nondecreasing.
- shadow is `none` or 1–4 comma-separated layers. A layer is optional `inset`,
  x-offset, y-offset, blur, optional spread, and one validated scrim/shadow
  color. Offsets/spread allow `-256..256px` or `-16..16rem|em`; blur allows
  `0..256px` or `0..16rem|em`. Unitless zero is allowed for any length. No
  other keyword or nested function is accepted.

All groups reject control characters, `url`, `var`, `calc`, `expression`,
`javascript:`, braces, semicolons, prototype keys, non-finite numbers, and
group-inappropriate units. Any out-of-range concrete value rejects the whole
custom Theme; no value is clamped or partially retained.

### 5. Pattern contract

`FacetPattern` is a named ordinary Facet tree with top-level `name`,
`description`, `useWhen`, optional `avoidWhen`, and ordinary `root`, `nodes`,
optional `screens`/`entry`/`data`. Its root/screen rules match `FacetTree`; it has
no Theme field, params, placeholders, nested Pattern reference, provenance, or
runtime node kind.

`validatePattern(raw, effectiveTheme)` is strict: it does not sanitize an invalid
Pattern into exposure. It validates the complete document, derives every base
and active Preset reference, and rejects the whole Pattern if any same-brick
Preset is absent. The maximum exposed exact list is 64; input length 65 rejects
the complete explicit list rather than truncating.

The default dataset exports `DEFAULT_PATTERNS`. First-party Patterns preferentially
use Theme Presets for repeatable single-brick roles and direct values for
Pattern-specific layout. Visual/tone-only variations become Presets, not extra
Patterns.

### 6. Asset and persistence shape

```ts
interface AssetDocuments {
  readonly theme?: unknown;
  readonly patterns?: unknown;
  readonly initialTree?: unknown;
  readonly issues?: readonly string[];
}

interface LoadedAssets {
  readonly theme: FacetTheme;
  readonly patterns: readonly FacetPattern[];
  readonly initialTree?: FacetTree;
  readonly issues: readonly string[];
}
```

Validation order is custom Theme → whole fallback to `DEFAULT_THEME` → Presets
→ Pattern list against the effective Theme → strict initial tree against the
same Theme. Missing `patterns` uses `DEFAULT_PATTERNS`; explicit `[]` exposes
none; a present non-array/unreadable/over-64 list exposes none with an issue.
Invalid/incompatible individual entries are hidden whole. The effective Theme,
Patterns, indexes, and validation context are deep-detached/frozen.

`FileAssets` hard-cuts to optional `theme.json`, optional exact-array
`patterns.json`, and optional `initial.tree.json`. A recognized retired Theme/
Composition/Catalog filename yields a bounded issue and is never read as a new
source. Node I/O stays behind `@facet/runtime/node`.

Postgres `facet_assets` has `agent_id`, nullable `theme`, nullable `patterns`,
nullable `initial_tree`, and `updated_at`. SQL `NULL` means the corresponding raw
asset property is absent; JSONB `null` means it is present but malformed. The
select path carries explicit presence flags so omitted, `[]`, and JSON `null`
remain distinct through `AssetDocuments`. `initSchema` creates the new table only
when absent. When the table exists, it reads the column set and throws a bounded
`migration required` error before normal use if the old or a mixed layout is
found; it never adds, renames, copies, or silently accepts old columns. The
release migration guide requires operators to back up/replace the old table and
manually write validated new-shape rows. They must also rewrite or clear old
`facet_stage`/file sessions, initial trees, and stale model summaries. Runtime
has no compatibility reader or converter.

### 7. Agent discovery and strict authoring

Every provider turn acquires one asset source result, validates/deep-snapshots it
once, and uses that same object for prompt indexes, reads, and mutation
validation. Static sources remain supported; a dynamic source may return a new
snapshot next turn, never mid-turn. Quickstart uses one static loaded snapshot;
no browser asset-fetch/hot-reload route is introduced.

The short system-prompt indexes are ordered Pattern → Preset → Brick:

- Pattern: name, description, useWhen;
- brick: type, description, useWhen;
- Preset: brick type, name, description, useWhen.

Generic mutation schemas stay small and tell the model to inspect unfamiliar
details. New read tools are:

- `get_brick_spec({type})`: one exact Brick. Returns its data fields, compact
  target/property paths, and state/applicability rules, never CSS.
- `get_style_choices({brick,target,property})`: one exact Brick-owned property
  domain with name/description/useWhen/avoidWhen metadata, never a global token
  lookup or CSS. Each Brick or choice observation is capped by test at 4,000
  characters.
- `get_preset({brick,name})`: one exact same-brick metadata/token bundle.
- `get_pattern({name})`: one exact compatible unresolved Pattern.

Hidden/missing/incompatible/malformed lookups all return the same bounded
`not_available`, with no existence leak. Reads are exact, read-only, deeply
frozen, and produce zero patch/message/pending/shadow change. There is no
`get_token`, `get_composition`, or `set_theme`.

`render_page`, `set_node`, and `append_node` run Core strict validation against
the turn snapshot before constructing a patch. Any invalid node field, Preset,
target, property, token/fixed choice, state, or input-kind applicability rejects
the complete call. The returned observation contains at most 16 safe entries:

```ts
interface AuthorIssue {
  readonly path: string;      // RFC-6901-like document path
  readonly message: string;   // bounded and control-char stripped
  readonly allowed?: readonly string[];
}
```

It also carries `omitted_error_count`, `status:"error"`,
`outcome:"rejected"`, `code:"invalid_authoring"`, and a repair-oriented
`next_action`. Error results have zero messages/patches/changed ids and the
identical shadow. Valid calls still emit only RFC-6902 patches and pass through
the unchanged shared fold. `applied_with_warnings` may remain for non-author
fold diagnostics, but a normal style mistake never succeeds with that outcome.

### 8. React resolution and colorMode

`StageRenderer` accepts one `theme?: FacetTheme` and defensively validates that
public input before use. An incomplete, cyclic, Proxy, or throwing Theme falls
back as a whole to byte-equivalent `DEFAULT_THEME`; no fragment is retained.
It also accepts
`colorMode?: ColorModePreference`; absent Theme uses `DEFAULT_THEME`, absent mode
uses `system`. `ColorModePreference` is `system|light|dark`; the effective
`ColorMode` and `ViewSnapshot.colorMode` are `light|dark`. SSR system fallback is
light until browser detection. Forced modes ignore media-query changes.

Resolution is property-wise:

1. Theme brick default;
2. base same-brick Preset;
3. base direct style;
4. when `activeWhen` matches, active Preset then active direct style;
5. current supported ephemeral state from the already-merged default/Preset/
   direct state definitions.

Ephemeral hover/pressed/focus uses renderer-owned classes and CSS variables;
checked/sorted/alternate uses renderer-known state attributes/indexes. Agent
data never becomes a selector or raw CSS. Brick role modules explicitly convert
each allowed property to CSS; no generic raw object merge exists. colorMode
selects only paint maps (color/shadow/gradient/scrim/highlight); all non-paint
token maps and layout output stay identical. A mode change may update
`onViewSnapshot`, the client's local view cache, and the effective mode carried
by the next ordinary outgoing event. It must not itself emit an immediate agent
event, `/record`, stage patch/data write, or LLM turn. Existing navigation and
toggle records remain unchanged; style evaluation adds no record.

The renderer remains an independent, silent last defense for bypassed raw data:
invalid properties/targets are ignored, unknown Presets use Theme default plus
valid direct choices, invalid nodes are skipped, valid siblings survive, and
throwing/proxy input never injects or throws. Agent repair is delivered by the
strict tool result; no new renderer-to-agent diagnostics channel is added.

## Fail-safe & Boundary Checklist

- malformed/empty/null tree/node/style/Theme/Pattern/store: strict author calls
  reject atomically; asset boundary falls back/none as specified; fold/renderer
  never throw and retain valid content. Covered WU-03,06,08,09,15,19.
- deep/cyclic/proxy/throwing inputs: existing bounded tree walk remains; strict
  wrappers and renderer closed reads catch access failures. Covered WU-03,08,09,
  15,19.
- offline asset source/provider: source rejection yields bounded load/snapshot
  issues and default/no-Preset behavior; a failed provider turn keeps stage
  unchanged. Covered WU-15,25,30.
- rapid/racing tool calls: each turn uses one immutable snapshot; invalid calls
  emit nothing; existing serial turn lane and shared fold ordering remain.
  Covered WU-25,27,30 and existing runtime/harness suites.
- rapid colorMode/hover/active/screen/toggle: React updates local paint/view only;
  forced mode ignores media changes; inert transition snapshots keep their
  captured Theme/view. Covered WU-18,23,24.
