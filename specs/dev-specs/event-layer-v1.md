## Overview
- Feature slug: `event-layer-v1`
- Intake brief: `specs/feature-intake/event-layer-v1.md`
- Context evidence: `specs/context/event-layer-v1.md`
- Affected `@facet/*` packages: `@facet/core`, `@facet/runtime`, `@facet/server`,
  `@facet/client`, `@facet/react`, `@facet/quickstart`, `@facet/store-postgres`,
  `@facet/bridge` (test-only literal migration),
  plus the (unpublished) `apps/playground` consumers.
- One-paragraph technical approach: Introduce a **three-layer** event model in the
  contract ‚Äî `trigger` (the visitor gesture) ‚äá `CollectedEvent` (the durable log
  currency: `visit | message | tap`) ‚äá `ClientEvent` (the forward subset the agent
  sees). Rename ONLY the `ClientEvent` envelope discriminant `kind:"action"` ‚Üí
  `kind:"tap"` (the agent-routed action envelope), leaving `FacetAction`/`AgentAction`
  and the `"agent"` guard untouched (RISK-API-6). Local `navigate`/`toggle` taps ‚Äî
  today handled entirely in the renderer and never logged ‚Äî now fire a **record-only**
  background send AFTER their optimistic view-state mutation, over the SAME serialized
  client channel that carries `/event`, stamped with a per-session monotonic `seq`.
  The server gains a `POST /record` endpoint that persists the collected `tap`
  (with its resolved `effect`) to the existing `Sink` **on the same per-visitor lane
  and the same runtime record queue** as `/event`, WITHOUT invoking the agent ‚Äî so
  `Sink.history()` is one durable, append-ordered, bidirectional timeline. No replay
  player, no new store, no retry: v1 is the *log*.

## Design detail (the contract that pins the WUs)

### Types (`@facet/core::protocol.ts`)
- `TapEffect = { readonly navigate: string } | { readonly toggle: NodeId }` ‚Äî the
  renderer-resolved local effect (mirrors `ClassifiedPress` navigate/toggle).
- `CollectedEvent` = the log currency:
  - `{ kind:"visit"; visitor: VisitorContext }`
  - `{ kind:"message"; text: string }`
  - `{ kind:"tap"; target?: NodeId; effect?: TapEffect; action?: FacetAction;
     fields?: Readonly<Record<string,string>> }` ‚Äî a **local** tap carries
    `effect` (+ `target`); a **forwarded agent** tap carries `action`. `name?`/
    `payload?` per DC-006 are read off `action`; the type unifies both shapes.
  - plus an optional per-session monotonic `seq?: number` on every variant (wire
    field, forward-compatible).
- `ClientEvent` = the FORWARD subset the agent sees + `/event` carries: `visit |
  message | { kind:"tap"; action: FacetAction; fields?; }` (the renamed action
  envelope) + optional `seq?`. It is **structurally assignable to** `CollectedEvent`
  (forward ‚äÜ collected), so `StoredEvent.event: CollectedEvent` accepts both a
  forwarded agent tap and a local tap. **The agent keeps receiving `ClientEvent`**
  (`FacetAgent`, `AgentEventFrame.event`, `FacetContext.event` unchanged ‚Äî RISK-API-4).
- `FacetTransport` gains an optional `record?(event: CollectedEvent): void` ‚Äî an
  explicit record method (not a raw fetch), additive so existing implementers/test
  doubles compile (RISK-INV-5).
- **Scope lock (RISK-API-6):** rename ONLY the `ClientEvent` envelope discriminant.
  `AgentAction.kind:"agent"`, `NavigateAction`/`ToggleAction`, the `onAction` prop
  name, `sanitizeActionPayload`, `validateTree`'s action normalization, and the
  server's `actionKind !== "agent"` guard all stay UNCHANGED.

### Ordering (RISK-INV-2 ‚Äî the deepest seam)
`/record` MUST ride the SAME two serial queues `/event`'s record rides:
1. Server `lane` (`server.ts:499`, per-visitor): `/record`'s handler acks 202 then
   enqueues its work via `lane(visitorId, ‚Ä¶)` ‚Äî NOT out-of-lane like rehydrate.
   Because the lane is serial, a slow `/event` turn's lane task completes (and, in
   `persist`, has already *synchronously enqueued* its Sink write on
   `serializeRecord`) before the following `/record` lane task runs.
2. Runtime `serializeRecord` (`runtime.ts:67`): `runtime.record()` enqueues its Sink
   write on the SAME queue as `persist`, so append order == send order regardless of
   async sink latency. **Ordering key = Sink append order, never `StoredEvent.at`.**

### Client channel + seq (RISK-INV-3 / RISK-INV-5)
`SseTransport` stamps a per-session monotonic `seq` at the SINGLE serialization
point and routes BOTH `/event` (`send`) and `/record` (`record`) through the SAME
pre-connect queue + `sendChain`. The pre-connect queue entry is tagged with its
endpoint; `seq` is stamped once, at the moment an event is committed to `sendChain`
(so queued-then-flushed events get a send-order seq, monotonic across both
endpoints and every caller ‚Äî never stamped in userland). A failed `/record` POST
mirrors the existing best-effort `catch` (`sse-transport.ts:52-56`): log + drop, NO
throw, NO retry (v1), NO callback into the renderer. `record` targets ONLY the
reference `/record` URL ‚Äî never an arbitrary/domain URL (invariant #7).

### Renderer seam (RISK-INV-1)
`StageRenderer` keeps the local view-state mutation as the authoritative,
SYNCHRONOUS, optimistic effect that runs FIRST and unconditionally. A NEW optional
prop `onRecord(tap: CollectedEvent)` (distinct from `onAction`) fires AFTER the
`setCurrentScreen`/`setVisibilityOverrides` in the `navigate`/`toggle` branches of
`handlePress`, carrying the already-resolved effect from `ClassifiedPress` plus the
pressed box's node id as `target` (the effect is captured here, NEVER re-derived
server-side). The record path is fire-and-forget so a failure can never unwind
`currentScreen`/`visibilityOverrides` (DC-003). With `onRecord` omitted, output is
byte-identical (RISK-API-3). The docstring at `StageRenderer.tsx:633-636` (which
currently asserts the very invariant being changed) is updated to describe the new
record-only channel.

### Legacy durable rows (RISK-API-2)
Existing on-disk/DB rows contain `{kind:"action"}`. The quickstart replay readers
`describeEvent` (`prompt.ts`) and the stub accept BOTH literals: a leading
normalizer coerces a legacy `kind:"action"` row into a `tap` before the switch, so
a historical tap still replays (not `"(unknown event)"`). Store readers'
`event: ClientEvent` casts are re-typed to `CollectedEvent` (`postgres-store.ts`;
`file-sink.ts::isStoredEvent` is shape-only and unchanged). A vitest pins that a
stored `{kind:"action"}` row replays as a tap.

## Fail-safe & Boundary Checklist
- **Malformed / empty / null `/record` body:** `isRecordBody` rejects ‚Üí 400, no
  `Sink.record`, no crash (DC-007, WU-5). Same `MAX_FIELDS_KEYS`/`MAX_FIELD_VALUE_CHARS`
  caps + `isFieldsRecord` as `/event`.
- **Spoofed local-effect tap on `/event`:** `isEventBody` KEEPS rejecting any
  `tap` whose `action.kind !== "agent"` ‚Äî navigate/toggle can't reach the agent
  (RISK-INV-4, WU-5; asserted in `server.test.ts`).
- **Record send fails (offline blip):** best-effort `catch` logs + drops; no throw,
  no retry, view-state (`currentScreen`/`visibilityOverrides`) unchanged (DC-003,
  WU-3/WU-4).
- **Deeply nested / cyclic tree at press time:** `onRecord` carries the
  already-resolved `ClassifiedPress` effect ‚Äî no tree walk at record time; the
  renderer's existing ancestor-set/depth/budget guards are untouched.
- **Rapid / racing events (`/event` slow, `/record` behind it):** shared
  `sendChain` (client) + shared `lane` (server) + shared `serializeRecord`
  (runtime) keep append order = send order; `seq` detects any gap (DC-002/DC-004).
- **Legacy durable `{kind:"action"}` row:** normalized to `tap` at the read
  boundary ‚Üí still replays (RISK-API-2, WU-6).
- **`onRecord` prop omitted:** every navigate/toggle branch no-ops the record call;
  rendered output byte-identical (static suite pins it ‚Äî RISK-API-3, WU-4).
