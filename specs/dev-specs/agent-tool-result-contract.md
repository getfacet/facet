- One-paragraph technical approach:
  Add a durable LLM-facing tool result contract, implement a provider-neutral
  structured observation formatter inside `@facet/agent-tools`, update executor
  and buffer outcomes to emit contract-shaped bounded JSON strings, teach the
  reference-agent prompt how to react to those outcomes, and update docs/tests so
  future tools cannot silently return ambiguous success.

## Fail-safe & Boundary Checklist

- Malformed / empty / null input: result is `status: "error"`,
  `outcome: "rejected"`, `applied: false`, no patch, unchanged shadow.
- Unknown tool: same rejected contract with available tool names bounded in the
  message.
- Invalid parent / non-box parent / missing node: rejected with concrete
  `next_action`.
- Pending closure buffer: `status: "pending"`, `outcome: "pending"`, no patch,
  bounded missing child ids, and next action to define missing child nodes.
- Sanitized full render: `status: "ok"`, `outcome: "applied_with_warnings"`,
  `warnings` populated, and next action to inspect/retry if needed.
- Applied but not visible: `status: "ok"`, `outcome: "applied_not_visible"`,
  `applied: true`, `visible_to_visitor: false`, and next action to attach or
  inspect.
- Deeply nested / cyclic tree: unchanged core validation/folding boundaries;
  observation warnings are bounded and never dump full stage JSON.
- Offline provider / provider retry: unchanged reference harness behavior; this
  feature changes tool-result observations after provider tool calls.
- Rapid/racing events: unchanged runtime serial queue; buffer still processes
  one provider step sequentially against the current local shadow.
