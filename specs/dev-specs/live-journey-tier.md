  **browser mechanics** (`journey.ts`) get a real red_check via a **stub-server

  the DOM-settle wait + the negative-fixture capture; the **server lifecycle +
  the `node dist/cli.js` bin smoke** (`harness.ts`, DC-005/DC-006) get a
  stub-boot red_check; and the **orchestration** (workflow JS + skill + config +
  docs) is wired last as justified-N/A tooling that *delegates every gating

### Decision A — verdict aggregation: one pure, unit-tested rule; the single source the workflow calls

`packages/quickstart/e2e/journey/verdict.ts` is the ONLY place the HARD/SOFT/quorum
policy lives. The workflow's synthesis step writes the collected votes to a JSON
file and runs `node .../verdict.js <votes.json>`, so the tested function is what
actually decides PASS/FAIL — no second, drift-prone copy of the rule in the
orchestration JS.

```ts
export type LensId = "safety" | "render" | "responsiveness" | "fidelity" | "diversity";
export type Severity = "hard" | "soft";
export type VoteVerdict = "pass" | "fail" | "abstain";

export interface Vote { readonly lens: LensId; readonly visitor: string; readonly verdict: VoteVerdict; }
export interface LensPolicy { readonly lens: LensId; readonly severity: Severity; readonly quorum: number; }
export interface TierVerdict {
  readonly result: "PASS" | "FAIL";
  readonly warnings: readonly string[];           // one per failed/insufficient SOFT lens×visitor
  readonly blocking: readonly string[];           // one per failed/insufficient HARD lens×visitor (names the visitor)
  readonly perLens: readonly { lens: LensId; visitor: string; outcome: "pass" | "fail" | "insufficient" }[];
}

export const DEFAULT_LENS_POLICY: readonly LensPolicy[] = [
  { lens: "safety",         severity: "hard", quorum: 2 }, // 3 adversarial votes, need ≥2 valid
  { lens: "render",         severity: "hard", quorum: 2 }, // 2 functional votes
  { lens: "responsiveness", severity: "hard", quorum: 2 }, // 2 functional votes
  { lens: "fidelity",       severity: "soft", quorum: 2 }, // 2 votes
  { lens: "diversity",      severity: "soft", quorum: 1 }, // 1 cross-visitor vote
];

export function aggregateVerdict(
  votes: readonly Vote[],
  policy?: readonly LensPolicy[],
): TierVerdict;
```

Rule (pure, deterministic — no I/O):

1. Group votes by `(lens, visitor)`. Valid votes = `verdict !== "abstain"`.
2. Per group: if `validCount < policy.quorum` ⇒ **`insufficient`** (DC-004: below
   quorum is NEVER a silent pass). Else majority of valid votes decides
   `pass`/`fail` (a tie counts as `fail` — conservative).
3. **HARD** lens with any `fail` or `insufficient` on ANY visitor ⇒ each such
   `(lens, visitor)` is pushed to `blocking` and `result = "FAIL"`.
4. **SOFT** lens with any `fail` or `insufficient` ⇒ pushed to `warnings`; it
   **never** sets `result = "FAIL"` (DC-003).
5. `result = "FAIL"` iff `blocking` is non-empty; otherwise `"PASS"` (possibly
   with warnings).
6. A lens with **zero** votes for a visitor that was expected to be judged is
   treated as `insufficient` for that `(lens, visitor)` (no silent pass from a
   missing judge — the workflow passes the expected `(lens, visitor)` matrix so
   a dead judge that produced no vote row still counts as below quorum).

`verdict.ts` also exports a tiny `main()` (guarded by an `import.meta` entry
check) that reads a votes JSON file argv, runs `aggregateVerdict`, prints the
verdict, and exits `0` on PASS / `1` on FAIL — this is what the workflow invokes.
Quora are policy params (Decision Lock: "adjustable"), defaulted per the vote
counts above.

### Decision B — journey harness + `node dist/cli.js` bin smoke (server lifecycle, DC-005/DC-006)

`packages/quickstart/e2e/journey/harness.ts`:

- `bootJourney({ agent, agentId }): Promise<RunningQuickstart>` — random free
  port + `EADDRINUSE` retry, mirroring `packages/quickstart/e2e/smoke.test.ts:74`
  and reusing `startQuickstart` (`server.ts:351`, `RunningQuickstart` at
  `server.ts:69` with `.url`/`.close()`). The **real** tier passes
  `createQuickstartAgent({ provider: resolveProvider(...) })`; the **self-test**
  passes `createStubAgent()` — one boot helper, two agents.
- `runBinSmoke(): Promise<BinSmokeResult>` (DC-005) — spawn
  `node <pkg>/dist/cli.js --stub` on a random port, poll `GET /health` for HTTP
  200 within a bounded timeout, then kill the child; returns
  `{ ok: boolean; detail: string }`. Under `--stub` no key is needed, so this is
  deterministic. In the dev monorepo this is where the documented
  workspace-resolution gap (`@facet/*` → `src/*.ts`) surfaces: `ok:false` +
  the captured stderr is reported (the point of DC-005). Requires
  `pnpm --filter @facet/quickstart build` first (shared preflight).
- **Teardown (DC-006):** every boot is paired with `running.close()` in a
  `finally`; the bin-smoke child is killed in a `finally`. The self-test asserts
  the port is free again after `close()` (no orphan).

### Decision C — the fixed Playwright journey + DOM-settle wait (OQ-1) + negative fixture (DC-002)

`packages/quickstart/e2e/journey/journey.ts`:

- `runJourney(page, opts): Promise<JourneyResult>` drives the FIXED journey and
  captures **≥4** screenshots into `opts.outDir`:
  1. `goto(url)` → `waitForSelector` the stage root → **shot 1** (load).
  2. type into the ChatDock "Add a section titled Pricing with three simple plan
     cards" → send → **settle** → **shot 2**.
  3. type "Now restyle the whole page with a playful, cat-themed look" → send →
     **settle** → **shot 3**.
  4. locate the most prominent pressable (first `[data-facet-*]` pressable /
     button in document order) → click → **settle** → **shot 4**.
- **OQ-1 resolved — DOM-settle, not a fixed sleep:** after each send/click,
  `settleDom(page)` polls the stage-root subtree and resolves when its serialized
  shape (child count + `innerHTML` length hash) is **unchanged across a quiet
  window** (default 400 ms) OR a **bounded max timeout** (default 15 s) elapses.
  A timeout is NOT a harness failure: the shot is captured anyway and a
  `stepTimedOut` flag is recorded — a visitor whose UI never updated is a signal
  for the render/fidelity judge, not a crash. Each step's own load/selector wait
  is separately bounded; a hard load failure marks THAT visitor failed and the
  workflow continues the others (policy row: "that visitor FAIL (recorded),
  others continue").
- **Negative mode (DC-002):** `--fixture broken` navigates to the committed
  `packages/quickstart/e2e/journey/fixtures/broken.html` (a deliberately broken
  page: a blank canvas + raw JSON dumped as text + an off-screen overlapping
  block) and captures it as the load shot. This gives the **safety judge a
  deterministic known-bad screenshot** to be exercised against, WITHOUT relying
  on the LLM to actually break — DC-002's claim is that the JUDGE catches a
  break, so the input is fixed and the judge/aggregation is what's under test.
- `journey.ts` exposes a `main()` CLI (`node journey.js --url <url> --visitor
  <id> --out <dir> [--fixture broken]`) that the workflow's visitor agents run
  via shell.
- **No-secrets (constraint):** screenshots are of the rendered page only; the
  page never renders the provider key, and the ephemeral `visitorId` is the only
  identifier — nothing beyond what the page itself shows is captured.

`journey.selftest.test.ts` is the **red_check**: boot the STUB server via
`bootJourney({ agent: createStubAgent() })`, run `runJourney` in **headless
chromium**, assert **≥4 screenshots** exist, that the post-chat settle observed a
DOM change (stub echoes the message), that a click was dispatched, and that
`--fixture broken` produces a shot of the broken page. Deterministic (stub), no
key; needs `playwright install chromium` (preflight).

### Decision D — the live-journey Workflow (fan-out → judge → synthesize)

`.claude/workflows/live-journey.js` mirrors the `spec-bridge.js` /
`code-review.js` shape (fan-out → panel → synthesize), all `fable`-model agents;
the workflow JS itself has no shell, so shell-bearing work is done by the agents
it spawns:

1. **Preflight agent** — boots ONE shared server (`harness.bootJourney` with the
   resolved real provider), runs `harness.runBinSmoke()` (DC-005). Boot fail OR
   bin-smoke fail ⇒ **preflight FAIL, no visitors run** (brief Example 2). Emits
   the shared `url` + the bin-smoke line.
2. **Visitor fan-out (3)** — each visitor agent runs `node journey.js` with a
   FRESH `visitorId` and its own artifacts subdir → ≥4 screenshots. 3 independent
   sessions/stages, no shared client state (proves per-visitor). A visitor whose
   journey fails is recorded; the others continue.
3. **Judge fan-out (per visitor × lens)** — vision judges vote `pass`/`fail`/
   `abstain` against LOOSE semantic criteria (NOT golden-image): **2** functional
   votes each for `render` + `responsiveness`, **3** adversarial votes for
   `safety`, **2** for `fidelity`, **1** cross-visitor `diversity` judge that sees
   all three load shots. A judge that dies/abstains counts as an abstain; the
   expected `(lens, visitor)` matrix is passed to `verdict.ts` so a missing vote
   is `insufficient`, never a silent pass. Judges inherit the session's
   vision-capable model (Decision Lock).
4. **Synthesize** — write `votes.json`, run `node verdict.js votes.json` →
   PASS/FAIL + warnings; assemble an optional GIF from one visitor's shots; then
   **teardown the shared server even on exception** (no orphan — DC-006). Report
   the per-lens table + the bin-smoke line + artifact paths.

The `/live-test` skill invokes this tier **after** the unchanged vitest Tiers
1a/1b/2. **No key present ⇒ SKIP-with-reason** (this tier is owner-run,
pre-merge/on-request, explicitly NOT a CI gate — brief non-goal), distinct from
the Tier-2 SKIPPED=FAIL rule which is unchanged.

### WU sizing rationale

4 WUs, split by testability seam and the ≤5-file cap: **WU-1** (verdict, 2
files) is the pure deterministic core and lands independently with a crisp
RED→GREEN; **WU-2** (harness, 2 files) owns server lifecycle + the bin smoke,
independent, stub-boot red_check; **WU-3** (journey + self-test + broken fixture +
the Playwright dep, 4 files) depends on WU-2's boot helper and gets its red_check
from a headless stub self-test; **WU-4** (workflow + skill + gitignore + AGENTS.md
note, 4 files) is justified-N/A orchestration/config/docs that delegates every
gating decision to WU-1's tested `verdict.ts` and is exercised by the recorded
owner-run tier runs (happy path + negative fixture). Every new file is in exactly
one WU; no orphans.

## Fail-safe & Boundary Checklist

Every entry maps to a WU test_plan item or a recorded WU-4 run:

- **Malformed / empty / null input (verdict):** empty vote list, all-abstain HARD
  lens, a `(lens, visitor)` with zero rows ⇒ `insufficient` ⇒ HARD FAIL (never a
  silent pass); SOFT `insufficient` ⇒ warning only — WU-1 (`verdict.test.ts`,
  DC-003/DC-004). The verdict CLI on a missing/garbage votes file exits FAIL
  (fail-closed).
- **Page won't load / step times out:** each step's selector/load wait is
  bounded; a hard load failure marks THAT visitor failed and the workflow
  continues the others; a settle timeout captures the shot + sets `stepTimedOut`
  (a judge signal, not a crash) — WU-3 (`journey.selftest.test.ts` settle/step
  behavior) + WU-4 (multi-visitor isolation).
- **Judge dies / abstains:** treated as abstain; the expected `(lens, visitor)`
  matrix ⇒ missing vote is `insufficient`; HARD below quorum ⇒ FAIL — WU-1 +
  WU-4.
- **Offline / server won't boot / bin smoke fails:** preflight FAIL, no visitors
  run; boot uses `EADDRINUSE` retry — WU-2 (`harness.test.ts`) + WU-4 preflight.
- **Teardown on exception (no orphan):** `running.close()` and the bin-smoke
  child kill run in `finally`; the self-test asserts the port frees — WU-2
  (`harness.test.ts`) + WU-4 (synthesize teardown, recorded no-listener check).
- **Chromium missing:** preflight `pnpm exec playwright install chromium`;
  absence surfaces as a preflight FAIL with the install guidance (recorded in
  WU-4) — not a silent pass.
- **Secret leakage:** screenshots/GIF capture only rendered page content; the
  page never renders the key; the ephemeral `visitorId` is the only identifier
  (brief constraint) — WU-3/WU-4.
