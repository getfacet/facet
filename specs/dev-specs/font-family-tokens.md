## Overview

- Feature slug: `font-family-tokens`
- Intake brief: `specs/feature-intake/font-family-tokens.md`
- Context evidence: `specs/context/font-family-tokens.md`
- Affected `@facet/*` packages: `@facet/core`, `@facet/assets`,
  `@facet/react`; quickstart prompt tests only.
- Technical approach: add a closed `FontFamily` token group (`sans`, `serif`,
  `mono`) to `@facet/core`; add `TextStyle.family?: FontFamily`; add a partial
  `FacetTheme.fontFamily` group validated as safe operator data; add
  `@facet/assets` default font-family values; and teach `@facet/react` to
  resolve/apply `fontFamily` through the same null-proto theme overlay path as
  all existing token groups. `STAGE_SPEC` advertises only token names, never raw
  font-family values. Quickstart/ChatDock app chrome remains scoped out for v1
  and continues inheriting/using its compatible system stack.

## Fail-safe & Boundary Checklist

- Malformed / empty / null tree input: unchanged; `validateTree` still returns a
  renderable empty tree and strips invalid style fields. New `family` junk is
  stripped with no throw.
- Malformed / empty / null theme input: unchanged never-throws boundary in
  `validateTheme`; dangerous `fontFamily` values produce errors and refuse the
  document.
- Hostile keys (`__proto__`, `constructor`, `prototype`): `validateGroup` drops
  forbidden keys; `resolveTheme` iterates allowed members only and creates
  null-proto maps.
- Deeply nested / cyclic tree: unchanged `validateTree` depth/cycle logic;
  font-family is a leaf token and does not affect traversal.
- Offline agent / rapid events / racing agent turn: no protocol/action/runtime
  changes. Existing patch ordering and renderer fail-safe behavior apply; the
  final full `pnpm test` gate remains the regression net for those unchanged
  paths.
- Quickstart app chrome: v1 scope is Stage text rendering. `ChatDock` and
  quickstart wrapper do not receive a new theme prop in this change; they remain
  on the compatible inherited/system stack.
