  `@facet/core`, `@facet/runtime`, `@facet/react`, `@facet/assets`,
  `@facet/agent-tools`, `@facet/reference-agent`, `@facet/quickstart`,
  `@facet/bridge`
- Technical approach: add the new agent-facing model additively. `@facet/core`
  remains the source of truth for primitive bricks, intrinsic component names,
  aliases, caps, catalog normalization, and composition validation. Existing
  `stamp` and `stat` surfaces remain valid compatibility APIs; new docs,
  prompts, catalog text, and default catalog prefer `Component` and `metric`.
  Composition components are validated as data assets and expanded through the
  existing safe stamp-like path into ordinary validated nodes. `@facet/react`
  gets private renderer-owned containment helpers so parent placement, child
  internal layout, and bounded overflow rules are enforced consistently.

## Invariant Fit Audit (Facet)

| # | Invariant | Touched? | Safe design |
|---|---|---|---|
| 1 | Responsibility boundary (no backend/domain in Facet) | Yes | `form`, `search`, and `filterBar` only collect/display UI data and reuse existing `FacetAction`; no schema field accepts fetch URLs, query expressions, data sources, or business logic. |
| 2 | Mechanism vs policy | Yes | Core defines the vocabulary and safe component/composition mechanism; operators choose catalog contents and optional component definitions. Defaults do not become a hidden domain policy library. |
| 3 | Fail-safe | Yes | Core validators bound arrays/strings, skip unknown/malformed component nodes, validate component definitions at asset load, and renderer unknowns still render `null` without throwing. Tests include hostile composition and live-render cases. |
| 4 | Declarative + tokens only | Yes | Stage nodes remain typed data. Composition definitions reuse typed node templates and slot strings only. Theme `recipes` remain token-only style recipes; raw HTML/JS/CSS/scalar style values are rejected or ignored. |
| 5 | Flow-only safety | Yes | No `position`, `zIndex`, arbitrary overlay, or free CSS is added. `modal`, `drawer`, and `menu` stay deferred. Layout contract enforces bounded flow roots and explicit renderer-owned scroll only. |
| 6 | Two-writers coherence | Yes | New components reuse existing `agent`, `navigate`, `toggle`, and field collection semantics. No new local component state writer or browser-side content mutation protocol is introduced. |
| 7 | Backend via agent tools, never client fetch | Yes | Data-bearing components render agent-supplied values only; filtering/search submission emits actions or collected fields for the agent/backend tools to handle. |

## Fail-safe & Boundary Checklist

- Malformed / empty / null input: `validateTree`, component validation, catalog
  validation, and asset loading return bounded issues and normalized safe output;
  renderer skips unknown or unusable nodes.
- Deeply nested / cyclic tree: existing tree traversal budgets stay in force and
  are covered by current `validate.test.ts`, `tree.test.ts`, and
  `StageRenderer.test.ts` cases; component-definition expansion uses existing id
  remapping and node caps before patches are emitted, with WU-2 adding
  composition-specific deep/cyclic coverage.
- Offline agent / rapid events / racing agent turn: no new writer is introduced;
  interaction components use existing serialized patch/event behavior covered by
  current `runtime.test.ts` queue/race cases; WU-4 keeps asset loading out of the
  event writer path.
- Hostile composition definition: invalid nodes, raw markup/style/code fields,
  unknown component names, duplicate ids, and too-large definitions are rejected
  at asset load or expansion; partial unsafe output is not emitted.
- Layout overflow: primitive and component roots get renderer-owned
  `box-sizing`, `min-width: 0`, `max-width: 100%`, wrapping, and scoped scroll
  ownership; tests use long text and wide tables/charts/media.
