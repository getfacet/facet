## Overview
- Feature slug: `composition-nesting`
- Intake brief: `specs/feature-intake/composition-nesting.md`
- Context evidence: `specs/context/composition-nesting.md`
- Affected `@facet/*` packages: `@facet/core`, `@facet/runtime`, `@facet/agent-tools`,
  `@facet/agent` (direct `expandComposition` consumer, added per RISK-PKG-1);
  `@facet/assets` is a compile-only consumer of the widened `FacetComposition`
  (no source edit — see Public API Impact).
- One-paragraph technical approach:
  A composition's node map may now hold a **closed reference node**
  `CompositionRef = { use: string; slots?: Record<string,string> }` alongside
  ordinary `FacetNode`s. The reference shape is admitted **only inside composition
  definitions** — gated behind a new `allowReference` flag on `SanitizeNodeOptions`
  that `validateComposition` passes and `validateTree` never does, so a reference
  node is structurally impossible in the live stage tree. Safety over the whole
  catalog is a **new load-time pass** `validateCompositionGraph(compositions)` in
  `@facet/core` (pure, compositions passed as data) that builds the reference
  graph and prunes any composition on a cycle, with a dangling reference, past the
  depth cap, or whose transitive expansion exceeds the node cap — wired into the
  runtime load path **after** the per-document loop assembles the array.
  `expandComposition` gains an optional `compositions` registry so
  `expandCompositionInner` resolves each `{ use }` reference **recursively to
  primitive/native nodes server-side** (threading the reference's `slots` as the
  child's params), keeping the load-time graph pass as the primary gate and a
  depth-cap + visited-set as the never-throw expand-time backstop. Both live
  call-sites thread the registry. Existing non-nested compositions are unchanged
  (additive).

## Decision Locks resolved (intake open questions)
- **Reference node shape** → LOCKED: `{ use: string; slots?: Record<string,string> }`,
  a closed shape where ONLY `use` and `slots` survive sanitation; lives solely in
  `FacetComposition.nodes` values, never in the `FacetNode` union nor any published
  brick/catalog list.
- **Dangling-reference policy** → LOCKED: **refuse** the referencing composition at
  load (default from the brief). Pruning iterates to a fixpoint (a composition that
  references a refused composition is itself refused).
- **Max nesting depth + slot threading** → LOCKED: `MAX_COMPOSITION_NEST_DEPTH = 8`;
  slot threading is **direct** — a reference's `slots` become the referenced
  composition's params (filled against the OUTER composition's params/defaults
  first, so an outer marker like `{{status}}` in a reference slot resolves before
  the child expands). No automatic multi-layer propagation, no DSL.

## Fail-safe & Boundary Checklist
- **Malformed / empty / null reference** (`{ use: 123 }`, `{}`, extra keys, forbidden field on a ref): `inspectCompositionNodes` + `sanitizeNode` reject/strip — a ref with a non-string `use`, missing `use`, or any `FORBIDDEN_COMPOSITION_FIELDS` key refuses the composition; unknown extra keys are dropped so only `{ use, slots }` survive. Never throws.
- **Deeply nested / cyclic graph**: load-time `validateCompositionGraph` prunes cycles (`a→b→a`, self `a→a`), dangling refs, chains `> MAX_COMPOSITION_NEST_DEPTH`, and DAGs whose memoized transitive node count `> MAX_COMPOSITION_NODES`, each with a bounded issue; the accepted subset is fixpoint-consistent. Expand-time carries a redundant depth-cap + visited-set so a hand-built registry can never infinite-loop.
- **Missing slot value**: falls through to the child slot's declared default (existing `fillString` behavior) — unchanged.
- **Reference used via the in-process Stage with no registry**: `expandComposition` with absent `compositions` fail-safe-skips the reference (drops it, bounded issue) and never throws; non-nested compositions are byte-for-byte unchanged.
- **Reference node authored into the live stage**: `validateTree` (no `allowReference`) drops it as an unknown node — fail-safe skip, never rendered.
- **Offline agent / rapid events / racing turn**: unaffected — expansion is synchronous server-side; ordering/buffer rules of `use_composition` are unchanged.
