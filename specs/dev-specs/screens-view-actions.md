## Overview

- Feature slug: `screens-view-actions`
- Intake brief: `specs/feature-intake/screens-view-actions.md`
- Context evidence: `specs/context/screens-view-actions.md`
- Affected `@facet/*` packages: `@facet/core`, `@facet/react`, `@facet/agent`, `@facet/cli`, `@facet/kit`, `@facet/bridge` (+ `apps/playground`)
- One-paragraph technical approach: The stage gains first-class named screens as
  **named roots into the existing flat node map** ‚Äî `FacetTree` grows two optional
  fields, `screens?: Readonly<Record<string, NodeId>>` (screen name ‚Üí the screen's
  root box id in `nodes`) and `entry?: string` ‚Äî so `/nodes/<id>` patch paths, the
  shared `applyPatch`, and every one of the 24 `FacetTree` consumers keep working
  unchanged (a legacy `{root, nodes}` tree IS the single-screen form: screens
  absent ‚áí render `root`). `FacetAction` becomes the discriminated union
  `AgentAction | NavigateAction | ToggleAction`, designed to be
  **source-compatible**: `AgentAction.kind` is optional (`kind?: "agent"`, encoding
  the locked "bare `{name}` ‚â° agent action" equivalence at the type level) and the
  new variants carry `name?: undefined` so existing `.name` probes on the union
  still compile and return the runtime truth. `validateTree` stays the single
  normalization point (stamps `kind: "agent"` on bare actions, strips unknown
  kinds/malformed shapes, sanitizes `screens`/`entry`, cycle/depth-breaks every
  screen root). The browser owns VIEW-STATE ONLY: `StageRenderer` holds React
  state `{ currentScreen, visibilityOverrides }`, executes `navigate`/`toggle`
  instantly with zero transport traffic, and forwards only agent-kind actions to
  `onAction`. Initial visibility is content-declared via a new
  `hidden?: boolean` on `BoxNode` (server-written default), overridden by the
  browser's toggle map after first interaction. `STAGE_SPEC`, the `Stage`
  authoring API, the CLI, kit, and the playground pick up the new vocabulary
  additively.

### Decision 1 ‚Äî exact stage shape (resolves the brief's open question)

`screens` maps **name ‚Üí NodeId** into the shared flat `nodes` map instead of
name ‚Üí per-screen tree. Rationale: (a) it preserves AGENTS.md invariant #2
("only patches travel", one flat map is what makes incremental `/nodes/<id>`
streaming work) ‚Äî a nested per-screen `nodes` map would break every existing
patch path, `Stage` op helper, and the raw client-side patch flow; (b) it is
**additive**: all 24 `FacetTree` consumer files keep compiling and behaving
because `root`/`nodes` remain the required base and a screenless tree is the
single-screen form (no materialized `screens: {main: ‚Ä¶}` ‚Äî materializing would
churn every stored stage and `toEqual` test for zero benefit); (c) nodes can be
shared across screens (one header box on every screen); (d) `validateTree`
normalization is a filter over one map, not a recursive per-screen re-validate.
`entry` is optional in the type (raw live-path trees are untrusted anyway) but
`validateTree` always emits a valid `entry` whenever it keeps a non-empty
`screens` map (falling back to the first kept key), and the renderer defends
independently.

### Decision 2 ‚Äî initial visibility (resolves the brief's open question)

`hidden?: boolean` on `BoxNode` is the content-declared DEFAULT; the browser's
toggle map is a per-node OVERRIDE that wins after first interaction (and resets
on refresh). A view-state-only "everything visible initially" model cannot
express the brief's own Example 2 ‚Äî a pre-drawn menu panel that is hidden on
FIRST paint with zero interactions ‚Äî without an agent round-trip, which defeats
the feature. Keeping the default in content and the override in view-state also
keeps invariant #6 clean: the server writes `hidden` (content), the browser
writes only the override map (view-state), never the same datum. It is minimal
(one optional boolean, on the one container brick ‚Äî panels are boxes; wrap
anything else in a box) and fail-safe (non-boolean `hidden` stripped by
`validateTree`; the renderer only honors literal `true`; `toggle` works on any
existing node id, defaulting to visible for nodes that cannot declare `hidden`).

### Decision 3 ‚Äî source-compatible union (migration strategy for 24 consumers)

`FacetAction = AgentAction | NavigateAction | ToggleAction` with:

```ts
export interface AgentAction {
  readonly kind?: "agent"; // optional: a bare {name} IS the (legacy) agent action
  readonly name: string;
  readonly payload?: Readonly<Record<string, string | number | boolean>>;
}
export interface NavigateAction {
  readonly kind: "navigate";
  readonly to: string;
  readonly name?: undefined; // keeps `.name` probes on the union compiling (truthfully undefined)
}
export interface ToggleAction {
  readonly kind: "toggle";
  readonly target: NodeId;
  readonly name?: undefined;
}
export type FacetAction = AgentAction | NavigateAction | ToggleAction;
```

Narrowing works (`kind === "navigate"` / `"toggle"` are exact literal
discriminants; the else-branch is `AgentAction`). A required `kind` would
atomically break ~12 consumer files (kit/playground/react-test literals plus
every `event.action.name` reader), which cannot fit a ‚â§5-file WU without a red
repo mid-sequence; the optional-kind + `name?: undefined` design makes WU-1
compile-neutral for the whole repo except one line
(`apps/playground/src/generated.tsx:43` assigns `action.name` ‚Äî now
`string | undefined` ‚Äî to a `string | null` state; fixed in WU-1).
`ClientEvent`/`StageRendererProps.onAction` keep their `FacetAction` signatures
(no protocol change); the runtime guarantee ‚Äî only agent-kind actions are ever
forwarded/sent ‚Äî is enforced by the renderer and pinned by DC-001/002 tests.
Canonical `kind: "agent"` is stamped by `validateTree` and by the renderer at
emit time; internal constructors migrate to explicit `kind: "agent"` in WU-6/7.

## Fail-safe & Boundary Checklist

- Malformed / empty / null input: `validateTree(null | 42 | "x")` still returns
  `EMPTY_TREE` (existing behavior, re-pinned). Malformed `screens` (non-object,
  numeric values, dangling ids, non-box targets), malformed `entry`, and
  malformed/unknown-kind actions are stripped with issues, never thrown.
  Renderer: junk `onPress` ‚áí non-pressable; junk `screens` on the raw path ‚áí
  plain `root` render; `hidden: "yes"` (non-boolean, raw path) ‚áí treated as
  visible (only literal `true` hides).
- Deeply nested / cyclic tree: existing gray/settled DFS + `MAX_DEPTH = 100` now
  runs from every screen root as well as `root` (shared settled set), so a cycle
  or 5000-deep chain inside a non-entry screen can neither recurse forever nor
  overflow ‚Äî and the renderer's own ancestor-set + depth guards still hold for
  raw-path trees.
- Offline agent / rapid events / racing agent turn: navigate/toggle are
  browser-local `setState` ‚Äî they work while the agent is offline or mid-turn
  and are idempotent under rapid clicks (no queue; double-click toggle =
  open+close, last-write-wins). An agent redraw landing mid-view re-derives the
  screen: alive ‚áí stay (DC-008), deleted ‚áí entry ‚áí first ‚áí plain root (DC-004).
  Content and view-state never contend by construction (invariant #6).
