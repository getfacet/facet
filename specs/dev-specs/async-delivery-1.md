## Overview

- Feature slug: `async-delivery-1`
- Intake brief: `specs/feature-intake/async-delivery-1.md` (Decision Lock FIXED
  — late results applied+delivered; SSE-native `Last-Event-ID` + per-session
  seq, join-fan-out-first; spawn cap default 4 via `FACET_MAX_CONCURRENT` +
  FIFO with per-visitor order kept; client POST abort timeout;
  behavior-centric HTTP harness scoped to this round).
- Context evidence: `specs/context/async-delivery-1.md` (all file:line anchors
  re-verified against `main` post hardening-1; RISK register §8 consumed below).
- Affected `@facet/*` packages: `@facet/core`, `@facet/runtime`,
  `@facet/server`, `@facet/client`, `@facet/bridge`.
- One-paragraph technical approach: The server stops discarding a remote
  agent's result when the per-event timeout fires — the timeout now resolves
  the HTTP wait with a **non-terminal interim `say`** and parks
  `{visitor, event}` in a bounded **late window** keyed by `requestId`
  (`server.ts:79-82` `Pending` gains both); a later `/agent/control` POST for a
  parked id re-injects the messages through a new additive runtime method
  `FacetRuntime.applyMessages(visitor, event, messages)` that reuses the
  existing per-visitor serial queue and per-op salvage (`runtime.ts:39,95-125`).
  Every browser frame gains a per-session monotonic **seq** carried ONLY in SSE
  `id:` framing (`id: <era>:<seq>` — never in `ServerMessage`), assigned inside
  a server-side **per-visitor delivery lane** (`createSerialQueue`) that
  brackets {apply → seq-assign → log → fan-out} as one ordered critical
  section, so apply-order equals delivery-order by construction (RISK-INV-1).
  Frames land in a server-local in-memory per-session **frame log** (ring ≤ 200
  frames, session map LRU-capped) — the `Sink` is NOT the replay store
  (RISK-API-1, resolved below). A reconnect with a valid `Last-Event-ID` joins
  the fan-out and replays the gap **synchronously** from the in-memory log (no
  await → no interleaving window); anything malformed/out-of-retention/wrong-era
  degrades to a full-snapshot rehydrate that runs OUTSIDE the lane (a token-less
  new tab never waits behind a mid-flight turn) yet still closes the RISK-HRD-4
  loss window: it captures the seq watermark before its store reads, then
  finalizes in one synchronous block — unstamped `{kind:"reset"}` frame, stamped
  snapshot + history says, a replay of any ring frames delivered during the
  reads, then the fan-out join. Stamping starts AT the snapshot (never the
  reset), so a resume token can only exist in a client that has received a
  stage base. The
  spawn bridge gains a global FIFO cap via a new node-free `createSemaphore` in
  `@facet/core` (next to `createSerialQueue`, barrel-exported), applied INSIDE
  the per-visitor serialization so a visitor's own events never reorder. The
  client's ordered POST chain gains a fixed 10s `AbortSignal.timeout`. No new
  dependencies; `STAGE_SPEC` and the brick/token/action vocabulary untouched;
  `LocalTransport`/playground in-process path untouched.

### Design sketch (the seq/lane mechanism, normative for WU-3)

- `sse(res, data, id?)` (`server.ts:90-92`) learns an optional `id` — when
  present it writes `id: <id>\n` before `data:`. Only browser-channel frames
  get ids; agent-channel frames are unchanged (`parseSseFrames` tolerance
  verified, context §5 — not that it matters, since the agent channel never
  carries one).
- Per-session frame log: `Map<visitorId, { era, nextSeq, frames: {seq, json}[] }>`.
  `era` is a random token minted when the log entry is created (server restart
  or LRU eviction ⇒ new era ⇒ stale resume tokens are safely rejected instead
  of replaying wrong-era frames); its alphabet MUST exclude `":"` (e.g. base36)
  so `<era>:<seq>` parses unambiguously. Ring bound `FRAME_LOG_LIMIT = 200` frames per
  session; session map LRU-capped at `MAX_FRAME_SESSIONS = 1000`
  (re-insert-on-touch, same pattern as `bridge.ts:149-157`). Named constants
  with operator comments. Entries are minted lazily by a single get-or-create
  helper (`logFor(visitorId)`) used by both `deliver()` and the rehydrate's
  watermark capture — whichever runs first creates the entry and its era.
- `deliver(visitorId, messages)` — **synchronous, no await inside**: per
  message assign `seq = nextSeq++`, append to the ring, write
  `id: <era>:<seq>` + `data:` to every connected `res`. Frames are logged even
  with zero connections (that is exactly the late-result-while-disconnected
  case DC-001/Example-2 needs replayed).
- Per-visitor **lane** (`createSerialQueue<void>` from `@facet/core`, keyed by
  visitorId, server-side): the `/event` path becomes
  `lane(visitorId, async () => { const m = await runtime.handle(...); deliver(visitorId, m); })`
  (replacing the bare `.then(pushToBrowser)` at `server.ts:364-366`; the
  existing error-say catch also delivers through the lane task). The late path
  is `lane(visitorId, async () => { const m = await runtime.applyMessages(...); deliver(visitorId, m); })`.
  The **full-snapshot fallback rehydrate runs OUTSIDE the lane** — riding the
  lane would make a token-less attach (new tab / second device / incognito,
  the COMMON no-`Last-Event-ID` case) blank for up to `agentTimeoutMs` behind a
  mid-flight turn, where today's rehydrate paints immediately. It still closes
  the RISK-HRD-4 window without the lane, by construction:
  1. synchronously capture the watermark `(era, N0 = lastAssignedSeq)` from the
     get-or-create frame-log entry BEFORE any await;
  2. `await stageFor` / `historyFor` (outside the lane, as today);
  3. one **synchronous finalization block**, which FIRST re-gets
     `logFor(visitorId)` and checks **watermark continuity**: the entry must be
     the same object/era captured in step 1 AND the ring must fully retain
     `(N0, nextSeq)` — a violation (the entry was LRU-evicted and re-minted
     while ≥1000 other sessions churned during step 2, or >`FRAME_LOG_LIMIT`
     frames were delivered during a slow store read and the gap's head fell off
     the ring) means the captured snapshot can no longer be stitched to the
     live stream, so the block writes nothing and `res.end()`s — the same
     fail-safe the store-error path uses; the client's retry re-rehydrates
     fresh. On continuity: write `{kind:"reset"}` with **NO `id:`** → snapshot
     full-replace stamped `id: <era>:<N0>` → history says stamped
     `id: <era>:<N0>` → replay ring frames with `seq > N0` (their own original
     ids) → join the fan-out set.
  `deliver()` and the finalization block are both synchronous, so they cannot
  interleave; anything delivered during step 2 has `seq > N0`, lives in the
  ring (continuity-checked above), and is replayed in step 3 before the join —
  no frame can be lost in the snapshot-read→join window. The
  `server.ts:311-322` RESIDUAL comment is rewritten accordingly. **Stamping
  starts at the snapshot, never the reset**: if step 2 fails (store error), the
  continuity check fails, or the connection drops before step 3, no stamped
  frame was ever written, the client holds no resume token, and the campaign-1
  `.catch` → `res.end()` retry path performs a FULL rehydrate — a token
  therefore implies, inductively, that its holder received a stage base (live
  frames only reach connections that completed a stamped snapshot or a valid
  resume). Documented residuals, both cosmetic and bounded to the reconnect
  instant: a disconnect mid-finalization loses the tail of the history says for
  that view — and because the tab then holds a valid resume token, later
  reconnects resume, so the tail reappears only on a later FULL rehydrate (era
  change / out-of-retention), not the next one; and a say delivered during
  step 2 can appear once in the history replay and once in the ring replay
  (one duplicated chat line). Ring frames replayed onto a snapshot that already
  contains their effects are safe in practice — NOT because `add` is
  upsert-idempotent (true for object members, false for `children/-` array
  appends, which would duplicate an id), but because the renderer dedupes
  sibling ids and a batch the client drops (fail-safe) describes changes the
  snapshot already holds. (Wording amended after review.)
- Resume path (valid `Last-Event-ID` header: parses as `<era>:<seq>`, era
  matches, `0 ≤ seq ≤ lastAssigned`, and the gap is fully retained in the
  ring): join the fan-out set and replay `seq' > seq` **in one synchronous
  block** — join-first is safe precisely because there is no await for a live
  frame to interleave into. Everything else (missing header, garbage, wrong
  era, out-of-retention, future seq) → fallback full rehydrate. `/stream` never
  answers 4xx/5xx for a resume token; the only 400 remains the missing
  `visitorId` (`server.ts:294-298`).
- Timeout/late mechanics: `Pending` (`server.ts:79-82`) gains
  `visitor: VisitorContext` and `event: ClientEvent`. The timer
  (`server.ts:230-233`) resolves the wait with the interim say (A-3 text below)
  and moves the entry to `lateWindow: Map<requestId, {visitor, event}>`
  (FIFO-capped `LATE_WINDOW_LIMIT = 100`; an evicted/unknown requestId keeps
  today's silent-202 behavior — the late guarantee is deliberately bounded).
  `dropAgent` (`server.ts:212-220`) parks its pending entries in the late
  window too, so an agent that reconnects and posts the finished work still
  lands it. `/agent/control` (`server.ts:416-439`): pending hit → in-time
  resolve exactly as today (DC-002); miss + late-window hit → late apply via
  the lane; miss + miss → 202 no-op. Timer-vs-control atomicity is inherent
  (single-threaded map mutation): exactly one of {in-time resolve, interim note
  + late apply} happens per requestId. Deliberate side-effect, accepted: an
  interim-timeout turn produces TWO sink entries for one event — `handleOne`
  records the interim say, then `applyMessages` records the real messages. A
  history replay therefore shows the interim note followed by the real answer
  (chronologically true), and `ForwardSink` consumers observe the event twice;
  deduplication would add cross-call state for no correctness gain this round.

## Assumption Resolutions (brief A-1/A-2/A-3)

- **A-1 — seq does NOT ride the `Sink`; it is server-local** (the brief's own
  named fallback: "a small ring buffer in the server"). Three reasons, each
  decisive: (a) `Sink` is a pluggable interface whose `NullSink`/`ForwardSink`
  reference implementations retain nothing (`sink.ts:33-38,62-78`) — replay
  built on sink history would silently break for exactly the consumers the
  interface exists for; (b) `sink.record` is fire-and-forget behind its own
  serial queue (`runtime.ts:44,89-91`), so history lags delivery and a replay
  read could miss frames whose seq was already delivered — re-opening the very
  window this round closes; (c) rehydrate full-replace frames and the reset
  frame are synthesized per connection and never exist in the sink. Postgres
  implication: **none** — `StoredEvent` (`sink.ts:5-10`), the `Sink` interface,
  and `store-postgres/postgres-store.ts` (fixed SQL columns, context §RISK-API-1)
  are all untouched; no migration. Cost accepted: replay retention is
  in-memory and per-process (ring of 200 frames; beyond it → full rehydrate);
  multi-process fan-out stays out of scope per AGENTS.md (pluggable-adapter
  territory).
- **A-2 — client POST timeout = fixed 10s via `AbortSignal.timeout`**, a named
  constant `POST_TIMEOUT_MS = 10_000` in `sse-transport.ts`, non-configurable
  this round (per the brief). Rationale: the POST is a fire-and-forget 202
  (`server.ts:362-363` answers before the turn runs), so 10s is pure network
  headroom, ~3 orders of magnitude above a healthy RTT; an abort hits the
  existing chain `.catch` (`sse-transport.ts:46-50`) — logged, chain continues,
  no retry (at-most-once preserved). `AbortSignal.timeout` is web-baseline
  (all evergreen browsers), no polyfill, no new dep.
- **A-3 — the interim note is a plain `{kind:"say"}`** (no new message kind;
  `ServerMessage` vocabulary untouched). The text changes from the
  terminal-sounding `"(agent timed out)"` (`server.ts:232`) to
  `"(still working — this is taking longer than usual; the answer will appear here when it's ready)"`
  because the pending entry now parks in the late window instead of dying: the
  note must read as non-terminal (brief constraint). The genuinely terminal
  offline paths (`dropAgent` reasons) keep their texts.

Additional scoped design decision (records brief scenario step 8's "waiting
visitors get a brief notice"): the notice is **the server's interim-timeout
say** — a spawn-capped turn that waits past `agentTimeoutMs` triggers it, and
the finished result then arrives via the late path. A bridge-originated
immediate notice would require a second control response per `requestId`
(an `@facet/agent-client` protocol change) and is deliberately out of this
round; DC-005 (the testable criterion) does not require it. Latency under
default knobs, stated plainly: a capped-out visitor sees nothing until
`agentTimeoutMs` (120s default) fires their interim note — operators wanting a
snappier queue notice lower `agentTimeoutMs`. The synergy is worth stating: the
spawn cap is only safe to add BECAUSE late delivery lands queued turns that
outlive the timeout.

## Fail-safe & Boundary Checklist

- **Malformed `Last-Event-ID`**: empty / non-`era:seq` shape / non-integer seq /
  negative / era mismatch / seq beyond `lastAssigned` / gap older than the
  200-frame ring → full-snapshot rehydrate (reset → replace → says). Never a
  4xx/5xx on `/stream`; missing-`visitorId` 400 (`server.ts:294-298`) is the
  only rejection, unchanged.
- **Offline agent**: no agent stream → `offlineFor` unchanged
  (`server.ts:207-210`). Agent dies mid-turn → `dropAgent` resolves the waits
  with the offline note AND parks the entries in the late window, so a
  reconnecting agent that posts the finished result still lands it; if it never
  does, the entry ages out of the FIFO cap harmlessly.
- **Rapid events**: client POSTs stay serialized per transport
  (`sse-transport.ts:21,38-51`, campaign 1) now with a 10s abort so one
  black-holed POST can't wedge the chain (DC-007); server-side the per-visitor
  lane orders apply+deliver, so seqs are strictly increasing per session and
  frames can't cross.
- **Late-result-vs-live-event race**: both are lane tasks — arrival order at
  the lane is apply order is seq order (invariant #6 row). The pending-map
  mutation is single-threaded: exactly one of {in-time resolve, interim note +
  late apply} per requestId.
- **Cap misconfig**: `FACET_MAX_CONCURRENT` non-integer / `< 1` → `cli.ts`
  prints the offending value and exits 1 (mirror of `parseBridgePort`,
  `env.ts:37-46` / `cli.ts:27-33`) (DC-006). Programmatic
  `BridgeOptions.maxConcurrent < 1` → `createSemaphore` throws at
  construction (config-time fail-fast, same posture as `parseBridgePort`) —
  never a mid-request crash.
- **Malformed bodies / frames**: `/agent/control` shape check (`isControlBody`)
  unchanged → 400; unknown requestId not in the late window → 202 no-op
  (today's behavior, now the *bounded* rather than universal outcome). Client
  `onmessage` JSON guard unchanged (`sse-transport.ts:71-77`); an `id:` line is
  invisible to it (browser-native) and to `parseSseFrames` (context §5).
- **Memory bounds**: frame ring ≤ 200/session, session log map LRU ≤ 1000,
  late window FIFO ≤ 100 — all named constants with comments; `browserStreams`
  pruning unchanged (`server.ts:302-308`).
- **Rehydrate read failure / mid-rehydrate disconnect**: `.catch` → log +
  `res.end()` so EventSource retries (campaign-1 fix preserved). Because the
  reset frame is unstamped and stamping starts at the snapshot, a connection
  that dies before the finalization block holds NO resume token — the retry
  performs a full rehydrate, never a false "caught-up" resume (WU-3 test).
  Cosmetic residuals, bounded to the reconnect instant: a mid-finalization
  disconnect can lose the tail of the history says for that view, and a say
  delivered during the rehydrate's store reads can render once from the history
  replay and once from the ring replay (one duplicated chat line).
- **In-process path**: `LocalTransport`/playground untouched — no reconnect, no
  timeout, no seq on that path.
