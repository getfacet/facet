## Overview

- Feature slug: `kits-themes-as-data`
- Intake brief: `specs/feature-intake/kits-themes-as-data.md`
- Context evidence: `specs/context/kits-themes-as-data.md`
- Affected `@facet/*` packages: `@facet/core`, `@facet/agent`, `@facet/react`,
  `@facet/runtime`, `@facet/kit`, `@facet/quickstart` (+ docs/changeset).
  `@facet/server`, `@facet/client`, `@facet/agent-client`, `@facet/cli`,
  `@facet/store-postgres`, `@facet/bridge`, `apps/playground` are deliberately
  UNCHANGED (RISK-PKG-2: the boot seam is quickstart's own HTML shell; STAGE_SPEC
  embedders inherit the theme line verbatim with no code change).
- One-paragraph technical approach: Four additive layers, one flow. **(1) Theme
  documents:** `@facet/core` gains `FacetTheme` (an operator-authored token-name
  → CSS-value document covering all six token groups) and `validateTheme` — the
  one validator every document passes before it can reach a page (format
  allowlist; `url()`/`var()`/`expression()`/`javascript:` denied as errors;
  dimensions clamped; hostile keys never resolve via null-prototype maps; WCAG
  contrast measured as a warning, never a rejection). `FacetTree` gains
  `theme?: string` — a NAME, never a value — kept-if-string by `validateTree`'s
  output construction (the one mandatory core change; without it
  `runtime.ts:157`'s save-time sanitize would strip the name on first save).
  **(2) Resolution & delivery:** `@facet/react` exports today's `theme.ts`
  values as the `DEFAULT_THEME` document, theme-parameterizes the style fns with
  a defaulted trailing argument (zero-extra-arg calls stay byte-identical), and
  gives `StageRenderer` an optional `themes` prop; `resolveTheme` maps any
  `tree.theme` (treated as `unknown` — the live patch path can put junk there)
  to a resolved null-proto token map, unknown/missing names falling back to the
  default. The validated map ships to the browser ONCE, inline in quickstart's
  own `SHELL_HTML` (`<`-escaped JSON) — no protocol change, `@facet/server` and
  `@facet/client` untouched. **(3) Assets registry:** `@facet/runtime` gains the
  `AssetsStore` adapter (themes + stamps + initial tree per agent — the exact
  `StageStore` posture: interface + `MemoryAssets` in the browser-safe barrel,
  `FileAssets` behind the existing `./node` subpath), a `loadAssets` step that
  runs the core validators and skips invalid documents with logged issues, and
  `withInitialStage` — a `StageStore` decorator that seeds fresh sessions from a
  validated, *seedable* initial tree inside the runtime's serialized write path
  (the `validateTree`-returns-`EMPTY_TREE` trap is refused up front, so an
  invalid tree falls back to model-first paint instead of silently seeding an
  empty stage). **(4) Agent-facing surface:** `STAGE_SPEC` teaches
  `theme?: "<name>"` select-by-name-only; `Stage` gains a `theme(name)` op (one
  top-level RFC 6902 `add`, the `screens()` precedent); quickstart's prompt
  layer ② injects theme names + one-line descriptions and stamp fragments with
  the mandatory id-prefix copy rule (byte-identical output when no assets
  exist); `@facet/kit`'s `Builder` gains per-instantiation id prefixes plus a
  `fragment()` graft API, and the presets double as `KIT_STAMPS`, the default
  stamp library.

### Decision 1 — `FacetTheme` + `validateTheme`: the one safety boundary for operator CSS

Raw CSS values enter the system for the first time — as **operator data**, never
as tree content or model output. They pass exactly one gate, in `@facet/core`
(`packages/core/src/theme.ts`, new):

```ts
export interface FacetTheme {
  readonly name: string;                                        // /^[a-zA-Z0-9][a-zA-Z0-9_-]{0,63}$/
  readonly description?: string;                                // one-liner, truncated at 200 chars (warning)
  readonly color?: Readonly<Partial<Record<Color, string>>>;
  readonly space?: Readonly<Partial<Record<Space, string>>>;
  readonly fontSize?: Readonly<Partial<Record<FontSize, string>>>;
  readonly fontWeight?: Readonly<Partial<Record<FontWeight, number>>>;
  readonly radius?: Readonly<Partial<Record<Radius, string>>>;
  readonly ratio?: Readonly<Partial<Record<Ratio, string>>>;
}
export interface ThemeIssue { readonly severity: "error" | "warning"; readonly message: string }
export interface ThemeValidationResult { readonly theme?: FacetTheme; readonly issues: readonly ThemeIssue[] }
export function validateTheme(input: unknown): ThemeValidationResult;
```

- **Documents are PARTIAL overrides** over the default theme (resolution merges;
  Decision 2). Token-group keys iterate the runtime arrays in
  `packages/core/src/tokens.ts` (`SPACES:16` … `RATIOS:64`) — the same
  membership-check precedent as `asToken` (`validate.ts:94-96`). Unknown group
  or token keys ⇒ dropped + warning. `__proto__`/`prototype`/`constructor`
  keys ⇒ dropped outright; every output group is rebuilt on
  `Object.create(null)` (the `react/src/theme.ts:28` pattern), so hostile keys
  can never resolve (DC-003).
- **Errors (document refused, `theme` undefined):** missing/malformed `name`;
  any value containing `url(`, `var(`, `expression(`, or `javascript:`
  (case-insensitive, whitespace-tolerant) or any of `;`, `{`, `}`, `<`, `>`,
  `\`, backticks, or control characters; a color value not matching the
  allowlist (`#rgb[a]`/`#rrggbb[aa]` hex, `rgb()/rgba()/hsl()/hsla()` with
  numeric/`%`/`deg` arguments, or a pure-ASCII-letter keyword ≤ 32 chars); a
  dimension (`space`/`fontSize`/`radius`) not matching `0` or
  `<number><px|rem|em>`; a non-number `fontWeight`; a `ratio` not matching
  `<n> / <m>` with positive finite numbers. Values are capped at 64 chars.
- **Clamps (document kept, warning):** dimension px-equivalents (rem/em × 16)
  clamped into [0, 512] for `space`/`fontSize` and [0, 9999] for `radius`
  (invariant #5: a theme cannot push content off-screen); `fontWeight` clamped
  into [1, 1000]; over-long `description` truncated.
- **Contrast = warning, never rejection (Decision Lock):** for the pairs
  (`fg`,`bg`), (`fg-muted`,`bg`), (`accent-fg`,`accent`) — when both members
  are present in the document and parse to sRGB (hex or numeric `rgb()/rgba()`)
  — compute the WCAG relative-luminance contrast ratio (pure arithmetic, no
  color library; `@facet/core` stays dependency-free, RISK-PKG-1); ratio < 4.5
  ⇒ a warning naming the pair and the ratio to 2 decimals (DC-004). Facet
  measures (mechanism); the caller decides policy (invariant #2).
- **Assumption confirmed (brief Decision Lock row):** theme documents cover all
  six token groups with per-group validation. Trimming later to colors-only
  would only shrink the validator, no API break.

`FacetTree` gains `theme?: string` (`packages/core/src/tree.ts:13-26` — one more
optional field; `isTreeShaped` at `tree.ts:40-46` checks only root/nodes, so
every shape guard passes it through, RISK-API-1). `validateTree`'s output
construction (`packages/core/src/validate.ts:465-475`, the explicit field list)
keeps `theme` iff `typeof input.theme === "string"`, dropping anything else with
an issue (DC-005) — `applyPatch` is field-agnostic and needs nothing;
`EMPTY_TREE`, kit `page()`, `offline.ts:4`, and the playground constructors need
nothing.

### Decision 2 — resolution & delivery: default-preserving react surface + the HTML-shell boot seam

**`@facet/react` (`packages/react/src/theme.ts`):**

```ts
export const DEFAULT_THEME: FacetTheme;   // name "default" — today's SPACE/FONT_SIZE/FONT_WEIGHT/RADIUS/COLOR/RATIO values as a document
export interface ResolvedTheme {           // full null-proto maps, every token present
  readonly space: Record<Space, string>;   readonly fontSize: Record<FontSize, string>;
  readonly fontWeight: Record<FontWeight, number>; readonly radius: Record<Radius, string>;
  readonly color: Record<Color, string>;   readonly ratio: Record<Ratio, string>;
}
export function resolveTheme(name: unknown, themes?: readonly FacetTheme[]): ResolvedTheme;
```

- `resolveTheme` is a **pure lookup** (invariant #6: the browser never writes
  stage state): non-string/missing/unknown `name` ⇒ the module-constant default
  resolved theme (zero allocation on the common path); a match ⇒ default values
  overlaid by the document's entries. Defense at the lookup site (DC-011): only
  **own** keys that are members of the token-group arrays with primitive values
  of the right type are copied, onto `Object.create(null)` maps — because a
  JSON round trip (shell → `JSON.parse`) restores a normal prototype, the
  null-proto guarantee must be re-established where lookups happen.
  `validateTheme` remains the security boundary for untrusted documents; hosts
  must validate before passing the prop.
- The style fns gain a **defaulted trailing parameter** —
  `boxStyle(style?: BoxStyle, theme?: ResolvedTheme)` (likewise
  `textStyle`/`imageStyle`/`fieldStyle`) — so every zero-extra-arg call stays
  byte-identical to today (RISK-API-2; pinned by the untouched
  `StageRenderer.test.ts` static suite + `theme.test.ts`). The current
  module-level maps become the default `ResolvedTheme`'s members; **`COLOR`
  stays exported unchanged** (it aliases the default resolved palette).
- `StageRenderer` gains one optional prop: `themes?: readonly FacetTheme[]`.
  It memoizes `resolveTheme((tree as { theme?: unknown }).theme, themes)` and
  threads the result into the style-fn call sites
  (`StageRenderer.tsx:394,401,406,414,432`). Existing narrow usages
  (`quickstart/src/page/main.tsx:66`, playground) compile unchanged.
- **ChatDock stays on the default document's palette** (the no-break option,
  decided): it keeps consuming the exported `COLOR`
  (`ChatDock.tsx:3,70-103`) and takes no theme prop in v1. The stage restyles;
  the chrome stays stable. Recorded as intentional.

**Boot seam (brief Open Question — RESOLVED per RISK-PKG-2):** the validated
theme map ships **inline in quickstart's `SHELL_HTML`**
(`packages/quickstart/src/server.ts:62-66`) as
`<script>window.__FACET_THEMES__ = <json></script>`, where the JSON is
`JSON.stringify(themes).replace(/</g, "\\u003c")` — the standard script-context
escape, pinned by a test with a hostile `</script>` description (defense in
depth; `validateTheme` already refuses `<` in values but descriptions are freer
text). `page/main.tsx` floor-guards the global (array of objects with string
`name`, else `undefined`) and passes it to `StageRenderer` as the `themes`
prop. Only `@facet/quickstart` + `@facet/react` change; `@facet/server`,
`@facet/client`, and the protocol (`core/src/protocol.ts:76-79`,
`server/src/server.ts:297-301`) are untouched — the SSE-snapshot alternative
would be a new protocol message the brief forbids and is a spec-review blocker
if reintroduced. Live switch = local re-resolution against the boot-shipped
map; registry edits need a reboot (Decision Lock, v1). Non-quickstart hosts
own their shell and pass the prop themselves.

### Decision 3 — assets registry: the `StageStore` adapter posture, exactly

`@facet/runtime` (`packages/runtime/src/assets.ts`, new; browser-safe — no Node
built-ins):

```ts
export interface AssetDocuments {          // RAW documents, straight from the backend
  readonly themes: readonly unknown[];
  readonly stamps: readonly unknown[];
  readonly initialTree?: unknown;
  readonly issues?: readonly string[];     // backend-level issues (unreadable file, bad JSON)
}
export interface AssetsStore { load(agentId: string): Promise<AssetDocuments> }
export class MemoryAssets implements AssetsStore { constructor(docs: AssetDocuments) }
export interface LoadedAssets {
  readonly themes: readonly FacetTheme[];
  readonly stamps: readonly FacetStamp[];
  readonly initialTree?: FacetTree;
  readonly issues: readonly string[];
}
export async function loadAssets(store: AssetsStore, agentId: string): Promise<LoadedAssets>;
export function isSeedableTree(tree: FacetTree): boolean;
export function withInitialStage(store: StageStore, initialStage: FacetTree): StageStore;
```

- `loadAssets` runs the core validators once, at boot (Decision Lock: no hot
  reload): each theme through `validateTheme` (error ⇒ skipped + issue;
  warnings ⇒ kept + issue; duplicate names ⇒ first wins + issue), each stamp
  through `validateStamp` (Decision 5), the initial tree through `validateTree`
  **plus `isSeedableTree`** (Decision 4). Invalid documents are skipped with a
  logged issue and boot proceeds — the `file-stage-store.ts:90-99` posture
  (DC-007).
- `FileAssets(dir)` (`packages/runtime/src/file-assets.ts`, new, exported via
  the existing node-only entry `runtime/src/node.ts` — the `runtime/package.json`
  `./node` subpath already exists, no manifest change): reads
  `*.theme.json`, `*.stamp.json` (sorted for determinism), and
  `initial.tree.json` from the directory; unreadable/unparseable files become
  `issues` entries, never throws past a file (RISK-PKG-1: interface + memory
  in `index.ts`, file behind `./node`, dependency direction unchanged).
- **Single-tenant references, per-agent interface:** `load(agentId)` carries
  the agent id for adapter parity (a Postgres adapter keys on it — the
  `@facet/store-postgres` precedent); the memory and file references serve
  their constructed documents/directory to whichever agent asks. Recorded as
  the v1 contract.

### Decision 4 — initial-tree seeding: a store decorator inside the serialized write path, with the EMPTY_TREE trap closed

- `withInitialStage(store, tree)` decorates `open()`: existing session ⇒
  returned untouched; fresh session ⇒ created with `stage: initialStage`
  (instead of `openSession`'s hardcoded `EMPTY_TREE`,
  `runtime/src/stage-store.ts:41`), saved, returned. `get`/`save` delegate.
  Because **all** opens run under `FacetRuntime`'s per-(agent, visitor) serial
  queue (`runtime/src/runtime.ts:72-76`) and `handleOne` opens the session
  BEFORE the agent call (`runtime.ts:98-105`), the seed is (a) inside the same
  serialized stage-write path (invariant #6, RISK-INV-3) and (b) visible to
  the agent's first turn ("refines the seeded stage"). The browser's first
  snapshot comes from `runtime.stageFor` via rehydrate
  (`server/src/server.ts:281,299-301`) — a stage seeded at open ships
  automatically; zero `@facet/server` change (DC-009).
- **The trap (RISK-INV-1, pinned by DC-009):** `validateTree(garbage)` returns
  `EMPTY_TREE` + issues (`validate.ts:352-357`), which would *silently seed an
  empty tree* — flipping `hasBuiltStage` (`server/src/offline.ts:29-35`) and
  changing the offline face for fresh visitors. So `loadAssets` accepts an
  initial tree only if, after `validateTree`, `isSeedableTree` holds: the
  render root resolves to a box with ≥ 1 child, or `screens` is non-empty
  (mirroring `hasBuiltStage` semantics). Anything else ⇒ `initialTree`
  undefined + issue ⇒ **no decorator is installed** ⇒ today's model-first
  paint, exactly as the brief promises.
- **Intentional interaction (recorded):** a *valid* seeded tree counts as
  "built", so `offlineFor` (`offline.ts:40-48`) will NOT overwrite it with the
  offline face — desired (the skeleton IS the page); documented on
  `withInitialStage`'s doc comment.
- Seed happens once per fresh session by construction (`open` is
  get-then-create); rapid duplicate visits are serialized by the runtime queue.

### Decision 5 — stamps phase 1: a validated fragment document, prompt-copy only

**Shape (brief Open Question — RESOLVED):** slots are NOT needed for phase-1
copy-only, and this shape doesn't preclude phase-3 server-side `{use, id, slots}`
macros (a future `slots` field is additive):

```ts
export interface FacetStamp {
  readonly name: string;                   // same rules as theme names
  readonly description?: string;           // one-liner for prompt ②
  readonly root: NodeId;                   // fragment root — must resolve in nodes
  readonly nodes: Readonly<Record<NodeId, FacetNode>>;
}
export interface StampValidationResult { readonly stamp?: FacetStamp; readonly issues: readonly string[] }
export function validateStamp(input: unknown): StampValidationResult;
```

- `validateStamp` lives in `packages/core/src/validate.ts` **next to
  `validateTree`** (recommended by context: `sanitizeNode`,
  `validate.ts:226-305`, is module-private and must be shared, not re-derived):
  same brick-shape + token-membership sanitization, same null-proto node map,
  same dangling-child/cycle discipline; the fragment root must resolve but need
  NOT be a box (a single-text stamp is legal); `screens`/`entry` are not part
  of a stamp. No usable root or zero surviving nodes ⇒ `stamp` undefined.
  `onPress` actions are allowed (a product card has a button) — the locked
  marketplace import rules (strip `onPress`, re-mint ids) apply to a future
  sharing feature, not to operator-local assets.
- **No expansion anywhere (permanent constraint):** stamps reach the LLM as
  prompt data only (Decision 6); the model copies nodes into ordinary
  patches/tools; `validateTree` server-side and the fail-safe renderer are
  unchanged. A stamp copied with a stale prefix replaces nodes (cosmetic,
  never structural — `set_node`/`add` upsert semantics), exactly the brief's
  Example 5.

### Decision 6 — the agent-facing selection surface: name in, never a value

- **`STAGE_SPEC`** (`packages/core/src/spec.ts:8`, the single prompt source):
  the tree-shape line gains `"theme"?: "<theme name>"` and one compact
  sentence — *set the top-level `"theme"` only to a theme name you have been
  given (a THEMES list, if any); you never write CSS values — styles stay
  tokens; an unknown name simply falls back to the default look.* Embedders
  (`apps/playground/src/generator.ts:7,11`, `bridge/src/bridge.ts:8,91`,
  `bridge/src/persistent.ts:11,37`) inherit the line verbatim — select-by-name
  needs no new tool on those paths (they emit raw patches; RISK-API-4).
  `spec.test.ts` gains the pin (DC-012).
- **`Stage.theme(name: string)`** (`packages/agent/src/stage.ts`): records
  `{ op: "add", path: "/theme", value: name }` — the `screens()` top-level
  upsert precedent (`stage.ts:56-62`). One normal RFC 6902 op through the
  normal event loop; the server stays the only stage writer.
- **Quickstart tool loop:** `TOOLS` (`quickstart/src/prompt.ts:73-122`) gains
  `set_theme` (`{ name: string }` — a NAME argument only; its description says
  to use only names from the THEMES section and to skip it when none are
  listed). `agent.ts`'s tool-execution switch maps it to `stage.theme(name)`;
  a non-string argument becomes an "error" observation, never a throw (the
  existing fail-safe arm posture).
- **Prompt layer ②** (`buildSystem`, `prompt.ts:56-63`) gains an optional
  second parameter:

  ```ts
  export interface PromptAssets { readonly themes: readonly FacetTheme[]; readonly stamps: readonly FacetStamp[] }
  export function buildSystem(guide: string, assets?: PromptAssets): string;
  ```

  With no assets (or both arrays empty) the output is **byte-identical** to
  today's `[intro, STAGE_SPEC, WORKFLOW, PAGE BRIEF].join("\n\n")` (DC-008;
  existing calls compile unchanged). Themes present ⇒ a `THEMES` section:
  name + one-line description per line, plus the select-by-name rule — names
  and descriptions ONLY, never values (DC-012, RISK-INV-2). Stamps present ⇒ a
  `STAMPS` section: per stamp `- <name>: <description>` + the `{root, nodes}`
  JSON, prefixed by the mandatory copy rule (*copy the fragment's nodes into
  the page and prefix EVERY id — the root and all child references — with a
  fresh instance prefix like `p1-`*). A stamp whose serialized JSON exceeds
  `MAX_STAMP_PROMPT_CHARS = 4000` is excluded with a `console.warn` — a
  quickstart POLICY, not a core rule (invariant #2).
- **Stub determinism (DC-010):** `createStubAgent` gains one deterministic
  behavior — a `message` whose text starts with `"theme "` runs
  `stage.theme(<rest>)` + `stage.say("stub: theme <rest>")` instead of the
  echo; all other behavior byte-identical. Zero randomness, zero clock reads.

### Decision 7 — kit namespacing: per-instantiation prefixes, `fragment()`, `KIT_STAMPS`

- `Builder` (`packages/kit/src/kit.ts:19-26`) gains a constructor prefix;
  `next()` mints `` `${prefix}k${n}` ``. `page()` passes no prefix — its
  generated ids and hardcoded `"root"` root stay exactly today's
  (`validateTree`'s root fallback expects `"root"`, `validate.ts:406-411`;
  `kit.test.ts:14` keeps pinning it). Within one `page()` call a preset
  instantiated twice already gets distinct ids (one Builder, monotonic
  counter) — now pinned by test (DC-006).
- New export — the graft API:

  ```ts
  export interface Fragment { readonly root: NodeId; readonly nodes: Readonly<Record<NodeId, FacetNode>> }
  export function fragment(block: Block, prefix: string): Fragment;
  ```

  Instantiates one `Block` on a fresh prefixed Builder: all ids are
  `` `${prefix}k${n}` `` — never `"root"`, never colliding with another
  prefix's ids — safe to merge into an existing tree's `nodes` (DC-006).
  Callers pass a non-empty prefix; kit does not police it (ids were never
  contractual — generated-id change waived pre-1.0 per the brief, RISK-API-3:
  the sweep shows no consumer or test pins `k*` ids).
- `KIT_STAMPS` (`packages/kit/src/stamps.ts`, new): the default stamp library
  derived from the presets via `fragment()` — v1 ships `hero`, `card`, and
  `cta-button`, each a named+described `FacetStamp` that passes `validateStamp`
  cleanly (pinned). Quickstart does NOT auto-inject them (DC-008
  byte-identity; no new quickstart→kit dependency) — they are data for hosts
  and operators to hand to the registry.

### Decision 8 — quickstart boot wiring (`--assets`)

- `facet-quickstart … [--assets <dir>]`: an **explicitly passed** directory
  that doesn't exist or can't be read ⇒ exit 1 naming the path (the `--guide`
  explicit-path precedent, `cli.ts:120-137`); no flag ⇒ no registry, today's
  boot exactly. No implicit default directory in v1 (no surprise loads;
  documented).
- Boot order in `runCli`: `FileAssets(dir)` (from `@facet/runtime/node`) →
  `loadAssets` → every issue logged as one concise `[facet-quickstart]` warn
  line → `createQuickstartAgent({ …, themes, stamps })` (layer ② built once at
  agent creation, `agent.ts:231` — "assets read once at boot") →
  `startQuickstart({ …, themes, initialStage })`.
- `QuickstartServerOptions` gains `themes?: readonly FacetTheme[]` (inlined
  into the shell per Decision 2) and `initialStage?: FacetTree` —
  `startQuickstart` wraps `options.stageStore ?? new MemoryStageStore()` with
  `withInitialStage` when present, then threads it into the internal
  `createFacetServer` (`server.ts:274-275` seam).
- `--stub` composes with assets: the shell still carries the theme map and the
  seed still applies (the stub is the Tier-1 vehicle for DC-009/DC-010).

### WU sizing rationale

11 WUs. The ≤5-file cap and package boundaries force the split: WU-1/WU-2 are
independent core halves (theme validator vs tree/stamp validator — disjoint
files) and everything narrows against their types; WU-3–WU-7 then fan out on
disjoint file sets (spec text / agent op / react resolution / runtime registry /
kit); the quickstart tail (WU-8 prompt+stub → WU-9 boot wiring → WU-10 live
gate) is inherently serial because each layer consumes the previous one's
exports; WU-11 (docs+changeset) closes after the gate proves the story.

## As-built deviations (recorded post-implementation, 2026-07-04)

The implementation matches this spec EXCEPT for the initial-tree delivery
mechanism, which the adversarial review + real-browser testing reshaped:

1. **Decision "the browser's first snapshot ships via the existing rehydrate
   path with zero server change" was WRONG** — rehydrate runs at stream-connect
   time, before the session exists. As built: `withInitialStage` reports a
   fresh seed once (optional `StageStore.takeSeeded`), and `FacetRuntime`
   prepends a stamped root-replace patch to that turn's messages
   (`pendingSeeds`, consumed only after persist succeeds, value = the current
   session stage). Invariant #2 holds: the seed travels the patch channel,
   same `applyPatch` both sides, replay-ring/Last-Event-ID safe.
2. **Instant first paint** (found by opening a real browser): the quickstart
   shell additionally ships the seed as `window.__FACET_INITIAL_STAGE__`
   (same escaped seam as `__FACET_THEMES__`) and `useFacet` accepts an
   optional `initialTree`, so page-open paints without waiting for the model;
   the wire seed frame then applies idempotently.
3. **Themed canvas**: `page/main.tsx` paints `document.body` bg/fg from the
   resolved theme (ChatDock stays default per Decision). Without this a dark
   theme changed almost nothing visibly.
4. **Hardening added beyond the spec**: GET / sits behind the DNS-rebinding
   Host guard (the shell now carries operator data); `tree.theme` enforces the
   shared `isValidThemeName` rule; single-parent-per-walk-root in
   `breakCycles` + a renderer render-budget close the pre-existing
   shared-child DAG path explosion (review round 5 P1).

5. **Convergence by construction (review rounds 7-9)**: the save-path
   sanitize/salvage asymmetry (server salvages per-op + validates; client
   applied raw batches) was first compensated with a corrective-frame
   heuristic, then REPLACED by the principled fix — one shared pure
   `foldPatchIntoStage` (apply + per-op salvage + `validateTree`) in
   `@facet/core`, executed identically by `FacetRuntime` and `useFacet`, so
   stored and live trees cannot diverge and the heuristic was deleted
   (runtime net -146 lines). `TurnResult.agentMutated` lets the transport
   distinguish real edits from the seed frame.

Tracked non-blocking residuals live in `docs/HARDENING-BACKLOG.md`
("Bundle B review residuals" section). ARCHITECTURE.md reflects the as-built
mechanism; this spec is kept as the historical plan + this addendum.
