## Overview

- Feature slug: `renderer-transitions`
- Intake brief: `specs/feature-intake/renderer-transitions.md`
- Context evidence: `specs/context/renderer-transitions.md`
- Affected `@facet/*` packages: `@facet/core`, `@facet/react`, `@facet/quickstart`
  (+ `apps/playground` consumer wiring)
- One-paragraph technical approach: Keep animation policy entirely renderer
  owned. `useFacet` gains an additive `transition` hint with a monotonic
  `revision` and `rootReplaced` bit derived from the shared stage fold result;
  no wire/server message changes and no core brick/token/protocol vocabulary
  change. `StageRenderer` gains an optional
  `transition` prop and an internal `motion.ts` stylesheet/constants module.
  The renderer computes visible brick ids with the same fail-safe, budgeted
  semantics as the render walk. First valid paint is instant. After that, new
  visible ids get a short enter class, disappearing ids render from the previous
  tree as inert exit visuals until the exit timer completes, same-id updates
  stay immediate, and root document writes or more than 8 total enter+exit ids
  use a stage-level crossfade instead of many brick animations. Existing
  `box.style.appear` remains a box-only explicit enter hint and keeps its
  current CSS/classes.

### Decision 1 — root-replace detection without protocol changes

`useFacet` is the React entrypoint that sees the patch fold result before
rendering (`packages/react/src/useFacet.ts:75-106`). Add:

```ts
export interface StageTransitionHint {
  readonly revision: number;
  readonly rootReplaced: boolean;
  readonly rootReplacedRevision?: number;
}
```

and `FacetState.transition: StageTransitionHint`. `revision` increments only
when `foldPatchIntoStage(...).mutated` is true; `rootReplaced` is true for that
revision only when the shared fold reports that an actually-applied operation
wrote the root document (`add|replace|copy|move` at `path: ""`). The core
addition is a fold-result metadata bit, not a wire/protocol field. `remove ""`
is not treated as `render_page`; it fails at the patch layer and should not
create a pretty error animation.

### Decision 2 — transition policy

- Initial render: no default lifecycle or stage motion.
- Same visible id before/after: no default motion, no remount.
- Newly visible id: brick enter class on the rendered host element.
- Disappearing visible id: render the previous host element as an inert exit
  visual, then remove it after `MOTION_EXIT_MS`.
- Root document write (`transition.rootReplaced` on a new revision): stage
  crossfade, no per-brick lifecycle classes for that frame.
- Many lifecycle changes: stage crossfade when `enterCount + exitCount > 8`.
- Reduced motion: static CSS media query suppresses animations; the final tree
  still renders and cleanup timers still complete.

### Decision 3 — exit visuals are renderer-only and inert

Exit visuals are not stage state. They are rendered from the previous tree with
an `exit` render mode:

- no `onPress`/`onHold` dispatch,
- no field `data-facet-field-id` attributes,
- `aria-hidden` and `pointerEvents: "none"` on the exiting host,
- topmost disappearing ancestor only, so removing a box exits the box subtree
  once rather than duplicating every descendant.

When a parent remains visible, its exiting child ids are inserted at their
previous sibling positions for the duration of the exit. If the parent also
disappears, the topmost disappearing parent exits at its own previous slot.

### Decision 4 — stage crossfade implementation

Do not use browser View Transitions or `react-dom/flushSync` in v1. `@facet/react`
currently peers only `react` (`packages/react/package.json:31-33`), so adding a
React DOM runtime dependency just for motion is too large for this feature.

For hosts that pass `transition`, `StageRenderer` keeps a stable internal current
subtree wrapper from the first client render and uses the existing stage root
scope wrapper when `onAction` is wired. On stage crossfade, the current tree
stays in normal flow and the previous tree is rendered once in inert mode as a
temporary previous-stage layer. The wrapper may become a temporary grid/relative
transition container only during the crossfade, but it must not create an
agent-authored overlay capability and must not block current-tree interactions.
Existing SSR/static output with no transition prop stays byte-identical.

### Decision 5 — stylesheet ownership

Add `packages/react/src/motion.ts` as an internal module, not exported through
`packages/react/src/index.ts`. It owns class names, durations, the threshold, and
one static `MOTION_CSS` string. `StageRenderer` injects motion CSS only while a
lifecycle/stage transition is active; `APPEAR_CSS` continues to inject only when
a reachable box uses `appear`.

## Fail-safe & Boundary Checklist

- Malformed / empty / null input: if `tree` is not renderable, `StageRenderer`
  returns `null` as today (`StageRenderer.tsx:723-727`) and clears active motion
  state. If a previous snapshot becomes unrenderable, it is dropped rather than
  rendered. Junk style/action fields do not affect transition classification.
- Deeply nested / cyclic tree: visible-id collection and exit rendering use the
  same `MAX_DEPTH`, ancestor-set, sibling-dedupe, and `RENDER_BUDGET` rules as
  `renderNode` (`StageRenderer.tsx:938-1015`). Budget exhaustion truncates motion
  candidates the same way rendering truncates nodes.
- Duplicate ids / dangling refs: first sibling occurrence wins, matching current
  render behavior. Dangling removed refs are skipped; no exit visual is created
  without a previous renderable node snapshot.
- Offline agent / rapid events / racing agent turn: transition visuals never
  block `useFacet` patch folding. If a patch removes an id and a later patch
  re-adds it before `MOTION_EXIT_MS`, the exit record is cancelled and the live
  current node wins. Stage crossfade snapshots are keyed by revision and replaced
  by newer crossfades.
- Field collection boundary: exit/stage-previous render mode never stamps
  `data-facet-field-id`; `collectFieldValues` therefore sees only current live
  fields under `stageRootRef`.
