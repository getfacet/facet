- One-paragraph technical approach:
  expand Facet's safe UI contract in `@facet/core` from primitive-only nodes to
  a deliberate v1 high-level brick vocabulary, add a validated `FacetCatalog`
  that controls which bricks/stamps/themes/variants the agent may use, add
  theme component recipes as validated operator theme data, extend default
  assets with a catalog/stamp metadata/theme recipes, enforce catalog policy in
  `@facet/agent-tools`, and wire the validated catalog through runtime,
  reference-agent, and quickstart. Rendering stays declarative and flow-only:
  agents still emit stage data and JSON Patch only; the browser never evaluates
  HTML/JS/CSS from the model and never fetches chart/table/catalog data.

## Concrete V1 Contract Decisions

### Catalog Contract

`@facet/core` adds a zero-dependency `FacetCatalog` contract and
`validateCatalog(input)`:

```ts
interface FacetCatalog {
  readonly name: string;
  readonly description?: string;
  readonly theme: {
    readonly active?: string;
    readonly switchPolicy: "locked" | "allowed";
    readonly allowed?: readonly string[];
  };
  readonly bricks: readonly CatalogBrick[];
  readonly stamps: { readonly mode: "all" } | { readonly mode: "allow"; readonly names: readonly string[] };
  readonly primitiveFallback: "allowed" | "discouraged";
  readonly policy: {
    readonly order: readonly ["stamp", "brick", "primitive"];
    readonly editBeforeAppend: boolean;
    readonly compactScreens: boolean;
    readonly maxScreenSections?: number;
  };
}

interface CatalogBrick {
  readonly type: FacetNode["type"];
  readonly variants?: readonly string[];
  readonly guidance?: string;
}
```

Validation is fail-soft: malformed names, unknown brick types, unbounded arrays,
and bad enum values are dropped or defaulted with issues. Missing catalog input
resolves to `DEFAULT_CATALOG`.

### High-Level Brick Contract

`FacetNode` becomes a closed union of the existing primitives plus these v1
bricks:

- `button`: leaf action brick with `label`, `variant?`, `tone?`, `disabled?`,
  `onPress?`, and `onHold?`.
- `section`: normal-flow container with `title?`, `eyebrow?`, `body?`,
  `variant?`, and `children`.
- `card`: normal-flow container with `title?`, `body?`, `variant?`, `tone?`,
  optional `onPress?`, and `children`.
- `tabs`: navigation wrapper over existing local `navigate` semantics; items are
  `{ label, to }` screen links. It does not mutate stage content.
- `table`: display-only tabular data with capped columns/rows/cells.
- `chart`: display-only SVG/data brick with capped bar/line/donut series data.
- `stat`: compact metric brick.
- `badge`: compact status label.
- `progress`: bounded progress display.
- `alert`: feedback/message block.
- `list`: capped list display.
- `divider`: visual/content separator.

Only `box`, `section`, and `card` are containers in v1. Tables/charts are
display-only; no client fetch, sort, filter, or backend binding. Overlay-class
modal/drawer/popover bricks are explicitly out of v1.

### Theme Recipes

`FacetTheme` gains optional `recipes`. Recipes are operator theme data, not
agent-authored style:

```ts
interface ComponentRecipe {
  readonly box?: BoxStyle;
  readonly text?: TextStyle;
  readonly media?: MediaStyle;
  readonly field?: FieldStyle;
}

type ComponentRecipes = Partial<
  Record<RecipeComponentName, Readonly<Record<string, ComponentRecipe>>>
>;
```

Recipe style values are the same token style values already allowed in trees;
the renderer resolves them through the active theme's CSS token maps. High-level
nodes choose `variant`/`tone`, not raw styles.

### Stamp Metadata

`FacetStamp` preserves existing fields and adds bounded `metadata`:

```ts
interface StampMetadata {
  readonly category?: string;
  readonly useWhen?: string;
  readonly avoidWhen?: string;
  readonly variants?: readonly string[];
  readonly tags?: readonly string[];
  readonly repeatable?: boolean;
  readonly preferredParent?: "root" | "box" | "section" | "card";
  readonly composedOf?: readonly FacetNode["type"][];
  readonly dataRequirements?: readonly string[];
  readonly followUpEdits?: readonly string[];
}
```

Prompt-kit exposes only whitelisted compact metadata. It never exposes stamp
`root`, `nodes`, slot default values, or unknown fields to the LLM.

## Fail-safe & Boundary Checklist

- Malformed / empty / null input:
  - `validateCatalog(null)` returns `DEFAULT_CATALOG` plus issues.
  - `validateTree` continues returning a safe tree.
  - `StageRenderer` skips malformed high-level node payloads instead of throwing.
  - Tests: WU-02, WU-03, WU-05, WU-10.
- Deeply nested / cyclic tree:
  - Existing traversal caps remain in `validateTree`.
  - Only `box`, `section`, and `card` are containers; shared child traversal
    helpers prevent each consumer from inventing divergent recursion.
  - Tests: WU-02 and WU-04.
- Oversized table/chart/list payloads:
  - Core validators cap rows, columns, cells, series, points, item counts, and
    label/value string lengths.
  - Renderer receives already-sanitized data and still guards raw-path payloads.
  - Tests: WU-02 and WU-05.
- Unknown catalog/variant/stamp/theme:
  - Tool executor rejects with `outcome: "rejected"`, `reason:
    "catalog_policy"`, no patch, and a `next_action` that points the model to
    inspect/use allowed catalog entries.
  - Tests: WU-10 and WU-11.
- Offline agent / provider failure:
  - Reference-agent retry/fallback behavior is unchanged; catalog is prompt/tool
    context, not a provider dependency.
  - Tests: WU-12 adds an explicit reference-agent provider-failure/catalog
    regression in `packages/agent-stack/reference-agent/src/agent.test.ts`.
- Rapid events / racing agent turn:
  - Runtime's existing per-visitor serialization stays unchanged.
  - High-level bricks still produce normal stage patches; local tabs do not
    write server content.
  - Tests: WU-13 adds an explicit quickstart E2E that sends rapid catalog-guided
    visitor events and verifies serialized, non-throwing stage updates.
