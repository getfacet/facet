## Overview
- Feature slug: `overlay`
- Intake brief: `specs/feature-intake/overlay.md`
- Context evidence: `specs/context/overlay.md`
- Affected `@facet/*` packages: `@facet/core`, `@facet/react` (NO `@facet/assets`
  change â€” see RISK-API-4 / RISK-PKG-1).
- One-paragraph technical approach:
  Add an `overlay?: { kind: "modal" | "drawer" }` slot to the module-private
  `Layered` concern pack that `BoxNode` already composes (alongside `backdrop`),
  backed by a closed-EXTENSIBLE `OVERLAY_KINDS` const tuple + `OverlayKind` type
  (mirrors `FIELD_INPUTS`/`MARK_KINDS`). `validateBox` gains a `sanitizeOverlay`
  that admits ONLY `raw.overlay.kind âˆˆ OVERLAY_KINDS`, strips every other key, and
  DROPS the whole descriptor on unknown/missing kind or wrong type (fail-safe,
  mirroring the `backdrop` guard). STAGE_SPEC teaches the capability as closed
  names only. In `@facet/react`, `layout-contract.ts` gains renderer-FIXED
  POSITIVE-z placement presets (modal = fixed screen-center, drawer = fixed
  logical end-edge) + a positive-z scrim â€” selected purely by `kind`, no author
  z/inset/position. When a rendered (already visibility-filtered) box carries a
  valid `overlay`, `renderer-render.tsx` wraps its `BoxElement` in a client
  `OverlayFrame` (focus-trap + Esc + scrim-click + ref-counted body scroll-lock).
  Every close affordance (Esc, scrim, and any author close-button) routes through
  the EXISTING toggle writer so `view.toggled` stays single-sourced; the
  renderer-owned close is an idempotent-HIDE (not a blind flip) so double-close
  cannot reopen. No new public `StageRenderer` prop is added.

## Fail-safe & Boundary Checklist
- **Malformed / empty / null overlay**: `overlay:{kind:"lightbox"}` / `overlay:{}` /
  `overlay:"modal"` / `overlay:42` â†’ descriptor dropped in `validateBox`, box
  renders inline; a bounded issue is pushed. Never throws (DC-003).
- **Extra author keys**: `overlay:{kind:"modal", z:999, top:10, position:"fixed"}`
  â†’ only `kind` survives; `z`/`top`/`position` never reach the tree or renderer
  (whitelist-copy + explicit strip, DC-004).
- **Dangling toggle target**: trigger `toggle` at a missing overlay id â†’ existing
  `handlePress` `hasOwnProperty`-guarded no-op (unchanged).
- **Deeply nested / cyclic tree**: `OverlayFrame` wraps the already-resolved
  `BoxElement`; children still route through the budget-bounded walk â€” an overlay
  box counts against `MAX_RENDER_NODES` exactly like any box. No new recursion.
- **Rapid / double close**: renderer-owned close is set-to-hidden (idempotent);
  after hide the box is visibility-filtered out and `OverlayFrame` unmounts, so a
  second Esc/scrim cannot fire (RISK-INV-4).
- **SSR / no-JS**: overlay content validates and exists in the tree; the frame's
  floating/scrim/focus/scroll-lock are browser-only effects that no-op under SSR
  (content still renders inside the frame element).
- **Focus / scroll-lock scope**: focus moves into the frame on mount and restores
  to the previously-focused element on unmount; body scroll-lock is ref-counted
  so nested/multiple overlays restore `document.body` overflow exactly once.
